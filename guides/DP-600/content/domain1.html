Domain 1 — Maintain a Data Analytics Solution 25–30%

Domain 1 covers the operational and governance responsibilities of a Fabric analytics solution: securing workspaces and data at every granularity, managing the trustworthiness of published assets, and controlling how code and semantic models move through the development lifecycle into production. Mastery here underpins safe, auditable enterprise analytics at scale.

1.1 Implement Security and Governance

1.1.1 Implement Workspace-Level Access Controls

Every Microsoft Fabric workspace comes with four built-in roles that govern what principals can do inside that workspace. Roles can be assigned to individuals, security groups, Microsoft 365 groups, distribution lists, or Microsoft Entra service principals. When a user belongs to multiple groups, Fabric grants the highest permission from any applicable membership — not the intersection.

Workspace roles are strictly scoped to a single workspace; they do not cascade across workspaces or to capacity administration. OneLake data-access roles can further refine data permissions within a workspace independently of these workspace roles.

Key facts

  • Four roles (highest → lowest): Admin, Member, Contributor, Viewer.
  • Admin only: delete or update the workspace itself, add other Admins, create a workspace identity, and connect the workspace to a Git repository.
  • Admin + Member: add Members or lower-privilege roles to the workspace.
  • Admin + Member + Contributor: create, write, and delete Fabric items; execute notebooks and pipelines; schedule refreshes via an on-premises gateway; read Lakehouse and Warehouse data via OneLake APIs and Spark (ReadAll permission).
  • All four roles (including Viewer): view item content; connect to the SQL analytics endpoint; read Lakehouse and Warehouse data via T-SQL through the TDS endpoint (ReadData permission).
  • Viewer cannot: write data, create items, or read data via OneLake APIs or Spark.
  • Admin and Member can edit OneLake security roles; Contributor and Viewer cannot.
  • Service principals (Entra app registrations) can be assigned workspace roles and behave like users for API-based operations.
  • By default, only Admins can connect or disconnect the workspace from Git. Admins can optionally enable a setting that allows Contributors/Members to also switch branches.
  • Workspace roles do not span workspaces; each workspace is an independent permission boundary.
Tip: When a user belongs to several groups with different roles, Fabric awards the highest permission, not the most restrictive. Verify effective permissions explicitly before assuming a user is limited to Viewer.
Key: RLS in a semantic model applies only to Viewer-level users. Admin, Member, and Contributor workspace roles bypass row-level security and always see all data.

Exam tips & gotchas

  • Viewers can read Lakehouse/Warehouse data through T-SQL (ReadData via the TDS endpoint), but they cannot read through OneLake APIs or Spark (ReadAll) — that requires Contributor or higher.
  • Only a workspace Admin can connect or disconnect the workspace from a Git repository — no exceptions unless the Admin explicitly enables branch-switching for Contributors/Members.
  • Contributors cannot add other Contributors or Members — only Admins and Members can add Members or lower roles.
  • OneLake security roles add a second permission layer; they can restrict data access even for Contributor-level users at the storage level.
  • Workspace roles are scoped to a single workspace; they grant no access to items in other workspaces.

Practice

Q1. A data analyst needs to run Spark notebooks against a Lakehouse and schedule pipeline refreshes, but must not be able to delete the workspace or modify workspace membership. What is the minimum sufficient workspace role?
Answer: Contributor. Why: Contributor can create and execute Fabric items (notebooks, pipelines) and read data via OneLake APIs (ReadAll). Admin and Member grant more rights than necessary (managing membership, connecting Git). Viewer cannot execute items or write data.
Q2. A user connects to a Fabric Lakehouse SQL analytics endpoint using SSMS. The user has only the Viewer workspace role and no OneLake security role. What level of data access does the user have?
Answer: Read-only access via T-SQL (ReadData through the TDS endpoint). Why: All four workspace roles — including Viewer — can connect to the SQL analytics endpoint and query data through T-SQL. Viewers cannot use OneLake APIs or Spark (ReadAll requires Contributor+), but ReadData via T-SQL is permitted for all roles.
Q3. A senior developer with the Member workspace role wants to connect the workspace to Azure DevOps. Can they do this without Admin involvement?
Answer: No. Why: Only workspace Admins can connect or disconnect a workspace from a Git repository. The Member role does not include this capability. The developer must request that an Admin performs the connection, or be elevated to Admin.

1.1.2 Implement Item-Level Access Controls

Item permissions control access to a single Fabric artifact — a semantic model, report, lakehouse, and so on — independently of workspace roles. A user who holds no workspace role can still be granted access to a specific item via a sharing link, reaching only that item without any workspace context. Item permissions and workspace roles coexist; Fabric evaluates both during access checks.

Access evaluation follows a three-layer model: first Microsoft Entra authentication, then Fabric item access (workspace role or item permission), and finally data-layer security such as RLS and OneLake roles.

Key facts

  • Read (default on share): the recipient can view the item's metadata and any reports built on it, but this permission alone does not grant access to underlying data via SQL or OneLake.
  • Build: lets the recipient create new reports, dashboards, and other downstream content on top of a shared semantic model — the key permission for self-service BI.
  • Write: lets the recipient modify the item itself (e.g., edit a semantic model's definition).
  • Reshare: lets the recipient grant access to other users.
  • DirectLake reports shared with Read-only item permission require the user to also hold OneLake data permissions on the underlying Delta tables to see actual data.
  • If a user holds a workspace Viewer role, revoking item-level permissions alone is insufficient — workspace role access remains. Both must be removed to fully block access.
  • The SQL analytics endpoint supports native T-SQL GRANT/DENY for granular column or schema-scoped control beyond item permissions.
  • Security evaluation order: (1) Entra authentication → (2) Fabric access (workspace role or item permission) → (3) data security (RLS, OLS, OneLake roles).
Gotcha: Sharing a semantic model with Read permission grants view access to existing reports but does not allow direct data queries via SQL or OneLake. Grant Build for self-service report creation; use SQL GRANT on the SQL analytics endpoint for direct data queries.

Exam tips & gotchas

  • Build permission is required for a consumer to create new reports on a shared semantic model — Read alone is not sufficient.
  • A user with no workspace role but with item-level Read can access the item via direct link without seeing the workspace.
  • Removing item-level permissions does not revoke access if the user also has a workspace role — both layers must be revoked.
  • The SQL analytics endpoint supports T-SQL GRANT/DENY independently, enabling column- or schema-scoped control that is separate from item-level permissions.

Practice

Q1. A business analyst in a different workspace needs to create new Power BI reports against a published semantic model. The analyst should not be able to modify the semantic model or browse other workspace items. What is the minimum permission needed on the semantic model?
Answer: Build. Why: Build permission lets the analyst create new reports and dashboards on a shared semantic model. It does not grant Write access (to modify the model) or visibility of the source workspace. Read alone does not enable new report creation.
Q2. A user was granted item-level Read permission on a report backed by a DirectLake semantic model. The report loads but shows no data. What additional permission is most likely missing?
Answer: OneLake data-access permission on the underlying Delta tables. Why: DirectLake reads directly from OneLake storage. Item-level Read on the report does not include OneLake storage access. The user needs an OneLake data-access role that grants Read on the relevant tables for data to be returned.
Q3. An admin revokes the item-level share permission from a user for a semantic model. The user still has a Viewer workspace role. Can the user still access the semantic model?
Answer: Yes. Why: Workspace roles are independent of item permissions. A Viewer role grants the right to see all items in the workspace. Removing the item-level share does not override the workspace role; both must be revoked to fully block access.

1.1.3 Implement Row-Level, Column-Level, Object-Level, and File-Level Access Control

Fabric provides four distinct data-layer security mechanisms that operate at different granularities. Row-Level Security (RLS) filters which rows a user sees within a table, using DAX expressions inside role definitions. Object-Level Security (OLS) hides entire tables or individual columns from certain roles, concealing both the data and the object's metadata name. Column-level restriction in the tabular model is technically a form of OLS (CLS via OLS). OneLake data-access roles enforce file- and folder-level RBAC at the storage layer, independently of any semantic model.

These controls activate after a user has passed Fabric item-access checks. They filter or block specific data for users who have already been granted the right to reach the item.

Key facts — RLS (Row-Level Security)

  • RLS restricts which rows a user sees by applying a DAX filter expression on one or more columns within a role definition.
  • Roles are defined in Power BI Desktop → Modeling → Manage Roles; published to the Power BI service, where members are added under the semantic model's Security option.
  • RLS applies only to workspace Viewer-level users (and users who accessed the model via item sharing only). Admin, Member, and Contributor workspace roles bypass RLS and always see all rows.
  • Static RLS: hard-coded filter value — e.g., restrict rows where Region equals a fixed string.
  • Dynamic RLS: uses USERPRINCIPALNAME() or USERNAME() so a single role adapts to each signed-in user via a mapping table — the standard production pattern.
  • Bi-directional cross-filter propagation with RLS must be enabled per relationship and can degrade query performance.
  • For Direct Lake semantic models, RLS is supported; DAX fallback to DirectQuery also respects the RLS filters.
  • Workflow: define roles in Desktop → publish → add members in the service → validate with Test as role or View as.

Key facts — OLS (Object-Level Security) and CLS (Column-Level Security)

  • OLS hides entire tables or individual columns from a role, concealing both the data and the object's metadata name — it appears as if the object does not exist.
  • Applies to tabular models at compatibility level 1400+ (Fabric/Premium, Azure AS, SSAS 2017+).
  • Configured via: TMDL view in Power BI Desktop, open-source Tabular Editor (GUI), TMSL JSON scripts, or TOM API. There is no dedicated OLS dialog in Power BI Desktop's Manage Roles UI.
  • The controlling property is metadataPermission on a tablePermissions or columnPermissions object: none = hidden; read = visible.
  • Column-Level Security (CLS) in the semantic model is implemented via columnPermissions with metadataPermission: none — the same OLS mechanism applied at column granularity.
  • Workspace Admin, Member, and Contributor roles bypass OLS — they can see all secured objects regardless.
  • OLS cannot be applied to a table that would break a required relationship chain — this causes a design-time error.
  • OLS is incompatible with Q&A visuals, Quick insights, Smart narrative, and the Excel Data Types gallery.
  • Combining RLS and OLS rules from different roles on the same user causes a query-time error.

Key facts — OneLake Data-Access Roles (File/Folder Level)

  • OneLake uses a deny-by-default RBAC model: users start with no data access unless explicitly granted by an OneLake role.
  • Each role specifies: Permission (Read or ReadWrite), Scope (specific tables, folders, or schemas), and Members (Entra identities/groups).
  • Supported Fabric items: Lakehouse, Azure Databricks Mirrored Catalog, Mirrored Databases.
  • Two default roles are auto-created per Lakehouse: DefaultReader (Read on all folders; assigned to all ReadAll-permission users) and DefaultReadWriter (ReadWrite on all folders; assigned to all Write-permission users). Modify or remove these to enforce custom access.
  • Workspace Admin, Member, and Contributor automatically have Write access to OneLake — OneLake roles cannot restrict them below this. Viewer has no OneLake access by default; an OneLake role must explicitly grant it.
  • Admin and Member can manage OneLake security roles; Contributor and Viewer cannot.
  • Supported engines (GA): Lakehouse, Spark notebooks, SQL Analytics Endpoint (user-identity mode), Direct Lake on OneLake. Eventhouse RLS: public preview.
  • For non-authorized engines where RLS/CLS is defined, the entire query is blocked rather than partially filtered, to prevent data leakage.
-- Static RLS: restrict rows to a single hard-coded region
-- Applied in the role filter for the Sales table:
[Region] = "West"

-- Dynamic RLS: one role covers all users via a mapping table
-- DimEmployee[UserEmail] must hold the user's UPN (e.g., from Entra ID)
[UserEmail] = USERPRINCIPALNAME()

-- Dynamic RLS with a related dimension:
-- Role filter placed on the Sales fact table, resolving via RELATED:
RELATED(DimEmployee[UserEmail]) = USERPRINCIPALNAME()
Gotcha: RLS applies only to Viewer-level consumers. Admin, Member, and Contributor workspace roles always see all rows — they are not filtered by any RLS role definition. To test RLS as a developer, use the View as / Test as role feature in the Fabric service.
Key distinction summary:
  • RLS — row filtering via DAX; defined in Desktop Manage Roles; bypassed by Admin/Member/Contributor.
  • CLS (semantic model) — column hiding via columnPermissions metadataPermission: none; same OLS mechanism, same bypass rules.
  • OLS — table and column hiding (data + metadata name); no dedicated Desktop UI — requires TMDL, Tabular Editor, TMSL, or TOM; Admin/Member/Contributor bypass it.
  • OneLake data-access roles — file/folder-level RBAC at the storage layer; deny-by-default; independent of semantic model roles; Admin/Member/Contributor still have Write access and cannot be restricted below it.

Exam tips & gotchas

  • OLS hides the metadata too — a secured column does not appear in field lists, unlike RLS which only filters rows.
  • OLS has no dedicated dialog in Power BI Desktop's Manage Roles UI; configure it via TMDL, Tabular Editor, TMSL, or TOM. RLS does have a "Manage Roles" dialog.
  • Admin, Member, and Contributor bypass both RLS and OLS. Security at these levels must be enforced by limiting who holds those roles.
  • OneLake data-access roles use deny-by-default; Viewers have no OneLake access unless an OneLake role explicitly grants it.
  • Combining RLS and OLS from different roles on the same user generates a query-time error — ensure a user is assigned to only one role that applies both if needed.
  • For Direct Lake models, RLS is enforced even when the query falls back to DirectQuery mode.

Practice

Q1. A report should show each salesperson only the rows for their own territory, dynamically based on their login. A single role must cover all salespeople. Which DAX function enables this pattern?
Answer: USERPRINCIPALNAME(). Why: Dynamic RLS uses USERPRINCIPALNAME() to return the signed-in user's UPN, matched against a column in the model mapping users to territories. One role definition handles all users without needing a role per person. USERNAME() is an alternative but returns the domain\user format.
Q2. A finance semantic model contains a Salary column. It should be completely invisible to most consumers — absent from field lists, report visuals, and metadata. Which security mechanism achieves this?
Answer: Object-Level Security (OLS) / Column-Level Security via columnPermissions with metadataPermission: none. Why: OLS hides both the data and the column name from the role, making it appear nonexistent. RLS would only filter rows — the column name would still be visible in field lists.
Q3. A developer with a Contributor workspace role opens a Power BI report that has RLS defined. Are the RLS filters applied to the developer's session?
Answer: No. Why: RLS applies only to Viewer-level consumers. Admin, Member, and Contributor workspace roles bypass RLS and always see all rows. The developer must use the "View as / Test as role" feature to simulate an end-user experience.
Q4. A Lakehouse owner wants to grant a third-party team read access to only the "SalesData/2024/" folder, not the entire Lakehouse. Which mechanism enables this?
Answer: A custom OneLake data-access role with a folder-scoped permission. Why: OneLake RBAC roles support folder-level scoping. The owner can modify or replace the default roles with a custom role that grants Read permission scoped specifically to "SalesData/2024/", leaving all other folders inaccessible.

1.1.4 Apply Sensitivity Labels to Items

Sensitivity labels in Microsoft Fabric are powered by Microsoft Purview Information Protection. Labels are created and published in the Microsoft Purview compliance portal, after which they surface in Fabric and Power BI Desktop to classify data asset confidentiality — for example, "Public", "Confidential", or "Highly Confidential". Labels can travel with data across supported export paths, extending governance outside the Fabric boundary.

Key facts

  • Labels must be created and published to users via a label policy in the Purview compliance portal before they appear in Fabric. Being in the "allowed users" list alone is insufficient.
  • Licensing: a Microsoft Purview Information Protection license plus a Power BI Pro or PPU license are required to apply labels to Power BI items.
  • Five label capabilities in Fabric:
    1. Manual labeling — user applies a label; supported for all Fabric items.
    2. Default labeling — label applied automatically on item creation or edit; all Fabric items (with limits for non-Power BI items).
    3. Mandatory labeling — users cannot save without a label; fully supported for Power BI items only.
    4. Programmatic labeling — via Power BI admin REST APIs; all Fabric items.
    5. Downstream inheritance — label propagates to dependent items; all Fabric items with limits.
  • Downstream label inheritance rules:
    • Power BI item → Power BI item: ✓ supported.
    • Fabric item → Fabric item: ✓ supported.
    • Fabric item → Power BI item: ✓ supported.
    • Power BI item → Fabric item: ✗ not supported.
  • Inheritance on creation: a notebook created from a Lakehouse inherits the Lakehouse label; a KQL Queryset created from a KQL Database inherits its label.
  • Source-data label inheritance is currently supported only for Power BI semantic models.
  • Supported export paths where labels travel with the file: Export to Excel, PDF, or PowerPoint; Analyze in Excel; PivotTable in Excel (requires Microsoft 365 E3+); download .pbix from Fabric.
  • CSV, TXT, and cross-tenant exports do not carry labels.
  • Sensitivity labels are not supported with Power BI Projects (.pbip) files — manage labels in the Fabric service post-publish.
  • Label coverage can be monitored in the OneLake catalog governance experience.
Gotcha: Sensitivity labels are not supported with the .pbip (Power BI Projects) file format. If your development workflow uses .pbip, labels must be applied or changed in the Fabric service after publishing — not in the project files.

Exam tips & gotchas

  • Labels must be published via a policy in Purview — simply creating a label does not make it available to users in Fabric.
  • Mandatory labeling (blocking save without a label) is fully supported only for Power BI items; other Fabric items have limited mandatory-labeling support.
  • Downstream inheritance does not flow from Power BI items to Fabric items — only the reverse direction (Fabric → Power BI) is supported.
  • CSV and TXT exports strip labels; only structured formats (Excel, PDF, PowerPoint, .pbix) carry labels out of Fabric.
  • .pbip files do not support sensitivity labels — this is a known limitation during the Power BI Projects preview.

Practice

Q1. A sensitivity label is applied to a published semantic model. A report author creates a new report from that semantic model in the same workspace. Will the report automatically inherit the label?
Answer: Yes, through downstream inheritance. Why: A Fabric semantic model to a Power BI report is a supported inheritance path. The new report inherits the label automatically when downstream inheritance is enabled (the default). The inheritance is applied at item creation.
Q2. A compliance officer wants to enforce that all Power BI reports must carry a sensitivity label before they can be saved. Which labeling capability satisfies this requirement?
Answer: Mandatory labeling. Why: Mandatory labeling prevents users from saving a Power BI item without first selecting a sensitivity label. It is configured through the Purview label policy and is fully supported for Power BI items.
Q3. A user exports a report page as a CSV file. Will the report's sensitivity label travel with the exported file?
Answer: No. Why: CSV and TXT exports are not in the list of supported export paths for label persistence. Only structured formats — Excel, PDF, PowerPoint, and .pbix downloads — carry the sensitivity label with the exported content.

1.1.5 Endorse Items

Endorsement is Fabric's mechanism for surfacing trusted, high-quality data assets to consumers. Three levels form a progressive trust hierarchy: Promoted (creator-assessed quality), Certified (organization-approved authority), and Master data (single source of truth for core reference data). Endorsed items display distinctive badges, appear higher in search and discovery lists, and show their badges in lineage view.

Key facts

  • Promoted: Any user with Write permission on the item can self-apply it. Signals the creator believes the item is production-ready and suitable for sharing.
  • Certified: Only users designated by a Fabric administrator (or a delegated domain administrator) can grant Certified status. Any user can request certification. Signals the item meets organizational quality standards and is authoritative.
  • Master data: Only users designated by a Fabric administrator can apply it. Any user can request it. Indicates the item is a core organizational reference source — the single source of truth (e.g., a product code list or customer master record).
  • Certification and Master data endorsement must first be enabled by a Fabric administrator in the admin portal. Without admin enablement, only Promoted is available.
  • Certification authority can be delegated to domain administrators, allowing per-domain certifiers without granting tenant-wide admin rights.
  • What can be endorsed:
    • Promoted and Certified: all Fabric and Power BI items, except Power BI dashboards.
    • Master data: only items that contain data (e.g., lakehouses, semantic models, warehouses). Cannot be applied to reports or dashboards.
  • Endorsement badges are visible in workspace lists, lineage view item cards, and OneLake catalog search results.
  • Endorsement is independent of sensitivity labels; both can coexist on the same item.
Key: Any user can request Certified or Master data status, but only admin-designated individuals can grant those badges. Promoted is the only tier that can be self-applied — by anyone with Write permission on the item.

Exam tips & gotchas

  • Power BI dashboards cannot be endorsed — neither Promoted, Certified, nor Master data applies to dashboards.
  • Master data cannot be applied to reports or dashboards — only to items that contain data.
  • Certified and Master data require admin enablement; if not enabled, only Promoted is available to all users.
  • Any user with Write permission on an item can apply Promoted — no admin involvement is needed for that level.
  • Certification authority can be delegated at the domain level, enabling per-domain governance without full tenant-admin privileges.

Practice

Q1. A data engineer wants to mark a Lakehouse as the authoritative source for all customer reference data across the organization. Which endorsement level is appropriate, and who can apply it?
Answer: Master data, applied by a Fabric administrator or an admin-designated user. Why: Master data is specifically designed for core organizational reference data representing the single source of truth. It cannot be self-applied — only admin-designated individuals can grant it, and the item must contain data (a Lakehouse qualifies).
Q2. A report author with Write permission on a semantic model wants to signal to colleagues that the model is production-ready and safe to use. Which endorsement can they apply without contacting an administrator?
Answer: Promoted. Why: Promoted is the only endorsement tier that can be self-applied by any user with Write permission on the item. Certified and Master data require an admin-designated approver — they cannot be self-granted.
Q3. Which of the following can receive the Master data endorsement: (a) a Power BI report, (b) a Fabric Lakehouse, (c) a Power BI dashboard?
Answer: (b) A Fabric Lakehouse only. Why: Master data applies only to items that contain data. Power BI reports and dashboards are visualization items; neither qualifies. A Lakehouse stores data and is a valid candidate for Master data endorsement.

1.2 Maintain the Analytics Development Lifecycle

1.2.1 Configure Version Control for a Workspace

Fabric Git integration connects a workspace to a branch in a cloud-hosted Git repository — either Azure DevOps or GitHub — so changes made in the workspace can be committed to source control, and changes made in source control can be pulled back into the workspace. Each workspace connects to exactly one branch and one folder within a repository, establishing a clear workspace-to-branch mapping.

The connection enables modern team practices: branching strategies, pull-request reviews, rollback to previous commits, and CI/CD automation. Semantic models are stored as TMDL files and reports as PBIR files, making them fully text-diffable during code review.

Key facts

  • Supported providers: Azure DevOps (cloud), GitHub (cloud), GitHub Enterprise (cloud). On-premises Azure DevOps Server is not supported.
  • Only a workspace Admin can connect or disconnect a workspace from Git.
  • Admins can optionally enable a workspace setting that allows Contributors/Members to also switch Git branches.
  • Two tenant switches must be enabled: (1) Users can synchronize workspace items with their Git repositories; (2) for GitHub, additionally Users can synchronize workspace items with GitHub repositories.
  • Authentication: Azure DevOps uses OAuth2 or Service Principal; GitHub requires a Personal Access Token (fine-grained with Contents read/write, or classic with repo scope).
  • Initial sync logic: if workspace is empty and Git branch has content → content copied to workspace; if both have content → user chooses direction.
  • Day-to-day operations: Commit (workspace → Git), Update from Git (Git → workspace), Undo (revert workspace items to last synced state).
  • Supported items: Lakehouses, Notebooks, Spark Job Definitions, Dataflow Gen2, Pipelines, Eventhouses, KQL Databases, Warehouses (preview), Power BI Reports (preview), Semantic models (preview). Unsupported items are silently ignored during sync — not deleted.
  • Source code format: semantic models use TMDL (YAML-like); reports use PBIR (JSON). Both are human-readable and suitable for line-by-line code review.
  • Workspace folder hierarchy is preserved in the repository.
Key: Git integration is workspace-to-branch, not workspace-to-repository. One workspace maps to exactly one branch (and one folder). To use a multi-branch workflow, map separate workspaces to separate branches and merge via standard Git pull requests.

Exam tips & gotchas

  • On-premises Azure DevOps Server is not supported — only cloud Git providers are supported.
  • Only workspace Admins can connect or disconnect from Git. Members and Contributors cannot, unless explicitly allowed to switch branches via the Admin-controlled workspace setting.
  • Unsupported item types are skipped silently during sync — they are not deleted from the workspace.
  • Semantic models → TMDL; reports → PBIR. Both are plain text and diffable.
  • The GitHub tenant switch must be enabled separately from the general Git sync tenant switch.

Practice

Q1. A team wants to connect their Fabric workspace to a self-hosted, on-premises Azure DevOps Server. Is this configuration supported?
Answer: No. Why: Fabric Git integration supports only cloud-hosted Git providers: Azure DevOps (cloud), GitHub (cloud), and GitHub Enterprise (cloud). On-premises Azure DevOps Server is explicitly not supported.
Q2. A developer with a Contributor workspace role wants to connect the workspace to a GitHub repository. What must happen first?
Answer: A workspace Admin must perform the connection. Why: Only workspace Admins can connect or disconnect a workspace from a Git repository. The Contributor role cannot perform this action regardless of the Git provider. An Admin must also ensure the required tenant switches are enabled.
Q3. What file formats does Fabric Git integration use to store semantic models and reports in source control, and why are those formats significant?
Answer: TMDL (Tabular Model Definition Language) for semantic models; PBIR (Power BI Enhanced Report format) for reports. Why: Both are plain-text, human-readable formats — TMDL is YAML-like and PBIR is JSON-based. They can be reviewed as readable text diffs in pull requests, enabling proper code review and change tracking for analytics assets.

1.2.2 Create and Manage a Power BI Desktop Project (.pbip)

A Power BI Project (.pbip) saves a report and its semantic model as a collection of individual plain-text files in a folder structure, rather than as a single binary .pbix. The format is designed for source-control workflows and team collaboration — files can be opened in any text editor, diffed in Git, and automated with scripts or programmatic tooling.

Key facts

  • Enabled via: File → Options and settings → Options → Preview features → Power BI Project (.pbip) save option.
  • Folder structure created on save:
    • ProjectName.Report/ — report pages and visuals in PBIR (JSON) format.
    • ProjectName.SemanticModel/ — model definition in TMDL (YAML-like) format.
    • .gitignore — auto-created; excludes local cache files (cache.abf, localSettings.json) from version control.
    • ProjectName.pbip — optional pointer file for opening the report in Desktop.
  • Multiple reports can share a single semantic model folder in the same project — each report has its own PBIR folder.
  • The .pbip pointer file is optional; reports can also be opened by opening the definition.pbir file directly.
  • TMDL — YAML-like syntax for the semantic model definition (tables, measures, relationships); supports batch edits.
  • PBIR — JSON format for report pages and visuals; enables programmatic report generation.
  • External tool edits (outside Desktop) require a Desktop restart to take effect — not picked up in real time.
  • Files must be saved in UTF-8 without BOM encoding; configure Git with autocrlf to avoid line-ending diffs.
  • Deploying .pbip to Fabric: via Fabric Git integration (recommended for teams), Fabric REST APIs, or Power BI Desktop Publish.
  • PBIX ↔ PBIP conversion is only possible in Power BI Desktop via File → Save As. No programmatic conversion tool exists.
  • Sensitivity labels are not supported with .pbip files — apply labels in the Fabric service after publishing.
  • Maximum Windows path length (260 characters) can be reached with long table or object names.
  • Cannot save .pbip directly to OneDrive or SharePoint cloud paths — use a locally synced OneDrive folder (with awareness of potential sync conflicts).
Gotcha: Sensitivity labels are not supported with the .pbip format. Labels must be applied in the Fabric service after publishing. Also, .pbip cannot be saved directly to a SharePoint or OneDrive cloud URL — use a locally synced folder.
Tip: Visual Studio Code is the recommended editor for .pbip files — it provides syntax highlighting, validation, IntelliSense for TMDL, and Git diff visualization out of the box.

Exam tips & gotchas

  • The .pbip format requires a Preview feature toggle in Desktop — it is not enabled by default.
  • Unlike .pbix, a .pbip project contains no data cache — only definitions. Data is loaded from sources when the file is opened.
  • External tool edits to TMDL or PBIR files are picked up only after restarting Power BI Desktop, not in real time.
  • Sensitivity labels cannot be applied to .pbip files — manage them post-publish in the Fabric service.
  • For team workflows, deploy to Fabric via Git integration or REST APIs rather than solely relying on the Desktop Publish button.

Practice

Q1. A developer edits a measure directly in the TMDL file of a .pbip project using VS Code while Power BI Desktop has the project open. When will Desktop reflect the change?
Answer: After restarting Power BI Desktop. Why: External edits to TMDL or PBIR files outside Desktop require a Desktop restart to be recognized. Changes made while Desktop is open are not picked up in real time.
Q2. A compliance team needs to apply sensitivity labels to a report being developed in .pbip format. Can labels be applied from within the .pbip project files?
Answer: No. Why: Sensitivity labels are not supported with the .pbip file format. Labels must be applied in the Fabric service after the project has been published. The .pbip files themselves contain no label metadata.
Q3. A team wants multiple report authors to independently develop reports that all use the same underlying semantic model — without duplicating the model definition. How does .pbip support this pattern?
Answer: Multiple report folders can share a single SemanticModel folder within the same .pbip project. Each report has its own PBIR folder pointing to the shared semantic model definition. This enables independent report development without duplicating the model.

1.2.3 Create and Configure Deployment Pipelines

Deployment pipelines provide a structured promotion workflow for Fabric items across environments — typically Development → Test → Production — where each stage is mapped to a distinct Fabric workspace. Content is copied from one stage to the next, and deployment rules allow you to configure stage-specific settings (such as different connection strings or parameter values) without modifying the source items themselves.

Key facts

  • A pipeline can have 2 to 10 stages; the default is 3 (Development, Test, Production). Stages can be added, removed, or renamed.
  • Each stage maps to a Fabric workspace. A Fabric capacity is required — shared capacity is not supported.
  • Deployment copies content (does not move it); paired items in the target stage are overwritten.
  • Item pairing occurs on the first deployment by name. Items added to the source workspace after the workspace was assigned to the pipeline are not auto-paired; they must be included in an explicit deployment to create the pairing.
  • Paired items remain linked even if renamed in one stage — differing names between stages are permitted.
  • Deployment can be triggered manually or via the Deployment Pipelines REST APIs for automation/CI-CD integration.
  • Workspace folder hierarchy is preserved during deployment; items can also be selected from a flat cross-folder list.
  • Deployment rules are configured on the target stage and take effect on the next deployment:
    • Data source rules: replace connection strings between stages (requires same data source type — e.g., SQL Server → SQL Server).
    • Parameter rules: override Power Query parameter values for the target stage.
    • Default lakehouse rules: specify the default Lakehouse for notebooks in the target stage.
  • Rules cannot be created in the Development stage — only Test and Production stages support rules.
  • You must be the item owner to create a deployment rule on that item.
  • If a data source or parameter referenced in a rule is changed or removed in the source stage, the rule becomes invalid and the next deployment fails.
  • Items that support data source rules: Dataflow Gen1, Semantic model, Paginated report, Mirrored database. Notebooks support only default lakehouse rules.
Gotcha: Deployment rules are defined on the target stage (e.g., Production), not the source. They do not take effect until the next deployment — comparing stages before deploying shows them in a "proposed" state.
Key: Rules cannot be created in the Development stage. If environment-specific configuration is needed in Development, use Power Query parameters in the source queries instead.

Exam tips & gotchas

  • A pipeline can have 2–10 stages, not just 3. The exam may test the valid range.
  • Deployment rules are set on the target stage and apply on the next deployment, not immediately.
  • Rules cannot be created in the Development stage; only Test and Production stages are eligible.
  • Data source rules require the same source type; cross-type swaps are not supported — use parameter rules instead for cross-type scenarios.
  • New items added to the source workspace after pipeline assignment are not auto-paired — they require an explicit first deployment to create the pairing.
  • A Fabric capacity is required; deployment pipelines are not available for shared-capacity workspaces.

Practice

Q1. A team wants the Production stage of a deployment pipeline to connect to a different Azure SQL Database than Development. Which deployment rule type enables this, and on which stage is the rule configured?
Answer: A data source rule, configured on the Production (target) stage. Why: Data source rules override connection strings when content is promoted from source to target. They are always defined on the target stage. The rule requires the same data source type in both stages (both must be Azure SQL Database).
Q2. After assigning a Development workspace to a pipeline and deploying once to Test, a developer adds a new notebook to Development. They then run another deployment to Test. The notebook does not appear in Test. Why?
Answer: The notebook was added after the workspace was assigned to the pipeline and has not been paired with the Test stage yet. Why: Items added to a workspace after pipeline assignment are not automatically paired. The developer must include the notebook in an explicit deployment to create the pairing for the first time.
Q3. A deployment pipeline has a data source rule on the Production stage. A developer changes the Development semantic model to use a completely different data source type. What happens at the next deployment?
Answer: The deployment fails. Why: If the data source referenced by a rule is removed or changed to a different type in the source stage, the rule becomes invalid. Fabric validates rules before deploying and will block the deployment with an error indicating the rule cannot be resolved.

1.2.4 Perform Impact Analysis of Downstream Dependencies

Lineage view and impact analysis together expose the full dependency graph of Fabric artifacts across workspaces. Lineage view shows data flowing from sources through lakehouses, dataflows, warehouses, and semantic models to downstream reports and dashboards. Impact analysis, available directly from a semantic model's lineage card, quantifies how many consumers would be affected by a model change and provides 30-day usage metrics to help prioritize communication and remediation effort.

Key facts

  • Access lineage view: workspace list view → View → Lineage.
  • Requires at least Contributor workspace role and a Power BI Pro license. Viewers cannot access lineage view.
  • Lineage view shows both upstream dependencies (data sources, dataflows, lakehouses, warehouses) and downstream dependencies (reports, dashboards) — including external items from other workspaces.
  • Item cards display: last refresh time, endorsement badges, data source type, and gateway information.
  • Selecting the double-arrows icon on a card highlights all related upstream/downstream artifacts and dims the rest.
  • Semantic model impact analysis (via the impact analysis button on the semantic model's card):
    • Shows counts of impacted workspaces, reports, and dashboards.
    • Displays 30-day usage metrics (excluding the current day): distinct viewer count and total view count per downstream artifact.
    • Helps prioritize which downstream consumers are most critical — a report with thousands of viewers warrants more urgent communication than one with a handful.
  • Notify contacts: sends an email under the analyst's name to the contact lists of all affected workspaces — useful for coordinating change communication.
  • Impact analysis can only be performed on semantic models in your own workspace; navigate to the source workspace to analyze external (cross-workspace) semantic models.
  • Items the user cannot access appear as "Limited access" in the impact panel — their existence is shown for count accuracy, but details are withheld.
  • From Power BI Desktop, republishing a changed semantic model displays a pre-publish warning with the count of impacted workspaces, reports, and dashboards.
  • Usage metrics are not supported for personal workspaces (My Workspace).
Tip: Before modifying a widely used semantic model, run impact analysis to identify high-traffic downstream reports. Use the "Notify contacts" feature to proactively inform workspace owners of all affected workspaces about the upcoming change.

Exam tips & gotchas

  • Lineage view requires at least Contributor role — Viewers are excluded.
  • Usage metrics cover the last 30 days, excluding the current day. Metrics older than 30 days are not shown.
  • To analyze an external (cross-workspace) semantic model, you must navigate to its source workspace — you cannot analyze models you do not own from another workspace.
  • Items the user lacks permission to view appear as "Limited access" — their count is included in totals but their details are hidden.
  • Usage metrics are unavailable for My Workspace (personal workspace) items.

Practice

Q1. A data engineer is about to make breaking changes to a shared semantic model. They want to know how many reports and dashboards would be affected and how heavily each is used. Which Fabric feature provides this information?
Answer: Semantic model impact analysis, accessed from the lineage view. Why: Impact analysis shows the count of downstream workspaces, reports, and dashboards, plus 30-day usage metrics (unique viewers and view counts per artifact). This enables the engineer to identify high-priority consumers and use "Notify contacts" for proactive communication.
Q2. A workspace Viewer wants to review the lineage of items in the workspace to understand data dependencies. Can they access lineage view?
Answer: No. Why: Lineage view requires at least a Contributor workspace role and a Power BI Pro license. The Viewer role is insufficient — Viewers are excluded from this feature.
Q3. In the impact analysis panel for a semantic model, several downstream items are listed as "Limited access." What does this indicate?
Answer: The user performing the analysis lacks permission to view those items. Why: Fabric includes these items in the impact count (to give an accurate total) but withholds their names, owners, and usage metrics because the requesting user does not have access to those artifacts.

1.2.5 Deploy and Manage Semantic Models by Using the XMLA Endpoint

The XMLA endpoint exposes the XML for Analysis (XMLA) protocol on Power BI semantic models hosted in Premium, Premium Per User (PPU), or Power BI Embedded workspaces. It enables external client tools — including Tabular Editor, SQL Server Management Studio (SSMS), DAX Studio, and ALM Toolkit — to connect using a standard Analysis Services–compatible connection string for querying, metadata scripting, programmatic refresh, and structural model management. All communication over the XMLA protocol is fully encrypted.

The endpoint is read-only by default. An administrator must explicitly enable read-write mode at the capacity level before tools can push model changes.

Key facts

  • Connection string format: powerbi://api.powerbi.com/v1.0/[tenant name]/[workspace name] — used as the server name in any Analysis Services–compatible client tool. Workspace names must be URI-encoded (spaces as %20).
  • Read-only is the default for all Premium/PPU workspaces. Allows: querying data, reading metadata, viewing model schema, profiling queries.
  • Read-write must be explicitly enabled by an administrator:
    • Premium capacity: Admin portal → Capacity settings → Power BI Premium → [capacity] → Workloads → XMLA Endpoint → select Read Write. Applies to all workspaces on that capacity.
    • PPU: Admin portal → Premium Per User → Semantic model workload settings → XMLA Endpoint → Read Write.
  • Read-write enables: deploying model changes via TMSL scripts, programmatic refresh/processing, advanced structural modifications, batch scripting.
  • Client tool requirements:
    • DAX Studio — read-only (DAX queries, performance profiling).
    • Tabular Editor 2.x — read-only for queries; read-write for metadata and model changes.
    • ALM Toolkit — read-only for comparison; read-write for deployment and model sync.
    • SSMS ≥ 18.9 — read-only for queries; read-write for TMSL scripting and object management.
    • SQL Server Profiler (with SSMS ≥ 18.9) — read-only (trace and event profiling).
    • Excel PivotTables — read-only.
    • Power BI Report Builder — read-only (paginated report design).
  • Client libraries used: MSOLAP, ADOMD.NET, AMO — the same libraries used for Azure AS and SSAS.
  • The Az.AnalysisServices PowerShell module is not supported for Power BI semantic models. Use the SqlServer PowerShell module (version ≥ 21.1.18256) with the powerbi:// connection string instead.
  • TMSL (Tabular Model Scripting Language): JSON-based language for model management operations (create, alter, refresh, process) executed via SSMS or XMLA scripts.
  • For models larger than ~1 GB, enabling the large models storage format significantly reduces write overhead during XMLA operations.
Gotcha: The XMLA endpoint is read-only by default. Tools like Tabular Editor and ALM Toolkit cannot push model changes until a capacity administrator explicitly enables Read Write in the Admin portal under Capacity settings.
Key: Never use the Az.AnalysisServices PowerShell module for Power BI semantic models — it is designed for Azure Analysis Services and is not compatible with Power BI. Use the SqlServer module (≥ 21.1.18256) instead.

Exam tips & gotchas

  • XMLA is read-only by default — read-write requires explicit admin enablement at the capacity level, not the workspace level.
  • The connection string uses the powerbi:// scheme, not a traditional server URL. Workspace names must be URI-encoded.
  • Az.AnalysisServices PowerShell module is not compatible with Power BI — use the SqlServer module.
  • DAX Studio and Excel PivotTables operate read-only. Tabular Editor, ALM Toolkit, and SSMS require read-write to push changes.
  • The XMLA endpoint is available on Premium capacity, PPU, and Power BI Embedded — not on workspaces using only shared/Pro capacity.

Practice

Q1. A capacity administrator wants to enable Tabular Editor to deploy model schema changes to a Premium capacity workspace. What configuration step is required?
Answer: In the Admin portal, navigate to Capacity settings → Power BI Premium → [capacity] → Workloads → XMLA Endpoint → select Read Write. Why: The XMLA endpoint is read-only by default. Enabling Read Write at the capacity level unlocks write operations for all tools connecting via XMLA to workspaces on that capacity, including Tabular Editor.
Q2. A developer connects to a Power BI Premium semantic model via the XMLA endpoint using DAX Studio. What operations are available?
Answer: Read-only operations — running DAX queries against the model, viewing metadata, and performance profiling. Why: DAX Studio uses read-only XMLA connectivity. It cannot modify the model structure or trigger programmatic refreshes. Those operations require a tool like Tabular Editor or SSMS with the endpoint set to Read Write.
Q3. A developer attempts to use the Az.AnalysisServices PowerShell module to refresh a Power BI Premium semantic model via the XMLA endpoint. What is the expected outcome?
Answer: Failure — the Az.AnalysisServices module is not supported for Power BI semantic models. Why: Az.AnalysisServices is designed for Azure Analysis Services and does not support the Power BI XMLA endpoint. The correct approach is to use the SqlServer PowerShell module (version ≥ 21.1.18256) with the powerbi:// connection string.

1.2.6 Create and Update Reusable Assets (.pbit, .pbids, Shared Semantic Models)

Power BI and Fabric provide three complementary mechanisms for reusing analytics assets across teams and projects. A .pbit (Power BI Template) packages a report's structure and data model — without any data — as a starting point for new reports. A .pbids (Power BI Data Source) file distributes connection specifications without credentials, directing report authors to the correct data endpoint. Shared semantic models publish a single governed model to the Fabric service so report authors across workspaces can build their own reports on top of it, supporting the managed self-service BI pattern.

Key facts — .pbit (Power BI Template)

  • Created via: File → Export → Power BI template in Power BI Desktop.
  • Contains: report pages and visuals, data model schema (tables, relationships, measures), all query definitions.
  • Does not contain data — no data cache is included. File size is smaller than a .pbix.
  • Opening a .pbit: Desktop prompts for parameter values (if defined), then credentials and data source location, then loads data. The result is a full .pbix with live data — the template is the starting point.
  • Useful for standardizing report layouts, measure naming, and data model structure across multiple report authors.

Key facts — .pbids (Power BI Data Source)

  • Contains: connection information only (protocol, server, database address). Contains no credentials and no table/schema definitions.
  • Opening a .pbids: Desktop prompts for credentials, then displays the Navigator to select tables/views.
  • Supports specifying "mode" as "DirectQuery" or "Import"; if absent, the user is prompted to choose.
  • Currently supports a single data source per file — specifying multiple sources causes an error.
  • Created by: exporting from an existing .pbix (File → Options → Data source settings → export as PBIDS), or manually authoring JSON in the DSR (Data Source Reference) format with a .pbids extension.
  • Useful for distributing standardized connection specs to report authors without sharing credentials.

Key facts — Shared Semantic Models

  • A semantic model published to the Fabric service can be shared with report authors across workspaces — the foundation of the managed self-service BI pattern.
  • Share via: OneLake catalog or data details page → Share → configure permission level.
  • Key permissions on a shared semantic model:
    • Read: view metadata and existing reports; cannot query underlying data or create new reports.
    • Build: create new reports and dashboards on top of the semantic model — the core self-service BI permission.
    • Write: modify the semantic model definition.
    • Reshare: grant access to other users.
  • Downstream reports built on shared semantic models are visible in lineage view, with the source workspace indicated on the semantic model card.
  • Impact analysis is especially valuable for widely shared semantic models, since changes can affect consumers across many workspaces simultaneously.
// Example .pbids file — Azure SQL Database connection
// Contains only the address; NO credentials, NO table selection
{
  "version": "0.1",
  "connections": [
    {
      "details": {
        "protocol": "tds",
        "address": {
          "server": "myserver.database.windows.net",
          "database": "SalesDB"
        }
      },
      "mode": "DirectQuery"
    }
  ]
}
Key distinction — all four formats:
  • .pbit — report structure + model schema; no data; prompts for credentials and source on open; distribution format.
  • .pbids — connection address only; no credentials, no schema; prompts for credentials and table selection on open; single data source per file.
  • .pbip — full project definition (model + report) as editable text files (TMDL + PBIR); designed for development and source control, not distribution.
  • Shared semantic model — live published model in the service; shared via Build permission; enables cross-workspace self-service BI; change impact is tracked via lineage.

Exam tips & gotchas

  • .pbit contains no data — only structure. Opening it always triggers data load from the live source.
  • .pbids contains no credentials and no schema — only the connection address. The user provides credentials and selects tables at open time.
  • .pbids supports only a single data source per file — attempting multiple sources causes an error.
  • Build permission on a shared semantic model is required for self-service report creation. Read alone is insufficient for authoring new reports.
  • The .pbip format is a development format (source control, team collaboration), not a distribution format like .pbit or .pbids.
  • Impact analysis is especially important for shared semantic models because one change can ripple across report consumers in many workspaces.

Practice

Q1. A BI team wants to distribute a standardized report template — with pre-built measures, a star schema, and consistent branding — to 10 regional report developers. Each developer will connect to their own regional database. Which file format is most appropriate?
Answer: .pbit (Power BI Template). Why: A .pbit packages the report structure, data model, and query definitions without any data. Each developer opens it, provides their own credentials and data source details, and immediately has a fully structured starting point. The template enforces consistent measures and layout without pre-loading any data.
Q2. A data architect wants to ensure all report developers in the company always connect to the correct Azure SQL Database server and database name, without distributing credentials. Which file format achieves this?
Answer: .pbids (Power BI Data Source file). Why: A .pbids file contains only the connection address (protocol, server, database) — no credentials and no table definitions. Authors open it, are prompted for credentials, and then select their own tables. The connection target is standardized without exposing any credentials.
Q3. A report author in Team B wants to build new Power BI reports on top of a semantic model owned by Team A in a different workspace. Team A shares the model but grants only Read permission. Can Team B's author create new reports?
Answer: No. Why: Creating new reports on a shared semantic model requires Build permission. Read permission only allows viewing metadata and existing reports — it does not enable report authoring. Team A must explicitly grant Build permission to enable self-service report creation on the shared model.
Q4. A developer opens a .pbids file and sees the Power BI Desktop Navigator dialog prompting them to select tables. What information did the .pbids file supply, and what did it intentionally omit?
Answer: The .pbids file supplied the connection address (protocol, server, and database name). It intentionally omitted credentials (the developer entered those when prompted) and table/schema definitions (the developer selects tables in the Navigator). The .pbids format is a connection specification only — no credentials, no schema, no data.
