2.1 Get data

2.1.1 Create a data connection

Before any Fabric item (Dataflow Gen2, pipeline, or semantic model) can pull from an external source, a connection must exist that stores the endpoint URL and credentials. Fabric centralises connection management at Settings → Manage connections and gateways, where administrators create, edit, share, and delete both cloud and on-premises connections.

Cloud sources that are publicly reachable (Azure SQL, ADLS Gen2, SharePoint Online, REST APIs) use a cloud connection with no additional software. Private or on-premises sources require a gateway installed on a machine that can reach the source.

Key facts

  • On-premises data gateway (standard mode) — multi-user; supports scheduled refresh, DirectQuery, and live connection. Requires gateway software on a customer-managed machine.
  • On-premises data gateway (personal mode) — single user only; scheduled refresh only; no DirectQuery or live connection support.
  • Virtual network (VNet) data gateway — Microsoft-managed SaaS; multi-user; requires Premium capacity or PPU; no on-premises software to install.
  • Cloud connections are created in the Connections tab (New → Cloud) for publicly accessible sources such as Azure SQL, ADLS Gen2, and SharePoint Online.
  • Authentication options: Basic, OAuth2, Service Principal (Entra app registration). OAuth2 caveat: long-running queries may fail when the access token expires; cross-tenant Entra accounts are not supported.
  • A connection can be made shareable: other workspace members can use it in their items without seeing the stored credentials.
  • Gateway version ≥ 3000.214.2 is required to support Fabric pipelines against on-premises sources; older versions support only Dataflows and semantic-model refresh.
  • Fabric Data Factory supports 26+ on-premises connectors via gateway (SQL Server, Oracle, SAP HANA, PostgreSQL, MySQL, Salesforce, SharePoint, ODBC, and more).
// Create a shareable cloud connection (Fabric REST API)
// POST https://api.fabric.microsoft.com/v1/connections

  "connectivityType": "ShareableCloud",
  "connectionDetails": {
    "type": "AzureSqlDatabase",
    "parameters": [
      { "name": "server",   "value": "myserver.database.windows.net" },

Tip: Use standard-mode gateway (not personal) whenever DirectQuery or multi-user scheduled refresh is required. Personal mode is a single-user development convenience and must not be used for shared production reports.

Exam tips & gotchas

  • Personal-mode gateway = single user + scheduled refresh only; no DirectQuery or live connection.
  • VNet gateway = Microsoft-managed, no software install, but requires Premium P or Fabric F capacity — it is not free.
  • OAuth2 tokens expire: long-running ETL jobs may fail mid-run when the token lifetime is exceeded.
  • Shareable connections hide credentials — members can use the connection but cannot view the password or SPN secret.
  • Fabric pipeline activities against on-premises sources require gateway ≥ 3000.214.2; older gateways block pipeline execution even when Dataflows work fine.

Practice

Q1. A team needs multiple analysts to refresh reports via DirectQuery against an on-premises SQL Server. Which gateway type is required?
Answer: On-premises data gateway (standard mode). Why: Standard mode supports DirectQuery and live connections for multiple simultaneous users. Personal mode supports only scheduled refresh for a single user and cannot be shared.
Q2. A new analyst needs to use an existing Azure SQL connection in their Dataflow without seeing the stored password. What feature enables this?
Answer: Marking the connection as shareable. Why: A shareable connection lets workspace members reference the connection by its ID in their Dataflows and pipelines without exposing the underlying credentials (password or SPN secret).
Q3. A data engineer installs an on-premises gateway running version 3000.200.0 and tries to use it in a Fabric pipeline Copy activity targeting an on-premises Oracle source. The activity fails. What is the most likely cause?
Answer: The gateway version is below the required minimum (3000.214.2). Why: Older gateway versions support only Dataflows and semantic model scheduled refresh — not pipeline activities. Upgrading the gateway software resolves the issue.

2.1.2 Discover data via OneLake catalog & Real-Time hub

Microsoft Fabric provides two first-class discovery experiences. The OneLake data catalog (previously OneLake data hub) is a searchable, organization-wide inventory of every Fabric item — lakehouses, warehouses, KQL databases, semantic models, Eventstreams, and more — that you have access to. It surfaces endorsement badges, lineage, descriptions, and contact information so teams can locate authoritative data without direct knowledge of workspace names.

The Real-Time hub is a streaming-data marketplace: it lists all Eventstream sources, Azure Event Hubs namespaces, MQTT topics, and other event producers that are accessible to you, and it provides one-click connections from those streams into Fabric workloads.

Key facts

  • OneLake catalog — search or browse by item type, workspace, domain, or endorsement; "All data" tab shows everything you have read access to across the entire tenant.
  • Endorsements: Promoted (workspace members can endorse) and Certified (tenant-admin-designated certifiers only). Certified items rank higher in search and have a gold badge.
  • Clicking any catalog item opens a detail panel with description, contact, freshness, sensitivity label, and lineage; users can request access directly from the panel if they lack permission.
  • Domains (logical groupings of workspaces) appear as filter facets in the catalog, enabling business-unit-level browsing.
  • Real-Time hub — surfaces all event producers visible to you; can create and connect Eventstreams, alert rules (Activator), or KQL eventhouses from a stream in one step.
  • Real-Time hub "My streams" tab shows Eventstreams you own; "All streams" shows streams from the entire organisation you have access to; "Microsoft sources" shows connectable Azure services (Event Hubs, IoT Hub, Service Bus, Azure Blob Storage events).
  • From Real-Time hub you can create an alert (Fabric Activator) directly on a stream without writing code.
  • Items without sensitivity labels set by workspace admins inherit the tenant default label configured in Microsoft Purview.
Key: The OneLake catalog is a read-only discovery layer; it does not itself store or move data. All items shown live in their respective workspaces; the catalog provides metadata and access pathways only.

Exam tips & gotchas

  • Only tenant-admin-designated certifiers can apply the Certified badge; any workspace member can Promote.
  • Real-Time hub is the correct Fabric entry point for connecting to Azure Event Hubs, IoT Hub, and MQTT topics — not the Fabric portal Settings.
  • The catalog searches item names and descriptions but does not index the actual data inside tables or files.
  • "Request access" in the catalog triggers a Fabric access workflow; the workspace admin must approve it — the request does not auto-grant permission.

Practice

Q1. A data steward wants to flag a lakehouse as the official source of customer data that all teams should trust. Which endorsement level is most appropriate, and who can apply it?
Answer: Certified, applied by a tenant-admin-designated certifier. Why: The Certified badge conveys org-wide authority; workspace members can only Promote, which signals recommendation without the same governance weight.
Q2. A business analyst searches the OneLake catalog for "sales" but cannot find the lakehouse owned by another team. What is the most likely reason?
Answer: The analyst does not have read access to that workspace. Why: The catalog shows only items the signed-in user has at least Viewer-level access to. Items in workspaces the user cannot see are hidden.
Q3. Which Fabric experience lets you create a one-click connection from an Azure Event Hubs namespace to an Eventhouse KQL database?
Answer: Real-Time hub. Why: Real-Time hub lists Azure Event Hubs under "Microsoft sources" and provides a wizard to route events directly to Eventhouse, Eventstream, or Activator alerts without custom pipeline code.

2.1.3 Ingest or access data

Fabric offers a layered set of ingestion options ranging from ad-hoc file upload to fully orchestrated pipelines. Choosing the right mechanism depends on data volume, latency needs, transformation complexity, and team skill set. A key architectural principle is that all Fabric storage items ultimately write data to OneLake in Delta-Parquet format, enabling read sharing across engines.

The medallion architecture (Bronze → Silver → Gold) is the recommended organisational pattern: raw data lands in the Bronze layer, cleaned and conformed data in Silver, and business-ready aggregates in Gold. Fabric tools map naturally to each layer.

Key facts

  • Upload — drag-and-drop small CSV/Parquet/JSON files directly into a Lakehouse Files section; not suitable for large or recurring loads.
  • Dataflows Gen2 — low-code Power Query M transforms; writes results to a destination (Lakehouse, Warehouse, Azure SQL, etc.); supports query folding for source-pushdown; scheduled up to 48× per day.
  • Pipelines (Data Factory) — orchestration engine; Copy activity for bulk ingest; supports 90+ connectors; parameterisable; error handling and retry; can call notebooks, Dataflows, stored procedures, and REST APIs.
  • Notebooks / Spark — PySpark or Scala for complex transforms; read Lakehouse files or Delta tables, transform with Spark DataFrame API, write back as Delta; best for large-scale or schema-on-read scenarios.
  • Shortcuts — virtual references to external storage (ADLS Gen2, S3, GCS, Dataverse, Google Drive, S3-compatible) without physically copying data; appear as tables or files in the Lakehouse and are read by any Fabric engine.
  • COPY INTO (Warehouse) — T-SQL bulk load command that reads Parquet or CSV from ADLS Gen2 or OneLake paths directly into Fabric Warehouse tables; no pipeline required for flat loads.
  • Eventstream — real-time ingestion from Event Hubs, IoT Hub, Kafka, CDC sources, or custom apps into Eventhouse KQL database, Lakehouse Delta tables, or Activator alerts with sub-second latency.
  • Pipelines support incremental copy via high-watermark patterns (store last-loaded timestamp in a parameter table, filter source with a WHERE clause in the Copy source query).
-- COPY INTO: bulk-load Parquet from ADLS Gen2 into Fabric Warehouse
COPY INTO dbo.SalesFact
FROM 'https://myaccount.dfs.core.windows.net/raw/sales/2024/*.parquet'
WITH (
    FILE_TYPE = 'PARQUET',
    CREDENTIAL = (IDENTITY = 'Managed Identity')
);

# PySpark: read CSV from Lakehouse Files, clean, write as Delta table
from pyspark.sql import functions as F

df = spark.read.option("header", True).csv(
    "Files/raw/sales/2024/*.csv"
)

df_clean = (df
    .withColumn("SaleDate", F.to_date("SaleDate", "yyyy-MM-dd"))
    .withColumn("Amount",   F.col("Amount").cast("double"))
    .dropna(subset=["OrderID", "SaleDate"])
)

df_clean.write.format("delta").mode("append").saveAsTable("SalesFact")
Tip: Use shortcuts when data already lives in ADLS Gen2 or S3 and the team needs access from multiple Fabric engines (Lakehouse, Warehouse, Spark) without duplicating storage costs. Shortcuts are zero-copy — only metadata is stored in OneLake.

Exam tips & gotchas

  • COPY INTO is a Fabric Warehouse–only command; it is not available at the Lakehouse SQL analytics endpoint.
  • Shortcuts are read-only views of external storage — you cannot INSERT or UPDATE data in the source via a shortcut.
  • Eventstream supports only append semantics; it cannot update existing Eventhouse rows.
  • Dataflows Gen2 can write to a Fabric Warehouse destination (DML upsert/overwrite handled by the runtime) — this is distinct from running COPY INTO yourself.
  • Pipeline Copy activity vs. COPY INTO: Copy activity works for any destination (Lakehouse, Warehouse, Azure SQL); COPY INTO is T-SQL only and targets the Warehouse.

Practice

Q1. A team wants to query petabytes of Parquet files stored in an Azure Data Lake Storage Gen2 account from a Fabric Lakehouse without duplicating the data. Which ingestion mechanism should they use?
Answer: Shortcut. Why: Shortcuts create a virtual reference to the ADLS Gen2 location; Spark, the SQL analytics endpoint, and OneLake all read the files without copying them into the Lakehouse's own storage.
Q2. A Fabric pipeline needs to load 50 GB of Parquet files from ADLS Gen2 directly into a Fabric Warehouse table with no transformation. Which is the most efficient method?
Answer: COPY INTO (T-SQL). Why: COPY INTO is a bulk-optimised T-SQL command that reads Parquet directly from ADLS Gen2/OneLake paths into a Warehouse table in parallel without per-row overhead.
Q3. Which Fabric tool is best suited for a recurring ETL job that must read from an on-premises SQL Server, join with a Lakehouse Delta table, and write the result to another Lakehouse table using complex business logic?
Answer: A Notebook (PySpark) called from a Pipeline. Why: The pipeline provides scheduling, retries, and the gateway-connected Copy activity to extract from on-premises SQL Server; the notebook handles the complex PySpark join and write logic that goes beyond what a Dataflow Gen2 or simple COPY INTO can express.

2.1.4 Choose between data stores

Fabric surfaces four primary analytical data stores, each optimised for a different workload profile. Choosing incorrectly leads to limitations that cannot be worked around — for example, a Lakehouse SQL analytics endpoint silently blocks DML writes, and an Eventhouse cannot perform row updates. Understanding the architectural contracts of each store is a core exam competency.

Key facts

  • Lakehouse — stores files and Delta tables in OneLake. Accessed via Spark (full read/write) or the auto-generated SQL analytics endpoint (read-only SELECT; CREATE VIEW, FUNCTION, PROCEDURE allowed; no INSERT/UPDATE/DELETE/CREATE TABLE via SQL endpoint). Best for big-data engineering with Spark.
  • Fabric Data Warehouse — full T-SQL analytical warehouse; supports INSERT, UPDATE, DELETE, MERGE, CREATE TABLE, CTAS, SELECT INTO. Data stored as Delta-Parquet in OneLake. Best for BI teams comfortable with T-SQL who need DML at scale.
  • Eventhouse (KQL database) — append-optimised columnar store for time-series, logs, and telemetry. Primary query language is KQL. Supports very high ingest throughput and sub-second freshness. Rows cannot be updated or deleted in the standard write path (use update policies or soft delete, not SQL-style UPDATE).
  • SQL database in Fabric — OLTP/translytical; full SQL Server-compatible DML; automatically replicates to OneLake Delta for analytical queries; designed for row-level transactional workloads, not large analytical scans.
  • Summary rule: Need UPDATE/MERGE at analytical scale → Warehouse. Need Spark ETL → Lakehouse. Need sub-second time-series streaming ingest + KQL → Eventhouse. Need OLTP row transactions → SQL database.
-- Fabric Warehouse: full DML supported
MERGE INTO dbo.Customer AS target
USING dbo.CustomerStaging AS src
  ON target.CustomerKey = src.CustomerKey
WHEN MATCHED THEN
  UPDATE SET target.Email = src.Email,
             target.UpdatedAt = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
  INSERT (CustomerKey, Name, Email, CreatedAt)
  VALUES (src.CustomerKey, src.Name, src.Email, GETDATE());

-- Lakehouse SQL analytics endpoint: DML is NOT supported
-- The statement below will be REJECTED at the analytics endpoint:
-- UPDATE dbo.Customer SET Email = 'new@example.com' WHERE CustomerKey = 1;
Gotcha: The Lakehouse SQL analytics endpoint is read-only. Writing T-SQL INSERT/UPDATE/DELETE/CREATE TABLE statements against it returns an error. If your workflow requires DML via SQL, you must use a Fabric Warehouse.

Exam tips & gotchas

  • Lakehouse SQL endpoint: SELECT + DDL for views/functions/procedures = OK. DML (INSERT/UPDATE/DELETE) + CREATE TABLE = blocked.
  • Warehouse stores data in OneLake as Delta-Parquet — both Spark and other engines can read Warehouse tables via shortcuts or SQL analytics endpoint.
  • Eventhouse is the only store with native KQL support; SQL can be used for basic queries but KQL is the primary and more powerful interface.
  • SQL database in Fabric auto-replicates to OneLake, but is intended for small-to-medium OLTP loads — not petabyte analytical workloads.
  • Both Lakehouse and Warehouse expose a SQL analytics endpoint, but only the Warehouse endpoint allows full DML.

Practice

Q1. A data engineer runs the T-SQL statement INSERT INTO dbo.Orders SELECT * FROM dbo.OrdersStaging against the Lakehouse SQL analytics endpoint. What happens?
Answer: The statement fails with an error. Why: The Lakehouse SQL analytics endpoint is read-only; INSERT, UPDATE, DELETE, and CREATE TABLE are not supported. The engineer must use a Fabric Data Warehouse for T-SQL DML operations.
Q2. A team ingests millions of IoT sensor events per minute and needs KQL-based time-series analysis with sub-second freshness. Which Fabric data store is the best fit?
Answer: Eventhouse (KQL database). Why: Eventhouse is purpose-built for high-throughput append-only streaming ingest and KQL-based time-series analysis. SQL databases and Data Warehouses are not optimised for sub-second streaming.
Q3. A BI team wants to use T-SQL MERGE to implement SCD Type 2 updates on a dimension table. Which Fabric store should they use?
Answer: Fabric Data Warehouse. Why: MERGE is a DML operation that requires write access. The Fabric Data Warehouse supports full T-SQL DML including MERGE; the Lakehouse SQL analytics endpoint does not.

2.1.5 OneLake integration for Eventhouse and semantic models

OneLake is Fabric's unified storage layer — a single logical data lake for the entire organisation, backed by ADLS Gen2 and accessible via DFS endpoint. All Fabric compute engines read from and write to OneLake, giving analytical items a shared physical foundation. Two items that require special understanding are Eventhouse and semantic models.

Eventhouse OneLake availability lets KQL database data be mirrored to OneLake as Delta-Parquet, making it queryable by Spark, Warehouse, and other engines without an Eventstream pipeline. Direct Lake is the semantic model storage mode in which Power BI reads Parquet files directly from OneLake rather than importing data into memory or relying on DirectQuery — combining the freshness of DirectQuery with near-Import query performance.

Key facts

  • OneLake availability for Eventhouse — enable per-table in the KQL database properties; tables are materialised as Delta-Parquet in OneLake; latency is typically minutes after ingestion.
  • Once enabled, Eventhouse data appears as Delta tables under the Lakehouse Files path and can be queried from Spark notebooks or the Lakehouse SQL analytics endpoint with no data movement.
  • Direct Lake mode — semantic model reads Delta-Parquet files from OneLake (Lakehouse or Warehouse) directly into columnar memory; no import copy or DirectQuery per-query round-trips.
  • Direct Lake has a framing concept: the model frame is anchored to a specific Delta checkpoint (v-order); a refresh operation advances the frame to the latest Delta version, making new data visible.
  • Direct Lake falls back to DirectQuery automatically when the data volume exceeds capacity limits (parquet file row count or column cardinality thresholds).
  • OneLake paths follow the pattern onelake://<workspace>/<item>.Lakehouse/Tables/<tablename>; ADLS Gen2 DFS URL: https://onelake.dfs.fabric.microsoft.com/<workspace-id>/....
  • Fabric Mirroring (for Azure SQL, Snowflake, Cosmos DB) also replicates data to OneLake as Delta-Parquet, enabling the same cross-engine access pattern as Eventhouse OneLake availability.
Key: Direct Lake is a semantic model storage mode, not an ingestion tool. It does not move or transform data — it reads existing Delta files in OneLake and requires no scheduled import refresh for data freshness (only a frame-advance refresh).

Exam tips & gotchas

  • Direct Lake fallback to DirectQuery happens automatically when parquet capacity limits are hit — the user sees slower queries but no errors.
  • Enabling OneLake availability for Eventhouse does not make Eventhouse writable from Spark; it is a one-direction mirror (KQL → Delta).
  • A Direct Lake semantic model does not need an import refresh to see new Lakehouse data — but it does need a frame refresh to advance the Delta checkpoint pointer.
  • OneLake exposes an ADLS Gen2 DFS endpoint, so any tool compatible with ADLS Gen2 (ADF, Synapse, external Spark clusters, local storage explorer) can read Fabric data.

Practice

Q1. A data engineer enables OneLake availability for an Eventhouse KQL database table. What happens to that data?
Answer: The table is mirrored to OneLake as Delta-Parquet and becomes queryable by Spark, the Lakehouse SQL analytics endpoint, and other Fabric engines. Why: OneLake availability exposes KQL table data as Delta files in OneLake without copying it to a separate Lakehouse item.
Q2. A Power BI semantic model in Direct Lake mode begins returning slower query responses than normal. No model changes were made. What is the most likely cause?
Answer: The model has fallen back to DirectQuery because the data volume exceeded the capacity parquet limits. Why: Direct Lake automatically falls back to DirectQuery when column cardinality or row thresholds are breached, resulting in slower per-query SQL round-trips to the underlying store.
Q3. Which statement correctly describes the relationship between a Direct Lake semantic model and its underlying Lakehouse Delta table?
Answer: The semantic model reads Delta-Parquet files directly from OneLake without importing data into memory or issuing per-query SQL. New data written to the Delta table is only visible to the model after a frame refresh that advances the model's Delta checkpoint pointer.

2.2 Transform data

2.2.1 Create views, functions, and stored procedures (T-SQL)

Fabric Warehouse supports the full set of T-SQL persistent database objects: views (named SELECT statements), user-defined functions (scalar and table-valued), and stored procedures (parameterised T-SQL programs). These objects encapsulate business logic, simplify report queries, and reduce duplicated code across notebooks and Dataflows.

The Lakehouse SQL analytics endpoint also allows CREATE VIEW, CREATE FUNCTION, and CREATE PROCEDURE — but it does not allow DML. Views created at the analytics endpoint are read-only projections over the underlying Delta tables.

Key facts

  • Views — virtualise a SELECT; no data stored; reflect current table state at query time; support SELECT, JOIN, WHERE, GROUP BY, subqueries, CTEs.
  • Inline table-valued functions (iTVF) — parameterised views; return a table; can be used in FROM clause; support predicate push-down (query folding from Dataflows).
  • Scalar functions — return a single value; useful in computed columns or expressions; may reduce query performance if applied row-by-row.
  • Stored procedures — T-SQL programs; support parameters (IN), conditional logic (IF/ELSE), loops, transactions, and error handling (TRY/CATCH); return result sets or output parameters.
  • Objects created at the Lakehouse SQL analytics endpoint are not visible from Spark — they live in the analytics-endpoint SQL catalog only.
  • Fabric Warehouse does not currently support EXECUTE AS OWNER or cross-database queries within the same warehouse SQL endpoint.
  • Schemas (CREATE SCHEMA) are supported and recommended to organise objects: dbo for core tables, rpt for report views, etl for staging objects.
-- Create a schema for report objects
CREATE SCHEMA rpt;
GO

-- Create a view that joins fact and dimension
CREATE OR ALTER VIEW rpt.vSalesByProduct AS
SELECT
    p.ProductName,
    p.CategoryName,
    SUM(f.SalesAmount)  AS TotalSales,
    COUNT(f.OrderID)    AS OrderCount
FROM dbo.SalesFact      AS f
JOIN dbo.DimProduct     AS p ON f.ProductKey = p.ProductKey
WHERE f.SalesAmount > 0
GROUP BY p.ProductName, p.CategoryName;
GO

-- Inline table-valued function (parameterised view)
CREATE OR ALTER FUNCTION dbo.fnSalesByRegion
    (@Region NVARCHAR(50))
RETURNS TABLE
AS
RETURN
(
    SELECT f.OrderID, f.SalesAmount, d.RegionName
    FROM   dbo.SalesFact   f
    JOIN   dbo.DimGeography d ON f.GeographyKey = d.GeographyKey
    WHERE  d.RegionName = @Region
);
GO

-- Simple stored procedure with error handling
CREATE OR ALTER PROCEDURE dbo.uspRefreshSalesSummary
    @LoadDate DATE
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRY
        DELETE FROM dbo.SalesSummary WHERE SaleDate = @LoadDate;
        INSERT INTO dbo.SalesSummary (SaleDate, TotalSales)
        SELECT SaleDate, SUM(SalesAmount)
        FROM   dbo.SalesFact
        WHERE  SaleDate = @LoadDate
        GROUP BY SaleDate;
    END TRY
    BEGIN CATCH
        THROW;
    END CATCH;
END;
GO

Exam tips & gotchas

  • Views and functions can be created at the Lakehouse SQL analytics endpoint, but DML inside them (INSERT/UPDATE) is blocked.
  • CREATE OR ALTER is the preferred idempotent syntax in Fabric Warehouse — avoids DROP/CREATE pairs that break dependent objects.
  • Inline TVFs support query folding from Dataflows Gen2 (the WHERE predicate on the TVF is pushed to the source); scalar functions do not fold.
  • Stored procedures are not supported as a Direct Lake data source — they must be called from a pipeline activity or a Dataflow's "invoke stored procedure" step.

Practice

Q1. A Dataflow Gen2 is configured to read from an iTVF in a Fabric Warehouse and filter on a date parameter. A data engineer wants to ensure the filter is applied at the Warehouse rather than in the Dataflow engine. What must be true for query folding to occur?
Answer: The Dataflow must connect to the Warehouse SQL endpoint via a foldable connector (e.g., SQL Server connector), and the filter step must be expressible as a WHERE clause compatible with the source SQL dialect. Inline TVFs support predicate push-down, so folding will occur as long as the connector supports it.
Q2. A data engineer creates a view at the Lakehouse SQL analytics endpoint that performs a JOIN between two Delta tables. Another engineer tries to query the view from a PySpark notebook. What is the result?
Answer: The query fails — or returns an error that the view is not found. Why: Views created at the SQL analytics endpoint exist only in the analytics-endpoint SQL catalog; they are not registered as Delta tables in the Lakehouse metastore and are not visible to Spark.
Q3. Why should a Fabric Warehouse developer prefer CREATE OR ALTER VIEW over DROP VIEW IF EXISTS; CREATE VIEW?
Answer: CREATE OR ALTER is atomic and preserves permissions on the existing object. DROP followed by CREATE breaks the permission chain — any GRANT on the old view object must be re-applied after re-creation.

2.2.2 Enrich data (add columns and tables)

Data enrichment adds computed columns, derived attributes, or reference data that are not in the source but increase analytical value. In Fabric, enrichment happens in three common layers: T-SQL computed columns or ALTER TABLE ADD COLUMN in the Warehouse, PySpark withColumn transforms in notebooks, and Power Query M column-addition steps in Dataflows Gen2.

Key facts

  • T-SQL — ALTER TABLE … ADD column AS expression persisted computed columns; or add a regular column and populate with UPDATE. Computed columns referencing other columns are evaluated at query time.
  • PySpark withColumn — adds a new column from an expression; non-destructive (existing columns are kept); chained calls are lazy-evaluated and optimised by Spark.
  • Power Query M — Table.AddColumn(table, "NewCol", each [OldCol] * 1.1) adds a computed column; evaluated in-engine unless the connector folds it back to the source.
  • Adding a reference/lookup table — import a small CSV or call an API in a pipeline → load to Lakehouse → join in Spark or Warehouse. Common pattern: postal code → territory mapping table enriching orders.
  • Enrichment columns should be added in the Silver or Gold layer, not Bronze — Bronze is the raw landing zone and should mirror the source structure.
# PySpark: enrich with computed columns
from pyspark.sql import functions as F

df_enriched = (df_orders
    .withColumn("ProfitMargin",
        F.round((F.col("Revenue") - F.col("Cost")) / F.col("Revenue"), 4))
    .withColumn("OrderYear",  F.year("OrderDate"))
    .withColumn("OrderMonth", F.month("OrderDate"))
    .withColumn("IsHighValue", (F.col("Revenue") > 10000).cast("boolean"))
)
df_enriched.write.format("delta").mode("overwrite").option("overwriteSchema","true") \
    .saveAsTable("gold.OrdersEnriched")

-- T-SQL: enrich via computed column (Fabric Warehouse)
ALTER TABLE dbo.SalesFact
    ADD GrossProfit AS (SalesAmount - CostAmount);

-- T-SQL: add regular column, then populate from lookup join
ALTER TABLE dbo.SalesFact ADD TerritoryName NVARCHAR(100);
UPDATE f
SET    f.TerritoryName = t.TerritoryName
FROM   dbo.SalesFact   f
JOIN   dbo.DimTerritory t ON f.TerritoryKey = t.TerritoryKey;

Exam tips & gotchas

  • Computed columns in T-SQL are evaluated at query time and cannot be directly indexed (non-persisted). Use persisted computed columns (AS expression PERSISTED) to make them indexable.
  • PySpark withColumn is lazy — it does not execute until an action (write, show, count) is triggered.
  • Power Query Table.AddColumn may fold to the source SQL if the expression is simple (arithmetic, string concat); complex expressions (regex, custom M functions) prevent folding.

Practice

Q1. A data engineer adds a column ProfitMargin to a Lakehouse Delta table using PySpark withColumn. Nothing is written to storage. What must the engineer do to persist the result?
Answer: Call a write action such as df.write.format("delta").mode("overwrite").saveAsTable(...). Why: PySpark transformations are lazy; withColumn only defines the transformation plan. No data is written until an action triggers execution.
Q2. A Dataflow Gen2 uses Table.AddColumn to compute the ratio of two columns already present in a SQL Server source table. Will this step fold?
Answer: Yes, if the SQL Server connector supports folding for arithmetic expressions — which it does. The Dataflow engine translates the column addition to a SQL SELECT col1 / col2 AS Ratio expression pushed to SQL Server, avoiding loading all rows into the Dataflow engine.

2.2.3 Implement a star schema

A star schema organises analytical data into one or more central fact tables surrounded by dimension tables, connected by foreign-key relationships. It is the standard design for the Gold layer of a Fabric lakehouse or warehouse because it simplifies DAX, reduces join complexity for report tools, and leverages VertiPaq compression efficiently in semantic models.

The schema name derives from its visual appearance: the fact table at the center with dimension tables radiating outward like star points. Choosing the right fact type and dimension patterns is a primary exam topic.

Key facts

  • Grain — the level of detail of one row in the fact table (e.g., one row per product per store per day). All measures in the fact must be consistent with the declared grain. Adding columns at a finer grain than declared corrupts the grain.
  • Fact types:
    • Transaction fact — one row per business event (sale, click, call); sparse; most common.
    • Periodic snapshot fact — one row per period per entity (e.g., daily account balance); dense; period inserted even when no transaction occurred.
    • Accumulating snapshot fact — one row per process instance that is updated as the process advances through lifecycle milestones (e.g., order → pick → ship → deliver); contains multiple date-FK columns for each milestone.
  • Surrogate key — integer key generated by the warehouse (IDENTITY or sequence), not from the source system; insulates the warehouse from source key changes and enables SCD history rows.
  • Conformed dimension — shared across multiple fact tables (e.g., DimDate used by both SalesFact and InventoryFact); enables cross-subject-area drill and consistent filtering.
  • Role-playing dimension — a single physical dimension table used multiple times in the same fact table for different date roles (e.g., DimDate used as OrderDate, ShipDate, DueDate); implemented in the semantic model as inactive relationships activated with USERELATIONSHIP().
  • SCD Type 1 — overwrite old value; no history kept; simpler; use when historical attribute is irrelevant (e.g., correcting a typo in a name).
  • SCD Type 2 — add new row with new surrogate key; old row marked inactive (IsCurrent = 0, ExpiredDate = <date>); preserves history; requires surrogate key design; most common for name/address changes.
-- Create DimProduct with surrogate key and SCD columns
CREATE TABLE dbo.DimProduct (
    ProductKey      INT IDENTITY(1,1) PRIMARY KEY,  -- surrogate key
    ProductID       NVARCHAR(20)  NOT NULL,           -- natural/business key
    ProductName     NVARCHAR(200) NOT NULL,
    CategoryName    NVARCHAR(100),
    UnitCost        DECIMAL(10,2),
    -- SCD Type 2 columns
    EffectiveDate   DATE NOT NULL,
    ExpiryDate      DATE NULL,       -- NULL = current row
    IsCurrent       BIT  NOT NULL DEFAULT 1
);

-- SCD Type 2 MERGE: expire old row, insert new row
MERGE INTO dbo.DimProduct AS tgt
USING (
    SELECT ProductID, ProductName, CategoryName, UnitCost
    FROM   stg.ProductStaging
) AS src ON tgt.ProductID = src.ProductID AND tgt.IsCurrent = 1
WHEN MATCHED AND (
        tgt.ProductName  <> src.ProductName   OR
        tgt.CategoryName <> src.CategoryName  OR
        tgt.UnitCost     <> src.UnitCost
) THEN
    UPDATE SET tgt.IsCurrent   = 0,
               tgt.ExpiryDate  = CAST(GETDATE() AS DATE)
WHEN NOT MATCHED BY TARGET THEN
    INSERT (ProductID, ProductName, CategoryName, UnitCost,
            EffectiveDate, ExpiryDate, IsCurrent)
    VALUES (src.ProductID, src.ProductName, src.CategoryName,
            src.UnitCost, CAST(GETDATE() AS DATE), NULL, 1);

-- After expiring old rows, INSERT new versions (for changed rows)
INSERT INTO dbo.DimProduct
    (ProductID, ProductName, CategoryName, UnitCost,
     EffectiveDate, ExpiryDate, IsCurrent)
SELECT src.ProductID, src.ProductName, src.CategoryName, src.UnitCost,
       CAST(GETDATE() AS DATE), NULL, 1
FROM   stg.ProductStaging src
JOIN   dbo.DimProduct      tgt
    ON  tgt.ProductID = src.ProductID AND tgt.IsCurrent = 0
    AND tgt.ExpiryDate = CAST(GETDATE() AS DATE); -- just expired today
Key: Accumulating snapshot facts contain multiple foreign keys to the same dimension (e.g., OrderDateKey, ShipDateKey, DeliveryDateKey — all pointing to DimDate). When implemented in a semantic model, each becomes a separate relationship, with only one active at a time.

Exam tips & gotchas

  • SCD Type 1 = overwrite (simpler, no history). SCD Type 2 = insert new row (history preserved, surrogate key required).
  • Grain must be declared before choosing measures — a "unit price" column at transaction grain is valid; including both transaction-level and period-snapshot measures in the same fact table violates grain consistency.
  • Surrogate key must be generated by the warehouse (IDENTITY), not sourced from the source system — source natural keys can change or be reused.
  • Role-playing dimensions: only one relationship between fact and dimension can be active in a semantic model at a time; the others must use USERELATIONSHIP() in DAX measures.
  • Periodic snapshot facts should include rows for every period even when no activity occurred — otherwise NULL gaps in time-intelligence calculations.

Practice

Q1. A fact table contains one row per product per store per day recording closing inventory level, including days when no sales occurred. What type of fact table is this?
Answer: Periodic snapshot. Why: A periodic snapshot captures the state at the end of each regular period (day, week, month) for every entity — rows exist for every period regardless of whether a transaction occurred. A transaction fact would only have rows when events happen.
Q2. A dimension table for Customer is used in both SalesFact and ServiceRequestFact. When the filter on Customer is applied in a report, it automatically filters both facts. What dimension pattern does this describe?
Answer: Conformed dimension. Why: A conformed dimension has the same keys and attributes across multiple fact tables, enabling consistent cross-subject filtering and drill-through in a single semantic model.
Q3. A product changes its name but keeps the same ProductID. The team needs to analyse historical sales under the old name and future sales under the new name. Which SCD type should be implemented?
Answer: SCD Type 2. Why: Type 2 preserves both old and new attribute values as separate rows with different surrogate keys and effective/expiry dates. Historical sales facts pointing to the old surrogate key will correctly resolve to the old name; new sales get the new row's surrogate key.
Q4. An order tracking fact table records OrderDateKey, PickDateKey, ShipDateKey, and DeliveryDateKey — all foreign keys to DimDate. What fact type is this, and what challenge does it create in a semantic model?
Answer: Accumulating snapshot fact. Challenge: A semantic model can only have one active relationship between the fact and DimDate at a time. The other three date relationships must be marked inactive and activated with USERELATIONSHIP() inside DAX measures.

2.2.4 Denormalize data

Denormalization intentionally relaxes third-normal-form (3NF) rules by merging related tables or pre-computing derived attributes into a single wider table. The goal is to reduce JOIN complexity at query time, which improves report performance at the cost of storage redundancy. In a Fabric Lakehouse or Warehouse Gold layer, denormalized tables are common inputs for Direct Lake semantic models or flat Power BI reports.

Key facts

  • Wide table / flattened fact — combines columns from fact and all related dimensions into one table; eliminates JOIN at query time; increases row width and storage, reduces query latency.
  • Pre-aggregated summary table — stores GROUP BY results; fastest query; must be refreshed on a schedule; risk of stale data.
  • Trade-offs: denormalization reduces query complexity but increases data duplication; update anomalies must be managed by rebuilding or MERGE-updating the denormalised table.
  • In Fabric, CTAS (CREATE TABLE AS SELECT) and SELECT INTO are the T-SQL patterns for creating a denormalised snapshot table from a multi-table JOIN.
  • PySpark equivalent: df_fact.join(df_dim, key).write.format("delta").saveAsTable("gold.WideSales").
-- Fabric Warehouse: create a denormalised wide table (CTAS pattern)
CREATE TABLE gold.WideSalesFact
AS
SELECT
    f.OrderID,
    f.OrderDate,
    f.SalesAmount,
    f.Quantity,
    p.ProductName,
    p.CategoryName,
    c.CustomerName,
    c.CountryRegion,
    d.CalendarYear,
    d.MonthName
FROM dbo.SalesFact      f
JOIN dbo.DimProduct     p ON f.ProductKey   = p.ProductKey
JOIN dbo.DimCustomer    c ON f.CustomerKey  = c.CustomerKey
JOIN dbo.DimDate        d ON f.DateKey      = d.DateKey;

Exam tips & gotchas

  • Denormalization trades storage space for query speed — always valid in analytical (OLAP) scenarios; never appropriate for OLTP writes (update anomalies).
  • A denormalised table must be rebuilt or merged whenever source dimension attributes change (e.g., product name changes); it does not stay in sync automatically.
  • CTAS in Fabric Warehouse (CREATE TABLE … AS SELECT) creates and populates a table in one statement — equivalent to INSERT … SELECT but also creates the table.

Practice

Q1. A reporting team complains that their Power BI report is slow because it must JOIN five dimension tables at runtime. A data engineer creates a denormalised wide table that pre-joins all dimensions. What is the primary trade-off?
Answer: Increased storage and risk of stale data. The wide table stores attribute values redundantly for every row (e.g., ProductName repeated for every sale). If a product name changes, the wide table must be rebuilt or updated; it does not reflect source changes automatically.
Q2. Which T-SQL statement creates a new denormalised Fabric Warehouse table and populates it in a single operation?
Answer: CTAS — CREATE TABLE … AS SELECT. This is the Fabric Warehouse equivalent of a "create-and-fill" operation; it combines table creation with a SELECT query into one atomic step.

2.2.5 Aggregate data

Aggregation summarises detailed data to a coarser grain using functions such as SUM, COUNT, AVG, MIN, MAX. In Fabric, aggregations are performed in T-SQL (GROUP BY with aggregate functions), PySpark (groupBy + agg), Power Query M (Table.Group), and KQL (summarize). Window functions extend aggregation by computing values within a partition without collapsing rows.

Key facts

  • T-SQL GROUP BY — every non-aggregate SELECT column must appear in GROUP BY; filter aggregates with HAVING (not WHERE); use ROLLUP for hierarchical subtotals, CUBE for all combinations.
  • Window functions — SUM() OVER (PARTITION BY … ORDER BY … ROWS …); returns one row per input row (no collapsing); ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, FIRST_VALUE, LAST_VALUE; highly optimised in Fabric Warehouse.
  • PySpark groupBy.agg — df.groupBy("col").agg(F.sum("amount"), F.count("*")); equivalent to GROUP BY; result is a new DataFrame.
  • Power Query Table.Group — Table.Group(table, {"Key"}, {{"Total", each List.Sum([Amount]), type number}}); folds to source GROUP BY when the connector supports it.
  • Aggregation tables in Power BI — pre-aggregated summary tables that Power BI automatically uses for high-level queries, falling back to detail tables for fine-grain queries.
-- T-SQL: GROUP BY with HAVING and ROLLUP
SELECT
    p.CategoryName,
    p.ProductName,
    SUM(f.SalesAmount) AS TotalSales,
    COUNT(DISTINCT f.OrderID) AS Orders
FROM dbo.SalesFact  f
JOIN dbo.DimProduct p ON f.ProductKey = p.ProductKey
GROUP BY ROLLUP(p.CategoryName, p.ProductName)
HAVING SUM(f.SalesAmount) > 1000
ORDER BY p.CategoryName, p.ProductName;

-- Window function: running total and rank within category
SELECT
    ProductName,
    CategoryName,
    SalesAmount,
    SUM(SalesAmount) OVER (PARTITION BY CategoryName
                           ORDER BY SaleDate
                           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
                           AS RunningTotal,
    RANK() OVER (PARTITION BY CategoryName ORDER BY SalesAmount DESC) AS SalesRank
FROM dbo.SalesFact
JOIN dbo.DimProduct ON SalesFact.ProductKey = DimProduct.ProductKey;

# PySpark: groupBy aggregation and window functions
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# Basic aggregation
df_summary = (df_sales
    .groupBy("CategoryName", "ProductName")
    .agg(
        F.sum("SalesAmount").alias("TotalSales"),
        F.countDistinct("OrderID").alias("Orders"),
        F.avg("SalesAmount").alias("AvgSale")
    )
    .filter(F.col("TotalSales") > 1000)
)

# Window function: running total per category
window_spec = Window.partitionBy("CategoryName").orderBy("SaleDate") \
    .rowsBetween(Window.unboundedPreceding, Window.currentRow)
df_running = df_sales.withColumn(
    "RunningTotal", F.sum("SalesAmount").over(window_spec)
)

// Power Query M: Table.Group aggregation
let
    Source = SalesFact,
    Grouped = Table.Group(
        Source,
        {"CategoryName"},
        {
          {"TotalSales",  each List.Sum([SalesAmount]),    type number},
          {"OrderCount",  each Table.RowCount(_),          type number},
          {"AvgSale",     each List.Average([SalesAmount]),type number}
        }
    )
in
    Grouped

Exam tips & gotchas

  • HAVING filters on aggregated values; WHERE filters on row values before aggregation — they are not interchangeable.
  • Window functions do not reduce row count — the original rows are preserved with the computed window value added. Use GROUP BY to collapse rows.
  • PySpark groupBy.agg is lazy; use .write or .show to trigger execution.
  • Table.Group in Power Query M will fold to source GROUP BY SQL if the source connector supports it; complex aggregation functions prevent folding.

Practice

Q1. A T-SQL query uses GROUP BY CategoryName and includes a column ProductName in the SELECT list that is not in GROUP BY. What is the result?
Answer: A syntax error — "Column 'ProductName' is invalid in the select list because it is not contained in either an aggregate function or the GROUP BY clause." Every non-aggregated SELECT column must appear in GROUP BY.
Q2. A report needs to show each sale row alongside a running total of sales for the same product category, without removing any detail rows. Which T-SQL feature is most appropriate?
Answer: A window function — SUM(SalesAmount) OVER (PARTITION BY CategoryName ORDER BY SaleDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW). Window functions compute per-row aggregates without collapsing rows.

2.2.6 Merge and join data

Join operations combine rows from two or more tables based on a matching key. In Fabric, joins are performed via T-SQL (INNER/LEFT/RIGHT/FULL OUTER JOIN, CROSS JOIN), PySpark DataFrame join API, and Power Query M (Table.Join / merge queries). The T-SQL MERGE statement extends this to a combined INSERT/UPDATE/DELETE upsert in a single pass, which is critical for dimension loading and incremental ETL.

Key facts

  • INNER JOIN — returns only rows where the key exists in both tables; rows with no match in either table are excluded.
  • LEFT OUTER JOIN — all rows from the left table; NULL for right-table columns when no match; used for "show all customers including those with no orders".
  • FULL OUTER JOIN — all rows from both tables; NULLs on either side when no match; useful for reconciliation.
  • CROSS JOIN — Cartesian product (every row of left × every row of right); no ON clause; use deliberately for calendar/product grid generation.
  • MERGE (upsert) — matches source to target; WHEN MATCHED → UPDATE; WHEN NOT MATCHED BY TARGET → INSERT; WHEN NOT MATCHED BY SOURCE → DELETE (soft-deletes pattern). Single-pass over target table.
  • PySpark join types: inner, left, right, full, left_semi (like INNER but returns only left columns), left_anti (rows in left with no match on right).
  • Power Query merge queries — visual LEFT/INNER/RIGHT/FULL/LEFT ANTI/RIGHT ANTI join; folds to SQL JOIN if source supports it.
-- T-SQL: LEFT JOIN with NULL handling
SELECT
    c.CustomerName,
    c.Email,
    COALESCE(SUM(f.SalesAmount), 0) AS LifetimeSales
FROM dbo.DimCustomer c
LEFT JOIN dbo.SalesFact f ON c.CustomerKey = f.CustomerKey
GROUP BY c.CustomerName, c.Email
ORDER BY LifetimeSales DESC;

-- T-SQL: MERGE upsert pattern (incremental load)
MERGE INTO dbo.DimCustomer AS tgt
USING stg.CustomerStaging  AS src
    ON tgt.CustomerID = src.CustomerID
WHEN MATCHED AND (
    tgt.Email <> src.Email OR tgt.Phone <> src.Phone
) THEN
    UPDATE SET tgt.Email       = src.Email,
               tgt.Phone       = src.Phone,
               tgt.UpdatedDate = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
    INSERT (CustomerID, CustomerName, Email, Phone, CreatedDate)
    VALUES (src.CustomerID, src.CustomerName, src.Email,
            src.Phone, GETDATE());

# PySpark: join types and left-anti for orphan detection
df_sales    = spark.table("silver.SalesFact")
df_customer = spark.table("silver.DimCustomer")

# INNER join: only matched rows
df_inner = df_sales.join(df_customer, "CustomerKey", "inner")

# LEFT join: all sales, NULL customer attributes if unmatched
df_left = df_sales.join(df_customer, "CustomerKey", "left")

# LEFT ANTI join: sales with no matching customer (data quality check)
df_orphans = df_sales.join(df_customer, "CustomerKey", "left_anti")
df_orphans.write.format("delta").mode("overwrite") \
    .saveAsTable("dq.OrphanSales")

Exam tips & gotchas

  • MERGE requires the source to produce at most one matching row per target row — if the source has duplicates on the join key, MERGE throws an error. Deduplicate staging before MERGE.
  • PySpark left_semi — returns left rows where a match exists in right, but only left-table columns; equivalent to SQL EXISTS subquery.
  • PySpark left_anti — returns left rows with no match in right; efficient for finding orphaned fact rows (missing dimension key).
  • Power Query merge creates an "expanded" column — you must explicitly choose which right-table columns to expand; unexpanded columns remain as nested Table objects.
  • CROSS JOIN performance: every row combination is produced, so 1,000 left rows × 365 date rows = 365,000 output rows. Always verify intent before using.

Practice

Q1. A data quality job needs to find all rows in SalesFact that have a CustomerKey value not present in DimCustomer. Which PySpark join type returns exactly these rows?
Answer: left_anti join. Why: df_sales.join(df_customer, "CustomerKey", "left_anti") returns only the sales rows for which there is no matching CustomerKey in the dimension — exactly the orphaned fact rows.
Q2. A MERGE statement fails with "The MERGE statement attempted to UPDATE or DELETE the same row more than once." What is the cause and the fix?
Answer: The source table contains duplicate rows on the join key. MERGE requires each target row to match at most one source row. Fix: deduplicate the source (e.g., SELECT DISTINCT or ROW_NUMBER() OVER (PARTITION BY key) = 1) before the MERGE.
Q3. A T-SQL query must return all customers and all orders, including customers with no orders and orders with no matching customer. Which JOIN type is required?
Answer: FULL OUTER JOIN. Why: LEFT JOIN would exclude orders with no customer; RIGHT JOIN would exclude customers with no orders. FULL OUTER JOIN returns all rows from both tables, placing NULLs on whichever side has no match.

2.2.7 Handle duplicates, missing values, and nulls

Real-world source data frequently contains duplicate rows, missing values (empty strings, sentinel values such as -1 or "N/A"), and SQL NULLs. Addressing these in the Silver transformation layer is a fundamental data engineering practice. Fabric supports dedicated operations across T-SQL (ISNULL, COALESCE, NOT EXISTS deduplication), PySpark (dropDuplicates, na.drop, na.fill), and Power Query M (Table.Distinct, Table.FillDown, Table.ReplaceValue).

Key facts

  • NULL semantics: Any arithmetic with NULL produces NULL. Any comparison with NULL using = or <> produces UNKNOWN (not TRUE/FALSE) — use IS NULL / IS NOT NULL. COUNT(*) counts all rows; COUNT(column) skips NULLs.
  • ISNULL(expr, replacement) — SQL Server/Fabric syntax; returns replacement if expr is NULL; limited to two arguments.
  • COALESCE(expr1, expr2, …) — ANSI SQL; returns the first non-NULL value from the list; preferred when checking multiple fallback values.
  • ROW_NUMBER() deduplication — assign a row number within a group (PARTITION BY natural key ORDER BY freshness DESC); keep row_num = 1; most flexible dedup pattern in T-SQL.
  • PySpark dropDuplicates([cols]) — removes duplicate rows; optional column list restricts dedup key; whole-row or subset dedup.
  • PySpark na.fill(value, subset=[cols]) — fills NULL with a constant or dict of column→value; na.drop(subset=[cols], thresh=n) removes rows with fewer than n non-null values in the subset.
  • Power Query Table.Distinct([cols]) — removes rows where specified columns are identical; without argument, deduplicates on all columns.
  • Table.FillDown — propagates last non-null value downward in a column; useful for sparse hierarchical data exports.
-- T-SQL: COALESCE and ISNULL for null handling
SELECT
    OrderID,
    COALESCE(ShipCity, BillingCity, 'Unknown') AS City,
    ISNULL(Discount, 0)                         AS Discount,
    ISNULL(Notes, '')                            AS Notes
FROM dbo.Orders;

-- T-SQL: ROW_NUMBER deduplication (keep most recent per OrderID)
WITH Dedup AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY OrderID
               ORDER BY LastModifiedDate DESC
           ) AS RowNum
    FROM dbo.OrdersStaging
)
DELETE FROM Dedup WHERE RowNum > 1;

-- T-SQL: filter out rows where critical column is NULL
SELECT * FROM dbo.SalesFact
WHERE CustomerKey IS NOT NULL
  AND ProductKey  IS NOT NULL;

# PySpark: dropDuplicates, na.fill, na.drop
from pyspark.sql import functions as F

df = spark.table("bronze.RawOrders")

# 1. Remove full-row duplicates
df_dedup = df.dropDuplicates()

# 2. Remove duplicates on business key, keeping latest
from pyspark.sql.window import Window
w = Window.partitionBy("OrderID").orderBy(F.col("LastModifiedDate").desc())
df_dedup_bk = (df
    .withColumn("_rn", F.row_number().over(w))
    .filter(F.col("_rn") == 1)
    .drop("_rn")
)

# 3. Fill nulls: constant for numeric, dict for mixed types
df_filled = df_dedup.na.fill(0, subset=["Discount", "TaxAmount"])
df_filled = df_filled.na.fill({"ShipCity": "Unknown", "Notes": ""})

# 4. Drop rows missing critical columns
df_clean = df_filled.na.drop(subset=["OrderID", "CustomerKey", "OrderDate"])

df_clean.write.format("delta").mode("overwrite").saveAsTable("silver.Orders")

Exam tips & gotchas

  • NULL ≠ empty string — WHERE Notes = '' does not catch NULLs; use WHERE Notes IS NULL OR Notes = ''.
  • COALESCE short-circuits evaluation (stops at first non-NULL); ISNULL does not — prefer COALESCE for multiple fallbacks.
  • PySpark dropDuplicates() with no arguments deduplicates on all columns — rows with the same business key but different timestamps will be kept.
  • COUNT(column) vs COUNT(*): COUNT(*) counts all rows including NULLs; COUNT(column) counts only non-NULL values in that column.
  • ROW_NUMBER deduplication in T-SQL inside a CTE can be deleted directly with DELETE FROM cte_name WHERE RowNum > 1 — Fabric Warehouse supports this pattern.

Practice

Q1. A T-SQL query filters rows with WHERE Discount <> 0. A row with Discount = NULL is present. Is it included in the result?
Answer: No. Why: NULL compared with any value using = or <> evaluates to UNKNOWN, which is treated as FALSE. The row is excluded. To include NULLs, use WHERE Discount <> 0 OR Discount IS NULL.
Q2. A PySpark job calls df.dropDuplicates() on a DataFrame with an OrderID column and a LastModifiedDate column. Two rows have the same OrderID but different LastModifiedDate values. Are both rows kept?
Answer: Yes, both are kept. Why: dropDuplicates() with no arguments deduplicates on all columns. Since the two rows differ on LastModifiedDate, they are not considered duplicates. Use dropDuplicates(["OrderID"]) to deduplicate by business key only (non-deterministic which row is kept).
Q3. A column ShipCity contains NULLs. A T-SQL expression must return the BillingCity if ShipCity is NULL, or the string 'Unknown' if BillingCity is also NULL. Which expression is correct?
Answer: COALESCE(ShipCity, BillingCity, 'Unknown'). Why: COALESCE evaluates its arguments left to right and returns the first non-NULL. ISNULL only accepts two arguments, so it cannot check three values in one expression.

2.2.8 Convert data types

Source data frequently arrives with incorrect or generic types: dates stored as strings, integers stored as VARCHAR, decimals stored as FLOAT. Converting to the correct analytical type improves storage efficiency, enables correct sorting and arithmetic, and prevents semantic errors in reports. Fabric supports type conversion in T-SQL (CAST, CONVERT, TRY_CAST, TRY_CONVERT), PySpark (cast, withColumn), and Power Query M (type annotations and conversion functions).

Key facts

  • CAST(expr AS type) — ANSI SQL standard; throws an error on invalid conversion (e.g., non-numeric string to INT).
  • TRY_CAST / TRY_CONVERT — "safe" variants; return NULL instead of an error on invalid conversion; essential for dirty data in ETL pipelines.
  • CONVERT(type, expr, style) — SQL Server extension; adds a style parameter for date/time formatting (e.g., style 103 = dd/mm/yyyy); not ANSI standard.
  • Implicit conversion — Fabric Warehouse will silently convert compatible types (e.g., INT to BIGINT); implicit conversion of incompatible types (e.g., VARCHAR to DATE) causes runtime errors.
  • PySpark cast — df.withColumn("Amount", col("Amount").cast("double")); for invalid values, returns NULL (safe by default in Spark).
  • Power Query M — each column has a declared type; changing type via UI generates Table.TransformColumnTypes(table, {{"Col", type date}}); errors can be replaced with Table.ReplaceErrorValues.
  • Date/time conversions are the most error-prone: string date format must exactly match the style parameter (or ISO 8601 yyyy-MM-dd); locale-specific formats (MM/dd/yyyy vs dd/MM/yyyy) cause silent wrong conversions.
-- T-SQL: CAST, TRY_CAST, TRY_CONVERT examples
SELECT
    OrderID,
    -- Safe conversion; returns NULL for invalid strings (not an error)
    TRY_CAST(OrderDateStr  AS DATE)          AS OrderDate,
    TRY_CAST(SalesAmountStr AS DECIMAL(10,2)) AS SalesAmount,
    -- CAST raises error on bad data — use when data is guaranteed clean
    CAST(Quantity AS INT)                    AS Quantity,
    -- CONVERT with style 103 = dd/mm/yyyy input format
    TRY_CONVERT(DATE, EUDateStr, 103)        AS EUDate
FROM dbo.OrdersStaging
WHERE TRY_CAST(SalesAmountStr AS DECIMAL(10,2)) IS NOT NULL;  -- filter bad rows

# PySpark: cast columns to correct types
from pyspark.sql import functions as F
from pyspark.sql.types import IntegerType, DoubleType, DateType

df = spark.table("bronze.RawOrders")

df_typed = (df
    .withColumn("OrderDate",   F.to_date("OrderDateStr", "yyyy-MM-dd"))
    .withColumn("SalesAmount", F.col("SalesAmountStr").cast(DoubleType()))
    .withColumn("Quantity",    F.col("QuantityStr").cast(IntegerType()))
    # Flag rows where cast produced NULL (i.e., invalid source values)
    .withColumn("HasTypeError",
        F.col("SalesAmount").isNull() | F.col("Quantity").isNull())
)

df_valid   = df_typed.filter(~F.col("HasTypeError"))
df_invalid = df_typed.filter( F.col("HasTypeError"))

df_valid.write.format("delta").mode("append").saveAsTable("silver.Orders")
df_invalid.write.format("delta").mode("append").saveAsTable("dq.TypeConversionErrors")

Exam tips & gotchas

  • CAST vs TRY_CAST: use TRY_CAST in ETL pipelines where source data quality cannot be guaranteed; use CAST only when data is validated.
  • FLOAT is an approximate type — avoid it for financial amounts where DECIMAL(p,s) is required for exact precision.
  • PySpark cast to DoubleType() returns NULL (not an error) for invalid string values — always check for unexpected NULLs post-cast.
  • Power Query M type changes applied via the UI generate Table.TransformColumnTypes; this step will throw a step-level error if a value cannot be converted, halting the entire Dataflow refresh.

Practice

Q1. A staging table column SalesAmountStr contains mostly numeric strings but occasionally the value 'N/A'. A T-SQL ETL pipeline must convert this column to DECIMAL(10,2) without failing on 'N/A' rows. Which function should be used?
Answer: TRY_CAST(SalesAmountStr AS DECIMAL(10,2)). Why: TRY_CAST returns NULL for invalid conversions instead of raising an error. CAST would fail the entire query for any 'N/A' value.
Q2. A PySpark notebook casts a column Price from StringType to DoubleType. After the cast, a data quality check finds unexpected NULLs in Price. What most likely caused this?
Answer: The source column contains invalid (non-numeric) string values that Spark could not convert. PySpark's cast returns NULL for values that fail conversion instead of throwing an error, so invalid strings silently become NULLs.

2.2.9 Filter data

Filtering reduces a dataset to only the rows needed for a given transformation step, report, or destination. Applying filters as early as possible in the processing pipeline — ideally at the source system via query pushdown — is the most important performance optimisation in data engineering. This principle is formalised as predicate pushdown in Spark and query folding in Power Query.

Key facts

  • T-SQL WHERE — evaluated after FROM/JOIN, before GROUP BY; can reference any column in scope; supports AND, OR, NOT, IN, BETWEEN, LIKE, IS NULL, EXISTS.
  • T-SQL HAVING — filters on aggregated expressions after GROUP BY; cannot reference aliases defined in SELECT (use CTE or subquery).
  • PySpark filter / where — equivalent methods; accepts Column expressions or SQL strings; lazy (no execution until action); Spark automatically pushes filters to the Delta scan (partition and file-level pruning).
  • Delta partition pruning — if the filter column matches a Delta partition column, Spark skips entire file partitions; dramatically reduces I/O on large tables.
  • Power Query Table.SelectRows — Table.SelectRows(table, each [Amount] > 0 and [Status] = "Active"); folds to source WHERE if connector supports it.
  • Query folding — Power Query sends the filter as a WHERE clause to the data source, reducing rows returned over the network; verify with "View native query" option.
-- T-SQL: WHERE with multiple conditions and IN list
SELECT OrderID, CustomerKey, SalesAmount, OrderDate
FROM   dbo.SalesFact
WHERE  SalesAmount > 0
  AND  OrderDate BETWEEN '2024-01-01' AND '2024-12-31'
  AND  StatusCode IN ('SHIP', 'COMP')
  AND  CancelledDate IS NULL;

-- T-SQL: filter using EXISTS (semi-join — efficient for large tables)
SELECT c.CustomerKey, c.CustomerName
FROM   dbo.DimCustomer c
WHERE  EXISTS (
    SELECT 1 FROM dbo.SalesFact f
    WHERE  f.CustomerKey = c.CustomerKey
      AND  f.OrderDate >= '2024-01-01'
);

# PySpark: filter with Column expressions (partition pruning)
from pyspark.sql import functions as F

df = spark.table("silver.SalesFact")  # partitioned by OrderYear

# Spark pushes the year filter to the Delta scan — skips other partitions
df_filtered = (df
    .filter(F.col("OrderYear") == 2024)           # partition pruning
    .filter(F.col("SalesAmount") > 0)
    .filter(F.col("StatusCode").isin("SHIP","COMP"))
    .filter(F.col("CancelledDate").isNull())
)

df_filtered.write.format("delta").mode("overwrite") \
    .saveAsTable("gold.Sales2024Active")

// Power Query M: Table.SelectRows with query folding
let
    Source    = Sql.Database("myserver.database.windows.net", "SalesDB"),
    SalesFact = Source{[Schema="dbo",Item="SalesFact"]}[Data],
    // This filter folds to WHERE SalesAmount > 0 AND StatusCode IN ('SHIP','COMP') in SQL
    Filtered  = Table.SelectRows(SalesFact,
                    each [SalesAmount] > 0
                    and List.Contains({"SHIP","COMP"}, [StatusCode])
                    and [CancelledDate] = null)
in
    Filtered
Tip: Always filter on Delta partition columns first in PySpark. If a table is partitioned by OrderYear, a filter on OrderYear = 2024 lets Spark skip all other year partitions at the file-system level — often a 10× or greater I/O reduction.

Exam tips & gotchas

  • WHERE vs HAVING: WHERE filters before aggregation; HAVING filters the result of GROUP BY. Using HAVING to filter on a non-aggregate column is legal but less efficient.
  • Query folding in Power Query: if any step breaks folding (e.g., adding a custom M column with a non-foldable function), all subsequent steps also lose folding — even simple filters.
  • PySpark filter and where are identical — choose either for readability.
  • Delta Z-Order clustering (if applied) also helps filter performance by collocating related data within Parquet files, reducing the number of row groups scanned.
  • EXISTS in T-SQL is usually faster than IN with a large subquery list because it short-circuits on the first match.

Practice

Q1. A Dataflow Gen2 reads from an Azure SQL Database and applies a Table.SelectRows filter. A data engineer then adds a custom M column using a regex function. What is the impact on query folding for the filter?
Answer: The filter loses query folding. Why: The custom M column uses a function that cannot be translated to SQL (regex is not a foldable operation). Any step that breaks folding causes all subsequent steps — including earlier filter steps that appear after the break — to execute in the Dataflow engine rather than at the source.
Q2. A PySpark job filters a Delta table on a column ProductCategory, but the table is partitioned by OrderYear. Will Spark apply partition pruning for the ProductCategory filter?
Answer: No. Why: Partition pruning skips entire partition directories based only on the partition columns (here, OrderYear). A filter on ProductCategory cannot skip partitions; instead, Spark uses column statistics and row-group filtering within Parquet files (data skipping), which is less aggressive than partition pruning.
Q3. Which T-SQL filtering technique is most efficient when checking whether a large fact table contains at least one row for each customer in a dimension table, without returning the fact rows themselves?
Answer: EXISTS — WHERE EXISTS (SELECT 1 FROM SalesFact WHERE CustomerKey = c.CustomerKey). Why: EXISTS short-circuits on the first match, so Fabric Warehouse does not need to read all matching fact rows. IN with a subquery or a JOIN that returns all fact rows requires processing more data.

2.3 Query and analyze data

2.3.1 Visual Query Editor and query folding

The Visual Query Editor is a no-code query builder available inside Fabric Warehouse and the Lakehouse SQL analytics endpoint. It presents a drag-and-drop canvas where engineers add tables, apply transforms (filter, group by, join, rename columns), and preview results — all without writing SQL. Each step is translated into a SQL query that is sent to the underlying engine, meaning the engine does all the heavy lifting rather than the client.

Query folding is the mechanism whereby Power Query (Dataflows Gen2, Power BI Desktop) translates M transform steps into a single native query — SQL, OData, SPARQL — that runs at the source rather than in the Mashup engine. Folded queries reduce data movement and use source-side indexes, parallelism, and pushdown capabilities.

Key facts

  • Visual Query Editor in Warehouse produces SQL that can be viewed and copied; the generated query is submitted to the Warehouse SQL engine — no separate compute is used.
  • Transforms available in Visual Query: filter rows, choose/rename/reorder columns, group by, sort, merge (JOIN), append (UNION), aggregate, add column (limited expressions).
  • Query folding indicators — in Power Query Editor, right-click a step: if "View native query" is available (not grayed out), the step folds. A grayed-out option means folding stopped at that step.
  • Connectors that always support folding: SQL Server, Azure SQL, Fabric Warehouse/Lakehouse, OData, SharePoint List, Salesforce.
  • Steps that commonly break folding: custom M functions, invoke custom function, add index column, Table.Buffer, Table.Sort followed by a non-foldable step, and any function without a native SQL equivalent.
  • Best practice: apply all foldable steps (filter, remove columns, rename, type) before any non-foldable step so the maximum filter is pushed to the source.
  • Visual Query results can be saved as a view in the Warehouse without writing SQL manually.
Key: Query folding is a Mashup engine feature, not a storage engine feature. It affects Dataflows Gen2 and Power BI Desktop (Direct Query or Import with Power Query steps) — it does not apply to T-SQL or KQL queries, which always run at the engine directly.

Exam tips & gotchas

  • If a step breaks folding, ALL subsequent steps also lose folding — even a simple rename step after a non-foldable step will execute in the Mashup engine.
  • "View native query" being grayed out confirms folding stopped; use this to diagnose Dataflow performance problems.
  • Visual Query Editor (Warehouse) vs Power Query Editor (Dataflows): both are no-code, but VQE is purely SQL-based and always runs at the engine; Power Query uses M and relies on folding to push work to the source.
  • Table.Buffer() forces data into memory and always breaks folding — avoid unless you have a specific reason (e.g., preventing multiple source evaluations for performance).

Practice

Q1. A Dataflow Gen2 connects to a SQL Server source and applies steps in this order: (1) filter rows, (2) remove columns, (3) add custom M column using Text.BetweenDelimiters, (4) rename columns. Which steps fold to SQL Server?
Answer: Steps 1 and 2 fold. Why: Step 3 (Text.BetweenDelimiters is a non-foldable M function) breaks folding. Step 4 (rename), although normally foldable, comes after the break and therefore also executes in the Mashup engine.
Q2. A data engineer right-clicks a filter step in Power Query Editor and finds "View native query" is grayed out. What does this indicate?
Answer: The filter step does not fold to the source — it executes in the Power Query Mashup engine (client-side). A previous step likely broke folding, causing all subsequent steps to run locally. The engineer should identify and relocate or remove the non-foldable step.
Q3. Which action in a Dataflow Gen2 step sequence always breaks query folding, regardless of the source connector?
Answer: Table.Buffer(). Why: Table.Buffer forces the data into memory in the Mashup engine, terminating the connection to the source query plan. All steps after Buffer execute entirely in memory within the Mashup engine.

2.3.2 Query data with SQL

Fabric Warehouse and the Lakehouse SQL analytics endpoint both expose T-SQL endpoints that clients connect to via TDS (SQL Server wire protocol). Standard SQL tooling — SQL Server Management Studio, Azure Data Studio, Power BI Desktop, and the Fabric web SQL editor — connects with no driver changes. Fabric's T-SQL dialect is based on Azure Synapse Dedicated SQL and supports nearly all ANSI SQL alongside common SQL Server extensions.

Key facts

  • SELECT … FROM … JOIN … WHERE … GROUP BY … HAVING … ORDER BY — all supported; CTE (WITH clause) fully supported.
  • Window functions — ROW_NUMBER, RANK, DENSE_RANK, NTILE, LAG, LEAD, FIRST_VALUE, LAST_VALUE, SUM/AVG/COUNT OVER (PARTITION BY … ORDER BY …) — all supported.
  • CROSS APPLY / OUTER APPLY — supported; used to lateral-join table-valued functions or JSON parsing expressions.
  • Common Fabric-specific unsupported T-SQL: no distributed transactions, no linked servers, no CLR objects, no XML methods on columns (as of GA). Check documentation for the latest support matrix.
  • Queries can reference tables across workspaces using cross-workspace shortcuts exposed as external tables (via shortcuts at the Lakehouse level).
  • Fabric Warehouse supports EXPLAIN (query plan) via the SQL editor; use it to diagnose full-table scans and suboptimal join orders.
  • SELECT … INTO creates a new table (not supported at Lakehouse SQL analytics endpoint; Warehouse only).
-- Window functions in Fabric Warehouse SQL
WITH RankedSales AS (
    SELECT
        c.CustomerName,
        p.CategoryName,
        SUM(f.SalesAmount)   AS TotalSales,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryName
            ORDER BY SUM(f.SalesAmount) DESC
        )                    AS CategoryRank,
        RANK() OVER (
            ORDER BY SUM(f.SalesAmount) DESC
        )                    AS OverallRank,
        LAG(SUM(f.SalesAmount), 1, 0) OVER (
            PARTITION BY p.CategoryName
            ORDER BY MIN(f.OrderDate)
        )                    AS PrevPeriodSales
    FROM dbo.SalesFact  f
    JOIN dbo.DimCustomer c ON f.CustomerKey = c.CustomerKey
    JOIN dbo.DimProduct  p ON f.ProductKey  = p.ProductKey
    GROUP BY c.CustomerName, p.CategoryName
)
SELECT *
FROM   RankedSales
WHERE  CategoryRank <= 10
ORDER  BY CategoryName, CategoryRank;

Exam tips & gotchas

  • ROW_NUMBER returns a unique integer per row (no ties); RANK skips numbers after ties; DENSE_RANK does not skip.
  • LAG/LEAD access previous/next row values within the partition — powerful for period-over-period comparison without a self-join.
  • The Lakehouse SQL analytics endpoint supports SELECT, CREATE VIEW/FUNCTION/PROCEDURE but not DML. The Warehouse supports full DML.
  • Window functions cannot be referenced directly in WHERE or HAVING — wrap in a CTE or subquery (as shown above with WHERE CategoryRank <= 10).

Practice

Q1. A SQL query using a window function assigns the same RANK value to two customers with equal TotalSales. What rank value does the next customer receive?
Answer: The next customer receives rank 3 (if two customers tied at rank 1) — i.e., RANK skips rank 2. Why: RANK assigns the same value to tied rows and then skips the next rank number(s). Use DENSE_RANK to avoid skipping.
Q2. A query must filter on a window function result (e.g., show only top-10 per category). A data engineer writes: SELECT ..., RANK() OVER (...) AS Rnk FROM ... WHERE Rnk <= 10. Does this work?
Answer: No — it throws an error. Why: Window functions are evaluated after WHERE, so they cannot be referenced in the WHERE clause of the same SELECT. Wrap the window function in a CTE or subquery and filter in the outer query.

2.3.3 Query data with KQL

Kusto Query Language (KQL) is the primary query language for Fabric Eventhouse (KQL databases). It uses a pipe (|) syntax where the output of each operator flows into the next, building a readable left-to-right transformation pipeline. KQL is optimised for time-series, log analysis, and real-time telemetry — operations that are cumbersome in SQL but natural in KQL.

KQL is available in the Eventhouse query editor, from a KQL queryset item, from Fabric notebooks via the %%kql magic, and programmatically through the Kusto REST API. Mastering the core operators and the bin() time-bucketing function is essential for the DP-600 exam.

Key facts

  • Pipe syntax — TableName | operator1 | operator2 | ...; each operator receives the previous output as its input table.
  • where — filters rows; equivalent to SQL WHERE; | where SalesAmount > 0 and Region == "West". Use == (not =) for equality comparison in KQL.
  • project — selects and optionally renames columns; drops all others; | project OrderID, Amount = SalesAmount.
  • project-away — drops named columns, keeps all others (opposite of project).
  • extend — adds computed columns without dropping others; | extend ProfitMargin = (Revenue - Cost) / Revenue.
  • summarize — aggregates; collapses rows to one per group-by key; | summarize TotalSales = sum(SalesAmount), Orders = count() by CategoryName, bin(EventTime, 1h). After summarize, only by-columns and aggregate expressions are available.
  • bin(column, interval) — floor to the nearest multiple of interval; essential for time-series bucketing; bin(StartTime, 1d) rounds to day.
  • sort by / order by — synonyms; | sort by TotalSales desc.
  • take / limit — return the first N rows (non-deterministic order unless sorted first).
  • render — chart directive; | render timechart; display hint only — does not change data output.
  • join — | join kind=inner (RightTable | project ...) on Key; join kinds: inner, leftouter, rightouter, fullouter, leftanti, rightanti, leftsemi.
  • make-series — creates time-series arrays per group; useful with time-series ML functions (series_decompose, series_outliers).
  • Type functions: toint(), tolong(), todouble(), tostring(), todatetime(), totimespan().
  • Time helpers: ago(30d), now(), startofday(), startofmonth(), datetime(2024-01-01).
// 1. Basic where + project + sort
SalesFact
| where EventTime > ago(30d)
    and SalesAmount > 0
    and Region == "West"
| project OrderID, CustomerID, SalesAmount,
          EventTime, Region
| sort by SalesAmount desc
| take 100

// 2. extend: add a computed column
SalesFact
| where EventTime between (datetime(2024-01-01) .. datetime(2024-12-31))
| extend ProfitMargin = round((Revenue - Cost) / Revenue, 4)
| project OrderID, Revenue, Cost, ProfitMargin

// 3. summarize with bin() for hourly time-series
SalesFact
| where EventTime > ago(7d)
| summarize
    TotalSales = sum(SalesAmount),
    Orders     = count()
  by CategoryName, bin(EventTime, 1h)
| sort by EventTime asc
| render timechart

// 4. join: enrich fact with dimension
SalesFact
| join kind=leftouter (
    DimProduct
    | project ProductID, ProductName, CategoryName
  ) on ProductID
| project OrderID, ProductName, CategoryName, SalesAmount, EventTime

// 5. Time-range filter patterns
SecurityEvents
| where TimeGenerated between (ago(24h) .. now())
| where EventID in (4624, 4625, 4648)
| summarize FailedLogons = countif(EventID == 4625),
            SuccessLogons = countif(EventID == 4624)
  by Computer, bin(TimeGenerated, 1h)
Gotcha: After summarize, only the columns in the by clause and the aggregation expressions exist in the output. Any attempt to reference a column that was present before summarize but not in the by list will cause an error in a downstream operator.

Exam tips & gotchas

  • KQL uses == for equality (not =). Using = causes a parse error.
  • project drops all columns not named; extend keeps all existing columns and adds new ones. Know which to use to avoid accidental column loss.
  • bin() is required to produce regular time buckets in summarize by — without it, each unique timestamp becomes its own group.
  • render is a display directive, not a data operation. It has no effect on the tabular output and does not exist in programmatic API calls.
  • take N is non-deterministic unless preceded by sort by — do not use take to get "the most recent N rows" without an explicit sort.
  • After summarize, you cannot project a column that was not in the by clause — it no longer exists in scope.

Practice

Q1. A KQL query must count the number of events per hour for the past seven days, grouped by server name. Which operators and functions are required?
Answer: | where TimeGenerated > ago(7d) to filter the time range, then | summarize EventCount = count() by ServerName, bin(TimeGenerated, 1h) to count per hour per server. bin(TimeGenerated, 1h) is essential to floor each timestamp to its hour boundary, creating regular one-hour buckets.
Q2. A data analyst writes this KQL query:
SalesFact | summarize Total = sum(SalesAmount) by CategoryName | project OrderID, CategoryName, Total. Why does this query fail?
Answer: The project references OrderID, which no longer exists after summarize. The summarize operator collapses rows to one per CategoryName and retains only the by-clause columns and aggregate expressions — OrderID is not in scope. Remove OrderID from the project list.
Q3. In KQL, what is the difference between project and extend?
Answer: project selects specific columns and drops all others (like SQL SELECT with explicit column list). extend adds new computed columns while preserving all existing columns (like SQL SELECT *, newcol). Use project when you want to narrow the schema; use extend when you want to enrich without losing columns.

2.3.4 Query data with DAX

Data Analysis Expressions (DAX) is the formula language of Power BI semantic models. For DP-600, DAX is tested at an introductory level in Domain 2 (Prepare data) — you must understand the core concepts of filter context, measures, and the most important functions. Deep DAX mastery (time intelligence, semi-additive measures, row context) belongs to Domain 3 (Analyse and visualise data).

Measures are DAX formulas stored in a semantic model that are evaluated dynamically based on the current filter context — the combination of slicer selections, visual filters, and page/report filters active at the moment the measure is called. Unlike calculated columns, measures produce no stored data and are recalculated on every query.

Key facts

  • Filter context — the set of filters applied to a visual at the moment DAX evaluates a measure; determined by slicers, visual-level filters, page filters, report filters, and cross-filter relationships.
  • CALCULATE(expression, [filter1], …) — the single most important DAX function; evaluates expression after modifying the filter context with the given filters; can add, remove, or replace filters.
  • FILTER(table, condition) — returns a filtered virtual table; used as an argument to CALCULATE; CALCULATE(SUM(Sales[Amount]), FILTER(Sales, Sales[Region] = "West")).
  • DIVIDE(numerator, denominator, [alternateResult]) — safe division; returns alternateResult (default: BLANK()) when denominator is 0; always prefer over the / operator to avoid division-by-zero errors.
  • ALL(table_or_column) — removes filters from the specified table or column; used inside CALCULATE to create "percent of total" measures.
  • Basic aggregations: SUM, AVERAGE, COUNT (counts rows where expression is non-blank), DISTINCTCOUNT, MIN, MAX, COUNTROWS (counts rows in a table).
  • Time intelligence: TOTALYTD, TOTALMTD, TOTALQTD require a properly marked Date table; SAMEPERIODLASTYEAR, DATEADD provide comparison periods.
  • DAX measures are defined in the semantic model (dataset), not in the Warehouse or Lakehouse — they do not affect underlying data storage.
  • A semantic model in Direct Lake mode can still have DAX measures; the measures execute against the in-memory framed data loaded from Delta files.
-- Basic measure definitions
TotalSales = SUM(SalesFact[SalesAmount])

OrderCount = COUNTROWS(SalesFact)

AvgOrderValue = DIVIDE(
    SUM(SalesFact[SalesAmount]),
    COUNTROWS(SalesFact),
    BLANK()    -- return BLANK instead of error when no rows
)

-- CALCULATE: modify filter context
WestRegionSales =
CALCULATE(
    SUM(SalesFact[SalesAmount]),
    DimGeography[Region] = "West"
)

-- Percent of total using ALL()
SalesShareOfTotal =
DIVIDE(
    SUM(SalesFact[SalesAmount]),
    CALCULATE(SUM(SalesFact[SalesAmount]), ALL(SalesFact)),
    0
)

-- FILTER inside CALCULATE for complex conditions
HighValueSales =
CALCULATE(
    SUM(SalesFact[SalesAmount]),
    FILTER(
        SalesFact,
        SalesFact[SalesAmount] > 10000
            && SalesFact[StatusCode] = "COMP"
    )
)

-- Time intelligence: year-to-date sales
SalesYTD =
TOTALYTD(
    SUM(SalesFact[SalesAmount]),
    DimDate[Date]
)
Key: CALCULATE is the gateway to advanced DAX. Every time intelligence function (TOTALYTD, SAMEPERIODLASTYEAR) is implemented by calling CALCULATE internally with a modified date filter. Understanding what CALCULATE does to the filter context is the foundation for understanding all complex DAX patterns.

Exam tips & gotchas

  • DIVIDE(x, y) is always preferred over x / y — the / operator returns a division-by-zero error when y = 0; DIVIDE returns BLANK() or an alternate result.
  • Measures are evaluated in filter context, not row context — a measure referencing SalesFact[SalesAmount] aggregates all matching rows, not a single row.
  • FILTER returns a table; it is used as an argument inside CALCULATE, not as a standalone query function.
  • ALL() removes filters from a column or table. ALLEXCEPT(table, col) removes all filters except the specified column — useful for "percent within category" patterns.
  • Time intelligence functions require a Date table marked as a Date table in the semantic model, with a contiguous date range covering all dates in the fact table.
  • Deep DAX (row context, context transition, EARLIER, RANKX, iterators) is tested in Domain 3 — Domain 2 only requires understanding the concepts above.

Practice

Q1. A measure calculates total sales only for the "West" region, regardless of any region filter applied in the report. Which DAX function is required and how is it structured?
Answer: CALCULATE(SUM(SalesFact[SalesAmount]), ALL(DimGeography[Region]), DimGeography[Region] = "West"). Why: CALCULATE first removes any existing Region filter with ALL(), then reapplies the fixed "West" filter. Without ALL(), the measure would still be affected by the report's active region filter, potentially returning BLANK() when a different region is selected.
Q2. A measure is defined as SharePct = SUM(SalesFact[SalesAmount]) / CALCULATE(SUM(SalesFact[SalesAmount]), ALL(SalesFact)). A report user selects only 2024 in a Year slicer. Does the denominator (total) change?
Answer: No. Why: ALL(SalesFact) removes all filters on the SalesFact table, including the Year filter. The denominator always returns the grand total across all years, making SharePct a true "share of total" regardless of slicer selection.
Q3. Why should a developer use DIVIDE(x, y) instead of x / y in a DAX measure?
Answer: DIVIDE handles division by zero gracefully by returning BLANK() (or a custom alternate result) instead of an error. The / operator raises a "Division by zero" error when the denominator evaluates to 0, which can break visuals. DIVIDE also propagates BLANK() correctly through calculations.
