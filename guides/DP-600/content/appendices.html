D. T-SQL cheat sheet (Fabric warehouse)

-- Clean + enrich into a curated table (CTAS)
SELECT o.order_id,
       ISNULL(c.region, 'Unknown')            AS region,   -- handle NULLs
       CAST(o.order_ts AS date)               AS order_date,
       o.qty * o.unit_price                   AS line_total
INTO   gold.fact_sales
FROM   staging.orders AS o
LEFT JOIN staging.customer AS c ON c.customer_id = o.customer_id
WHERE  o.qty > 0;

-- Deduplicate (keep latest)
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY updated_ts DESC) AS rn
  FROM staging.orders)
DELETE FROM ranked WHERE rn > 1;

-- Upsert with MERGE (incremental load)
MERGE gold.dim_customer AS tgt
USING staging.customer AS src ON tgt.customer_id = src.customer_id
WHEN MATCHED THEN UPDATE SET tgt.region = src.region
WHEN NOT MATCHED THEN INSERT (customer_id, region) VALUES (src.customer_id, src.region);

-- Reusable objects
CREATE VIEW gold.v_active_sales AS SELECT * FROM gold.fact_sales WHERE line_total > 0;
CREATE PROCEDURE gold.load_sales AS BEGIN /* ETL steps */ END;

-- Fast bulk ingest from files
COPY INTO gold.fact_sales FROM 'https://.../sales/*.parquet' WITH (FILE_TYPE = 'PARQUET');
Tip: The lakehouse SQL analytics endpoint is read-only — these DML/DDL statements run in a warehouse (or via Spark for the lakehouse).

E. KQL cheat sheet (eventhouse / KQL database)

Sales
| where OrderDate >= ago(30d)            // filter rows
| where Region == "Midwest"
| extend LineTotal = Qty * UnitPrice     // add a column
| summarize Revenue = sum(LineTotal),    // aggregate
            Orders  = count()
    by bin(OrderDate, 1d), Region        // group + time bucket
| project OrderDate, Region, Revenue, Orders   // select columns
| order by OrderDate asc
| top 100 by Revenue                     // limit

// Common operators: where, project, extend, summarize, count, distinct,
// join kind=inner, union, make-series (time series), ago(), bin(), between()
Remember: KQL is pipe-based, case-sensitive for string equality (==), and reads top-to-bottom — each | feeds the next operator.

F. DAX cheat sheet

-- Basic measures
Revenue      = SUM ( Sales[LineTotal] )
Profit       = [Revenue] - [Cost]
Margin %     = DIVIDE ( [Profit], [Revenue] )          -- safe divide (no /0 error)

-- Variables (readable, faster — evaluated once)
Sales YoY % =
VAR Cur  = [Revenue]
VAR Prev = CALCULATE ( [Revenue], SAMEPERIODLASTYEAR ( 'Date'[Date] ) )
RETURN DIVIDE ( Cur - Prev, Prev )

-- Iterator (row-by-row, then aggregate)
Weighted Sales = SUMX ( Sales, Sales[Qty] * Sales[UnitPrice] )

-- Modify filter context
Sales All Regions = CALCULATE ( [Revenue], ALL ( 'Region' ) )
Sales Big Orders  = CALCULATE ( [Revenue], FILTER ( Sales, Sales[LineTotal] > 1000 ) )

-- Information / dynamic security
Current User = USERPRINCIPALNAME()
Picked       = SELECTEDVALUE ( 'Product'[Category], "All" )

-- Windowing / time intelligence
YTD Revenue  = TOTALYTD ( [Revenue], 'Date'[Date] )
Prev Row     = CALCULATE ( [Revenue], OFFSET ( -1, ORDERBY ( 'Date'[Date] ) ) )
Gotcha: Row context (in calculated columns/iterators) is not the same as filter context (from slicers/relationships). CALCULATE is what transitions and overrides filter context.

G. "Which one when?" rapid-fire

  • UPDATE/MERGE needed → Warehouse (not lakehouse SQL endpoint).
  • Huge data, avoid copies, fast refresh → Direct Lake.
  • Need composite / Import + Direct Lake together → Direct Lake on OneLake.
  • Need SQL-endpoint RLS honored in Direct Lake → Direct Lake on SQL endpoint (accepts DirectQuery fallback).
  • Model > 10 GB or scale-out or XMLA write → enable Large semantic model storage format.
  • Refresh only recent data of a big table → Incremental refresh (RangeStart/RangeEnd).
  • One measure, many time patterns (MTD/YTD/PY) → Calculation group.
  • Let users pick a measure/dimension in a slicer → Field parameters.
  • Many-to-many between two dimensions → Bridge table.
  • Git, code review, CI/CD for reports/models → .pbip project + workspace Git.
  • Reuse a report/model structure without data → .pbit template. Standardize a connection → .pbids.
  • Deploy Dev→Test→Prod inside Fabric → Deployment pipelines. Programmatic/external deploy → XMLA endpoint (read-write).
  • Assess blast radius of a change → Impact analysis / lineage view.
  • Signal a trusted dataset → Endorsement (Promoted → Certified). Classify/protect → Sensitivity label.

H. Glossary

  • OneLake — the single, tenant-wide data lake underlying all Fabric items ("one copy" of data in Delta Parquet).
  • Shortcut — a pointer to data in another location (OneLake, ADLS Gen2, S3) with no copy.
  • Medallion architecture — Bronze (raw) → Silver (cleaned) → Gold (curated/modeled) layering.
  • Delta table — open Parquet + transaction log; the standard OneLake table format.
  • SQL analytics endpoint — auto-created read-only T-SQL surface over a lakehouse's Delta tables.
  • Framing — Direct Lake's metadata-only refresh that repoints the model to the latest Delta files.
  • Transcoding — on-demand loading of Delta columns into VertiPaq memory for Direct Lake.
  • Star schema — central fact table(s) surrounded by denormalized dimension tables.
  • SCD — Slowly Changing Dimension; Type 1 overwrites, Type 2 keeps history (new row + validity).
  • Calculation group — a reusable set of calculation items (e.g., YTD, PY) applied to measures.
  • Field parameter — lets report users swap measures/columns on a visual via a slicer.
  • XMLA endpoint — the Analysis Services protocol surface for external tools to read/write semantic models.
  • .pbip / .pbit / .pbids — Power BI project (Git-friendly) / template (structure, no data) / data-source connection file.
  • Real-Time hub — central place to discover and subscribe to streaming data (eventstreams, KQL).
  • Endorsement — trust signal: Promoted, Certified, or Master data.
  • Query folding — Power Query pushing transformation steps down to the source system.
