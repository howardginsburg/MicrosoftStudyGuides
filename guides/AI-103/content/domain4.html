Domain 4 — Implement text analysis solutions 10–15%

This domain covers how to extract structured information from text and audio using Azure AI Language, Azure OpenAI, Azure Translator, and Azure AI Speech. Topics span entity extraction, sentiment analysis, PII detection, machine and LLM-powered translation, speech recognition, text-to-speech synthesis, and audio-grounded multimodal reasoning—capabilities that underpin agentic, compliant, and voice-enabled AI solutions.

4.1 Apply language model text analysis

4.1.1 Extract entities, topics, summaries, and structured JSON outputs

Azure AI Language provides prebuilt NLP capabilities for entity recognition and summarization that require no model training. Named Entity Recognition (NER) identifies and categorizes entities such as people, places, organizations, and quantities from unstructured text using a preset taxonomy. A custom NER variant allows training on domain-specific entity types that go beyond the standard list—useful for legal, medical, or financial text.

Language Summarization delivers three modes matched to input genre: text summarization for prose blocks, conversation summarization for multi-turn transcripts, and native document summarization for .txt, .pdf, and .docx files. Extractive summarization selects and ranks the most salient original sentences—returning a rank score and positional data per sentence. Abstractive summarization generates novel, concise sentences that may not appear verbatim in the source. Both modes purge results within 24 hours.

For generative extraction with a guaranteed structure, Azure OpenAI Structured Outputs enforces a JSON Schema by setting response_format to json_schema with strict: true. This guarantees schema compliance, unlike JSON mode, which only guarantees valid JSON syntax. The Python openai + pydantic stack lets you declare a BaseModel schema and have the SDK validate the parsed response automatically before returning it.

Key facts

  • Structured Outputs require API version 2024-08-01-preview or later (GA: 2024-10-21).
  • Supported models: gpt-4o (≥ 2024-08-06), gpt-4o-mini (≥ 2024-07-18), gpt-4.1, o4-mini, o3.
  • Structured Outputs are not supported with gpt-4o-audio-preview, bring-your-own-data scenarios, the Assistants API, or Azure AI Agents Service.
  • JSON mode guarantees valid JSON; Structured Outputs (strict: true) guarantee schema shape compliance.
  • Summarization results are purged after 24 hours—retrieve and store results before this window closes.
  • Extractive summarization returns rank score and positional information per selected sentence.
  • Custom NER training requires labeled examples created in Language Studio or via REST API.
from pydantic import BaseModel, Field
from openai import AzureOpenAI

class DocumentExtract(BaseModel):
    title: str
    summary: str = Field(..., description="One-sentence summary")
    entities: list[str] = Field(..., description="Key named entities")

client = AzureOpenAI(
    azure_endpoint="https://<resource>.openai.azure.com/",
    api_key="<key>",
    api_version="2024-10-21",
)

completion = client.beta.chat.completions.parse(
    model="gpt-4o",
    messages=[{
        "role": "user",
        "content": "Extract entities and summarize: <document text here>"
    }],
    response_format=DocumentExtract,
)
result: DocumentExtract = completion.choices[0].message.parsed
print(result.title, result.entities)
Tip: Use Structured Outputs (strict: true) when downstream pipelines depend on a guaranteed schema shape. Fall back to JSON mode only when you need valid JSON but the exact key set is flexible.

Exam tips & gotchas

  • JSON mode ≠ Structured Outputs: JSON mode guarantees valid JSON, not schema shape. Structured Outputs guarantee both.
  • Structured Outputs do not work with audio-preview models, BYOD, the Assistants API, or Azure AI Agents Service.
  • Summarization results expire in 24 hours—always persist retrieved summaries immediately.
  • Prebuilt NER uses a fixed taxonomy; Custom NER requires labeled training data but covers any domain entity type.
  • Abstractive summaries generate new sentences; extractive summaries return ranked original sentences from the source.

Practice

Q1. A developer needs an Azure OpenAI response that always contains the fields risk_score (int) and summary (str) — no extra fields allowed. Which feature and setting should they use?
Answer: Azure OpenAI Structured Outputs — set response_format to json_schema with strict: true and define a matching Pydantic BaseModel. Why: JSON mode guarantees valid JSON but not field presence or type; Structured Outputs with strict: true enforce schema compliance exactly.
Q2. A compliance system must summarize .pdf audit documents and recognize domain-specific clause entity types not covered by the standard NER taxonomy. Which two Azure AI Language features address these requirements?
Answer: Native document summarization (accepts .pdf natively) and Custom NER (trained on domain-specific clause labels). Why: Prebuilt NER only covers the standard taxonomy; document-based summarization handles .pdf without preprocessing.
Q3. When using extractive summarization, what two metadata values does Azure AI Language return for each selected sentence?
Answer: Rank score and positional information. Why: These allow downstream systems to sort sentences by relevance and locate them within the source document.

4.1.2 Configure detection of sentiment, tone, safety issues, and sensitive content

Azure AI Language Sentiment Analysis assigns positive, neutral, or negative labels at both sentence and document level, returning confidence scores between 0 and 1 for each label—the label with the highest score is selected. Opinion Mining (aspect-based sentiment analysis) extends this by linking detected sentiment to specific target words or attributes, providing feature-level insight beyond aggregate document polarity.

PII Detection identifies sensitive personal data across three input formats matched to the data structure: Text PII handles synchronous string payloads such as messages and logs; Conversation PII processes turn-based chat and transcript structures; Document-based PII asynchronously processes .pdf, .docx, and .txt files while preserving document layout. All three variants return entity category, confidence score, and configurable redacted output with adjustable masking style.

These capabilities are surfaced through the shared azure-ai-textanalytics (Python) / Azure.AI.TextAnalytics (.NET) SDK and can be combined with other Language features in a single asynchronous batch request. For on-premises or strict data-residency scenarios, both sentiment analysis and PII detection support Docker container deployment.

Key facts

  • Sentiment labels are confidence scores 0–1; the label with the highest score is the assigned sentiment.
  • Opinion Mining is enabled by passing show_opinion_mining=True — it links sentiment to target words (aspects).
  • PII variants by format: Text PII (synchronous strings), Conversation PII (turn-based chat), Document PII (async .pdf/.docx/.txt).
  • PII returns redacted text, entity categories, and confidence scores per detected span.
  • Classic Language sentiment analysis retires March 31, 2029; new projects should migrate to Azure AI Foundry models.
  • Docker container deployment is supported for both sentiment analysis and PII detection for on-premises processing.
  • All Language features share the same SDK: azure-ai-textanalytics (Python) / Azure.AI.TextAnalytics (.NET).
from azure.ai.textanalytics import TextAnalyticsClient
from azure.core.credentials import AzureKeyCredential

client = TextAnalyticsClient(
    "https://<resource>.cognitiveservices.azure.com/",
    AzureKeyCredential("<key>")
)

# Sentiment with opinion mining (aspect-level feedback)
reviews = ["The delivery was fast but the product quality was disappointing."]
sentiment_results = client.analyze_sentiment(reviews, show_opinion_mining=True)
for doc in sentiment_results:
    print(doc.sentiment, doc.confidence_scores)

# PII redaction from a text payload
pii_docs = ["Please reach out to Sofia Hernandez at sofia.h@example.com."]
pii_results = client.recognize_pii_entities(pii_docs)
for doc in pii_results:
    print(doc.redacted_text)
Gotcha: Classic Azure AI Language sentiment analysis is scheduled to retire March 31, 2029. Begin planning migration timelines now for applications with long operational lifetimes.

Exam tips & gotchas

  • Opinion Mining is a mode of sentiment analysis (pass show_opinion_mining=True), not a separate API call.
  • The choice between Text PII, Conversation PII, and Document PII depends on input format, not content sensitivity.
  • Docker containers allow Language features to run entirely on-premises — data never leaves the local environment.
  • PII detection returns a redacted_text property and a list of detected entity spans with categories and confidence scores.

Practice

Q1. A retail company wants to identify which specific product features customers mention negatively in reviews, rather than just knowing the overall review sentiment. Which Language feature should they enable?
Answer: Opinion Mining (aspect-based sentiment analysis), enabled by passing show_opinion_mining=True to analyze_sentiment. Why: Standard sentiment returns document/sentence polarity only; Opinion Mining links sentiment polarity to specific target words—the product feature in this case.
Q2. A financial services firm must redact PII from multi-turn customer support chat logs before long-term storage. Which PII feature type is appropriate and why?
Answer: Conversation PII, which is designed for turn-based chat and transcript data structures. Why: Text PII processes single synchronous string payloads; Conversation PII understands the multi-turn structure of chat logs and preserves speaker-turn context.
Q3. A regulated financial institution cannot send data to Azure cloud endpoints. Which deployment option for PII detection meets this requirement?
Answer: Docker container deployment. Why: Azure AI Language supports running sentiment analysis and PII detection in Docker containers on-premises, keeping data within the organization's network boundary.

4.1.3 Translate text using Azure Translator in Foundry Tools or LLM-powered translation flows

Azure Translator is a cloud-based neural machine translation (NMT) service accessible via REST API. Beginning with the 2026-06-06 GA API release, individual translation requests can optionally route through a large language model (for example, GPT-5.4) for context-aware, tonally adaptive translation. Standard NMT continues to work with existing Translator or multi-service resources; LLM-powered translation requires a Foundry resource.

The two paths have distinct service limits and billing models. NMT supports up to 1,000 array elements at 50,000 characters each per request; LLM translation is capped at 50 array elements and 5,000 characters per element. NMT is billed by source character count, while LLM translation is billed by input and output token consumption under Azure OpenAI pricing—a significantly different cost profile at scale.

The Foundry portal Translator playground (available from November 2025) allows side-by-side comparison of Azure-MT (NMT), GPT-4o, and GPT-4o mini outputs and supports customization by gender, tone, and domain terminology without writing code. The Transliterate endpoint converts text between writing systems (for example, Japanese kana to Latin romanization) without translating the language itself.

Key facts

  • NMT limits: 1,000 array elements / 50,000 characters per element per request.
  • LLM translation limits: 50 array elements / 5,000 characters per element per request.
  • NMT billed by source character count; LLM translation billed by input + output tokens (OpenAI pricing).
  • LLM translation requires a Foundry resource; standalone Translator resources are insufficient.
  • Foundry playground model options: Azure-MT, GPT-4o, GPT-4o mini.
  • Transliterate converts scripts/alphabets (e.g., kana → Latin) — distinct from translation; the source language is preserved.
  • For high-volume, cost-sensitive workloads, route to NMT; reserve LLM for tone control or domain-specific contextual handling.
import requests, uuid

endpoint = "https://api.cognitive.microsofttranslator.com"
headers = {
    "Ocp-Apim-Subscription-Key": "<key>",
    "Ocp-Apim-Subscription-Region": "<region>",
    "Content-Type": "application/json",
    "X-ClientTraceId": str(uuid.uuid4()),
}
# NMT: translate Italian to English and French in one call
body = [{"text": "Questo documento è riservato."}]
params = {"api-version": "3.0", "from": "it", "to": ["en", "fr"]}
response = requests.post(
    f"{endpoint}/translate", headers=headers, params=params, json=body
)
for translation in response.json()[0]["translations"]:
    print(f"[{translation['to']}] {translation['text']}")
Key: Route to NMT for high-volume baseline traffic. Use LLM translation only when tone control, gender agreement, or rich domain adaptation is required—and confirm the Foundry resource is provisioned before calling the 2026-06-06 API endpoint.

Exam tips & gotchas

  • NMT and LLM translation have different element-count and character limits — exceeding LLM limits causes errors not encountered with NMT.
  • Transliterate ≠ Translate: transliterate changes the script but keeps the source language; translate changes the language.
  • LLM translation is billed per token, not per character — can be significantly more expensive at high volume.
  • The 2026-06-06 API version is required to select an LLM per request; earlier API versions support NMT only.

Practice

Q1. A legal firm needs to translate contracts requiring precise tone and gender agreement. They have an existing standalone Translator resource. What must they provision to enable LLM-powered translation?
Answer: A Foundry resource (not a standalone Translator resource), and they must call API version 2026-06-06. Why: LLM translation is only available via the Foundry resource; existing standalone Translator resources cannot route requests through an LLM.
Q2. A solution needs to translate 800 text strings in a single API call, each up to 40,000 characters. Which translation path is compatible?
Answer: NMT (standard Azure Translator). Why: NMT supports up to 1,000 elements and 50,000 characters per element per call; LLM translation is limited to 50 elements and 5,000 characters per element.

4.1.4 Customize language model outputs for domain tasks such as compliance summarization and domain extraction

Domain customization spans both classical Language Service features and generative model configuration. Custom NER allows labeling and training on entity types specific to a business domain—contract clause types, pharmaceutical drug names, financial instruments—that the prebuilt taxonomy does not cover. Custom Translator trains domain-adapted machine translation models using parallel bilingual document pairs; a Category ID is assigned to the trained model and supplied in Translator API calls to route requests through it.

For conversation-based compliance workflows, Language Summarization supports domain-specific aspect configuration: recap, issue/resolution (call-center QA), chapter title, and narrative. These aspects are tailored to the structure of compliance monitoring, meeting notes, and audit-trail generation. The 2026-06-06 API also introduces adaptive custom translation for select language pairs (English, French, German, Spanish), enabling terminology fine-tuning without full model retraining.

Generative model customization uses Pydantic-based Structured Outputs to enforce domain-specific JSON shapes from an LLM. Declaring fields such as clause_type, risk_level, and action_required in a BaseModel ensures schema-validated responses for downstream pipeline reliability. The SDK's analyze_actions method batches NER, summarization, key phrase extraction, and PII redaction into a single asynchronous request—important for throughput-sensitive compliance workloads.

Key facts

  • Custom NER requires labeled training data; annotate in Language Studio or via REST API before training.
  • Custom Translator: assign a Category ID to the trained model; pass it in Translator API calls to route through the domain-adapted model.
  • Adaptive custom translation (2026-06-06 API) tunes terminology for select language pairs without full retraining.
  • Conversation summarization aspects: recap, issue/resolution, chapter title, narrative.
  • Document-based PII and summarization accept .pdf and .docx natively, eliminating preprocessing for document-heavy workflows.
  • begin_analyze_actions combines multiple Language tasks (NER, summarization, key phrases, PII) in a single async batch.
from azure.ai.textanalytics import (
    TextAnalyticsClient,
    ExtractSummaryAction,
    RecognizeEntitiesAction,
)
from azure.core.credentials import AzureKeyCredential

client = TextAnalyticsClient(
    "https://<resource>.cognitiveservices.azure.com/",
    AzureKeyCredential("<key>")
)

docs = ["This service agreement shall remain in effect for thirty-six months from execution..."]

poller = client.begin_analyze_actions(
    docs,
    actions=[
        ExtractSummaryAction(max_sentence_count=2),
        RecognizeEntitiesAction(),
    ],
)
for result_page in poller.result():
    for action_result in result_page:
        print(action_result)

Exam tips & gotchas

  • Custom NER vs. prebuilt NER: prebuilt covers general entity types with no training; Custom NER requires labeled data but covers any domain entity type.
  • Custom Translator requires the Category ID in each API call — without it, requests fall back to the base NMT model silently.
  • Adaptive custom translation covers only select language pairs—verify pair support before designing routing logic.
  • begin_analyze_actions is async; iterate over the poller's result pages to retrieve each action's output.

Practice

Q1. A compliance team wants to summarize call-center transcripts with structured output that separates the problem identified from the resolution applied. Which conversation summarization aspect should they configure?
Answer: The issue/resolution aspect. Why: This aspect is purpose-built for call-center quality assurance, generating summaries that explicitly separate problem identification from outcome/resolution.
Q2. A pharmaceutical company trains a Custom Translator model on drug approval documentation. How do they ensure live translation API requests use this model instead of the base NMT model?
Answer: Pass the Category ID of the trained custom model in the Translator API request. Why: Without the Category ID, requests are silently routed through the default base NMT model, bypassing domain-specific training.
Q3. A developer needs to run entity recognition, extractive summarization, and PII redaction on a set of legal documents in the fewest possible API calls. Which SDK approach enables this?
Answer: Use begin_analyze_actions with ExtractSummaryAction, RecognizeEntitiesAction, and RecognizePiiEntitiesAction in a single asynchronous batch. Why: analyze_actions combines multiple Language tasks in one call, reducing latency and API overhead for compliance batch workflows.

4.2 Implement speech solutions

4.2.1 Convert speech to text and text to speech for agentic interactions

The Azure AI Speech SDK provides a unified API surface for speech-to-text (STT), text-to-speech (TTS), speech translation, and related audio capabilities. It is available for C#, C++, Java, JavaScript, Python, Go, Objective-C, and Swift across Windows, Linux, macOS, Android, iOS, and browser environments, making it well-suited for embedding voice modalities into agent applications across diverse deployment targets.

Real-time STT is built from a SpeechRecognizer composed with a SpeechConfig (subscription key and region) and an AudioConfig (microphone or audio file). recognize_once_async() handles single utterances; continuous recognition with event callbacks handles streaming agent interactions. For TTS, a SpeechSynthesizer configured with a voice name and language renders agent text responses to audio output using speak_text_async().

For batch transcription of large audio archives, the Speech REST API is the correct interface—not the SDK. When LLM-enhanced transcription accuracy is required, the Speech Transcription SDK (azure-ai-speech-transcription package) provides an alternative high-accuracy path. Real-time and file-based scenarios share the same SDK API surface; only AudioConfig differs between them.

Key facts

  • Speech SDK languages: C#, C++, Java, JavaScript, Python, Go, Objective-C, Swift.
  • Real-time STT: SpeechRecognizer + recognize_once_async() for single utterances; event-based continuous recognition for streaming.
  • TTS: SpeechSynthesizer + speak_text_async(); voice configured via speech_config.speech_synthesis_voice_name.
  • Batch transcription and custom model management use the Speech REST API, not the SDK.
  • LLM-enhanced transcription uses the separate Speech Transcription SDK (azure-ai-speech-transcription).
  • Only AudioConfig differs between microphone and file-based scenarios; SpeechConfig is identical.
import azure.cognitiveservices.speech as speechsdk

speech_config = speechsdk.SpeechConfig(subscription="<key>", region="<region>")
speech_config.speech_synthesis_voice_name = "en-US-JennyNeural"

# TTS: render agent's text response as speech
synthesizer = speechsdk.SpeechSynthesizer(speech_config=speech_config)
tts_result = synthesizer.speak_text_async("The package will arrive by Friday.").get()

# STT: capture user's spoken query
audio_config = speechsdk.AudioConfig(use_default_microphone=True)
recognizer = speechsdk.SpeechRecognizer(
    speech_config=speech_config, audio_config=audio_config
)
stt_result = recognizer.recognize_once_async().get()
print("User said:", stt_result.text)

Exam tips & gotchas

  • Batch transcription uses the Speech REST API, not the SDK — a common source of confusion on the exam.
  • The AudioConfig is the only component that changes between microphone and file inputs; SpeechConfig remains identical.
  • Voice name is set on SpeechConfig (speech_synthesis_voice_name), not on the SpeechSynthesizer object directly.
  • LLM-enhanced transcription requires the separate azure-ai-speech-transcription package, not the standard SDK package.

Practice

Q1. An agent application must transcribe 15,000 stored audio files overnight in a cost-efficient manner. Which Azure Speech interface should it use?
Answer: The Speech REST API (batch transcription endpoint). Why: Batch transcription of large audio archives is designed for the REST API, not the real-time Speech SDK, which targets streaming or single-utterance scenarios.
Q2. A developer sets speech_config.speech_synthesis_voice_name = "en-US-AriaNeural". What additional object do they need to create, and which method renders the text as speech?
Answer: A SpeechSynthesizer object, instantiated with the SpeechConfig. Call speak_text_async("…").get() to perform synthesis. Why: SpeechConfig carries configuration; the synthesizer holds the synthesis logic and exposes the async method.

4.2.2 Integrate speech as an agent modality, including custom speech models

Custom Speech improves recognition accuracy beyond the universal base model by training on domain-specific vocabulary and audio conditions. Training inputs include text transcripts, audio files paired with reference transcriptions, structured text for pattern-based recognition, and custom pronunciation files. This is especially valuable in agents deployed in specialized domains—legal, medical, technical—where base models produce high error rates on niche terminology.

The Custom Speech workflow follows five sequential steps: create a Speech Studio project (or use the API), upload training and test data, train the custom model, evaluate accuracy using Word Error Rate (WER), then deploy the approved model to a custom endpoint. Real-time STT with a custom model requires a dedicated custom endpoint; batch transcription can reference a custom model directly without hosting a persistent endpoint, reducing ongoing costs for bulk offline processing.

Custom endpoints remain pinned to the model version they were trained on, providing stable, predictable recognition quality even as Microsoft releases new base model versions. When a domain spans distinct sub-areas with different vocabularies, separate per-sub-area custom models outperform a single generalized custom model. Integration is seamless: set speech_config.endpoint_id to the custom endpoint URL; no other SDK changes are required.

Key facts

  • Custom Speech training data types: text transcripts, audio + reference transcriptions, structured text, custom pronunciations.
  • Accuracy is measured by Word Error Rate (WER) — lower WER indicates better recognition quality.
  • Real-time STT with a custom model requires a custom endpoint; batch transcription does not.
  • Custom endpoints are version-pinned — recognition quality remains consistent even when Microsoft releases newer base models.
  • Training cost applies if the base model was created on or after October 1, 2023; endpoint hosting cost applies regardless of base model date.
  • Use speech_config.endpoint_id = "<custom-endpoint-id>" to route SDK calls to the custom model.
  • Multiple domain-specific models outperform a single generalized model when vocabularies between sub-areas are distinctly different.
import azure.cognitiveservices.speech as speechsdk

speech_config = speechsdk.SpeechConfig(subscription="<key>", region="<region>")
# Route SDK calls to the trained custom speech endpoint
speech_config.endpoint_id = "<custom-endpoint-id>"

audio_config = speechsdk.AudioConfig(filename="legal_deposition.wav")
recognizer = speechsdk.SpeechRecognizer(
    speech_config=speech_config,
    audio_config=audio_config,
)
result = recognizer.recognize_once_async().get()
print("Transcript:", result.text)
Tip: Evaluate WER on held-out audio that is representative of actual production conditions, not just training data. A low WER on training audio but high WER in production typically indicates a distribution mismatch—collect more diverse samples before retraining.

Exam tips & gotchas

  • A custom endpoint is required for real-time STT with a custom model — this is not optional.
  • Batch transcription can use a custom model without a dedicated hosted endpoint — use this to avoid hosting charges for offline workloads.
  • Endpoint hosting costs apply regardless of base model date; only the training cost varies based on the October 1, 2023 cutoff.
  • Only endpoint_id needs to change in the SDK to switch from the base model to a custom model.

Practice

Q1. A hospital voice agent uses Azure Speech for live physician dictation. The standard model frequently misidentifies drug names. What is the recommended solution?
Answer: Train a Custom Speech model using medical transcripts and custom pronunciation files, deploy it to a custom endpoint, and set speech_config.endpoint_id to point to it. Why: The universal base model does not cover specialized medical terminology; Custom Speech training on domain data reduces WER on niche vocabulary.
Q2. A team needs to run overnight batch transcription using a custom speech model while minimizing costs. Do they need to host a custom endpoint?
Answer: No. Batch transcription can reference a custom model directly without a dedicated hosted endpoint. Why: Only real-time STT requires a hosted custom endpoint; batch transcription references the model ID directly, avoiding ongoing endpoint hosting charges.

4.2.3 Enable multimodal reasoning from audio inputs

Audio-capable Azure OpenAI models — gpt-4o-audio-preview and gpt-4o-mini-audio-preview — accept audio as a content part inside a standard chat completions message, alongside text and images, enabling the model to reason directly over spoken content without a separate STT preprocessing step. Audio is base64-encoded and submitted as an input_audio content type alongside a user text instruction.

For low-latency conversational agents, the GPT Realtime API provides a persistent WebSocket session supporting "speech in, speech out" interactions. The openai Python library exposes a RealtimeClient helper for managing audio delta streams and model response events. Sessions can include an ISO-639-1 language hint (for example, "en") to improve transcription accuracy for a known input language. The gpt-realtime-translate Foundry model targets real-time speech translation over this same WebSocket channel.

An important constraint: Structured Outputs (schema-enforced JSON) are not supported with gpt-4o-audio-preview or gpt-4o-mini-audio-preview. Pipelines that need schema-validated JSON output must transcribe audio to text first, then apply Structured Outputs in a subsequent text-model call. LLM Speech (Speech Transcription SDK) provides an alternative path supporting approximately 25 input languages in multi-lingual mode by default.

Key facts

  • gpt-4o-audio-preview and gpt-4o-mini-audio-preview accept input_audio content parts in chat messages—no separate STT step needed.
  • GPT Realtime API uses a persistent WebSocket for low-latency "speech in, speech out" conversational interactions.
  • gpt-realtime-translate: Global Standard (pay-as-you-go) deployment; billed hourly under Audio Models pricing.
  • Structured Outputs are NOT supported with gpt-4o-audio-preview / gpt-4o-mini-audio-preview (version 2024-12-17).
  • Language hints in Realtime API sessions use ISO-639-1 codes (e.g., "en"); optional but improve accuracy for known locales.
  • LLM Speech (Speech Transcription SDK) supports ~25 languages, multi-lingual mode by default — no locale specification required.
import base64
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint="https://<resource>.openai.azure.com/",
    api_key="<key>",
    api_version="2025-01-01-preview",
)

with open("meeting_clip.wav", "rb") as f:
    audio_b64 = base64.b64encode(f.read()).decode()

response = client.chat.completions.create(
    model="gpt-4o-audio-preview",
    messages=[{
        "role": "user",
        "content": [
            {"type": "text", "text": "List any action items mentioned in this recording."},
            {"type": "input_audio", "input_audio": {"data": audio_b64, "format": "wav"}},
        ],
    }],
)
print(response.choices[0].message.content)
Gotcha: Structured Outputs (strict: true JSON schema enforcement) are not supported with audio-preview models. If your pipeline requires schema-validated JSON, transcribe the audio to text first, then call a text-capable model with Structured Outputs in a second request.

Exam tips & gotchas

  • Audio-preview models accept audio directly in chat—no STT preprocessing required—but no Structured Outputs either.
  • gpt-realtime-translate is billed hourly, not per token or character—factor this into high-volume cost planning.
  • LLM Speech operates in multi-lingual mode by default; no input locale needs to be specified, unlike classic STT.
  • The Realtime API is WebSocket-based and stateful per session—not compatible with stateless HTTP REST patterns.

Practice

Q1. A developer submits an audio clip to gpt-4o-audio-preview and needs the response as a JSON object with specific fields for downstream processing. What must they do?
Answer: Transcribe the audio to text first (using STT or a non-audio model path), then submit the text to a supported model (e.g., gpt-4o) with response_format = json_schema and strict: true. Why: Structured Outputs are not supported with gpt-4o-audio-preview; schema enforcement requires a text-based model.
Q2. Which deployment type and billing model applies to gpt-realtime-translate?
Answer: Global Standard (pay-as-you-go) deployment, billed hourly under the Audio Models pricing tier. Why: This model is specialized for real-time speech translation and has a distinct hourly pricing structure separate from token-based model billing.
Q3. What is the key advantage of LLM Speech (Speech Transcription SDK) over classic Azure Speech STT for multilingual input scenarios?
Answer: LLM Speech operates in multi-lingual mode by default across approximately 25 languages without requiring a pre-specified input locale, whereas classic STT requires you to declare the recognition language upfront. Why: This simplifies multilingual agent pipelines where the input language may vary between speakers or turns.

4.2.4 Translate speech into other languages using language models and Foundry Tools

Azure AI Speech Translation provides real-time speech-to-text and speech-to-speech translation via the Speech SDK or CLI. The standard Speech Translation API supports up to two target languages in a single call; producing output in more than two languages requires a multi-service Foundry resource, with Azure Translator pricing applied for each additional target language beyond the second. Outputs include STT translation, speech-to-speech synthesis, and multi-lingual conversation translation.

Live Interpreter continuously identifies the spoken input language without requiring a pre-configured locale setting, then synthesizes translated output in a natural voice that preserves the speaker's style and tone. This makes it well-suited for international conferences, Teams meetings, contact centers, and educational events where the input language is dynamic. Domain-specific translation accuracy can be improved by supplying a Custom Translator Category ID in the SpeechTranslationConfig, routing translated output through a domain-adapted MT model while the STT step remains unchanged.

For LLM-powered speech translation, gpt-realtime-translate is a dedicated Foundry model delivering multilingual speech-to-speech translation over a persistent WebSocket session. The Speech Transcription SDK (LLM Speech) supports approximately 25 input languages for both transcription and translation tasks in multi-lingual mode by default, with optional ISO-639-1 locale hints available for specific languages.

Key facts

  • Standard Speech Translation API: up to 2 target languages per call; more requires a multi-service Foundry resource.
  • Additional languages beyond the second incur Azure Translator character-based pricing.
  • Live Interpreter: automatic language identification (no pre-set locale), speech-to-speech synthesis with speaker style preservation.
  • Custom Translator integration: supply Category ID in SpeechTranslationConfig to apply domain-adapted MT to translated output.
  • gpt-realtime-translate: Foundry model, Global Standard deployment, billed hourly.
  • LLM Speech: ~25 input languages, multi-lingual mode by default, optional ISO-639-1 locale hints.
  • Total billing = STT transcription cost + Translator character fees per additional target language beyond the second.
import azure.cognitiveservices.speech as speechsdk

translation_config = speechsdk.translation.SpeechTranslationConfig(
    subscription="<key>", region="<region>"
)
translation_config.speech_recognition_language = "en-US"
translation_config.add_target_language("es")  # target language 1
translation_config.add_target_language("ja")  # target language 2

audio_config = speechsdk.AudioConfig(use_default_microphone=True)
recognizer = speechsdk.translation.TranslationRecognizer(
    translation_config=translation_config,
    audio_config=audio_config,
)

def on_recognized(evt):
    for lang, text in evt.result.translations.items():
        print(f"[{lang}] {text}")

recognizer.recognized.connect(on_recognized)
recognizer.start_continuous_recognition()
input("Press Enter to stop...\n")
recognizer.stop_continuous_recognition()
Key: When more than two simultaneous target languages are required, provision a multi-service Foundry resource before making the API call. Translator character-based pricing applies per additional language—estimate character volume carefully for high-throughput scenarios.

Exam tips & gotchas

  • The 2-language limit is a per-request limit, not an account limit — you cannot circumvent it without a multi-service Foundry resource.
  • Live Interpreter does not require a pre-set input locale — automatic language detection is its defining capability.
  • Custom Translator Category ID goes on SpeechTranslationConfig, not on SpeechConfig.
  • gpt-realtime-translate is billed hourly — significantly different from character-based NMT pricing at high audio volume.

Practice

Q1. A conference organizer needs real-time speech translation into five simultaneous target languages. They have a standalone Azure Speech resource. What must they provision to support this?
Answer: A multi-service Foundry resource. Why: The standard Speech Translation API supports only two target languages per call; exceeding this limit requires a multi-service Foundry resource, with Translator character pricing applied for each additional language.
Q2. A developer wants to apply domain-adapted medical terminology to the speech translation output without changing the STT step. How should they configure this?
Answer: Train a Custom Translator model on medical parallel corpora to obtain a Category ID, then supply that Category ID in the SpeechTranslationConfig. Why: This routes only the translation component through the domain-adapted model; the STT step is unaffected, improving medical terminology accuracy in translated output.
Q3. Which speech translation feature automatically identifies the input language without a pre-configured locale, and in which scenarios is it most appropriate?
Answer: Live Interpreter. It is most appropriate for multilingual conferences, international Teams meetings, contact centers, and educational events where speakers switch languages dynamically. Why: Live Interpreter continuously detects the spoken language and synthesizes translated output, eliminating the need to pre-declare a recognition language.
