1.1 Choose the appropriate Foundry services for generative AI and agents

1.1.1 Choose an appropriate model for each task

The Azure AI Foundry model catalog aggregates more than 1,900 models from Microsoft, OpenAI, Anthropic, Meta, Mistral, DeepSeek, xAI, and the broader Hugging Face community. Models fall into two top-level catalog categories: Foundry Models sold by Azure (hosted and billed by Azure, covered by Microsoft SLAs, using fungible PTU quota) and Foundry Models from partners and community (governed by third-party terms, available via pay-as-you-go serverless or managed compute).

Selecting the right model begins with understanding the nature of the task — its complexity, latency requirements, context window needs, and whether the input is text-only or multimodal. Large language models (LLMs) handle complex, multi-step reasoning; small language models (SLMs) suit resource-constrained or edge scenarios; embedding models convert text to vectors for retrieval; and multimodal models process both text and images in a single inference call.

The Model Router deployment is a special option that dispatches each incoming prompt at runtime to the most cost-effective model capable of handling it (selecting from GPT-4.1, GPT-4.1-mini, GPT-4.1-nano, or o4-mini), exposed as a single Chat Completions API endpoint. This reduces manual tuning of model selection logic when query complexity varies widely across users.

Key facts

  • GPT-5 / gpt-5.6-sol/terra/luna: highest-capability models; support multimodal input (text + image); up to 1 M-token context window.
  • GPT-4.1 family (gpt-4.1, gpt-4.1-mini, gpt-4.1-nano): production-grade; mini/nano variants optimised for low-latency, high-throughput workloads.
  • o-series reasoning models (o4-mini, o3, o1): extended thinking for complex problem-solving; higher latency but deeper chain-of-thought.
  • Claude (Anthropic): strong for advanced reasoning, code generation, and multimodal tasks.
  • DeepSeek-R1: open-weight reasoning model at scale.
  • Mistral families: multilingual text, code generation, general-purpose chat.
  • SLMs — Phi-4: Microsoft SLM with strong reasoning at an efficient footprint; Phi-4 Mini Instruct for even smaller deployments.
  • SLMs — Meta Llama 1B/3B: on-device and edge inferencing.
  • Multimodal — GPT-4o: text + image input by design.
  • Multimodal — Llama 4 Scout / Maverick: mixture-of-experts (MoE) architecture; 128 K-token context; text + image.
  • Audio — Whisper: speech-to-text; GPT-4o audio models support speech-in/speech-out.
  • Embedding models: text-embedding-3-small, text-embedding-3-large, text-embedding-ada-002 — convert text to dense vectors for similarity search and RAG.
  • Industry / domain models (NVIDIA NIMs, Bayer, Rockwell, Cerence, etc.) require managed compute on AI Hub resources.
  • Model Router: single deployment endpoint that routes to GPT-4.1 / mini / nano / o4-mini based on query complexity; reduces cost without sacrificing quality on complex queries.

from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient

project = AIProjectClient(
    endpoint="https://<resource>.ai.azure.com/api/projects/<project>",

openai_client = project.get_openai_client()

# Phi-4 is well-suited for summarisation tasks with efficient compute
response = openai_client.chat.completions.create(
    model="phi-4",
    messages=[{"role": "user", "content": "Summarise this article in three bullet points."}],
    max_tokens=300,

print(response.choices[0].message.content)

Exam tips & gotchas

  • SLMs (Phi-4, Llama 1B/3B) are the right choice for edge, on-device, or resource-constrained scenarios — not for complex reasoning tasks.
  • The o-series models use extended thinking/chain-of-thought; they have higher latency than standard chat models and are not optimal for low-latency streaming.
  • Model Router is a single deployment endpoint — the underlying model is selected automatically, not by the caller.
  • Partner/community models (serverless API) operate under third-party terms and do not support custom content filters or keyless authentication.
  • Embedding models are distinct from completion/chat models — they output vectors, not text, and are selected based on dimension requirements and the downstream vector store.

Answer: A Model Router deployment. Why: Model Router dispatches each prompt to the best-fit model (GPT-4.1, mini, nano, or o4-mini) based on complexity at runtime, exposed as a single Chat Completions endpoint — no client-side routing logic is needed.

1.1.2 Choose the appropriate Foundry services for generative tasks, grounding, vector search, agent workflows, or multimodal processing

Microsoft Foundry is Azure's unified PaaS platform for AI. It consolidates model inference, agent hosting, evaluation, fine-tuning, and observability under a single resource hierarchy rooted in a Microsoft.CognitiveServices/accounts resource of kind AIServices. Rather than assembling separate AI services manually, teams compose purpose-built Foundry capabilities for each layer of their solution.

Choosing the right Foundry service begins with the question: what is this component doing? Generating text responses calls for Azure OpenAI in Foundry Models; grounding responses in enterprise data calls for Azure AI Search or Foundry IQ; orchestrating multi-step tasks calls for the Foundry Agent Service; processing images alongside text calls for multimodal models. Each service has a specific role and is accessed through the same Foundry project.

Foundry Tools (the former Azure AI Services umbrella) encompasses Speech, Vision, Language, Translator, and Azure AI Content Understanding — all accessible within the same Foundry resource, allowing a single project to combine generative and classic AI capabilities without managing separate resource keys.

Key facts

  • Generative text / chat: Azure OpenAI in Foundry Models — Chat Completions API or Responses API.
  • RAG grounding: Azure AI Search (retrieval) feeding context into an LLM completion; Foundry IQ (managed knowledge layer) for permission-aware enterprise knowledge bases.
  • Vector search: Azure AI Search — supports full-text (BM25), vector, hybrid, and multimodal queries; natively underpins Foundry IQ and the Agent File Search tool.
  • Agent workflows: Foundry Agent Service (GA May 2025) — Responses API with Conversations, Items, and Runs primitives.
  • Multimodal (image + text): GPT-4o, GPT-4.1, Llama 4 Scout/Maverick; audio: Whisper or GPT-4o audio.
  • Content moderation: Azure AI Content Safety (part of Foundry Tools) — real-time harm classification for text and images; integrates as guardrails on model deployments.
  • Agent grounding tools: File Search (RAG over Azure AI Search / Blob Storage), Grounding with Bing Search (live web), Grounding with Bing Custom Search (scoped web), SharePoint tool, Fabric Data Agent (structured data Q&A).
  • Multi-agent orchestration: Connected agents (GA May 2025) — task-specific sub-agents interacting with a primary agent via A2A endpoints (preview).
  • Foundry Tools: Speech, Vision, Language, Translator, Content Understanding — all available within the same Foundry resource.
Key: For live web grounding use Grounding with Bing Search; for internal SharePoint documents use the SharePoint tool; for structured Fabric data use the Fabric Data Agent — they are not interchangeable.

Exam tips & gotchas

  • Foundry IQ provides a managed, permission-aware knowledge layer — choose it over DIY RAG when you need automatic access-control inheritance.
  • Azure AI Search is the underlying retrieval engine for both Foundry IQ and the Agent File Search tool.
  • The Foundry Agent Service is GA; Connected agents (A2A endpoints) are in preview — know the difference in scenarios requiring multi-agent coordination.
  • Audio processing (speech-to-text) uses Whisper or GPT-4o audio — these are distinct from standard text multimodal models.
  • Foundry Tools (Azure AI Services) can be accessed within the same Foundry resource — no separate connection or key required when using managed identity.

1.1.3 Choose an appropriate method for retrieval and indexing

Azure AI Search offers two retrieval modes that address different agent patterns. Classic search is index-first: queries run against a single index with predictable, low-latency results. It supports full-text BM25 ranking, vector similarity, hybrid combinations, and semantic re-ranking. Agentic retrieval uses a knowledge base abstraction backed by an LLM query planner that decomposes complex questions into sub-queries, runs them in parallel across multiple knowledge sources, and returns a synthesised answer with references — suited for agents that answer open-ended, multi-document questions.

Integrated vectorization automates the chunking and embedding steps inside the indexing pipeline, eliminating the need for a separate preprocessing service. It works through a combination of an indexer (pulls data from source), a skillset (splits text and calls an embedding model), and a vector field in the index schema. At query time, a configured vectorizer converts the incoming text query to a vector automatically — using the same embedding model as the skillset.

Choosing the right retrieval method depends on the desired balance of recall and precision, latency requirements, and whether the downstream consumer is a classic application or an agentic system. Hybrid search (BM25 + vector, fused via Reciprocal Rank Fusion) with optional semantic re-ranking delivers the best combined recall and precision for most RAG use cases.

Key facts

  • Hybrid search: BM25 + vector scores fused via Reciprocal Rank Fusion (RRF); provides better recall + precision than either alone.
  • Semantic ranking: cross-encoder re-ranker scores top 50 BM25 candidates by intent relevance (0–4 scale, @search.rerankerScore); adds caption and answer extraction.
  • Integrated vectorization components: indexer (data source pull) → skillset (Text Split + AzureOpenAIEmbedding or AML skill) → vector field in index → query-time vectorizer.
  • Supported embedding skills: AzureOpenAIEmbeddingSkill (text-embedding-ada-002, text-embedding-3-small, text-embedding-3-large), AML skill (Foundry catalog models), Azure Vision multimodal embeddings skill.
  • Vectorizer must match embedding model: the vectorizer applied at query time must use the same embedding model as the indexing skillset — a mismatch silently degrades cosine similarity scores.
  • Agentic retrieval / knowledge base: abstracts multiple knowledge sources; LLM query planner decomposes queries into sub-queries and synthesises results; referenced by agents via the knowledge base, not by direct index queries.
  • Supported data sources for indexers: Azure Blob Storage, Azure Data Lake Storage Gen2, Azure SQL, Azure Cosmos DB, SharePoint Online, Microsoft Fabric OneLake.
  • Pricing: dedicated (Search Units/hr) or serverless (CU/hr + per-GB storage; preview).

Exam tips & gotchas

  • The vectorizer at query time must use the same embedding model as the skillset used during indexing — this is a common exam scenario.
  • Semantic ranking re-ranks only the top 50 BM25 candidates — it does not scan the full index.
  • Agentic retrieval uses a knowledge base, not a direct index query — the difference matters for agent design.
  • Integrated vectorization requires an indexer; it cannot be used with the push API (direct document upload to the index).
  • Reciprocal Rank Fusion (RRF) is the fusion algorithm for hybrid search — scores are not simply added.

Practice

Q1. After deploying a RAG solution, users report that keyword searches return accurate results but semantic/intent-based queries perform poorly. What is the most likely cause?
Answer: The query-time vectorizer is misconfigured or does not match the embedding model used during indexing. Why: Cosine similarity between a query vector and document vectors only produces meaningful scores when both are generated by the same embedding model with the same dimension settings.
Q2. A team wants the best possible recall for a RAG retrieval step without manually selecting keyword or vector mode. What configuration achieves this?
Answer: Hybrid search (BM25 + vector with RRF fusion) plus semantic ranking. Why: Hybrid RRF combines the lexical strengths of BM25 with semantic vector matching, then semantic ranking re-scores the top 50 results by intent relevance — providing the highest overall recall and precision.
Q3. An agent needs to answer multi-document questions across several knowledge sources with automatic query decomposition. Which Azure AI Search capability should be used?
Answer: Agentic retrieval backed by a knowledge base. Why: Agentic retrieval uses an LLM query planner to decompose complex questions into sub-queries, run them across multiple sources in parallel, and return a synthesised result — classic single-index queries cannot do this automatically.

1.1.4 Choose appropriate memory, tool, and knowledge integration services for agent solutions

The Foundry Agent Service (GA May 2025) is the managed runtime for AI agents. It exposes three core primitives via the Responses API: Conversations (persistent message threads), Items (individual messages or structured objects within a conversation), and Runs (execution cycles where the agent reads context, invokes tools, and produces a response). Each Run generates Run Steps that record every individual tool call or message produced during that cycle.

Agents have access to two memory scopes. Short-term memory is the in-thread conversation history — automatically maintained within the active session and truncated to fit the model's context window. Long-term memory is provided by the Foundry Agent Service Memory Store (preview), a persistent managed store that extracts and consolidates knowledge across sessions so agents can recall user preferences and prior facts on future runs.

Tools available to agents are divided into knowledge tools (retrieve information) and action tools (perform operations). The right combination depends on the agent's purpose. Knowledge tools are grounded in specific data sources — choosing the wrong tool means the agent cannot access its required information. Action tools like Azure Logic Apps triggers and Code Interpreter enable the agent to act on the world.

Key facts

  • Foundry Agent Service primitives: Conversation/Thread (persists message history), Run (one execution cycle), Run Steps (per-tool-call detail), Agent Version (versioned definition: model, instructions, tools).
  • Short-term memory: conversation context within the current session; automatic truncation to fit model context window.
  • Long-term memory: Memory Store (preview) — persists knowledge across sessions (user preferences, history), queried automatically at the start of each run.
  • BYO thread storage: Standard Agent Setup supports Azure Cosmos DB for NoSQL as bring-your-own thread store — all message data stays in customer resources.
  • Knowledge tools: File Search (RAG over Azure AI Search, Blob Storage, local files), Grounding with Bing Search (live web), Grounding with Bing Custom Search (scoped web), SharePoint (internal docs), Fabric Data Agent (structured Fabric data), Tripadvisor / Morningstar (licensed third-party).
  • Action tools: Code Interpreter, custom function tools, Azure Logic Apps triggers, 1,400+ tools via the tool catalog.
  • Tool-level override: tools can be attached at agent, thread, or run level. Run-level tools override thread-level, which override agent-level.
  • One instance per knowledge tool type per run: e.g., only one File Search and one Bing Grounding tool per run. Use connected agents for multiple index sources.
  • Connected agents (GA May 2025): task-specific sub-agents linked to a primary agent — enables multi-agent patterns without an external orchestrator.

        "tools": [

Exam tips & gotchas

  • In-thread memory (short-term) is lost when the session ends; the Memory Store (long-term) persists across sessions — pick based on whether inter-session recall is needed.
  • Basic Agent Setup uses Microsoft-managed storage; Standard Agent Setup uses customer-provided Cosmos DB and Storage — critical for compliance scenarios.
  • Tool resolution priority: run-level > thread-level > agent-level — know this for conflict scenarios.
  • Connected agents (GA) vs. multi-agent with external orchestrator (preview A2A endpoints) — exam may ask which is GA.
  • The Fabric Data Agent is the correct tool for structured Q&A over Microsoft Fabric data; it is not the same as File Search or Bing Grounding.

Practice

Q1. An agent must remember a user's dietary preferences across multiple separate chat sessions. Which memory capability addresses this requirement?
Answer: Foundry Agent Service Memory Store (long-term memory, preview). Why: In-thread short-term memory is cleared when the session ends. The Memory Store persists extracted knowledge (like user preferences) across sessions and is queried automatically at the start of each new run.
Q2. A single agent run needs to search two separate Azure AI Search indexes. How should the solution be designed?
Answer: Use connected agents — one primary agent plus two sub-agents, each configured with their own File Search tool pointed at a different index. Why: Only one File Search tool instance is allowed per run. Connected agents allow each sub-agent to hold a separate File Search tool, and the primary agent delegates to them.
Q3. A regulatory requirement mandates that all agent conversation history must reside in the customer's own Azure subscription. Which agent setup satisfies this?
Answer: Standard Agent Setup, configured with a customer-owned Azure Cosmos DB for NoSQL as the thread store. Why: Standard Agent Setup provisions all conversation thread data in customer-controlled resources; Basic Agent Setup uses Microsoft-managed multitenant storage.

1.2 Set up AI solutions in Foundry

1.2.1 Design Azure infrastructure for AI apps and agent-based solutions

The top-level Azure resource for Foundry solutions is a Microsoft.CognitiveServices/accounts resource with kind AIServices. This resource hosts model deployments, connections to dependent services, security settings, and one or more Projects. Projects (Microsoft.CognitiveServices/accounts/projects) are scoped development boundaries — teams build agents, run evaluations, and store artefacts within a project, reusing the parent Foundry resource's deployments and connections.

Several independent Azure services are linked as connected resources: Azure Storage (artefact and file storage), Azure Key Vault (secrets for API-key-based connections), Azure AI Search (retrieval), and Azure Application Insights (observability). Each connected resource maintains its own governance boundary and access controls. Foundry's managed identity is granted roles on these resources to avoid credential management.

For the Foundry Agent Service, infrastructure design also determines whether agent thread data lands in Microsoft-managed storage (Basic Agent Setup, fastest to start) or in customer-owned Cosmos DB and Storage (Standard Agent Setup, required for data isolation and compliance). Network topology choices — customer-managed VNet vs. platform-managed (Managed VNet) — affect connectivity to connected resources and agent compute.

Key facts

  • Resource hierarchy: Foundry resource → Projects → Project assets (agents, evaluations, files, connections).
  • Connected resources: Azure Storage, Azure Key Vault (one covers all project and resource-level connection secrets), Azure AI Search, Azure Application Insights.
  • Hub-based projects (classic): AI Hub (Microsoft.MachineLearningServices/workspaces) with AI projects — still supported; required for serverless API and managed compute deployments.
  • Basic vs Standard Agent Setup: Basic = Microsoft-managed multitenant storage (fastest start); Standard = customer-owned Cosmos DB + Storage + AI Search (full data isolation).
  • RBAC scoping: assignments at resource level apply to all projects; assignments at project level are scoped to that project only. Common pattern: assign Foundry User to developers at resource scope.
  • Networking options: Customer-managed VNet (BYO subnet, delegated to Microsoft.App/environments) or Managed VNet (platform-managed). Private endpoints block all public access — portal management then requires SDK/CLI.
  • Encryption: default Microsoft-managed keys (FIPS 140-2, AES-256). CMK (customer-managed keys) requires Key Vault in the same region with soft delete + purge protection, and managed identity with Key Vault Crypto User role.

  • One Key Vault covers secrets for all projects in the Foundry resource — it is not per-project.
  • CMK requires Key Vault in the same region as the Foundry resource — cross-region CMK is not supported.
  • When public access is disabled (private endpoint only), portal management for some settings is unavailable — use CLI or SDK.
  • Hub-based projects (classic) are still needed for managed compute and serverless API deployments; the new Foundry resource model does not yet support those.
  • Standard Agent Setup must be provisioned at creation time — you cannot migrate a Basic setup to Standard after deployment.

Q1. A GDPR-governed application requires that all agent conversation threads remain exclusively within the customer's Azure subscription. Which infrastructure choice is required?
Answer: Standard Agent Setup, with customer-owned Azure Cosmos DB for NoSQL as the thread store and customer-owned Azure Storage. Answer: Assign the Foundry User role at the individual project scope (not the resource scope). Why: Role assignments at project scope restrict data-plane access to that project only, preventing the user from interacting with agents or resources in sibling projects.

1.2.2 Choose appropriate deployment options

Foundry offers three families of deployment for Azure OpenAI models: pay-per-token deployments (Global Standard, Data Zone Standard, Regional Standard), provisioned deployments (Global Provisioned, Data Zone Provisioned, Regional Provisioned — all PTU-based), and Global Batch for async bulk processing. For partner and community models, Serverless API and Managed Compute are available via AI Hub (classic).

The primary trade-offs are cost predictability, throughput guarantees, and data residency. Pay-per-token deployments are flexible and suit variable traffic; provisioned deployments reserve compute capacity (billed hourly per PTU) and suit predictable high-volume production workloads. Data residency requirements determine whether to choose Global (Azure-managed cross-region routing), Data Zone (US or EU zone), or Regional (single-region) variants.

A key operational feature is spillover: when a provisioned deployment is fully utilised, excess requests can automatically overflow to a linked standard deployment in the same Foundry resource — preventing hard failures without manual intervention.

Key facts

  • Global Standard: pay-per-token; Azure-managed cross-region routing; best for dev/test and variable traffic.
  • Data Zone Standard / Provisioned: data stays within US or EU zone depending on configuration.
  • Regional Standard / Provisioned: strictest single-region data residency.
  • Global Provisioned (PTU): hourly billing per PTU; cross-region; high-scale production with predictable traffic.
  • Global Batch: discounted batch rate; async, no latency SLA; cross-region.
  • Serverless API (AI Hub): pay-per-token for partner models (Llama, Mistral, Cohere, etc.); no provisioned compute; regional only; does not support keyless auth or custom content filters.
  • Managed Compute (AI Hub): per-core-hour billing; required for Hugging Face, NVIDIA NIMs, industry models, custom models; regional only.
  • PTU quota vs. capacity: PTU quota is a policy limit (no cost); physical compute capacity must also be available at deployment time — having quota does not guarantee capacity.
  • PTU sizing: use the Foundry portal capacity calculator at https://ai.azure.com/resource/calculator.
  • Spillover: provisioned deployment overflow routes to a linked standard deployment; configured per deployment or via x-ms-spillover-deployment request header.
  • Azure Reservations: 1-month or 1-year commitment discounts the hourly PTU meter.

Exam tips & gotchas

  • PTU quota is a policy limit (no charge); PTU capacity is the physical compute — having quota does not guarantee a deployment will succeed.
  • Global Batch is the cheapest option but has no latency SLA — only for offline, non-interactive workloads.
  • Only Standard/Foundry-resource deployments support custom content filtering and Entra ID keyless auth — not Serverless API.
  • Spillover prevents hard 429 errors for provisioned deployments — it is configured on the provisioned deployment, not the standard one.
  • Managed Compute requires AI Hub (classic) resources — it is not available in the new Foundry resource model.

Q2. An enterprise needs to deploy Llama 3 with pay-per-token pricing and no dedicated compute to manage. Which deployment option is correct?
Answer: Serverless API (via AI Hub). Why: Serverless API provides pay-as-you-go access to partner and community models like Llama without requiring provisioned compute or capacity reservations.
Q3. A provisioned deployment returns 429 errors during peak load. Which feature should be enabled to prevent user-visible failures?
Answer: Spillover, configured to overflow to a linked Standard deployment in the same Foundry resource. Why: Spillover automatically routes excess requests to a standard deployment when provisioned capacity is saturated, preventing hard 429 errors from reaching users.

1.2.3 Configure model and agent deployments

Model deployments are created in the Foundry portal under Deployments → Deploy model → Deploy base model, or programmatically via Azure CLI, REST API, or Bicep/AVM patterns.   • Model deployment parameters: model name + version, deployment name, SKU/type, TPM capacity, content filter name, auto-update policy.

  • Guardrail assignment: one named guardrail per model deployment; agent guardrail fully overrides model guardrail.
  • Hosted agents: container-based persistent HTTP endpoint; managed by Foundry platform; deployed via azd deploy or Foundry portal.
  • VS Code Extension: allows deploying and configuring agents directly from the IDE without switching to the portal.

1.3.1 Manage quotas, scaling, rate limits, and cost footprints for model and agent workloads

Tokens per Minute (TPM) is the primary quota unit for Azure OpenAI in Foundry Models. Quota is assigned per subscription, per region, per model, and per deployment type — Global Standard, Data Zone Standard, and Regional Standard each have separate quota pools. When a deployment is created, the assigned TPM directly sets the model's TPM rate limit and a proportional RPM (Requests per Minute) limit. The total TPM allocated across all deployments of the same model in a region cannot exceed the subscription's regional quota.

For provisioned deployments, Provisioned Throughput Units (PTUs) are the reservation unit. PTU quota is a policy limit (no charge); the physical compute capacity must also be available in the chosen region and deployment type at creation time. Having PTU quota is necessary but not sufficient — always verify capacity availability before committing to provisioned reservations, using the Foundry portal capacity calculator.

Cost management for AI workloads involves tracking pay-per-token consumption (via Microsoft Cost Management) and managing hourly PTU billing. The Model Router deployment and spillover configurations are operational mechanisms to control costs — Model Router routes simple queries to cheaper mini/nano variants, while spillover limits the blast radius of provisioned capacity exhaustion.

Key facts

  • TPM quota pools: separate per subscription, region, model, and deployment type (Global / Data Zone / Regional are independent pools).
  • RPM ratio examples: o4-mini: 1 RPM per 1,000 TPM; older chat models: 6 RPM per 1,000 TPM.
  • PTU quota vs. capacity: quota is a policy limit (no charge); capacity is the compute that must be available at deployment time — distinct concepts.
  • Three PTU quota pools: GlobalProvisionedManaged, DataZoneProvisionedManaged, ProvisionedManaged — each is independent.
  • Capacity calculator: Foundry portal at https://ai.azure.com/resource/calculator — required for PTU sizing.
  • Quota increase requests: submit at https://aka.ms/oai/stuquotarequest; quota increases for partner/community models are not supported.
  • Viewing quota: Foundry portal → Management → Quota. Cognitive Services Usages Reader role (subscription scope) provides minimum permissions to read quota across multiple subscriptions.
  • PTU billing: hourly per PTU regardless of token consumption; reduce cost via Azure Reservations (1-month or 1-year).
  • Spillover: overflow from saturated provisioned deployments routes to a linked standard deployment — prevents hard failures.
  • Model Router: routes simple queries to cheaper model variants at runtime — reduces overall token cost without client-side routing logic.
  • Max resources per region: 30 Azure OpenAI resources per subscription per region.
# View TPM quota usage for gpt-4 family in East US
az cognitiveservices usage list \
  --location eastus \
  --query "[?contains(name.value, 'gpt-4')]" \
  --output table

Exam tips & gotchas

  • PTU quota and PTU capacity are different — a common exam trap is assuming quota approval guarantees deployment success.
  • The Cognitive Services Usages Reader role must be assigned at subscription scope to see quota across all resources in the subscription.
  • Each deployment type (Global, Data Zone, Regional) has a separate quota pool — deploying more Global Provisioned does not reduce your Regional Provisioned quota.
  • PTU billing continues hourly even when no tokens are consumed — provisioned deployments are a reservation, not a consumption meter.
  • Model Router is a deployment-level feature — it requires no client code changes; the routing is transparent to callers.

Practice

Why: PTU quota is a policy limit (no compute charge); actual compute capacity must be available at deployment time, and the two can be out of sync — always verify capacity before purchasing reservations.
Q2. A central platform team needs to audit Azure OpenAI quota consumption across all subscriptions in an organisation without granting admin access. What minimum role is required?
Answer: Cognitive Services Usages Reader role, assigned at subscription scope for each subscription. Why: This built-in role provides read-only access to quota usage data without granting any management or data-plane permissions.

Scheduled evaluations (preview) run evaluation templates on a schedule to compare current performance against baselines. 1.4.1 Configure safety filters, guardrails, risk detection, and content moderation

In Azure AI Foundry, guardrails are named collections of controls that define a three-part policy: the risk category to detect, the point in the conversation flow to intercept (intervention point), and the action to take when the risk is detected. Guardrails are configured at the Foundry resource or project level and then assigned to model deployments or agents. All Foundry Models sold by Azure are assigned the Microsoft.DefaultV2 guardrail automatically if no custom guardrail is assigned.

There are four intervention points where scanning can occur: User Input (the prompt sent to the model or agent), Tool Call (preview — the action the agent proposes to send to a tool), Tool Response (preview — content returned from a tool to the agent), and Output (the final completion returned to the caller). The tool-level intervention points are exclusive to agents and enable scanning of the agent's planned actions before they execute.

Severity levels control how aggressively the filter fires: Off (approved customers only, requires Limited Access Review), Low (flags at low severity and above), Medium, and High (flags only the most severe content). When a risk is detected, the action can be Annotate (log only, available for models) or Annotate and block (returns an error to the caller). Azure AI Content Safety provides the underlying harm-classification models that power guardrail controls and Groundedness Pro evaluation.

Key facts

  • Guardrail structure: risk category + intervention point + action = one control. Multiple controls form a named guardrail.
  • Four intervention points: User Input (prompt), Tool Call (preview, agents only), Tool Response (preview, agents only), Output (completion).
  • Risk categories — both models and agents: Hate, Sexual, Self-harm, Violence, User Prompt Attacks (jailbreak), Indirect Attacks (XPIA), Protected Material for text/code.
  • Risk categories — agents only: PII (preview), Task Adherence (preview), Prohibited Actions, Sensitive Data Leakage.
  • Risk categories — models only: Groundedness (preview), Spotlighting (preview).
  • Severity levels: Off (Limited Access Review required), Low, Medium, High.
  • Actions: Annotate (log only, models only) or Annotate and block (returns error to caller; both models and agents).
  • Default guardrail: Microsoft.DefaultV2 — applied automatically when no custom guardrail is assigned.
  • Agent guardrail overrides model guardrail: completely supersedes the model's guardrail for all agent traffic.
  • Role to modify guardrails: Foundry Account Owner.
  • Disabling content filters: requires submitting the Limited Access Review form (separate form for Azure Government).
  • Applies to: all Foundry Models sold by Azure (except audio models like Whisper); agents in Foundry Agent Service only (not externally registered agents).

            "severity": "Low",
            "action": "AnnotateAndBlock",
        },
    ],
}
# POST guardrail_body to Foundry guardrails endpoint for the project
print(json.dumps(guardrail, indent=2))
Gotcha: The agent's guardrail completely overrides the model's guardrail. If you only configure the model's guardrail and assign a different (or no) guardrail to the agent, the model's controls do not apply to agent traffic.

Exam tips & gotchas

  • Tool Call and Tool Response intervention points are agents only — they do not apply to standalone model deployments.
  • To turn content filters completely off, a Limited Access Review form must be submitted and approved — this cannot be done unilaterally via the portal.
  • Annotate action (log only, no block) is available for model deployments only — agents always block when the action is triggered.
  • Whisper (audio) is exempt from guardrail controls — content safety does not apply to audio transcription output.
  • Groundedness and Spotlighting controls are for model deployments only — they do not appear in agent-specific guardrail configurations.

Practice

Q1. An agent must block harmful content not only from user prompts but also from data returned by a web search tool before it reaches the model. Which intervention point must be configured in the agent's guardrail?
Answer: Tool Response (preview). Why: Tool Response is the intervention point that scans content returned from a tool (such as a web search result) to the agent before the agent processes it further. This prevents poisoned or harmful content retrieved from external sources from influencing the agent's response.
Q2. A business requires that a specific model deployment log potential violations for review but not block any responses. Which action setting achieves this?
Answer: Annotate (log only). Why: The Annotate action records the detected risk without blocking the response — available for model deployments. Note that Annotate is not available for agent guardrails (agents always block on intervention).
Q3. A compliance team wants to completely disable content filters for a research model deployment. What is required beyond changing the guardrail configuration?
Answer: Approval via the Limited Access Review form submitted to Microsoft. Why: Turning off content filters entirely (severity = Off) is a Limited Access feature — it requires an approved application form from Microsoft and cannot be self-served through the portal.

1.4.2 Apply responsible AI instrumentation (evaluators, safety evaluations, explanation tooling)

Azure AI Foundry provides a rich library of built-in evaluators that measure different quality and safety dimensions of generative AI outputs. Evaluators are organised into categories: General Purpose (coherence, fluency), Textual Similarity (BLEU, ROUGE, METEOR for n-gram overlap), RAG quality (Groundedness, Relevance, Retrieval, Response Completeness), Risk and Safety (hate, violence, XPIA, code vulnerability), and Agent-specific (Task Adherence, Tool Call Accuracy, Intent Resolution, Customer Satisfaction). Custom evaluators (preview) and the Rubric evaluator allow teams to define business-specific scoring criteria.

Evaluations can run at two levels: turn (individual response — the default) and conversation (entire multi-turn interaction). All evaluators in a single run must use the same level. For agent evaluation, conversation-level evaluation is critical when the agent's behaviour depends on earlier turns in the dialogue.

The Microsoft Responsible AI Standard structures the overall approach into three phases: Discover (red-team before deployment, run adversarial tests), Protect (guardrails, content filters, tool-access restrictions applied in production), and Govern (trace logging, continuous monitoring, and compliance integrations). Safety evaluations can also be run on fine-tuned models before deployment to detect harmful content risks introduced during fine-tuning.

Key facts

  • General purpose evaluators: Coherence (logical flow), Fluency (natural language quality).
  • Textual similarity: F1 Score, BLEU, GLEU, ROUGE, METEOR — all measure n-gram overlap between response and ground truth; suited for translation and summarisation benchmarks.
  • RAG evaluators: Groundedness (1–5 model-based), Groundedness Pro (binary, uses Content Safety), Relevance, Retrieval, Response Completeness (preview).
  • Risk and safety evaluators: Hate and Unfairness, Sexual, Violence, Self-Harm, Protected Materials, Indirect Attack (XPIA), Code Vulnerability, Ungrounded Attributes, Prohibited Actions, Sensitive Data Leakage.
  • Agent evaluators: Task Adherence, Task Completion, Tool Call Accuracy, Tool Selection, Tool Input Accuracy, Tool Output Utilisation, Tool Call Success, Intent Resolution, Customer Satisfaction (preview, 6 dimensions), Quality Grader (preview, bundles 5 dimensions).
  • Evaluation levels: turn (default, single response) or conversation (full multi-turn interaction) — all evaluators in one run must share the same level.
  • RAG recommended evaluator combination: Retrieval + Groundedness + Relevance + Content Safety.
  • Agent recommended evaluator combination: Tool Call Accuracy + Task Adherence + Intent Resolution + Rubric + Content Safety.
  • Custom evaluators (preview): define custom scoring logic, validation rules, and quality metrics.
  • Rubric evaluator (preview): LLM-as-judge with custom weighted criteria; returns weighted average 0–1 with per-dimension reasoning.
  • Safety evaluations on fine-tuned models: detect harmful content risks introduced during fine-tuning before production deployment.
  • Discover → Protect → Govern: Microsoft RAI Standard three-phase framework.

Exam tips & gotchas

  • Groundedness Pro uses Content Safety (not a generative model) — it does not consume model token quota, unlike the standard Groundedness evaluator.
  • All evaluators in a single evaluation run must use the same level (turn or conversation) — mixing levels in one run is not supported.
  • Textual similarity evaluators (BLEU, ROUGE, etc.) require ground-truth reference answers — they cannot assess open-ended responses without a reference.
  • Safety evaluations can be applied to fine-tuned models before deployment — important for detecting risks introduced during fine-tuning that didn't exist in the base model.
  • The Rubric evaluator allows arbitrary business criteria — use it when no built-in evaluator maps to the organisation's quality definition.

Practice

Q1. A team wants to evaluate a summarisation model using automated metrics that compare generated summaries against reference human-written summaries. Which evaluator category is most appropriate?
Answer: Textual similarity evaluators (ROUGE, BLEU, METEOR, or F1 Score). Why: These evaluators measure n-gram overlap between the generated output and a reference ground truth — they are specifically designed for tasks like summarisation and translation where a reference answer is available.
Q2. An agent that handles multi-turn customer service conversations must be evaluated holistically across the full conversation, not turn by turn. Which evaluation level should be configured?
Answer: Conversation-level evaluation. Why: Conversation-level evaluation assesses the complete multi-turn interaction, enabling evaluation of agent behaviour that depends on conversational context — turn-level evaluation would miss issues that only emerge across multiple exchanges.
Q3. Before deploying a fine-tuned customer service model to production, a responsible AI review is required. Which evaluation type specifically addresses risks introduced during fine-tuning?
Answer: Safety evaluations run against the fine-tuned model pre-deployment. Why: Fine-tuning can inadvertently amplify harmful patterns from training data. Safety evaluations test for hate, violence, self-harm, and other risks in the fine-tuned model's outputs, identifying issues before the model reaches users.

1.4.3 Implement auditing (trace logging, provenance metadata, approval workflows)

Foundry tracing is built on OpenTelemetry (OTel) — the industry-standard observability framework. Traces are stored in the Azure Monitor Application Insights instance connected to the Foundry project. A trace captures the complete journey of a request: user inputs, model outputs, tool invocations and results, token consumption, latency at each step, and retry events. Each trace is composed of spans (individual operations, nestable in parent-child hierarchies) and attributes (key-value metadata on each span, such as function parameters and return values).

Foundry follows the OTel semantic conventions for generative AI, using the gen-ai.* namespace for span attributes. For multi-agent systems, standardised span types include execute_task, agent_to_agent_interaction, agent_planning, and agent_state.management. Cross-service trace correlation uses the W3C Trace Context standard (the traceparent header), which allows trace IDs to be propagated across calls to different services, SDKs, and agent frameworks (LangChain, LangGraph, OpenAI Agents SDK, Microsoft Agent Framework).

Trace data serves as the primary audit trail: each agent run's trace links outputs to the exact tool calls, retrieved documents, and model invocations that produced them — enabling lineage tracking. For high-risk agentic actions, Foundry's "Ask AI" approval flow inserts a human review step: the agent proposes the action, the user reviews it in the interface, and only then does execution proceed. Approval scopes (System access vs. none) can be configured per session.

Key facts

  • Tracing stack: OpenTelemetry → Azure Monitor Application Insights (connected to Foundry project).
  • Trace components: Traces (full request journey), Spans (individual operations, nestable), Attributes (key-value metadata on spans).
  • Semantic conventions: gen-ai.* OTel namespace for generative AI; multi-agent span types: execute_task, agent_to_agent_interaction, agent_planning, agent_state.management.
  • Multi-agent trace data per run: user inputs, agent outputs, tool invocations + results, token consumption, latency, model call spans, evaluation events with name and error type.
  • W3C Trace Context: traceparent header propagates correlation across distributed agent calls and service boundaries.
  • Setup: connect Application Insights to the Foundry project → enable tracing → assign Log Analytics Reader role (+ Privileged Monitoring Data Reader if tables are protected).
  • Data retention: follows Application Insights configuration — review for compliance requirements.
  • Approval workflows: "Ask AI" feature proposes an action, user reviews and approves before execution; pre-approval scopes configurable per session.
  • Provenance: traces link each output to the specific tool calls, retrieved documents, and model calls that produced it.
  • PII security: redact sensitive data from span attributes before it reaches telemetry; do not store credentials or secrets in span attributes.
  • Framework integrations: LangChain, LangGraph, OpenAI Agents SDK, Microsoft Agent Framework — all support the same OTel conventions.
# Enable tracing for all agent runs in a project
import os
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient
from azure.ai.projects.telemetry import enable_telemetry

project = AIProjectClient(
    endpoint="https://<resource>.ai.azure.com/api/projects/<project>",
    credential=DefaultAzureCredential(),
)

# All subsequent agent runs are traced to Application Insights
enable_telemetry(
    project,
    connection_string=os.environ["APPLICATIONINSIGHTS_CONNECTION_STRING"],
)

print("Tracing enabled — agent runs will appear in Application Insights.")
Gotcha: Trace data is stored in Application Insights logs — PII, credentials, and sensitive prompt content must be redacted before they enter the telemetry pipeline. Apply the same access controls and retention policies to trace data as you would to production logs.

Exam tips & gotchas

  • W3C Trace Context (traceparent header) is the mechanism for propagating trace correlation across multi-service, multi-agent boundaries.
  • Tracing requires the project to be connected to Application Insights — no connection = no traces.
  • Approval workflows ("Ask AI") are a human-in-the-loop control, not a guardrail — they pause execution for user review, not filter content.
  • Data retention in Application Insights must be explicitly configured for compliance — the default may not meet regulatory retention requirements.
  • The Privileged Monitoring Data Reader role is only needed if Log Analytics tables are protected — standard log access uses Log Analytics Reader.

Practice

Q1. A compliance audit requires that every agent response can be traced back to the exact retrieved documents and tool calls that produced it. Which Foundry capability provides this lineage?
Answer: Foundry Tracing (OpenTelemetry-based) with traces stored in Application Insights. Why: Each agent run generates a trace that links the final output to every span — including tool invocations, retrieved documents, and model calls — providing full provenance for compliance auditing.
Q2. A distributed agent solution spans three separate microservices. An analyst notices traces from different services cannot be correlated. What standard must be implemented to fix this?
Answer: W3C Trace Context — the traceparent header must be propagated across all HTTP calls between services. Why: W3C Trace Context is the standard mechanism for distributed trace correlation. Without it, each service generates independent trace IDs that cannot be joined into a single end-to-end trace.
Q3. An agent's span attributes contain raw customer query text including email addresses. What should be done before the trace data reaches Application Insights?
Answer: Implement PII redaction or scrubbing in the telemetry pipeline before span attributes are emitted to Application Insights. Why: Application Insights logs are accessible to anyone with Log Analytics Reader access — storing PII (email addresses, names, etc.) in traces violates data protection requirements and creates unnecessary privacy risk.

1.4.4 Govern agent behavior (oversight modes, constraints, tool-access controls)

Governing agent behaviour requires applying the Microsoft Responsible AI framework across three phases: Discover (red-team and adversarially test agents before deployment to find risks like jailbreaks, data leakage, or scope violations), Protect (apply guardrails with tool call and tool response intervention points, restrict tool access to least-privilege, scan proposed agent actions), and Govern (maintain trace logs, run continuous evaluation, integrate with compliance and audit systems).

Tool-access controls enforce the principle of least privilege: each agent should only have access to the specific data sources and tools it needs for its defined function. When an agent acts on behalf of a user, it should inherit the user's permissions — not use broad shared credentials that exceed individual user rights. Microsoft Entra Agent ID provides a distinct managed identity for each agent, enabling RBAC-scoped tool access and a per-agent audit trail in Azure activity logs.

For high-risk or irreversible actions, agents should include human-in-the-loop approval steps — via Azure Logic Apps triggers, custom function tools that pause execution pending approval, or the Foundry "Ask AI" approval flow. System-level constraints are enforced through the agent's instructions field (system prompt), guardrail assignment (controlling what the agent can say and do), and the tool catalog (controlling what actions the agent can take).

Key facts

  • Discover → Protect → Govern: Microsoft RAI Standard framework for agent lifecycle; discover risks early, protect in production, govern continuously.
  • Least-privilege tool access: grant each agent access only to the specific data sources it requires for its function.
  • User identity inheritance: when agents access data on behalf of a user, they should pass the user's identity/token to downstream systems — not use broad shared credentials.
  • Microsoft Entra Agent ID: distinct managed identity per agent; enables RBAC-scoped access and per-agent audit trails.
  • Human-in-the-loop: Azure Logic Apps triggers, custom function tools that pause execution, or Foundry "Ask AI" approval flow for high-risk actions.
  • Transparency: disclose AI involvement clearly in agent interfaces; notify users when the agent is acting on their behalf.
  • Azure Policy: enforce infrastructure governance (private endpoints, CMK, allowed model SKUs) using Microsoft.CognitiveServices aliases.
  • Microsoft Purview: apply DLP policies and sensitivity labels to data sources accessed by agents.
  • System prompt constraints: define agent persona, scope, and output format restrictions via the instructions field — the first line of behavioural governance.
  • Agent guardrail: include Tool Call and Tool Response intervention points (preview) to scan the agent's planned actions and inbound tool data.
  • Connected agents scope: use narrowly scoped sub-agents for specific subtasks — reduces blast radius if one agent is compromised or misbehaves.
  • Human accountability: designate a responsible human owner for AI outcomes; use Foundry evaluations (bias, fairness) and continuous monitoring to ensure ongoing compliance.
# Agent with constrained system prompt for governance
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project = AIProjectClient(
    endpoint="https://<resource>.ai.azure.com/api/projects/<project>",
    credential=DefaultAzureCredential(),
)

agent = project.agents.create_version(
    agent_name="hr-policy-agent",
    definition={
        "model": "gpt-4.1",
        "instructions": (
            "You are an HR policy assistant. "
            "Only answer questions about the company HR policies contained in the knowledge base. "
            "Do not access, modify, or take actions on any HR systems. "
            "Do not answer questions outside of HR policy scope. "
            "If asked to perform any action beyond retrieving policy information, "
            "respond that you are not authorised to do so and suggest contacting the HR team directly."
        ),
        "tools": [{"type": "file_search"}],  # read-only knowledge retrieval only
    }
)
print(f"Constrained HR agent created: {agent.id}")
Key: The agent's instructions field is the first layer of behavioural control — but it is not sufficient alone. Pair it with a guardrail that includes Tool Call and Tool Response intervention points, and restrict the tool catalog to only what the agent needs.

Exam tips & gotchas

  • System prompt constraints can be overridden by adversarial users — they must be backed by guardrails with Tool Call intervention points for robust governance.
  • Connected sub-agents with narrow tool sets reduce blast radius — if one agent is compromised, it cannot reach resources outside its assigned scope.
  • Entra Agent ID provides a per-agent managed identity — this is important for audit trails when multiple agents share the same Foundry resource.
  • Human-in-the-loop approval is critical for irreversible actions (data deletion, financial transactions, system configuration changes) — agents should pause, not proceed, without approval.
  • Microsoft Purview DLP policies apply to the data sources accessed by agents — they do not replace guardrails on agent outputs.

Practice

Q1. An agent accesses a company SharePoint library using a shared service account with read access to all documents. A user asks the agent about documents they are not supposed to see. What governance principle is being violated?
Answer: User identity inheritance / least-privilege. Why: The agent should pass the requesting user's identity to SharePoint so that access control is enforced at the data layer. Using a broad shared service account allows the agent to surface documents beyond the user's own permissions.
Q2. A financial services agent must confirm with a human operator before executing any wire transfer action. Which mechanism should be implemented in the agent workflow?
Answer: A human-in-the-loop approval step — either using a custom function tool that pauses execution pending human approval, an Azure Logic Apps trigger that routes to an approval workflow, or the Foundry "Ask AI" approval flow. Why: Wire transfers are irreversible high-risk actions; human oversight before execution is a core responsible AI control for such scenarios.
Q3. An organisation wants each of its ten deployed agents to have a distinct audit trail in Azure activity logs. What should be provisioned for each agent?
Answer: A separate Microsoft Entra Agent ID (distinct managed identity) for each agent. Why: Each agent having its own managed identity means Azure activity logs record actions against that specific identity — enabling per-agent audit trails that distinguish which agent performed which action.
