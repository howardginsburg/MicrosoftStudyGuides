2.1 Build generative applications by using Foundry

2.1.1 Deploy and consume LLMs, small models, code models, and multimodal models

Azure AI Foundry's model catalog consolidates large language models (LLMs), small language models (SLMs), code-generation models, and multimodal models (text, image, and audio) into a single browsable surface. Once you select a model, you choose one of two deployment pathways before your application can call it.

Serverless API deployments are pay-per-token, fully managed, and require no provisioned compute infrastructure — ideal for variable or unpredictable workloads. Managed Compute deployments run on a dedicated container endpoint, required for custom fine-tuned models, regulatory data-residency isolation, or when predictable throughput matters; autoscaling for managed compute is configured through Azure Machine Learning online endpoint scaling rules.

Four official SDKs support model inference: the OpenAI SDK, the Azure OpenAI SDK, the Azure AI Inference package (azure-ai-inference), and the Azure AI Projects package (azure-ai-projects). The azure-ai-inference package is the recommended cross-model abstraction — it targets the Azure AI Model Inference REST API and lets you swap the underlying model without changing application code.

Key facts

  • Deployment types: Serverless API (pay-per-token, no provisioning) and Managed Compute (dedicated endpoint; mandatory for fine-tuned models).
  • Four inference SDKs: OpenAI SDK, Azure OpenAI SDK, azure-ai-inference, azure-ai-projects.
  • azure-ai-inference is the recommended cross-model abstraction targeting the Azure AI Model Inference REST API.
  • Project inference endpoint: https://<resource>.services.ai.azure.com/models; OpenAI-compatible endpoint: https://<resource>.openai.azure.com/openai/v1.
  • Multimodal content (images, audio) is passed via image_url or input_audio typed blocks inside the content array of a message.
  • Managed compute autoscaling is configured in Azure Machine Learning online endpoint scaling rules, not the Foundry portal directly.
  • The portal Test and Consume tabs on a deployment provide ready-to-use code samples and the target endpoint URI.
import os
from azure.ai.inference import ChatCompletionsClient
from azure.ai.inference.models import SystemMessage, UserMessage
from azure.core.credentials import AzureKeyCredential

client = ChatCompletionsClient(
    endpoint=os.environ["AZURE_INFERENCE_ENDPOINT"],
    credential=AzureKeyCredential(os.environ["AZURE_INFERENCE_KEY"]),
)

response = client.complete(
    model="gpt-4o",
    messages=[
        SystemMessage(content="You are a concise technical writer."),
        UserMessage(content="Explain transformer attention in two sentences."),
    ],
    temperature=0.3,
    max_tokens=256,
)
print(response.choices[0].message.content)
Tip: Prefer azure-ai-inference for new applications — its model-agnostic interface means you can switch from GPT-4o to a Phi-4 SLM by changing only the model parameter, not the client code.

Exam tips & gotchas

  • Fine-tuned models and regulatory isolation both require Managed Compute — Serverless API cannot host custom weights.
  • azure-ai-inference is for cross-model portability; azure-ai-projects is needed for Foundry-native features like agents, evaluations, and connections.
  • Multimodal content uses typed blocks inside the content array — not separate API parameters.
  • Managed compute autoscaling is an Azure Machine Learning concept; don't confuse it with Foundry-level settings.

Practice

Q1. A team has fine-tuned a GPT-4o model on proprietary data and needs a dedicated endpoint with predictable throughput. Which deployment type should they choose, and why?
Answer: Managed Compute. Why: Serverless API deployments do not support custom fine-tuned model weights. Managed Compute deploys the model on a dedicated container endpoint that can host custom weights and provides consistent throughput through Azure ML scaling rules.
Q2. A developer wants inference code that works against both GPT-4o and a Phi-4 SLM without changing the client class. Which SDK package enables this?
Answer: azure-ai-inference (ChatCompletionsClient). Why: The Azure AI Inference package targets the Azure AI Model Inference REST API, which provides a uniform interface across model families. Only the model parameter changes between calls.
Q3. An application needs to send an image and a text question to a multimodal model. How should the image be included in the request body?
Answer: As an image_url typed block inside the content array of the user message. Why: Multimodal models accept mixed content via typed blocks — image_url for images, input_audio for audio — within the messages content array, not as top-level API parameters.

2.1.2 Implement retrieval-augmented generation (RAG) in an application

Retrieval-augmented generation (RAG) grounds LLM responses in external knowledge rather than relying solely on training data. The pipeline has three stages: Retrieve relevant document chunks from an index, Augment the user query by injecting those chunks as context, then Generate a grounded response through the LLM. Explicitly instructing the model to answer only from the provided context is the critical guardrail against fabrication.

Azure AI Foundry integrates with Azure AI Search as its primary indexing and retrieval backend. Indexes must have searchable text fields; for semantic similarity, they also need vector fields of type Collection(Edm.Single). Content preparation capabilities include automatic chunking, 50+ language analyzers, OCR for PDFs and images, and integrated vectorization through Azure OpenAI or Azure Vision.

Two retrieval approaches are available in Azure AI Search: Classic RAG (hybrid search + semantic ranking, generally available, fine-grained control) and Agentic Retrieval (LLM-assisted query planning, multi-source, structured response with citations, preview). For managed knowledge bases in the Foundry portal, Foundry IQ builds permission-aware knowledge bases on top of Azure AI Search for use by agents.

Key facts

  • RAG pipeline: Retrieve → Augment → Generate.
  • Two retrieval modes: Classic RAG (hybrid search + semantic ranking, GA) and Agentic Retrieval (LLM-assisted, preview).
  • Best-practice search configuration: hybrid search (vector + keyword) plus semantic ranking for query-intent reranking.
  • Vector fields in Azure AI Search must be type Collection(Edm.Single).
  • Code pattern: chunk and index data → create Foundry project connection to search service → retrieve with search SDK → inject context into LLM prompt.
  • Groundedness evaluator is the primary metric for detecting RAG fabrications; it measures whether responses are supported by retrieved context.
  • Foundry IQ is the managed knowledge-base layer on top of Azure AI Search, designed for agent-facing knowledge bases.
import os
from azure.search.documents import SearchClient
from azure.core.credentials import AzureKeyCredential
from azure.ai.inference import ChatCompletionsClient
from azure.ai.inference.models import SystemMessage, UserMessage

search_client = SearchClient(
    endpoint=os.environ["SEARCH_ENDPOINT"],
    index_name="product-docs",
    credential=AzureKeyCredential(os.environ["SEARCH_KEY"]),
)

def rag_answer(question: str) -> str:
    # Retrieve top-3 relevant chunks
    results = search_client.search(search_text=question, top=3)
    context = "\n".join(r["content"] for r in results)

    # Augment + Generate
    chat = ChatCompletionsClient(
        endpoint=os.environ["AZURE_INFERENCE_ENDPOINT"],
        credential=AzureKeyCredential(os.environ["AZURE_INFERENCE_KEY"]),
    )
    resp = chat.complete(
        model="gpt-4o",
        messages=[
            SystemMessage(content=f"Answer only using the context below.\n\n{context}"),
            UserMessage(content=question),
        ],
    )
    return resp.choices[0].message.content
Key: Instruct the LLM explicitly to answer only from the provided context. Without this constraint the model may blend retrieved facts with training data, undermining grounding quality.

Exam tips & gotchas

  • The Groundedness evaluator is the correct metric for fabrication detection in RAG — not Relevance or Fluency.
  • Hybrid search (vector + keyword) outperforms either alone; adding semantic ranking further improves intent-based reranking.
  • Vector fields in Azure AI Search must be Collection(Edm.Single) — a common index configuration mistake is using the wrong field type.
  • Agentic Retrieval is still preview; Classic RAG (hybrid + semantic) is GA and the safer default for production scenarios.

Practice

Q1. A RAG application returns answers that include facts not present in any retrieved document. Which evaluator should you run to measure and detect this problem?
Answer: Groundedness evaluator. Why: Groundedness measures whether each claim in a response is supported by the provided context. A low score identifies fabrication — content generated from model training data rather than retrieved chunks.
Q2. A team wants the best out-of-box relevance for RAG queries over a mixed document corpus. What Azure AI Search configuration should they use?
Answer: Hybrid search (vector + keyword) with semantic ranking enabled. Why: Hybrid search combines BM25 keyword matching with vector similarity for broad recall; semantic ranking reorders results by query intent, improving precision beyond what either method achieves alone.
Q3. After chunking and indexing documents in Azure AI Search, what must be done in the Foundry project before an agent tool can query that index?
Answer: Create a project connection from the Foundry project to the Azure AI Search service. Why: Foundry agents access external services through configured connections. Without a connection, the agent tool cannot authenticate or query the index.

2.1.3 Design workflows, tool-augmented flows, and multistep reasoning pipelines

Tool-augmented flows extend an LLM beyond text generation by coupling reasoning with external actions — search queries, code execution, API calls. The model emits a structured tool-call request; application code executes the tool and returns the result; the model continues reasoning with the new information. This request–execute–continue loop repeats until the model produces a final response with no pending tool calls.

The current recommended entry point for Foundry workflow orchestration is the Responses API, which handles streaming, multi-turn conversation management, and the complete tool-call lifecycle. Prompt flow, the earlier Foundry workflow surface, is now deprecated with retirement scheduled for April 20, 2027; new development should migrate to the Microsoft Agent Framework and the Responses API.

Multiple orchestration frameworks integrate natively with the Foundry project endpoint: the Agent Framework for hosted code-based agents, LangGraph, the OpenAI Agents SDK, and Semantic Kernel. The Foundry Toolbox feature lets teams define, version, and expose a curated set of tools through a single MCP-compatible endpoint, making those tools consumable by any of these frameworks without per-framework integration work.

Key facts

  • Prompt flow is deprecated — retirement April 20, 2027; migrate to the Agent Framework and Responses API.
  • The Responses API is the single recommended entry point for Foundry agent and tool orchestration; supports streaming and multi-turn management.
  • Compatible orchestration frameworks: Agent Framework, LangGraph, OpenAI Agents SDK, Semantic Kernel.
  • Built-in platform tools via the Responses API: file search, code interpreter, web search, memory, SharePoint, WorkIQ, Fabric IQ, remote MCP servers.
  • Rule of thumb: single agent + multiple tools for sequential tasks; multi-agent pattern for parallel or specialized subtasks.
  • Foundry Toolbox versions and publishes tools via a single MCP-compatible endpoint; no per-framework integration needed.
import os
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_KEY"],
    api_version="2025-01-01-preview",
)

tools = [{
    "type": "function",
    "function": {
        "name": "get_inventory",
        "description": "Return current stock count for a product SKU.",
        "parameters": {
            "type": "object",
            "properties": {"sku": {"type": "string"}},
            "required": ["sku"],
        },
    },
}]

messages = [{"role": "user", "content": "Is SKU-42 in stock?"}]
response = client.chat.completions.create(
    model="gpt-4o", messages=messages, tools=tools
)
# When finish_reason == "tool_calls", execute the function, append result, and re-call
Gotcha: Prompt flow is deprecated. Exam questions about production best practices for new workflows should favor the Agent Framework and Responses API, not prompt flow.

Exam tips & gotchas

  • Know prompt flow's retirement date (April 20, 2027) and its replacement (Agent Framework + Responses API).
  • Tool-augmented flows require an explicit execution loop — detect finish_reason == "tool_calls", execute, append result, and re-invoke the model.
  • The Responses API — not the Completions API — is the recommended orchestration entry point for Foundry agents.
  • Foundry Toolbox's MCP-compatible endpoint eliminates per-framework tool integration code.

Practice

Q1. A team built an orchestration pipeline using Prompt flow. What should they migrate to, and by when must they act?
Answer: Migrate to the Microsoft Agent Framework and Responses API before April 20, 2027. Why: Prompt flow is officially deprecated with retirement on April 20, 2027. The Agent Framework and Responses API are its designated replacements in Azure AI Foundry.
Q2. An application calls a weather API and a database query tool in a multi-step reasoning loop. When should the application re-invoke the model?
Answer: After each tool execution — append the tool result as a tool-role message, then call the model again. Why: The LLM emits a tool-call request and produces finish_reason == "tool_calls". The application must execute the tool, append the result, and re-invoke the model so it can continue reasoning with the new data.
Q3. Five internal tools need to be available to LangGraph, Semantic Kernel, and a custom agent without separate integration per framework. What Foundry feature solves this?
Answer: Foundry Toolbox. Why: Foundry Toolbox publishes versioned tools behind a single MCP-compatible endpoint. Any MCP-compatible framework can consume the tools without custom per-framework wiring.

2.1.4 Evaluate models and apps, including detecting fabrications, relevance, quality, and safety

The Foundry Evaluation service runs model or agent outputs against built-in or custom evaluators and surfaces scores in the portal's Evaluations tab. Evaluations can be triggered from the portal, the SDK, or on a schedule. The service supports four evaluation targets: individual Model turns, full Agent conversations or individual agent turns, pre-computed Datasets, and historical Traces from Application Insights.

Evaluators fall into three categories. Quality evaluators are AI-assisted (they require a judge model, such as gpt-4.1-mini) and measure properties like groundedness, relevance, coherence, fluency, response completeness, customer satisfaction, and task completion. Safety evaluators detect harmful content categories — violence, sexual, self-harm, and hate/unfairness — and apply only to individual turns. Agent evaluators measure tool usage effectiveness: intent resolution, task adherence, tool call success, tool selection accuracy, tool output utilization, and tool input accuracy.

Programmatic evaluation uses the azure-ai-projects SDK: create an evaluation run targeting a dataset or trace store, poll for completion, then retrieve the report_url to review per-sample scores. Simulated data generation can create synthetic multi-turn conversations from scenario descriptions when production traffic is unavailable.

Key facts

  • Evaluation targets: Model (individual turn), Agent (conversation or turn), Dataset (pre-computed outputs), Traces (Application Insights).
  • Quality evaluators require a judge model (e.g., gpt-4.1-mini): Groundedness, Relevance, Coherence, Fluency, Response Completeness, Customer Satisfaction, Task Completion.
  • Groundedness is the primary metric for detecting RAG fabrications — it measures whether responses are supported by provided context.
  • Safety evaluators (individual turns only): Violence, Sexual, Self-harm, Hate/Unfairness.
  • Agent evaluators: Intent Resolution, Task Adherence, Tool Call Success, Tool Selection Accuracy, Tool Output Utilization, Tool Input Accuracy.
  • Simulated (synthetic) multi-turn conversations can be generated from scenario descriptions for evaluation data coverage.
import os
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project_client = AIProjectClient(
    endpoint=os.environ["PROJECT_ENDPOINT"],
    credential=DefaultAzureCredential(),
)

# Trigger an evaluation run with quality evaluators; judge model must be deployed
eval_run = project_client.evals.runs.create(
    evaluation_id="<eval-config-id>",
    data_source={"type": "dataset", "dataset_id": "<dataset-id>"},
    evaluators=["groundedness", "relevance", "coherence"],
)
print(f"Evaluation run ID: {eval_run.id}")
# Poll until complete, then open eval_run.report_url in the portal
Key: Safety evaluators operate on individual turns only, not full conversation threads. For conversation-level safety analysis, evaluate each turn independently or use agent evaluators for tool-usage safety.

Exam tips & gotchas

  • AI-assisted quality evaluators require a judge model — a common misconfiguration is running them without a deployed judge.
  • Safety evaluators are restricted to individual turns; they cannot evaluate a full conversation thread as a single unit.
  • Groundedness is correct for fabrication detection; Relevance measures topical fit, not grounding support.
  • Agent evaluators (Tool Call Success, Intent Resolution, Task Adherence) are distinct from quality evaluators and need agent trace data, not just model outputs.

Practice

Q1. Your RAG agent generates answers that reference facts not found in any retrieved document. Which evaluator measures this problem?
Answer: Groundedness evaluator. Why: Groundedness measures whether each claim in a response is supported by the provided context. Low groundedness scores identify fabrication — content sourced from the model's training data rather than retrieved chunks.
Q2. A compliance team needs to verify no generated content contains self-harm language before deployment. Which evaluator category applies?
Answer: Safety evaluators — specifically the Self-harm safety evaluator. Why: Safety evaluators detect harmful content categories (Violence, Sexual, Self-harm, Hate/Unfairness) and are designed for content policy validation. They operate on individual turns.
Q3. An evaluation run using the Groundedness evaluator throws an error about a missing model. What is misconfigured?
Answer: No judge model is specified or deployed. Why: AI-assisted quality evaluators (including Groundedness) use a judge LLM to score outputs. A model such as gpt-4.1-mini must be deployed and referenced in the evaluation configuration.

2.1.5 Integrate generative workflows into applications by using Foundry SDKs and connectors

The Foundry SDK (azure-ai-projects ≥ 2.0.0) is the core thin-client package that wraps the Foundry project endpoint. It is the foundation on which higher-level sub-clients — agents, evaluations, inference, connections — are built. The AIProjectClient class is the primary entry point, exposing all Foundry-native operations through a single authenticated session.

Connections link external services to a Foundry project. They can be created via the Foundry portal (Management Center → Connected resources), via the Azure CLI (az ml connection create), or via the azure-ai-ml Python package. Supported connector types include Azure AI Search, Azure Blob Storage, Azure OpenAI, custom HTTPS endpoints, and third-party APIs via OpenAPI definitions. An agent cannot access any external service that does not have a corresponding configured connection.

Streaming responses are supported through the Responses API and the OpenAI SDK's stream=True parameter; partial message deltas arrive as server-sent events. The Foundry MCP server integration exposes Foundry tools and agents to any MCP-compatible client, removing the need for per-framework SDK integration code.

Key facts

  • SDK selection: azure-ai-projects for agents, evaluations, and Foundry-native features; OpenAI SDK for maximum OpenAI API compatibility or lowest latency; Agent Framework package for hosted multi-agent systems; Foundry Tools SDKs for Vision, Speech, Content Safety.
  • AIProjectClient provides sub-clients: agents, evaluations, connections, inference — all through one authenticated session.
  • Connection types: Azure AI Search, Azure Blob Storage, Azure OpenAI, custom HTTPS, third-party APIs via OpenAPI.
  • Connections created via portal, az ml connection create CLI, or azure-ai-ml package.
  • Streaming uses server-sent events; the response is an iterator of partial delta chunks.
  • Foundry MCP server exposes Foundry tools and agents to any MCP-compatible client without custom integration code.
import os
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project_client = AIProjectClient(
    endpoint=os.environ["PROJECT_ENDPOINT"],
    credential=DefaultAzureCredential(),
)

# List all connections configured in the project
connections = project_client.connections.list()
for conn in connections:
    print(f"{conn.name}  type={conn.type}")

# Get an OpenAI-compatible client scoped to this project with streaming
openai_client = project_client.inference.get_azure_openai_client(
    api_version="2025-01-01-preview"
)
stream = openai_client.chat.completions.create(
    model=os.environ["MODEL_DEPLOYMENT_NAME"],
    messages=[{"role": "user", "content": "Summarize the quarterly report."}],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content or "", end="")

Exam tips & gotchas

  • Use azure-ai-projects when you need connections listing, agents, or evaluations; use the openai package when you only need raw chat completions.
  • Missing connections are a common deployment failure — every external service an agent tool references needs a configured connection in the project.
  • The MCP server integration removes custom per-framework tool integration code — one endpoint for all MCP-compatible consumers.
  • Streaming responses are iterators of partial delta chunks, not complete message objects — handle None content in each chunk.

Practice

Q1. A developer needs to list all connections in a Foundry project and obtain an OpenAI-compatible client in the same session. Which class provides both capabilities?
Answer: AIProjectClient from azure-ai-projects. Why: AIProjectClient wraps the project endpoint and exposes sub-clients for connections (project_client.connections.list()) and inference (project_client.inference.get_azure_openai_client()) within a single authenticated session.
Q2. An enterprise wants Foundry agents consumable by a third-party orchestration tool that uses the MCP protocol. What Foundry capability enables this?
Answer: The Foundry MCP server integration. Why: The MCP server exposes Foundry tools and agents via the MCP protocol, allowing any MCP-compatible client to invoke them without writing custom SDK integration code for each consumer.

2.1.6 Configure an application to connect to a Foundry project

Every Foundry project exposes a single, stable project endpoint in the format https://<resource-name>.services.ai.azure.com/api/projects/<project-name>. All Foundry-native operations — agents, evaluations, inference, and connections — flow through this endpoint. Applications that use the OpenAI SDK directly connect via the OpenAI-compatible route (/openai/v1) on the same resource.

Authentication defaults to DefaultAzureCredential from azure-identity, which chains through credential providers automatically — Managed Identity in Azure-hosted environments, a service principal via environment variables in CI/CD, and the az login token in local development — all without any code changes between environments. API keys can be used for the /openai/v1 routes but are secondary to credential-based auth.

Access is governed by Foundry RBAC roles. Foundry User is the least-privilege role for inference and agent invocation; Foundry Project Manager is needed for project administration. Enabling full prompt and response content recording in traces requires setting the environment variable AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true, which is off by default due to potential PII exposure.

Key facts

  • Project endpoint format: https://<resource-name>.services.ai.azure.com/api/projects/<project-name>.
  • DefaultAzureCredential selects: Managed Identity (Azure hosting), service principal (CI/CD via env vars), or az login token (local dev) — zero code changes across environments.
  • RBAC roles: Foundry User (least-privilege: inference + invocation), Foundry Project Manager (admin), Foundry Owner, Foundry Account Owner.
  • Key environment variables: PROJECT_ENDPOINT, AZURE_OPENAI_ENDPOINT, APPLICATION_INSIGHTS_CONNECTION_STRING, MODEL_DEPLOYMENT_NAME.
  • Set AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true to capture prompt and response text in traces (disabled by default — may expose PII).
import os
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

# DefaultAzureCredential: Managed Identity (Azure), service principal (CI),
# or az login token (local dev) — no code changes across environments.
project_client = AIProjectClient(
    endpoint=os.environ["PROJECT_ENDPOINT"],
    credential=DefaultAzureCredential(),
)

# Obtain an OpenAI-compatible client scoped to the Foundry project
openai_client = project_client.inference.get_azure_openai_client(
    api_version="2025-01-01-preview"
)
response = openai_client.chat.completions.create(
    model=os.environ["MODEL_DEPLOYMENT_NAME"],
    messages=[{"role": "user", "content": "Hello from Foundry!"}],
)
print(response.choices[0].message.content)
Tip: Store PROJECT_ENDPOINT and MODEL_DEPLOYMENT_NAME as environment variables or Azure App Configuration entries — never hard-code them. This makes the same application binary work across dev, staging, and production Foundry projects.

Exam tips & gotchas

  • DefaultAzureCredential is the recommended auth mechanism; API keys are secondary and reduce audit traceability.
  • The Foundry User role is sufficient for most developer tasks — do not over-grant Project Manager or Owner to developers.
  • Content recording (AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true) is off by default because it may capture PII in prompt and response text.
  • The Foundry project endpoint (.../api/projects/<name>) is distinct from the raw resource endpoint used directly by the OpenAI SDK.

Practice

Q1. An app must authenticate with az login on a developer laptop, a service principal in GitHub Actions, and Managed Identity in Azure App Service — all using the same code. What credential class should it use?
Answer: DefaultAzureCredential from azure-identity. Why: DefaultAzureCredential chains through credential providers in priority order (environment variables → workload identity → managed identity → CLI token), automatically selecting the right credential for the current environment.
Q2. What is the minimum RBAC role a developer needs to invoke an agent in a Foundry project?
Answer: Foundry User. Why: Foundry User is the least-privilege role granting inference and agent invocation rights. Foundry Project Manager and Owner are for administration tasks and should not be assigned to application developers by default.
Q3. An operations team needs to capture full prompt and response text in distributed traces for debugging. What change is required?
Answer: Set the environment variable AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true. Why: By default, traces capture span metadata (timestamps, token counts, model name) but not actual content, to avoid PII exposure. This variable opts in to full content recording.

2.2 Build agents by using Foundry

2.2.1 Define agent roles, goals, conversation-tracking approach, and tool schemas

A Foundry agent is defined by three components: a model (selected from the catalog), instructions (role, constraints, scope, escalation rules, and tone), and tools (capabilities such as search, code execution, APIs, or custom functions). Two agent types exist: Prompt agents (fully managed, no runtime code to maintain, authored in the portal or via SDK) and Hosted agents (your code packaged as a container, with Foundry providing the endpoint, scaling, and identity).

Threads represent persistent conversation state. A thread holds the ordered message history between a user and an agent across multiple turns, eliminating the need for the application to track and resend message history manually. Each new user message is appended to the thread before starting a run; each agent response is automatically added as well.

Tool schemas follow the JSON Schema format: each tool has a name, description, and parameters object with typed properties and a required list. The model relies entirely on the description field to decide when and why to call a tool — precise, specific descriptions dramatically improve tool selection accuracy. Agents can be published to stable Agent Application endpoints and given a dedicated Microsoft Entra identity for RBAC-governed downstream resource access without shared keys.

Key facts

  • Agent components: model + instructions + tools.
  • Agent types: Prompt agents (fully managed, portal/SDK-authored) and Hosted agents (containerized code, Foundry-managed endpoint).
  • Threads hold ordered message history across turns — no manual history tracking in application code.
  • Tool schemas use JSON Schema: name, description, parameters (object with typed properties + required list).
  • The model uses the description field to decide when to invoke a tool — quality of descriptions directly affects tool selection accuracy.
  • Agents can be published as Agent Applications with stable endpoints and a dedicated Microsoft Entra identity.
  • The Agents Playground in the portal enables interactive testing of agent behavior, tool connectivity, and prompt variations.
import os
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project_client = AIProjectClient(
    endpoint=os.environ["PROJECT_ENDPOINT"],
    credential=DefaultAzureCredential(),
)

# Create a prompt agent with role, goal, and a tool schema
agent = project_client.agents.create_agent(
    model=os.environ["MODEL_DEPLOYMENT_NAME"],
    name="inventory-agent",
    instructions=(
        "You are an inventory assistant. Answer questions about stock levels. "
        "Use get_inventory for any SKU lookup. Never fabricate quantities."
    ),
    tools=[{
        "type": "function",
        "function": {
            "name": "get_inventory",
            "description": "Return current stock count for a product SKU.",
            "parameters": {
                "type": "object",
                "properties": {
                    "sku": {"type": "string", "description": "Product SKU identifier"}
                },
                "required": ["sku"],
            },
        },
    }],
)
print(f"Agent ID: {agent.id}")

# Create a thread to track conversation across turns
thread = project_client.agents.threads.create()
print(f"Thread ID: {thread.id}")

Exam tips & gotchas

2.2.2 Build agents that integrate retrieval, function-calling, and conversation memory

Function calling enables an agent to emit structured tool-call requests (tool_calls in the run response) that your application code must execute and return. When the agent needs a function result, the run enters requires_action status and waits. The application polls for this status, executes the function locally, submits the outputs via submit_tool_outputs, and then continues polling to completion. Tool outputs expire 10 minutes after run creation — results must be submitted within this window.

For document retrieval without custom code, the file search built-in tool indexes uploaded PDFs, Word documents, and text files, allowing the agent to retrieve and cite content during a conversation. For persistent cross-session state, the Memory tool (preview) stores and retrieves key facts scoped to a user or session, eliminating the need for a custom database.

Conversation continuity across turns is maintained automatically within a thread — each turn's user message and agent response are appended to the thread's ordered history. The agent has full visibility of this history on every run without the application sending it explicitly.

Key facts

  • Function calling lifecycle: define schema → create agent with tool → create thread + message → start run → poll status → detect requires_action → execute function → submit tool_outputs → poll to completion.
  • Run enters requires_action status when the agent needs a function result; the application must act before the 10-minute expiry window.
  • Tool outputs expire 10 minutes after run creation — submit function results before this deadline or the run fails.
  • File search: built-in tool that indexes uploaded documents; the agent retrieves and cites content during conversation.
  • Memory (preview): platform tool for persisting key facts across sessions without custom database code.
  • Conversation memory within a session is maintained automatically via threads — no manual history management required.
import json, time
from azure.ai.projects.models import MessageRole, RunStatus

# Add user message to existing thread
project_client.agents.messages.create(
    thread_id=thread.id, role=MessageRole.USER, content="Is SKU-42 in stock?"
)

# Start a run and poll until complete or requires_action
run = project_client.agents.runs.create(thread_id=thread.id, agent_id=agent.id)

while run.status in (RunStatus.QUEUED, RunStatus.IN_PROGRESS, RunStatus.REQUIRES_ACTION):
    time.sleep(1)
    run = project_client.agents.runs.get(thread_id=thread.id, run_id=run.id)
    if run.status == RunStatus.REQUIRES_ACTION:
        tool_calls = run.required_action.submit_tool_outputs.tool_calls
        outputs = []
        for tc in tool_calls:
            args = json.loads(tc.function.arguments)
            # Execute function locally; submit result before 10-minute expiry
            result = {"sku": args["sku"], "quantity": 150}
            outputs.append({"tool_call_id": tc.id, "output": json.dumps(result)})
        run = project_client.agents.runs.submit_tool_outputs(
            thread_id=thread.id, run_id=run.id, tool_outputs=outputs
        )

# Retrieve the final agent response
msg = project_client.agents.messages.get_last_message_by_role(
    thread_id=thread.id, role=MessageRole.AGENT
)
print(msg.text_messages[0].text.value)
Gotcha: Tool outputs expire 10 minutes after run creation. If your function takes longer than that, the run will fail. Design functions to complete quickly or implement a timeout-and-retry strategy.

Exam tips & gotchas

  • The run status sequence is: queued → in_progress → requires_action (if tool needed) → back to in_progress → completed (or failed).
  • Tool outputs have a hard 10-minute expiry from run creation — not from when requires_action is detected.
  • File search indexes uploaded files for retrieval; use it when documents are known ahead of time. Use function calling for real-time or dynamic data.
  • Memory tool (preview) persists cross-session state; within a single session, threads already maintain conversation continuity.

2.2.4 Implement orchestrated multi-agent solutions

The Connected Agents pattern (classic API 2025-05-15-preview) implements a primary/subagent hierarchy: a main orchestrating agent delegates tasks to specialized subagents using natural language routing — no hardcoded decision logic. The ConnectedAgentTool wraps a subagent and registers it as a callable tool on the main agent. The main agent decides which subagent to invoke based on the tool's description; responses flow back to the main agent, which synthesizes a final answer for the user.

The connected agents pattern enforces a maximum depth of 2: a main agent can have multiple sibling subagents, but subagents cannot themselves have subagents. Exceeding this depth produces an Assistant Tool Call Depth Error. The name field for a connected agent tool must use only letters and underscores — no spaces or special characters.

For production deployments, publish connected agents and the main agent separately as Agent Applications, each with its own stable endpoint and Entra identity. The newer Workflows API (2025-11-15-preview) is the emerging recommended approach for declarative multi-agent orchestration; for code-based graph topologies, LangGraph and the OpenAI Agents SDK integrate with the Foundry Responses API.

Key facts

  • ConnectedAgentTool requires: id (subagent ID), name (letters and underscores only), description (routing guidance for the main agent).
  • Maximum depth is 2: main agent → subagents; subagents cannot have their own subagents. Violation produces Assistant Tool Call Depth Error.
  • Connected agent responses go to the main agent only — the main agent synthesizes the final user-facing response.
  • Production pattern: publish each agent as an Agent Application with its own stable endpoint and Entra identity.
  • The Workflows API (2025-11-15-preview) is the newer recommended approach for declarative multi-agent orchestration.
  • LangGraph and OpenAI Agents SDK integrate with the Foundry Responses API for code-based graph topologies.

from azure.ai.projects import AIProjectClient

)

# Create specialized subagent

2.2.5 Build autonomous or semiautonomous workflows with safeguards and approval flow controls

Fully autonomous agents carry risk when they take irreversible or high-impact actions without human oversight. The Human-in-the-Loop (HITL) pattern, implemented via the AG-UI protocol, addresses this: the agent generates tool calls, the server sends approval requests to the client UI, the user approves or rejects, and the server proceeds accordingly. Two tool-level approval modes govern this: always_require (every invocation needs explicit human sign-off) and never_require (fully autonomous, no approval).

The Task Adherence guardrail (preview) provides automated reflection at the tool-call level, detecting when tool invocations deviate from user intent — wrong tool, incorrect inputs, or misaligned responses — and producing a signal to block the invocation or escalate to HITL review. Foundry guardrails apply input/output content filtering across four severity dimensions (Violence, Sexual, Self-harm, Hate/Unfairness) and include Spotlighting to mitigate prompt injection, including cross-prompt injection attacks (XPIA). If no guardrail is explicitly assigned to an agent, the model's default guardrail applies.

Network-level isolation is available for compliance-sensitive workflows: virtual network injection for hosted agents and private endpoints for prompt agents provide full network isolation. RBAC further restricts who can create, invoke, or modify agents: Foundry User for invocation, Foundry Project Manager for management.

Key facts

  • AG-UI protocol: implements HITL — agent generates tool calls → server sends approval request → user approves/rejects → server proceeds.
  • Tool approval modes: always_require (every invocation needs human approval) and never_require (fully autonomous).
  • Use always_require for irreversible or high-risk actions (deletions, payments, external emails).
  • Task Adherence (preview): automated guardrail detecting misaligned tool invocations — wrong tool, bad inputs, or intent mismatch.
  • Foundry guardrails filter: Violence, Sexual, Self-harm, Hate/Unfairness (four severity dimensions).
  • Spotlighting: guardrail technique to mitigate prompt injection and cross-prompt injection attacks (XPIA).
  • If no guardrail is assigned to an agent, the model's default guardrail applies automatically.
  • Network isolation: VNet injection (hosted agents), private endpoints (prompt agents).
# AG-UI Human-in-the-Loop pattern (requires AG-UI server-side library)
from agui import tool, ToolApprovalMode

@tool(approval_mode=ToolApprovalMode.ALWAYS_REQUIRE)
def delete_record(record_id: str) -> str:
    """Delete a customer record. ALWAYS requires human approval before execution."""
    return f"Record {record_id} deleted."

@tool(approval_mode=ToolApprovalMode.NEVER_REQUIRE)
def read_record(record_id: str) -> dict:
    """Read a customer record — read-only, safe to run autonomously."""
    return {"id": record_id, "name": "Example Customer"}
Key: Apply always_require approval to any tool whose action is irreversible or has external impact (deletions, financial transactions, outbound communications). Use never_require only for read-only, low-risk operations.

Exam tips & gotchas

  • Know the two HITL approval modes: always_require (human sign-off every invocation) vs. never_require (fully autonomous).
  • Task Adherence is a preview guardrail for automated tool-call alignment checking — distinct from content safety filtering.
  • Spotlighting mitigates XPIA (cross-prompt injection attacks) — a specific guardrail technique, not a general content filter.
  • If no guardrail is explicitly assigned, the default model guardrail applies — verify assignment when debugging unexpected content filter behavior.

Practice

Q1. An agent has a tool that sends an email to a customer. The team wants a human to approve every invocation. What approval mode should be set on that tool?
Answer: always_require. Why: always_require sends an approval request to the user interface via the AG-UI protocol for every invocation, regardless of context. This is appropriate for outbound communications that are external and potentially irreversible.
Q2. An agent is invoking the wrong tool for certain user requests even though the correct tool is available. Which preview guardrail signal would surface this deviation?
Answer: Task Adherence. Why: Task Adherence (preview) detects misaligned tool invocations — including wrong tool selection, incorrect inputs, and responses inconsistent with user intent — providing a signal to block or escalate.
Q3. A security audit finds that an agent's content guardrail is not blocking expected harmful content. The agent has no guardrail explicitly assigned. What is the active guardrail?
Answer: The model's default guardrail. Why: If no guardrail is explicitly assigned to an agent, the underlying model's default guardrail is automatically applied. Verify whether the default guardrail covers the required severity thresholds and assign a custom one if not.

2.3 Optimize and operationalize generative AI systems

2.3.1 Tune generation behavior, such as prompt engineering and adjusting model parameters

The system message is the single most impactful lever for shaping model output — it defines the model's persona, response style, scope constraints, and format requirements — and should be tuned before adjusting any numeric parameters. Prompt engineering techniques include role assignment ("You are a…"), few-shot examples (providing input/output pairs directly in the prompt), explicit format constraints ("Respond in JSON"), and chain-of-thought triggering ("Think step by step before answering").

Numeric generation parameters control output diversity and length. Temperature (range 0–2, default 1) governs randomness: lower values (0.1–0.3) produce deterministic, focused responses; higher values (0.7–1.0) increase creativity and diversity. top_p (nucleus sampling, 0–1) restricts generation to the smallest token set whose cumulative probability exceeds the threshold — an alternative randomness control that should not be adjusted simultaneously with temperature. max_tokens sets the hard upper limit on response length in tokens.

Structured output enforces valid JSON in the response: activate with response_format={"type": "json_object"} for any valid JSON, or {"type": "json_schema", "json_schema": {...}} for strict schema adherence in newer API versions. For code generation models, use lower temperature (0.0–0.2) and include language and style constraints in the system message.

Key facts

  • System message is the most impactful output-shaping lever; tune it before touching numeric parameters.
  • Temperature: range 0–2, default 1. Low (0.1–0.3) = deterministic; high (0.7–1.0) = creative.
  • top_p: range 0–1 (nucleus sampling). Alternative randomness control — do not adjust both temperature and top_p simultaneously.
  • max_tokens (max_completion_tokens): hard upper limit on response length; one token ≈ 4 English characters.
  • GPT-4-class context window: up to 128,000 tokens shared across prompt and response.
  • Structured output: response_format={"type": "json_object"} for any valid JSON; json_schema for strict schema enforcement.
  • Prompt engineering techniques: role assignment, few-shot examples, format constraints, chain-of-thought triggering.
import os
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_KEY"],
    api_version="2025-01-01-preview",
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": (
            "You are a concise financial analyst. "
            "Answer in exactly three bullet points."
        )},
        {"role": "user", "content": "What are the key risks of rising interest rates?"},
    ],
    temperature=0.3,                             # low randomness for factual consistency
    top_p=1.0,                                   # full nucleus; not adjusted alongside temp
    max_tokens=300,                              # cap response length
    response_format={"type": "json_object"},     # force valid JSON output
)
Gotcha: Never adjust both temperature and top_p simultaneously. They are alternative randomness controls — tuning both at once produces unpredictable compounding effects. Pick one to tune; leave the other at its default.

Exam tips & gotchas

  • Temperature range is 0–2 (default 1); top_p range is 0–1 — know the defaults and ranges.
  • Adjusting temperature AND top_p simultaneously is explicitly discouraged — a common gotcha on the exam.
  • The system message is the first thing to tune for output quality — not temperature or top_p.
  • GPT-4-class models share a context window (up to 128 k tokens) across prompt + response — large prompts leave less room for responses.

Why: Both parameters independently restrict the token distribution; combining them produces compounding, unpredictable effects. Best practice is to adjust one and leave the other at its default (temperature at 1, top_p at 1).
Q2. An agent is generating responses with too much variation — similar questions get very different answers. What parameter change would produce more consistent outputs?

Answer: {"type": "json_schema", "json_schema": {...}} for strict schema enforcement, or {"type": "json_object"} for any valid JSON. Why: JSON mode forces the model to emit syntactically valid JSON. Using json_schema additionally validates against a specific schema definition in newer API versions.

2.3.2 Implement model reflection, chain-of-thought evaluations, and self-critique loops

Chain-of-thought (CoT) prompting instructs the model to reason through intermediate steps before producing a final answer, significantly improving accuracy on multi-step tasks. Common triggers include the phrase "Think step by step before answering" or providing few-shot examples that include explicit reasoning traces. This technique is most effective for math, logic, and multi-condition decision problems where intermediate reasoning steps matter.

Reasoning models (the o1, o3, and o4-mini series) perform internal multi-step reasoning without needing explicit CoT prompting — they are optimized for math, code, and complex inference tasks. For standard GPT-class models, CoT must be explicitly triggered in the prompt. Self-critique loops implement reflection programmatically: a model generates an initial response, a second call asks the model to identify errors or improvements in that response, and a final call incorporates the critique. This can be implemented as a multi-turn conversation or separate API calls.

For RAG applications specifically, a self-grounding check after generation prompts the model to identify which retrieved chunks support each claim in its response. Claims without supporting chunks are surfaced as potential hallucinations for correction or removal before the response reaches the user.

Key facts

  • CoT triggering phrases: "Think step by step before answering" or few-shot examples with explicit reasoning traces.
  • Reasoning models (o1, o3, o4-mini series): perform internal multi-step reasoning; no explicit CoT needed — best for math, logic, code.
  • Self-critique loop: (1) generate initial response, (2) critique the response with a second LLM call, (3) revise based on critique.
  • Self-critique can be implemented as a multi-turn conversation or three sequential API calls.
  • Agentic reflection: agent calls a validation tool or evaluator function after generating output, then revises before responding to the user.
  • Task Adherence (Foundry guardrail): provides automated reflection at the tool-call level for agents.
  • RAG self-grounding check: after generation, prompt the model to map each claim to a retrieved chunk — unmapped claims indicate potential hallucination.
import os
from azure.ai.inference import ChatCompletionsClient
from azure.ai.inference.models import SystemMessage, UserMessage
from azure.core.credentials import AzureKeyCredential

client = ChatCompletionsClient(
    endpoint=os.environ["AZURE_INFERENCE_ENDPOINT"],
    credential=AzureKeyCredential(os.environ["AZURE_INFERENCE_KEY"]),
)

# Step 1: Initial response with chain-of-thought trigger
r1 = client.complete(
    model="gpt-4o",
    messages=[
        SystemMessage(content="Think step by step, then provide a final answer."),
        UserMessage(content="How many prime numbers exist below 50?"),
    ],
    temperature=0.0,
)
initial = r1.choices[0].message.content

# Step 2: Self-critique — identify any errors in the first response
r2 = client.complete(
    model="gpt-4o",
    messages=[
        SystemMessage(content="You are a meticulous reviewer. Identify errors and provide a corrected answer."),
        UserMessage(content=f"Review this response:\n{initial}"),
    ],
    temperature=0.0,
)
print(r2.choices[0].message.content)
Tip: Use reasoning models (o1, o3, o4-mini) when the task requires deep inference (math proofs, complex code logic). For straightforward generation, standard GPT-class models with CoT prompting are more cost-effective.

Exam tips & gotchas

  • CoT requires an explicit trigger for GPT-class models; reasoning models (o1, o3, o4-mini) perform internal reasoning automatically.
  • Self-critique loops are multi-call patterns — two or three API calls per user request; understand the tradeoff between quality and latency/cost.
  • Self-grounding check (for RAG): post-generation step, not pre-retrieval — it validates output alignment with retrieved chunks.
  • Task Adherence (agent-level) is Foundry's automated form of reflection for tool calls — it is a guardrail, not a prompt technique.

Practice

Q1. A team wants the model to catch its own arithmetic errors before responding. What pattern should they implement?
Answer: A self-critique loop. Why: The self-critique pattern generates an initial response, then makes a second LLM call asking the model to identify and correct errors, and optionally a third call to produce the revised final answer. This catches reasoning mistakes before the user sees the response.
Q2. An application needs to solve advanced mathematical proofs. Should it use a GPT-4o model with CoT prompting or an o3 reasoning model?
Answer: An o3 reasoning model. Why: Reasoning models (o1, o3, o4-mini) perform internal multi-step reasoning without explicit CoT prompting and are specifically optimized for math, logic, and complex inference. GPT-4o with CoT prompting can help but is less reliable for advanced proof tasks.
Q3. In a RAG application, how can a self-grounding check reduce hallucinations in the final response?
Answer: After generating a response, prompt the model to identify which retrieved chunks support each claim. Claims with no chunk support are flagged as potential hallucinations and can be removed or corrected before delivery. Why: Self-grounding provides a post-generation validation step that surfaces unsupported claims, enabling targeted correction rather than rewriting the entire response.

2.3.3 Set up observability by implementing tracing, token analytics, safety signals, and latency breakdowns

Foundry observability is built on OpenTelemetry (OTeL). Every model call, tool invocation, and agent decision generates a span within a distributed trace. These spans carry attributes such as model name, token counts, latency, and tool call results, and are exportable to Application Insights or any OTeL-compatible backend. Configure the Python stack with opentelemetry-sdk, azure-core-tracing-opentelemetry, and the azure-monitor-opentelemetry package (which bundles exporter and configuration).

To export to Application Insights: retrieve the connection string from project_client.telemetry.get_connection_string() and pass it to configure_azure_monitor(). The Foundry Tracing UI (portal → project → Tracing) provides a visual trace explorer — filter and inspect individual spans, view model inputs and outputs, token counts, latencies per span, and tool call results. Set AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true to capture actual prompt and response text in spans (disabled by default due to potential PII).

Token analytics are available in the Agent Monitoring Dashboard, broken down per model, per run, and over time — essential for cost attribution and quota planning. Safety signals (content filter hits, guardrail activations, XPIA detections) appear as span attributes in traces and as aggregate counts in the monitoring dashboard. Latency breakdowns use span start/end timestamps to identify which pipeline step — retrieval, model inference, function execution — contributes most to end-to-end latency.

Key facts

  • Foundry observability is built on OpenTelemetry (OTeL); every call becomes a span within a distributed trace.
  • Python OTeL packages: opentelemetry-sdk, azure-core-tracing-opentelemetry, azure-monitor-opentelemetry (all-in-one).
  • Connection string for Application Insights: project_client.telemetry.get_connection_string().
  • Set AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true to record prompt/response text in spans (disabled by default — may capture PII).
  • Foundry Tracing UI: portal → project → Tracing; filter and inspect spans with token counts, latencies, and tool call results.
  • Safety signals (filter hits, guardrail activations, XPIA detections) appear as span attributes and as dashboard aggregate counts.
  • Latency breakdowns: span timestamps identify which step (retrieval, inference, function) dominates end-to-end latency.
import os
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from azure.monitor.opentelemetry.exporter import AzureMonitorTraceExporter

# Configure OTeL to export spans to Application Insights
exporter = AzureMonitorTraceExporter(
    connection_string=os.environ["APPLICATION_INSIGHTS_CONNECTION_STRING"]
)
provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(exporter))
trace.set_tracer_provider(provider)

tracer = trace.get_tracer("rag-app")

with tracer.start_as_current_span("rag-pipeline") as span:
    span.set_attribute("user_query", "What is the refund policy?")
    # ... retrieval and generation calls happen here ...
    span.set_attribute("retrieved_chunks", 3)
    span.set_attribute("completion_tokens", 142)
    span.set_attribute("latency_ms", 850)
Gotcha: AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true records actual prompt and response text in traces. This can expose PII — it is disabled by default and should only be enabled in environments with appropriate data handling controls.

Exam tips & gotchas

  • Foundry tracing is OTeL-native — know the required Python packages and how to retrieve the Application Insights connection string.
  • Content recording is off by default because it may capture PII — know the environment variable that enables it.
  • Latency breakdowns require no special configuration — span timestamps are captured automatically; query them in Application Insights.
  • Safety signals (filter hits, XPIA) appear as span attributes — use Application Insights queries to aggregate them across runs.

Practice

Q1. A team wants to view full prompt and response text in the Foundry Tracing UI for debugging. What must they configure?
Answer: Set the environment variable AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true. Why: By default, traces capture span metadata but not content text. Enabling this variable opts in to full content recording. Teams should ensure appropriate data handling controls are in place since prompts may contain PII.
Q2. A Python application needs to send agent run spans to Application Insights. What is the correct way to retrieve the Application Insights connection string in Foundry?
Answer: Call project_client.telemetry.get_connection_string() and pass the result to configure_azure_monitor(). Why: The AIProjectClient exposes a telemetry sub-client that retrieves the Application Insights connection string configured for the Foundry project, ensuring you connect to the correct monitoring resource.
Q3. End-to-end latency for a RAG pipeline is high. How would you use observability tooling to identify the slowest step?
Answer: Inspect the distributed trace spans in the Foundry Tracing UI or Application Insights. Each span has start and end timestamps, enabling you to compare retrieval span duration vs. LLM inference span duration vs. function execution duration and pinpoint the bottleneck. Why: OTeL spans capture individual step latencies; comparing span durations identifies which component dominates end-to-end latency without code changes.

2.3.4 Orchestrate multiple models, flows, or hybrid LLM and rules engines

The Model Router is a deployable Foundry model that selects the optimal underlying LLM per turn based on query complexity, cost, and capability — exposed as a single named deployment endpoint. Applications call the model router deployment just as they would any other model; the router invisibly dispatches to the best underlying model. Three routing modes are available: Quality (frontier models for complex tasks), Balanced (mixed capability/cost), and Cost (fast, inexpensive models for classification or triage). The actual model selected is available in response.model.

For code-based graph topologies, Semantic Kernel and LangGraph both integrate with the Foundry Responses API, enabling different graph nodes to invoke different models, tools, or rule engines. A cost-efficient multi-model pattern uses a small model (e.g., gpt-4.1-mini) for intent classification and routing, then a larger frontier model only for tasks requiring deep reasoning or generation.

Hybrid LLM + rules engine patterns combine deterministic guards with LLM calls: keyword or regex checks before the LLM call block off-topic or prohibited queries; schema or business rule validation after the LLM call ensures outputs conform to downstream system requirements. The declarative Workflows API (2025-11-15-preview) supports visual composition of model steps, tool calls, and branching logic in the Foundry portal for teams who prefer a configuration-driven approach.

Key facts

  • Model Router: single deployment endpoint; auto-selects the optimal model per turn; no routing code in the application.
  • Model Router routing modes: Quality (frontier), Balanced (mixed), Cost (fast, cheap).
  • The selected model is returned in response.model — useful for cost attribution and debugging.
  • Model Router integrates with agents: assign a model router deployment as the agent's model; the router picks cheaper models for simple turns.
  • Semantic Kernel and LangGraph integrate with the Foundry Responses API for graph-based multi-model orchestration.
  • Hybrid pattern: deterministic guard (keyword/regex/schema) before LLM call → LLM generation → output validation against business rules.
  • Workflows API (2025-11-15-preview): declarative multi-agent and multi-model orchestration in the Foundry portal.
import os
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_KEY"],
    api_version="2025-01-01-preview",
)

def smart_complete(user_message: str) -> str:
    """Single call to model-router deployment; it auto-selects the best model."""
    response = client.chat.completions.create(
        model="model-router",                    # model router deployment name
        messages=[{"role": "user", "content": user_message}],
    )
    print(f"Routed to: {response.model}")        # reveals which model was chosen
    return response.choices[0].message.content

# Hybrid: keyword guard + LLM generation
BLOCKED = {"competitor", "lawsuit"}
def safe_complete(user_message: str) -> str:
    if any(kw in user_message.lower() for kw in BLOCKED):
        return "I cannot assist with that topic."
    return smart_complete(user_message)
Tip: When using Model Router with agents, assign the model router deployment as the agent's model. The router will automatically select cheaper models for greeting turns and frontier models for complex reasoning or tool-calling chains — reducing cost without changing code.

Exam tips & gotchas

  • Model Router requires no routing code in the application — the routing decision is made server-side by the Foundry service.
  • Know the three routing modes: Quality, Balanced, Cost — and which scenarios each targets.
  • response.model returns the actual model selected by the router — useful for cost attribution dashboards.
  • Hybrid LLM + rules: apply guards before the LLM call (reduce unnecessary compute) and after (validate output correctness).

Practice

Q1. A developer wants to reduce inference costs by automatically using cheaper models for simple classification turns and frontier models for complex reasoning — without writing routing logic. What Foundry feature enables this?
Answer: Model Router. Why: Model Router is a deployable Foundry model that selects the optimal underlying LLM per turn based on complexity, cost, and capability. The application calls a single deployment endpoint; all routing decisions are made server-side.
Q2. An application uses the Model Router deployment and needs to log which underlying model was selected for cost attribution. Where is this information available?
Answer: In response.model of the API response. Why: The Model Router returns the name of the actual underlying model selected in the model field of the response object. Log this value for per-turn cost attribution.
Q3. A customer service application must never send LLM calls for messages containing competitor names (a compliance rule). How should this be implemented in a hybrid LLM + rules pattern?
Answer: Add a keyword/regex guard before the LLM call. If any blocked term is detected, return a canned response immediately without invoking the model. Why: Deterministic pre-LLM guards eliminate unnecessary compute costs and ensure compliance rules are enforced before any model invocation, regardless of how the model might respond.
