Appendices — Quick Reference & Cheat Sheets

Fast-recall material for the final days before the exam. Skim these until the distinctions between similar Azure AI services are automatic.

A. Model selection quick-reference

Model typeBest forTrade-off
Large language model (LLM)Complex reasoning, broad knowledge, long context, high-quality generationHigher latency and cost per token
Small language model (SLM)Low-latency, high-volume, cost-sensitive, or edge/on-device tasksLess capable on hard reasoning
Reasoning modelMulti-step problem solving, planning, math/code with deliberate "thinking"Slower, more tokens spent reasoning
Multimodal modelInputs that mix text with images (or audio) — captioning, visual Q&ALarger, pricier than text-only
Embedding modelVectorizing text for vector/hybrid search and RAG groundingNot a generator — pair with an LLM
Image / video modelGenerating or editing images and video from prompts and reference mediaContent-safety and provenance apply
Foundry Tool / AI servicePrebuilt skills: speech, translation, document/content understanding, searchLess flexible than a raw model
Decider: Need a prebuilt capability (speech, OCR, translation)? → a Foundry Tool / AI service. Need to search your own data? → an embedding model + Azure AI Search. Everything else is a size/quality trade-off between SLM and LLM (or a reasoning model for hard multi-step work).

C. Which grounding / extraction service?

ServiceUse it forSignature output
Azure AI SearchIndexing your corpus and retrieving relevant chunks (keyword + vector + semantic) for RAGRanked search results / grounding data
Azure AI Document IntelligenceStructured extraction from forms/documents — OCR, layout, tables, key-value fieldsStructured JSON of fields & tables
Azure AI Content UnderstandingTurning any modality (doc, image, audio, video) into clean structured or markdown output via analyzersAnalyzer results (JSON / markdown)
Foundry Agent ServiceOrchestrating a model with tools, memory (threads), and knowledge sourcesAgent responses + tool calls
Azure AI Content SafetyScreening prompts and outputs (text & images) for harmful content and prompt-injectionCategory + severity verdicts
Decider: Retrieval over many docs → AI Search. Pull fields from a form → Document Intelligence. Normalize mixed media into grounding data → Content Understanding. Orchestrate tools + memory → Agent Service.

E. Responsible AI & safety layers

LayerControlsWhere
Content filtersHate, sexual, violence, self-harm categories at severity thresholds; prompt shields for jailbreak/indirect injection; protected-material detectionAzure AI Content Safety / model deployment filters
EvaluatorsGroundedness, relevance, coherence, fluency, similarity, and risk/safety scoresFoundry evaluations (SDK & portal)
AuditingTrace logging, provenance metadata, approval workflows for sensitive actionsTracing + Azure Monitor
Agent oversightHuman-in-the-loop approval, constraints, and tool-access restrictionsAgent configuration
Gotcha: Content filtering covers both the input prompt and the generated output. Indirect prompt injection can hide instructions inside retrieved documents or text embedded in images — prompt shields and grounding hygiene mitigate it.

G. "Which one when?" rapid-fire

Snap decisions

  • Need up-to-date answers grounded on your data? → RAG with Azure AI Search (don't fine-tune just for knowledge).
  • Need the model to take an action or call a system? → function calling / OpenAPI tool in an agent.
  • Combine keyword precision with semantic recall? → hybrid search, then semantic reranking.
  • Extract fixed fields from invoices/forms? → Document Intelligence.
  • Normalize PDFs, images, audio, video into grounding data? → Content Understanding analyzers.
  • Predictable latency at steady high volume? → provisioned (PTU) deployment.
  • Reduce hallucinations you can measure? → groundedness evaluator + stronger retrieval.
  • Voice agent? → Speech to text in, LLM reasoning, text to speech out (speech translation for other languages).
  • Block harmful prompts and outputs? → content filters + prompt shields on the deployment.
  • No secrets in code? → managed identity + Entra RBAC (keyless).

H. Glossary

Key terms

  • Azure AI Foundry — the platform (portal + SDK) for building, deploying, and managing AI apps and agents; organized into a Foundry resource with projects.
  • Foundry project — a workspace that owns model deployments, connections, agents, and evaluations.
  • Foundry Agent Service — managed service that runs agents (model + instructions + tools + threads).
  • Thread — the stateful conversation an agent uses to remember prior turns.
  • Tool schema — the name + JSON parameter definition that tells a model how to call a function/tool.
  • RAG — retrieval-augmented generation: retrieve relevant context and add it to the prompt before generating.
  • Embedding — a numeric vector representing text/media meaning, used for vector search.
  • Hybrid search — combining keyword (BM25) and vector search, often fused with RRF and re-ranked.
  • Semantic ranker — a re-ranking step that reorders top results by deeper relevance.
  • Skillset — an Azure AI Search enrichment pipeline (OCR, splitting, embedding, custom skills).
  • Integrated vectorization — generating embeddings inside the indexing pipeline automatically.
  • Content Understanding analyzer — a configured extractor that turns a modality into structured/markdown output.
  • PTU — provisioned throughput units: reserved capacity for predictable latency and cost.
  • TPM — tokens per minute, the rate-limit/quota unit for pay-per-token deployments.
  • Prompt shield — a filter that detects jailbreak and indirect prompt-injection attempts.
  • Groundedness — an evaluator measuring whether an answer is supported by the provided context.
  • Managed identity — an Entra identity for a resource so it can authenticate without stored secrets.
