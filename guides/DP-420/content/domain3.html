Domain 3 — Integrate an Azure Cosmos DB Solution 5–10%

Azure Cosmos DB rarely lives in isolation. This domain covers how you connect it to the wider Azure analytics and application ecosystem: replicating operational data into Microsoft Fabric OneLake with no-ETL mirroring, running large-scale analytics through the column-oriented analytical store and Azure Synapse Link, and reading or writing data with the Spark connector. It also covers the event-driven patterns built on the change feed and Azure Functions — denormalization, referential integrity, materialized aggregates, and archival — plus integrating Azure Event Hubs and Azure AI Search. Although only 5–10% of the exam, these topics reward you for knowing which integration path fits a scenario and which ones cost RUs versus which run RU-free.

3.1 Enable Azure Cosmos DB Analytical Workloads

3.1.1 Configure Azure Cosmos DB Mirroring for Microsoft Fabric

Mirroring in Microsoft Fabric continuously replicates an Azure Cosmos DB database into Fabric OneLake in near real-time, giving you a no-ETL path to analytics. The replica is stored in the open-source Delta Lake / Parquet format, so every Fabric engine — Power BI (Direct Lake), the SQL analytics endpoint, and Spark — can read it without you building or maintaining a pipeline.

The critical selling points for the exam: replication is free (Fabric compute for replication and OneLake storage cost nothing), it has no performance impact on your transactional workload, and it doesn't consume Request Units. Mirroring is GA and is now the recommended replacement for Azure Synapse Link.

Key facts

  • Data replicates into OneLake in near real-time as Delta/Parquet; no ETL and no RU consumption on the source account.
  • Continuous backup is a prerequisite — enable 7-day (free, recommended) or 30-day continuous backup before mirroring.
  • Each mirrored item auto-creates a read-only SQL analytics endpoint and a default Power BI semantic model; the source is viewable read-only in Data Explorer.
  • Replication compute and OneLake storage are free; you pay only for query compute (SQL, Power BI, Spark) against Fabric capacity.
  • Currently supports API for NoSQL only; not in sovereign clouds; multiple write regions are not supported.
  • Mirroring does not use the analytical store or change feed as its source — those remain independently available.
  • Nested data surfaces as JSON strings in the SQL endpoint; expand with OPENJSON, CROSS APPLY, or OUTER APPLY.
Key: Mirroring gives HTAP analytics with zero RU cost and zero transactional impact. If a scenario asks for near real-time BI on Cosmos DB data without ETL and without touching provisioned throughput, mirroring is the answer.

Exam tips & gotchas

  • Continuous backup is mandatory — mirroring won't start without it. 7-day is free.
  • NoSQL only, single write region only. Multi-write-region accounts can't be mirrored.
  • The SQL analytics endpoint is read-only — you cannot write to Cosmos DB from Fabric.
  • Mirroring replaces Synapse Link for new projects but is a separate mechanism from analytical store/change feed.

Practice

Q1. A team needs near real-time Power BI reporting over a Cosmos DB for NoSQL container but cannot afford any extra RU pressure on the production workload. What should they configure?
Answer: Azure Cosmos DB mirroring for Microsoft Fabric. Why: Mirroring replicates the data into OneLake in near real-time with no RU consumption and no transactional impact, and Power BI can read it in Direct Lake mode — no ETL pipeline required.
Q2. An engineer tries to enable mirroring but the option is unavailable. The account is API for NoSQL with a single write region. What prerequisite is most likely missing?
Answer: Continuous backup is not enabled. Why: Mirroring requires continuous backup (7-day or 30-day) on the Cosmos DB account; enabling 7-day continuous backup, which is free, satisfies the prerequisite.
Q3. Can an application write results back to Cosmos DB through the mirrored database's SQL analytics endpoint?
Answer: No. Why: The mirrored copy in OneLake and its SQL analytics endpoint are read-only; writes to the source must go through the Cosmos DB APIs/SDKs directly.

3.1.2 Choose Between Azure Cosmos DB Mirroring and the Azure Cosmos DB Spark Connector

Both bring Cosmos DB data into an analytics environment, but they solve different problems. Mirroring is fully managed and no-code: it replicates the whole database into OneLake for read-only analytics, free of RU cost. The Spark connector is code-first and can both read and write — but reading the transactional store (cosmos.oltp) consumes RUs.

Choose mirroring for BI and analytics with no pipeline to maintain. Choose the Spark connector when you need programmatic transformations, data-engineering/ML workflows, or the ability to write results back to Cosmos DB.

Key facts

  • Mirroring: managed, no-code, near real-time, read-only, free compute, no RU cost; best for BI/reporting.
  • Spark connector: code-first; the cosmos.oltp format reads/writes the transactional store and consumes RUs; best for data engineering, ML, and write-back.
  • The Spark cosmos.olap format reads the analytical store (via Synapse Link) — read-only and RU-free — an alternative to cosmos.oltp when you don't need the freshest transactional data.
  • Only the Spark connector can write data back to Cosmos DB; mirroring is one-way and read-only.

Exam tips & gotchas

  • Need write-back or custom transformations? Spark connector. Need managed, RU-free BI? Mirroring.
  • cosmos.oltp = transactional store, costs RUs. cosmos.olap = analytical store, RU-free, read-only.
  • Mirroring is one-directional into OneLake; don't pick it when the scenario needs to update Cosmos DB.

Practice

Q1. A data-engineering job must read Cosmos DB data, transform it in code, and write aggregated results back into a Cosmos DB container. Mirroring or Spark connector?
Answer: Spark connector. Why: Only the Spark connector supports write-back to the transactional store (cosmos.oltp); mirroring is read-only into OneLake.
Q2. A BI team wants Power BI dashboards over Cosmos DB with the least operational overhead and no RU impact. Which do you recommend?
Answer: Mirroring. Why: It is fully managed, no-code, RU-free, and feeds Power BI Direct Lake directly from OneLake with no pipeline to maintain.

3.1.3 Enable the Analytical Store on a Container

The analytical store is a fully isolated, auto-managed column store that mirrors your container's operational data in a format optimized for large analytical scans and aggregations. Because it is physically separate from the row-based transactional store, analytical queries have no impact on transactional performance and require no RU allocation.

You enable it per container by setting Analytical TTL (ATTL) to -1 (infinite retention) or a positive number of seconds; 0 disables it. Enabling analytical store depends on Azure Synapse Link being enabled at the account level first.

Key facts

  • Enable via Analytical TTL: -1 = keep all history; positive n = expire after n seconds; 0 = disabled.
  • Auto-sync propagates inserts/updates/deletes to the column store, usually within ~2 minutes (up to ~5 min for shared-throughput databases with many containers).
  • No RU cost and no transactional impact — the column store is billed separately (storage + analytical read/write operations).
  • Turning on Synapse Link at the account does not auto-enable analytical store; you set ATTL on each container.
  • Configure via portal, Azure CLI (--analytical-storage-ttl), PowerShell (-AnalyticalStorageTtl), ARM, or SDK (AnalyticalStoreTimeToLiveInSeconds / analytical_storage_ttl).
  • Data tiering: set ATTL ≥ transactional TTL to keep longer history cheaply in analytical store while hot data expires from the transactional store.
  • Schema limits: 1000 properties, 127 nesting levels; columns can't be removed; no schema reset.
# Create a Synapse Link-enabled container with analytical store (infinite retention)
container = db.create_container(
    id="orders",
    partition_key=PartitionKey(path="/customerId"),
    analytical_storage_ttl=-1   # -1 = keep all history; positive n = expire after n seconds
)
Tip: ATTL is set at the container level only. Transactional TTL can be per-item or per-container, but analytical TTL is container-wide.

Exam tips & gotchas

  • ATTL -1 = infinite, 0 = disabled, positive n = seconds of retention. Memorize these three values.
  • Analytical store adds no RU cost — distractors claiming it consumes provisioned throughput are wrong.
  • Enabling Synapse Link on the account is a prerequisite but is not enough on its own; set ATTL per container.
  • For NoSQL, full-fidelity vs well-defined schema is chosen when Synapse Link is first enabled and can't be changed later.

Practice

Q1. You want operational data to expire from the transactional store after 30 days but retain two years of history for analytics. How do you configure TTL?
Answer: Set transactional TTL to 30 days and analytical TTL to two years (ATTL ≥ TTTL). Why: Analytical TTL controls the column store's retention independently, so history persists cheaply in analytical store after it expires from the transactional store.
Q2. After enabling Synapse Link on the account, a developer expects analytical queries to work but the analytical store is empty. What's missing?
Answer: Analytical store wasn't enabled on the container — analytical TTL is still unset. Why: Enabling Synapse Link at the account level doesn't turn on analytical store; you must set analytical TTL (-1 or a positive value) on each container.
Q3. Does enabling analytical store increase the RU/s you must provision for transactional queries?
Answer: No. Why: The analytical store is isolated and billed separately (storage plus analytical operations); auto-sync doesn't consume the container's provisioned RUs.

3.1.4 Enable a Connection to an Analytical Store and Query from Azure Synapse Spark or Azure Synapse SQL

Azure Synapse Link is the connection that lets Azure Synapse Analytics query the analytical store in near real-time. After enabling Synapse Link on the account and analytical store on the containers, you connect the Cosmos DB database to a Synapse workspace (creating a linked service) and query from either Synapse Spark or the serverless SQL pool.

Both runtimes are read-only against the analytical store — only auto-sync writes to it — and neither consumes RUs.

Key facts

  • Workflow: enable Synapse Link (account) → enable analytical store (container) → connect the DB to a Synapse workspace → query from Spark or serverless SQL.
  • Spark: read the analytical store with spark.read.format("cosmos.olap") into a DataFrame, or create a Spark table using cosmos.olap.
  • Serverless SQL: query with OPENROWSET(PROVIDER = 'CosmosDB', ...); auto-updating views can also be created.
  • DataFrames cache a snapshot at creation; Spark tables re-read the latest state on every query.
  • Contributor role to enable Synapse Link at account level; Operator to enable it on containers.
  • Both runtimes read only; RU-free. No writes to analytical store except auto-sync.
-- Synapse serverless SQL pool querying the analytical store (no RU cost)
SELECT TOP 100 *
FROM OPENROWSET(
    PROVIDER = 'CosmosDB',
    CONNECTION = 'Account=myaccount;Database=retail',
    OBJECT = 'orders'
) AS orders;

Exam tips & gotchas

  • Spark reads analytical store with cosmos.olap; serverless SQL uses OPENROWSET with PROVIDER = 'CosmosDB'.
  • Analytical store is read-only from Synapse — you can't UPDATE/DELETE it directly.
  • Contributor (account) vs Operator (container) — know which role each step needs.

Practice

Q1. Which Synapse serverless SQL construct reads the Cosmos DB analytical store without impacting the transactional workload?
Answer: OPENROWSET with PROVIDER = 'CosmosDB'. Why: It reads the isolated analytical store over Synapse Link, so there's no RU cost and no transactional impact.
Q2. A Spark DataFrame created from the analytical store doesn't reflect data changed after it was created. Why, and how do you always get the latest?
Answer: A DataFrame caches a snapshot of the metadata at creation. Why: Create a Spark table with cosmos.olap instead — Spark tables re-read the latest analytical store state on each query.

3.1.5 Perform a Query Against the Transactional Store from Spark

When you need the freshest data or capabilities the analytical store can't provide, Spark can read the transactional store directly using the cosmos.oltp format. Unlike cosmos.olap, reading the transactional store consumes provisioned RUs and can affect OLTP performance, so use it deliberately.

Key facts

  • spark.read.format("cosmos.oltp") reads the transactional store directly — it consumes RUs.
  • cosmos.olap (analytical store) is RU-free and read-only; cosmos.oltp (transactional) costs RUs but returns live data and supports writes.
  • Structured streaming from the change feed uses cosmos.oltp.changeFeed with options like spark.cosmos.changeFeed.startFrom and spark.cosmos.changeFeed.mode.
  • Choose cosmos.oltp for freshest data / write-back; cosmos.olap for RU-free analytics on synced data.
# Read the transactional store directly (consumes RUs)
df = (spark.read.format("cosmos.oltp")
      .option("spark.synapse.linkedService", "myCosmosLinkedService")
      .option("spark.cosmos.container", "orders")
      .load())
df.show(10)
Gotcha: cosmos.oltp reads hit the transactional store and burn RUs. If a scenario stresses "no impact on the transactional workload," the intended answer is the analytical store (cosmos.olap), not cosmos.oltp.

Exam tips & gotchas

  • cosmos.oltp = transactional store, costs RUs, live data. cosmos.olap = analytical store, RU-free, near real-time.
  • Change-feed streaming source is cosmos.oltp.changeFeed.

Practice

Q1. A Spark job must read the most up-to-the-second state of a container. Which format, and what's the trade-off?
Answer: cosmos.oltp. Why: It reads the transactional store for live data, but it consumes provisioned RUs and can affect transactional performance (the analytical store lags by up to ~2 minutes and is RU-free).
Q2. Which Spark format lets you consume the change feed as a structured-streaming source?
Answer: cosmos.oltp.changeFeed. Why: It streams incremental changes from the transactional store's change feed, configurable with start position and mode options.

3.1.6 Write Data Back to the Transactional Store from Spark

Spark can also ingest data into Cosmos DB. Writes always go through the transactional store using the cosmos.oltp format with an append mode; if Synapse Link is enabled, the new inserts/updates/deletes then auto-sync to the analytical store. Writing consumes RUs on the target container.

Key facts

  • Write with df.write.format("cosmos.oltp").mode("append") (Scala SaveMode.Append).
  • All ingestion goes through the transactional store; analytical store is updated only by auto-sync afterward.
  • Writing consumes RUs provisioned on the target container/shared database and impacts transactional performance.
  • Streaming writes use .writeStream.format("cosmos.oltp") with a checkpointLocation.
# Write a DataFrame into a Cosmos DB container (through the transactional store; consumes RUs)
(dataframe.write.format("cosmos.oltp")
    .option("spark.synapse.linkedService", "myCosmosLinkedService")
    .option("spark.cosmos.container", "orders")
    .mode("append")
    .save())

Exam tips & gotchas

  • You can only write to Cosmos DB via cosmos.oltp — you cannot write to the analytical store (cosmos.olap) directly.
  • Writes cost RUs; the analytical copy updates automatically via auto-sync, not by a separate write.

Practice

Q1. A Spark job writes transformed data back into a Cosmos DB container. Which store receives the write, and what updates the analytical store?
Answer: The transactional store receives the write (via cosmos.oltp); the analytical store is updated automatically by auto-sync. Why: All ingestion flows through the transactional store — you can't write directly to the analytical column store.
Q2. Does writing from Spark to a container consume Request Units?
Answer: Yes. Why: Writes hit the transactional store and consume RUs provisioned on the target container or shared database, unlike RU-free analytical reads.

3.1.7 Implement Change Data Capture in the Azure Cosmos DB Analytical Store

Change data capture (CDC) in the analytical store gives you a continuous, incremental feed of inserted, updated, and deleted data — consumed through Azure Synapse or Azure Data Factory mapping data flows with a no-code experience. Because it reads the analytical store, CDC doesn't consume RUs, doesn't affect transactional workloads, and offers lower latency and TCO.

Unlike the transactional change feed, analytical CDC natively captures deletes and TTL removals and manages checkpoints internally (GLSN-based), so each change appears exactly once with no records missed.

Key facts

  • Consumed via ADF / Synapse mapping data flows to any supported sink; no RU cost and no transactional impact.
  • Captures deletes, intermediate updates, and TTL operations (TTL treated as a delete); use {_rid} as the sink key to reflect updates/deletes.
  • Internally managed checkpoints (GLSN, not _ts): exactly-once, none missed; no fixed retention limit for availability.
  • Start options: from beginning, from a timestamp, or from now.
  • Filter by operation type (Insert/Update/Delete/TTL); apply filters, projections, transformations via a source query pushed down to the column store.
  • Multiple independent CDC processes can run in parallel with different sinks and transformations.
-- Source query for analytical-store CDC: filter, project, and transform (pushed to column store)
SELECT ProductId, Product, Segment,
       concat(Manufacturer, '-', Category) AS ManufacturerCategory
FROM c
WHERE Category = 'Urban'
Key: Analytical-store CDC is the RU-free way to capture deletes and intermediate updates. If a scenario needs a delete-aware incremental feed without touching provisioned throughput, it's analytical CDC — not the transactional change feed (latest-version mode misses deletes).

Exam tips & gotchas

  • Analytical CDC captures deletes and TTL removals natively; transactional change feed latest-version mode does not.
  • No RU cost, internally managed checkpoints (exactly-once), no fixed retention limit.
  • Consumed through ADF/Synapse data flows; use {_rid} as the sink key column for update/delete propagation.

Practice

Q1. You need an incremental feed that includes deletes and TTL removals, writing to a data lake, without consuming RUs. Which capability fits?
Answer: Change data capture in the analytical store (via ADF/Synapse data flows). Why: It natively captures deletes and TTL operations, runs RU-free off the analytical store, and manages checkpoints internally.
Q2. Why is analytical CDC more reliable than a custom _ts-based incremental read?
Answer: It uses internally managed GLSN-based checkpoints, guaranteeing each change is captured exactly once. Why: Auto-sync doesn't guarantee _ts ordering, so _ts filters can miss records; the GLSN checkpoint approach doesn't rely on _ts.

3.1.8 Implement Time Travel in Warehouse in Microsoft Fabric

Time travel in a Microsoft Fabric Warehouse (and SQL analytics endpoint) lets you query data as it existed at a past point in time using the T-SQL query hint OPTION (FOR TIMESTAMP AS OF '...'). For a mirrored Cosmos DB database whose data lands in a Fabric warehouse/endpoint, this enables stable reporting, auditing, and historical comparisons over prior versions of the data.

Key facts

  • Syntax: OPTION (FOR TIMESTAMP AS OF 'YYYY-MM-DDTHH:MM:SS[.fff]'); applies to the whole SELECT (all joined tables), once per statement.
  • Results are read-only — INSERT/UPDATE/DELETE aren't allowed with the hint.
  • Retention is automatic and configurable from 1 to 120 days (default 30); you can only travel within the retention window.
  • Timestamps are UTC only, max 3 fractional-second digits, and must be deterministic.
  • Returns the latest table schema; a point-in-time query before a schema change works only if referenced columns already existed then.
  • Table-level history is also available via CLONE TABLE.
-- Query a warehouse table as it existed at a past UTC timestamp
SELECT *
FROM dbo.orders
OPTION (FOR TIMESTAMP AS OF '2026-03-13T19:39:35.280');

Exam tips & gotchas

  • Default retention is 30 days; configurable 1–120. Outside the window, time travel fails.
  • UTC only, up to 3 fractional-second digits; the hint is read-only and used once per SELECT.
  • Time travel returns the current schema — reference only columns that existed at that past time.

Practice

Q1. An analyst must reproduce last night's report from a Fabric Warehouse while ETL is running. Which feature and syntax do they use?
Answer: Time travel with OPTION (FOR TIMESTAMP AS OF '...'). Why: It returns the data as it existed at the specified UTC timestamp, giving stable reporting even while background ETL modifies the same tables.
Q2. A time-travel query to a timestamp 200 days ago fails. The default retention hasn't been changed. Why?
Answer: The point in time is outside the retention window. Why: Default retention is 30 days (configurable up to 120); you can only time-travel within the configured retention period.

3.2 Implement Solutions Across Services

3.2.1 Integrate Events with Other Applications by Using Azure Functions and Azure Event Hubs

The Azure Functions trigger for Azure Cosmos DB fires on each change in a container's change feed, giving you the simplest way to build reactive, serverless event pipelines. It's built on the change feed processor, so you get automatic checkpointing, at-least-once delivery, and dynamic scaling without managing worker infrastructure.

A common cross-service pattern chains this to Azure Event Hubs: the Function reads each change and publishes an event to Event Hubs (via an output binding), decoupling Cosmos DB from downstream applications that consume the stream.

Key facts

  • The Functions trigger reacts to the change feed automatically; no worker infrastructure to manage (uses the change feed processor).
  • Requires a monitored container and a lease container (holds processing state for scaling; /id partition key; auto-create with CreateLeaseContainerIfNotExists).
  • Chain to Event Hubs with an output binding to fan events out to other applications; Event Hubs retains data up to ~90 days.
  • Supports latest-version mode (default) and all-versions-and-deletes mode (needs continuous backups; .NET isolated worker).
  • The Cosmos DB trigger currently supports API for NoSQL only; you can develop/test locally with the emulator.
// Azure Function triggered by the Cosmos DB change feed, forwarding events to Event Hubs
[Function("OrdersToEventHub")]
[EventHubOutput("orders-events", Connection = "EventHubConnection")]
public string Run(
    [CosmosDBTrigger(
        databaseName: "retail",
        containerName: "orders",
        Connection = "CosmosConnection",
        LeaseContainerName = "leases",
        CreateLeaseContainerIfNotExists = true)] IReadOnlyList<Order> changes)
{
    return JsonSerializer.Serialize(changes);
}

Exam tips & gotchas

  • The trigger needs a lease container for state/scaling — a monitored container alone isn't enough.
  • Cosmos DB Functions trigger = API for NoSQL only.
  • Event Hubs decouples producers/consumers for high-throughput streaming; pair it with the change feed for lambda architectures.

Practice

Q1. A solution must push every Cosmos DB item change to several downstream systems via a high-throughput event stream. What's a clean serverless design?
Answer: An Azure Functions Cosmos DB trigger that publishes each change to Azure Event Hubs. Why: The trigger reliably reads the change feed with automatic scaling/checkpointing, and Event Hubs fans the events out to multiple decoupled consumers.
Q2. Why does an Azure Functions Cosmos DB trigger require a lease container?
Answer: The lease container stores change-feed processing state, enabling checkpointing and dynamic scaling across function instances. Why: Without it, the trigger can't track progress or coordinate parallel processing.

3.2.2 Denormalize Data by Using Change Feed and Azure Functions

Cosmos DB data models favor denormalization — duplicating data to optimize reads. The change feed plus Azure Functions is the recommended way to keep those duplicated copies in sync: when a source item changes, a Function propagates the update to the embedded/referenced copies in other items or containers.

Key facts

  • A change to a source item triggers a Function that updates denormalized copies across items, partitions, or containers.
  • Example: propagate a changed product name into every order document that embedded it.
  • Change-feed replication guarantees only eventual consistency; monitor lag with the change feed estimator.
Tip: Denormalization trades write amplification (fan-out updates) for cheaper, single-read queries. The change feed is what makes maintaining those duplicates practical.

Exam tips & gotchas

  • Use the change feed to keep denormalized copies consistent — expect eventual consistency, not immediate.
  • Fan-out updates increase write cost; that's the accepted trade-off for fast reads.

Practice

Q1. Product names are embedded (denormalized) in thousands of order documents. When a product is renamed, how do you keep the orders current?
Answer: Use the change feed with an Azure Function that, on a product change, updates the embedded product name in the affected order documents. Why: The change feed reliably surfaces the source change and the Function fans the update out to the denormalized copies (eventually consistent).
Q2. What consistency guarantee does change-feed-based denormalization provide?
Answer: Eventual consistency. Why: There is a propagation delay between the source write and the update of the duplicated copies; monitor lag with the change feed estimator.

3.2.3 Enforce Referential Integrity by Using Change Feed and Azure Functions

Cosmos DB has no foreign keys, so referential integrity is enforced in the application layer. The standard approach is the change feed plus Azure Functions: when a referenced item changes or is deleted, a Function cascades the necessary updates or cleanup to related items to keep references valid.

Key facts

  • No foreign keys — integrity is enforced by application logic reacting to the change feed.
  • Latest-version mode doesn't capture deletes. Use a soft-delete flag (e.g., deleted: true, optionally with TTL) so the deletion appears as an update — or use all versions and deletes mode (requires continuous backups) to capture real deletes.
  • Order is guaranteed within a partition key value, not across partition keys — choose a partition key that preserves meaningful ordering for cascaded events.
Gotcha: A hard delete vanishes from the change feed in latest-version mode, so cascade logic never fires. Model deletes as a soft-delete flag (or switch to all-versions-and-deletes mode) to enforce referential cleanup.

Exam tips & gotchas

  • No foreign keys in Cosmos DB — integrity is application-enforced via change feed + Functions.
  • Soft-delete flags make deletes visible in latest-version mode; all-versions-and-deletes mode needs continuous backups.
  • Ordering holds only within a partition key — pick the key so related events stay in order.

Practice

Q1. When a customer is deleted, related order documents must be cleaned up. Using the latest-version change feed, why might the cleanup never run, and how do you fix it?
Answer: Latest-version mode doesn't emit deletes, so a hard delete never appears in the feed. Why: Use a soft-delete flag (set deleted: true, optionally with TTL) so the deletion shows up as an update the Function can act on — or use all-versions-and-deletes mode with continuous backups.
Q2. Why does the partition key matter when cascading referential updates via the change feed?
Answer: Ordering is guaranteed only within a partition key value. Why: If related events must be processed in sequence, they should share a partition key so the change feed delivers them in the correct order.

3.2.4 Aggregate Data by Using Change Feed and Azure Functions, Including Reporting

The change feed lets you maintain materialized views and pre-aggregated summaries for fast reporting. An Azure Function processes each change, updates a running aggregate (counts, totals, leaderboards), and persists it back to Cosmos DB (or another store), so reports avoid expensive full-container scans at read time.

Key facts

  • Build materialized views / real-time aggregates (e.g., leaderboards, running totals) from the change feed and persist them back for cheap reads.
  • Processing is per-partition ordered and at-least-once, so design aggregate updates to be idempotent.
  • Results are eventually consistent with the source container.
Tip: Because the change feed processor is at-least-once, an item can be delivered more than once. Make aggregate updates idempotent (e.g., recompute from source values) so retries don't double-count.

Exam tips & gotchas

  • Pre-aggregating via the change feed turns expensive read-time scans into cheap point reads of a materialized view.
  • At-least-once delivery ⇒ make aggregation idempotent to avoid double counting.

Practice

Q1. A gaming app needs a real-time leaderboard without scanning all score documents on every read. What pattern fits?
Answer: Use the change feed with an Azure Function to maintain a materialized leaderboard, writing aggregated scores back to Cosmos DB. Why: Reads then hit a small pre-aggregated view instead of scanning the full container.
Q2. Why must change-feed aggregation logic be idempotent?
Answer: The change feed processor guarantees at-least-once delivery, so a change can be processed more than once. Why: Idempotent updates ensure retries don't double-count and corrupt the aggregate.

3.2.5 Archive Data by Using Change Feed and Azure Functions

The change feed enables application-level data tiering and archival: keep "hot" data in Cosmos DB and age "cold" data out to cheaper storage such as Azure Blob Storage. An Azure Function reads the change feed and copies items to the archive; combined with TTL, hot data can then expire from the transactional store after it's safely archived.

Key facts

  • Read the change feed with a Function to copy data to cheaper cold storage (e.g., Azure Blob Storage).
  • Pair with TTL to expire hot data from the transactional store after archival, controlling cost.
  • The same data-movement pattern also powers zero-downtime migrations and syncing caches, search indexes, and data warehouses.

Exam tips & gotchas

  • Archival = change feed Function → Blob Storage, then TTL to remove hot data. Order matters: archive before expiring.
  • The change feed is the general-purpose engine for data movement (migration, cache/search/warehouse sync, tiering).

Practice

Q1. To reduce Cosmos DB storage costs, older records must be moved to Blob Storage and removed from the container. Outline a change-feed-based approach.
Answer: An Azure Function reads the change feed and writes items to Azure Blob Storage; set a TTL so the archived items later expire from the transactional store. Why: This tiers cold data to cheap storage while keeping hot data lean, and TTL automates cleanup after archival.
Q2. Besides archival, name two other data-movement uses of the change feed.
Answer: Zero-downtime migration to another container/account, and keeping a cache, search index, or data warehouse in sync. Why: The change feed is a general mechanism for propagating changes to external systems.

3.2.6 Implement Azure AI Search for an Azure Cosmos DB Solution

Azure AI Search makes Cosmos DB content full-text and vector searchable by pulling it in with an indexer. The workflow has three parts: create a data source pointing at the Cosmos DB account/database/container, create a search index, and create an indexer that reads the source and populates the index. (Cosmos DB indexing and AI Search indexing are separate concepts.)

Key facts

  • Data source uses "type": "cosmosdb" with a connection string (AccountEndpoint;AccountKey;Database) and a container; the indexer runs on creation and can be scheduled or run on demand.
  • Change detection: HighWaterMarkChangeDetectionPolicy on _ts enables incremental indexing of new/changed documents.
  • Deletion detection: only soft-delete is supported (SoftDeleteColumnDeletionDetectionPolicy with a Boolean flag such as isDeleted).
  • An optional data-source query can flatten nested JSON, project fields, join arrays, or filter — but DISTINCT and GROUP BY aren't supported (they break continuation-token pagination).
  • Auth via full-access key or a managed identity assigned Cosmos DB Account Reader + Cosmos DB Built-in Data Reader; the container needs the default Consistent automatic indexing policy.
  • Default document key is _rid (renamed rid, Base64-encoded) for partitioned collections.
{
  "name": "orders-cosmosdb-ds",
  "type": "cosmosdb",
  "credentials": {
    "connectionString": "AccountEndpoint=https://myacct.documents.azure.com;AccountKey=<key>;Database=retail"
  },
  "container": { "name": "orders" },
  "dataChangeDetectionPolicy": {
    "@odata.type": "#Microsoft.Azure.Search.HighWaterMarkChangeDetectionPolicy",
    "highWaterMarkColumnName": "_ts"
  },
  "dataDeletionDetectionPolicy": {
    "@odata.type": "#Microsoft.Azure.Search.SoftDeleteColumnDeletionDetectionPolicy",
    "softDeleteColumnName": "isDeleted",
    "softDeleteMarkerValue": "true"
  }
}

Exam tips & gotchas

  • Incremental indexing relies on the _ts high-water-mark policy; deletions require a soft-delete flag (hard deletes aren't detected).
  • DISTINCT and GROUP BY in the data-source query aren't supported.
  • Managed identity path needs Cosmos DB Account Reader + Built-in Data Reader roles.

Practice

Q1. An AI Search indexer over Cosmos DB should only reprocess new and changed documents on each run. What must the data source specify?
Answer: A HighWaterMarkChangeDetectionPolicy using the _ts column. Why: It's the only supported change-detection policy for Cosmos DB indexers and lets the indexer incrementally pick up items whose _ts advanced.
Q2. Deleted Cosmos DB items still appear in search results. What's the likely cause and fix?
Answer: No deletion detection is configured; hard deletes aren't detected. Why: Add a SoftDeleteColumnDeletionDetectionPolicy with a Boolean flag (e.g., isDeleted = true) so the indexer removes those documents from the index.
Q3. Why does a data-source query using GROUP BY fail with the Cosmos DB indexer?
Answer: DISTINCT and GROUP BY aren't supported because they're incompatible with the continuation tokens AI Search uses for pagination. Why: The indexer must fully enumerate results via SQL pagination, which those clauses break.
