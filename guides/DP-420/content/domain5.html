Maintenance is the largest single domain on DP-420, and it spans the whole operational lifecycle of a running account. You will monitor and troubleshoot with Azure Monitor metrics and resource logs (Normalized RU Consumption, server-side latency, 429s, and per-partition storage and RU data); protect data with backup and restore (periodic versus continuous point-in-time restore); lock down the account with layered security (encryption at rest, network access, control-plane and data-plane Azure RBAC, CORS, Key Vault, customer-managed keys, and Always Encrypted); move data in and out with the right data-movement tool (SDK bulk, Data Factory/Synapse, Kafka, Stream Analytics, Spark, IoT Hub); and operate repeatably with a DevOps process (declarative ARM/Bicep versus imperative CLI/PowerShell). 5.1 Monitor and Troubleshoot an Azure Cosmos DB Solution

5.1.1 Evaluate Response Status Code and Failure Metrics

Azure Monitor collects platform metrics for every Cosmos DB account automatically — no configuration, one-minute granularity, and short-term retention out of the box. The most useful failure signal is the Total Requests metric split by the StatusCode dimension, which lets you see exactly which HTTP responses your account is returning and how often.

Server-side telemetry (throughput, storage, availability, latency, consistency) answers "is the service healthy?"; client-side diagnostics (request charge, activity ID, substatus, diagnostic string) answer "why did this request fail?".

Key facts

  • Metrics are collected with no setup at 1-minute granularity; dimension values such as container names are case-insensitive.
  • Split Total Requests by StatusCode to count failures over time.
  • Status codes to know: 429 (request rate too large / RU budget exceeded), 401/403 (auth or firewall), 404 (not found), 449 (retriable transient write conflict), 410 (partition split — retry), 503 (service unavailable).
  • A 429 is expected under bursts; the SDK auto-retries, but sustained 429s mean you need more RU/s or a better partition key.
  • Client diagnostics expose activity ID, substatus code, and a diagnostics string that pinpoints the failing region/partition.
// Count data-plane responses by status code over the last hour
CDBDataPlaneRequests
| where TimeGenerated > ago(1h)
| summarize Requests = count() by StatusCode
| order by Requests desc
Key: A rising 429 count is the number-one troubleshooting signal — it means at least one physical partition ran out of its per-second RU budget.

Exam tips & gotchas

  • 429 = throttling (add RU/s or fix a hot partition); 449 = retriable write conflict (safe to retry); don't confuse them.
  • Metrics need no diagnostic setting; resource logs do — that distinction is a common trap.
  • A 403 can be a firewall/network block, not just bad credentials.

Practice

Q1. Users report intermittent failures during traffic spikes and the app logs show HTTP 429. What does this indicate and what is the first remediation?
Answer: Requests are being rate-limited because provisioned RU/s were exceeded on a partition. First remediation: increase throughput (or enable autoscale) and/or fix a hot partition key. Why: 429 means the RU budget for that second on a physical partition was exhausted.
Q2. You want to chart how many of each HTTP status code your account returns without enabling any logging. What do you use?
Answer: The Total Requests metric in Azure Monitor Metrics, split by the StatusCode dimension. Why: Metrics are collected automatically; only resource logs require a diagnostic setting.
Q3. A write returns 449 sporadically under concurrency. Should the client fail the operation?
Answer: No — 449 is a retriable transient write conflict; retry the operation. Why: 449 signals a concurrent metadata/write conflict that typically succeeds on retry.

5.1.2 Monitor the Normalized RU Consumption Metric by Using Azure Monitor

Normalized RU Consumption is a 0–100% metric that reports how close you are to fully using provisioned throughput. Crucially, it is the maximum utilization across all partition key ranges (and regions) — not an average — so it exposes a single hot partition even when the account as a whole looks idle.

Key facts

  • Value 0–100%; 100% means at least one physical partition's per-second RU budget is fully consumed and further requests to it are throttled (429).
  • Because it's a MAX, a single hot partition can pin it at 100% while total account RU usage is low → classic skew signal.
  • Split by PhysicalPartitionId and Region to locate the hot partition.
  • Sustained near-100% with 429s → scale up, switch to autoscale, or redesign the partition key.
// Correlate normalized RU with per-logical-partition RU spend
CDBPartitionKeyRUConsumption
| where TimeGenerated > ago(1h)
| summarize TotalRU = sum(RequestCharge) by PartitionKey
| top 10 by TotalRU desc
Tip: If Normalized RU Consumption is high but Total Request Units is low, you have a hot partition — investigate data/access skew rather than adding throughput blindly.

Exam tips & gotchas

  • It's a maximum across partitions, not an average — the exam loves this distinction.
  • Adding RU/s does not help a hot partition much; each physical partition gets an equal slice of total RU/s.
  • Split by PhysicalPartitionId to prove skew.

Practice

Q1. Normalized RU Consumption sits at 100% and clients see 429s, yet the account is provisioned well above its average total RU usage. What is the likely cause?
Answer: A hot logical/physical partition. Why: The metric is the maximum utilization across partitions, so one overloaded partition can hit 100% while total usage is low.
Q2. Which dimension do you split Normalized RU Consumption by to identify which partition is hot?
Answer: PhysicalPartitionId (and Region for multi-region). Why: It pinpoints the specific partition and location driving the utilization.

5.1.3 Monitor Server-Side Latency Metrics by Using Azure Monitor

The server-side latency metric measures time spent inside the Cosmos DB service and deliberately excludes network round-trip. Comparing server-side latency to the total latency your client observes tells you whether a slowdown is the database's fault or the network/SDK's.

Key facts

  • Split by operation type, region, and connection mode.
  • High server-side latency → cross-partition/fan-out queries, unindexed queries, or large response payloads.
  • Low server-side latency but high client latency → network, SDK config, gateway vs direct mode, or client-side resource pressure.
  • Use it alongside query metrics and RU charge to tune the offending query.
// Find the slowest data-plane operations server-side
CDBDataPlaneRequests
| where TimeGenerated > ago(1h)
| summarize p99 = percentile(DurationMs, 99), reqs = count() by OperationName
| order by p99 desc

Exam tips & gotchas

  • Server-side latency excludes network — don't blame the service for client/network latency.
  • Direct mode usually lowers latency versus gateway mode for high-throughput apps.
  • Cross-partition queries and unindexed scans are the usual server-side latency culprits.

Practice

Q1. Clients report ~120 ms request latency, but the server-side latency metric is ~4 ms. Where should you look?
Answer: The network/SDK/client — not the service. Why: Server-side latency excludes network time, so a large gap points to client region distance, gateway mode, or client resource pressure.
Q2. Server-side latency spikes for one query type. What database-side factors most likely explain it?
Answer: Cross-partition fan-out, missing indexes causing scans, or large response sizes. Why: These force more work inside the engine, raising server-side latency and RU charge.

5.1.4 Monitor Data Replication in Relation to Latency and Availability

In a multi-region account, data replicates asynchronously from the write region to read regions. You monitor the availability metric to confirm the account meets its SLA and watch replication latency to confirm read regions stay close to the write region — especially important under bounded staleness, which publishes explicit staleness bounds.

Key facts

  • Availability metric confirms uptime against the SLA; a "Region failed over" activity fires when a region fails over.
  • Replication (P2P) latency between write and read regions should be monitored to validate consistency guarantees.
  • Bounded staleness surfaces explicit staleness (time/operation) bounds — monitor to ensure they hold.
  • Combine with per-region latency metrics to spot a lagging or unhealthy region.

Exam tips & gotchas

  • Availability is a metric; a region failover is an activity-log event you can alert on.
  • Higher consistency (strong/bounded staleness) trades latency and RU for freshness — monitor accordingly.

Practice

Q1. You must be notified whenever the account performs a regional failover. What monitoring construct fits best?
Answer: An alert on the "Region failed over" activity/metric signal. Why: Failovers are surfaced as an activity/metric event that alerting can watch.
Q2. Under bounded staleness, which signal confirms read regions are keeping up?
Answer: Replication latency between write and read regions (plus availability). Why: It validates that staleness bounds are being honored across regions.

5.1.5 Configure Azure Monitor Alerts for Azure Cosmos DB

Alerts turn passive metrics and logs into proactive notifications. Cosmos DB supports three alert types: metric alerts (static or dynamic thresholds), log (KQL) alerts over Log Analytics, and activity-log alerts for control-plane events. Azure even ships recommended alert rules for the most common failure modes.

Key facts

  • Recommended rules: rate limiting on RU (metric alert on StatusCode = 429), region failed over (count > threshold), and account keys rotated (activity-log alert).
  • Metric alerts fire on thresholds or dynamic baselines; log alerts run a saved KQL query on a schedule.
  • A dedicated how-to covers alerting when a logical partition key approaches the 20-GB storage limit.
  • Alerts route to action groups (email, SMS, webhook, Logic Apps, Functions).
# Metric alert: fire when 429 (throttled) requests exceed a threshold
$acct = Get-AzResource -ResourceGroupName "rg-cosmos" `
  -ResourceType "Microsoft.DocumentDB/databaseAccounts" -Name "cosmos-prod"

$condition = New-AzMetricAlertRuleV2Criteria `
  -MetricName "TotalRequests" `
  -DimensionSelection (New-AzMetricAlertRuleV2DimensionSelection `
      -DimensionName "StatusCode" -ValuesToInclude "429") `
  -TimeAggregation Total -Operator GreaterThan -Threshold 100

Add-AzMetricAlertRuleV2 -Name "cosmos-429-alert" -ResourceGroupName "rg-cosmos" `
  -WindowSize 00:05:00 -Frequency 00:01:00 -TargetResourceId $acct.ResourceId `
  -Condition $condition -Severity 2 -ActionGroupId $actionGroupId

Exam tips & gotchas

  • Key rotation and region failover are control-plane events → use activity-log alerts, not metric alerts.
  • Throttling (429) is a metric alert on Total Requests split by StatusCode.
  • Log alerts require a diagnostic setting sending logs to Log Analytics first.

Practice

Q1. You want to be paged when someone rotates the account keys. Which alert type do you configure?
Answer: An activity-log alert on the key-rotation operation. Why: Key rotation is a control-plane event captured in the activity log, not a metric.
Q2. You need an alert when throttling becomes frequent. What signal and split do you use?
Answer: A metric alert on Total Requests split by StatusCode = 429. Why: Throttling surfaces as 429 responses in the request metric.
Q3. A logical partition is nearing its storage ceiling. How do you get warned early?
Answer: Configure the alert on logical partition key storage size approaching 20 GB (using PartitionKeyStatistics logs). Why: The 20-GB logical partition limit is hard; alerting early prevents write failures.

5.1.6 Implement and Query Azure Monitor Resource Logs for Azure Cosmos DB

Unlike metrics, resource (data-plane) logs are not collected until you create a diagnostic setting that routes them to Log Analytics, a storage account, or Event Hubs. Once flowing to Log Analytics you query them with KQL to inspect individual requests, query runtime statistics, and per-partition data.

Key facts

  • Create a diagnostic setting to enable log categories: DataPlaneRequests, QueryRuntimeStatistics, PartitionKeyStatistics, PartitionKeyRUConsumption, ControlPlaneRequests, and more.
  • Resource-specific mode (recommended) writes to dedicated tables (CDBDataPlaneRequests, CDBPartitionKeyStatistics, CDBPartitionKeyRUConsumption, CDBControlPlaneRequests); the legacy alternative is the shared AzureDiagnostics table.
  • The full-text query feature deobfuscates query text in logs (extra cost — turn off after troubleshooting).
  • Log Analytics retention and cost are separate from the account.
# Route Cosmos DB resource logs to Log Analytics (resource-specific tables)
az monitor diagnostic-settings create `
  --name "cosmos-diag" `
  --resource $(az cosmosdb show -g rg-cosmos -n cosmos-prod --query id -o tsv) `
  --workspace $(az monitor log-analytics workspace show -g rg-cosmos -n law-cosmos --query id -o tsv) `
  --logs '[{"category":"DataPlaneRequests","enabled":true},{"category":"PartitionKeyStatistics","enabled":true},{"category":"PartitionKeyRUConsumption","enabled":true}]' `
  --export-to-resource-specific true
// Top RU-consuming operations by collection in the last day
CDBDataPlaneRequests
| where TimeGenerated > ago(1d)
| summarize TotalRU = sum(RequestCharge) by CollectionName, OperationName
| top 15 by TotalRU desc
Gotcha: No diagnostic setting = no logs. Metrics flow automatically, but resource logs must be explicitly enabled and routed to a destination.

Exam tips & gotchas

  • Prefer resource-specific mode (dedicated CDB* tables) over legacy AzureDiagnostics.
  • PartitionKeyStatistics and PartitionKeyRUConsumption are the go-to categories for partition analysis.
  • Full-text query logging exposes query text — disable it when you're done to control cost/exposure.

Practice

Q1. A team enabled metrics but can't find any data-plane request logs to run KQL against. What did they miss?
Answer: They never created a diagnostic setting to send resource logs to Log Analytics. Why: Logs require an explicit diagnostic setting; only metrics are collected automatically.
Q2. Which log mode should you choose for cleaner, cheaper KQL queries with dedicated tables?
Answer: Resource-specific mode (CDB* tables). Why: It writes to typed, per-category tables rather than the shared AzureDiagnostics table.

5.1.7 Monitor Throughput Across Partitions

To find hot partitions — logical partition keys that consume a disproportionate share of RUs — query the PartitionKeyRUConsumption resource log. Combined with Normalized RU Consumption split by physical partition, it reveals whether throughput is being spread evenly or funneled into a few keys.

Key facts

  • PartitionKeyRUConsumption reports RU consumed per logical partition key → identifies hot keys.
  • Normalized RU Consumption split by PhysicalPartitionId shows physical-partition utilization.
  • Even RU distribution is the goal; concentration signals a partition-key design problem.
// Which logical partition keys burn the most RU?
CDBPartitionKeyRUConsumption
| where TimeGenerated > ago(6h)
| summarize RU = sum(RequestCharge) by PartitionKey, CollectionName
| top 20 by RU desc

Exam tips & gotchas

  • PartitionKeyRUConsumption = throughput per key; PartitionKeyStatistics = storage per key. Don't mix them up.
  • A hot partition is a design fix (better key), not just a scale-up.

Practice

Q1. You suspect one tenant's key is eating most of your RUs. Which resource log confirms it?
Answer: PartitionKeyRUConsumption. Why: It aggregates RU charge per logical partition key, exposing hot keys.
Q2. Throughput is concentrated on a few keys. Is adding RU/s the correct long-term fix?
Answer: No — redesign the partition key for even distribution. Why: RU/s is split evenly across physical partitions, so a hot key stays throttled regardless of total RU/s.

5.1.8 Monitor Distribution of Data Across Partitions

Storage skew is as dangerous as throughput skew: a single logical partition key cannot exceed 20 GB. The PartitionKeyStatistics resource log reports the storage size of your largest logical partition keys so you can detect skew and act before a key hits the ceiling.

Key facts

  • PartitionKeyStatistics reports storage size for the top logical partition keys.
  • Hard limit: a logical partition is capped at 20 GB — approaching it blocks further writes for that key.
  • Use it to detect skew early and to drive the "approaching 20 GB" alert.
// Largest logical partitions by storage (bytes)
CDBPartitionKeyStatistics
| where TimeGenerated > ago(1d)
| summarize MaxBytes = max(SizeKb) * 1024 by PartitionKey, CollectionName
| top 20 by MaxBytes desc

Exam tips & gotchas

  • 20 GB is the logical partition storage ceiling — memorize it.
  • PartitionKeyStatistics = storage; pair it with an alert to avoid write failures.
  • Fixing storage skew means a higher-cardinality or synthetic/hierarchical partition key.

Practice

Q1. Which log shows whether a single partition key is approaching its storage limit?
Answer: PartitionKeyStatistics. Why: It reports per-logical-partition storage size, so you can spot a key nearing 20 GB.
Q2. A device-per-day partition key will grow forever for busy devices. What redesign prevents hitting 20 GB?
Answer: A synthetic or hierarchical partition key (e.g., deviceId + time bucket) to bound each logical partition. Why: It caps growth per key and spreads storage across more partitions.

5.1.9 Monitor Security by Using Logging and Auditing

Security auditing focuses on the control plane — who changed throughput, firewall rules, network settings, or keys. The ControlPlaneRequests resource log records these operations, and disabling key-based metadata writes forces every metadata change through auditable ARM/RBAC paths.

Key facts

  • ControlPlaneRequests audits throughput changes, firewall/network changes, and key operations — enable it via a diagnostic setting.
  • disableKeyBasedMetadataWriteAccess blocks metadata changes made with account keys, forcing ARM/RBAC (which are auditable).
  • The Azure activity log records point-in-time restore actions and key rotation.
  • Pair control-plane logs with activity-log alerts for a complete audit trail.
# Force metadata changes through ARM/RBAC (disable key-based metadata writes)
az cosmosdb update `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --disable-key-based-metadata-write-access true

Exam tips & gotchas

  • ControlPlaneRequests audits management operations; DataPlaneRequests audits document operations.
  • Disabling key-based metadata writes hardens auditing by removing an unauditable path.

Practice

Q1. Security wants an audit trail of every throughput and firewall change. Which log category do you enable?
Answer: ControlPlaneRequests. Why: Control-plane operations (throughput, firewall, keys) are recorded there.
Q2. How do you ensure metadata changes can't bypass RBAC via account keys?
Answer: Set disableKeyBasedMetadataWriteAccess to true. Why: It forces metadata writes through ARM/RBAC, which are auditable.

5.2 Implement Backup and Restore for an Azure Cosmos DB Solution

5.2.1 Choose Between Periodic and Continuous Backup

Cosmos DB backs up your data automatically in one of two modes. Periodic (the default) takes full snapshots on an interval and requires a Microsoft support ticket to restore. Continuous gives you self-service point-in-time restore (PITR) to any second within the retention window. Both always restore into a new account.

Key facts

  • Periodic: full backup every 4 hours by default; last two copies retained (interval/retention configurable); stored in geo-redundant blob storage; restore ONLY via a support request; deleted db/container snapshots kept 30 days.
  • Continuous: self-service PITR to any second; two tiers — 7-day (free) and 30-day (paid); you trigger restore via portal/CLI/PowerShell/ARM.
  • Restore always targets a new account (you cannot overwrite the source in place).
  • Backups don't consume your provisioned RUs.
Key: Need self-service, "restore to any second" recovery? Choose continuous backup. Periodic requires opening a support ticket.

Exam tips & gotchas

  • Continuous = self-service PITR; periodic = support-ticket restore. This is the classic exam decision.
  • Restore always creates a new account, in both modes.
  • Continuous tiers are 7-day (free) and 30-day (paid) — you can't restore beyond the window.

Practice

Q1. A team must recover to the exact second before an accidental bulk delete, without contacting support. Which backup mode?
Answer: Continuous backup with point-in-time restore. Why: Only continuous mode provides self-service restore to an arbitrary timestamp.
Q2. With periodic backup, how is a restore initiated?
Answer: By opening a Microsoft support ticket. Why: Periodic-mode restores are performed by Microsoft support, into a new account.
Q3. After any restore, where does the data land?
Answer: In a new Cosmos DB account. Why: Both backup modes restore into a new account, never overwriting the source.

5.2.2 Configure Periodic Backup

For periodic mode you tune two knobs — the backup interval and the retention (how many copies to keep) — plus the storage redundancy of the backup blobs. Two backup copies are free; extra copies are billed, and backups never touch your RU budget.

Key facts

  • Configure interval + retention via portal, CLI, PowerShell, or ARM.
  • Backup storage redundancy: Geo (default), Zone, or Local.
  • What a restore does not bring back: account keys (new ones are generated), firewall/VNet/private-endpoint config, RBAC assignments, stored procedures/triggers/UDFs, TTL-expired documents, analytical store, and materialized views. The restored account is single-region (the write region).
# Set periodic backup interval, retention, and redundancy
az cosmosdb update `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --backup-interval 240 `
  --backup-retention 16 `
  --backup-redundancy Zone

Exam tips & gotchas

  • Interval is in minutes; retention is a count of copies (or hours depending on tool) — read the parameter carefully.
  • Restored accounts don't include keys, network rules, RBAC, or server-side scripts — you reconfigure those.
  • Backup redundancy (Geo/Zone/Local) is independent of the account's own data redundancy.

Practice

Q1. After a periodic restore, apps fail to authenticate with the old connection string. Why?
Answer: The restored account has new keys; account keys are not restored. Why: Restores generate fresh keys, so connection strings must be updated.
Q2. You must keep backup blobs within a single region for data-residency reasons. Which redundancy do you pick?
Answer: Local (locally redundant) backup storage. Why: It keeps backup copies within the region rather than geo-replicating them.

5.2.3 Configure Continuous Backup and Point-in-Time Restore

Continuous backup is enabled at account creation or by a one-way migration from periodic. You pick the 7-day (free) or 30-day (paid) tier, and mutations are captured asynchronously (within roughly 100 seconds), enabling restore to any point in the retention window.

Key facts

  • Enable at creation or migrate periodic → continuous (irreversible). You can switch tiers (7 ↔ 30 day).
  • Mutations are backed up asynchronously within ~100 seconds.
  • Supported APIs: NoSQL, MongoDB, Gremlin, Table — not Cassandra.
  • Not restored: firewall/VNet/private-endpoint config, data-plane RBAC, stored procedures/triggers/UDFs, and all source regions (restore is single-region).
  • CMK + continuous backup requires a user- or system-assigned managed identity (not the first-party identity).
# Create an account with continuous 7-day backup
az cosmosdb create `
  --resource-group rg-cosmos `
  --name cosmos-pitr `
  --locations regionName=eastus `
  --backup-policy-type Continuous `
  --continuous-tier Continuous7Days

# Migrate an existing periodic account to continuous (one-way)
az cosmosdb update `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --backup-policy-type Continuous
Gotcha: Migrating periodic → continuous is one-way; you cannot switch back to periodic. And Cassandra API does not support continuous backup.

Exam tips & gotchas

  • Continuous supports NoSQL, MongoDB, Gremlin, Table — memorize that Cassandra is excluded.
  • Migration to continuous is irreversible.
  • CMK accounts need a managed identity to use continuous backup.

Practice

Q1. Which API cannot use continuous backup?
Answer: API for Apache Cassandra. Why: Continuous backup supports NoSQL, MongoDB, Gremlin, and Table only.
Q2. A team migrated to continuous backup and now wants to revert to periodic. Can they?
Answer: No — the migration is one-way. Why: Periodic → continuous cannot be reversed.

5.2.4 Locate a Restore Point for a Point-in-Time Restore

Before restoring, you find a valid timestamp. Best practice is to get the latest restorable timestamp for the container, or use the event feed to see create/replace/delete events on databases and containers so you can restore to the moment just before an unwanted change.

Key facts

  • Get the latest restorable timestamp for a container before restoring.
  • The event feed (portal "Click here" link) or enumerating restorable resources via CLI/PowerShell shows create/replace/delete events.
  • Choose a timestamp just before the destructive operation you want to undo.
# List restorable database accounts and their restorable resources
az cosmosdb restorable-database-account list `
  --query "[?contains(accountName,'cosmos-prod')].{name:accountName, id:id}" -o table

# Enumerate restorable SQL containers to find create/delete timestamps
az cosmosdb sql restorable-container list `
  --location eastus `
  --instance-id <restorable-account-instance-id> `
  --database-rid <database-rid>

Exam tips & gotchas

  • Always confirm the latest restorable timestamp before kicking off a restore.
  • The event feed helps you pick a timestamp just before a delete/replace.

Practice

Q1. Before a PITR you want to confirm the newest point you can restore to. What do you retrieve?
Answer: The latest restorable timestamp for the container. Why: It defines the most recent valid restore point within the window.
Q2. How do you find the exact moment a container was deleted so you can restore just before it?
Answer: Use the event feed / enumerate restorable resources to see the delete event timestamp. Why: It lists create/replace/delete events with timestamps.

5.2.5 Restore a Database or Container from a Restore Point

With continuous backup you self-service a restore of a live or deleted account — the whole account or selected databases/containers — into a new account in the same region. Note that TTL is restored by default, so pass --disable-ttl to stop restored documents from immediately expiring.

Key facts

  • Restore a live or deleted account; entire account or chosen databases/containers, into a NEW account in the same region.
  • TTL is restored by default → use --disable-ttl True to prevent instant expiry of restored data.
  • A subset of containers under a shared-throughput database cannot be restored alone — you restore the whole database.
  • You can restrict public network access on the target with --public-network-access Disabled.
# Point-in-time restore of specific databases/containers into a new account
az cosmosdb restore `
  --target-database-account-name cosmos-restored `
  --account-name cosmos-prod `
  --resource-group rg-cosmos `
  --restore-timestamp "2026-07-14T22:05:00Z" `
  --location eastus `
  --databases-to-restore name=SalesDB collections=Orders Customers `
  --disable-ttl True
Tip: If restored data seems to vanish, TTL likely expired it on arrival — restore again with --disable-ttl True.

Exam tips & gotchas

  • TTL restores on by default — a classic reason restored data "disappears."
  • Shared-throughput databases restore as a whole, not container-by-container.
  • The target is always a new account; update connection strings, network rules, and RBAC afterward.

Practice

Q1. After restoring, documents with a short TTL vanish almost immediately. How do you preserve them next time?
Answer: Restore with --disable-ttl True. Why: TTL is restored by default, so aged documents expire on arrival unless TTL is disabled during restore.
Q2. You only need one container back, but it lives under a database with shared (database-level) throughput. What can you restore?
Answer: The entire shared-throughput database (not just the single container). Why: Containers sharing database throughput can't be restored individually.

5.3 Implement Security for an Azure Cosmos DB Solution

5.3.1 Choose Between Service-Managed and Customer-Managed Encryption Keys

Encryption at rest is always on in Cosmos DB (AES-256) with no toggle and no extra cost, using service-managed keys by default. If your compliance regime demands that you control the key lifecycle, you can add customer-managed keys (CMK) stored in Azure Key Vault, giving a second layer of encryption you manage.

Key facts

  • Default: service-managed keys (Microsoft-managed) — encryption at rest is automatic and free.
  • CMK adds a second encryption layer with keys you hold in Key Vault; must be enabled at account creation (or via an existing-accounts flow) and applies at account-level granularity.
  • CMK gives you key rotation and revocation control, at the cost of a small RU overhead and Key Vault dependency.

Exam tips & gotchas

  • You can't turn encryption at rest off; the choice is only who manages the key.
  • CMK is enabled at creation and is account-scoped.
  • Choose CMK only when compliance requires customer key control — it adds operational burden.

Practice

Q1. A compliance policy requires the organization to control and revoke the encryption key. Which option satisfies it?
Answer: Customer-managed keys (CMK) in Azure Key Vault. Why: CMK gives the customer control over key rotation and revocation; service-managed keys do not.
Q2. If you do nothing, is your data encrypted at rest?
Answer: Yes — with service-managed keys, automatically and at no extra cost. Why: Encryption at rest is always enabled and defaults to Microsoft-managed keys.

5.3.2 Configure Network-Level Access Control for Azure Cosmos DB

Network controls decide from where requests may reach the account, layered on top of (never instead of) the auth token. The three mechanisms are the IP firewall, VNet service endpoints, and Private Link / private endpoints, with publicNetworkAccess = Disabled as the strongest lockdown.

Key facts

  • IP firewall (ipRules / CIDR): non-listed IPs get 403. "Accept connections from within Azure datacenters" adds 0.0.0.0 (needed for some Azure services with no fixed IP).
  • VNet service endpoints allow specific subnets to reach the account over the Azure backbone.
  • Private endpoints (Private Link) assign a private IP in your subnet; combine with publicNetworkAccess = Disabled, which takes precedence over all firewall rules.
  • A valid auth token is ALWAYS required regardless of network rules — network controls are additive.
# Restrict to specific IP ranges and Azure services, keep public access on
az cosmosdb update `
  --resource-group rg-cosmos --name cosmos-prod `
  --ip-range-filter "13.88.56.148,40.65.0.0/16,0.0.0.0"

# Lock down entirely: disable public network access (use private endpoints)
az cosmosdb update `
  --resource-group rg-cosmos --name cosmos-prod `
  --public-network-access Disabled
Key: Network rules never replace authentication — even from an allowed IP or private endpoint, a request still needs a valid key or Entra token.

Exam tips & gotchas

  • publicNetworkAccess=Disabled overrides firewall rules — nothing public gets in.
  • 0.0.0.0 in the IP filter is the special "allow Azure datacenter traffic" entry, not "allow the world."
  • Private endpoints give private IPs; service endpoints keep public IPs but restrict by subnet.

Practice

Q1. You want the account reachable only through a private IP in your VNet, with no public exposure. What do you configure?
Answer: A private endpoint (Private Link) plus publicNetworkAccess = Disabled. Why: The private endpoint provides a private IP, and disabling public access blocks all public routes regardless of firewall rules.
Q2. An Azure service with no fixed outbound IP must reach Cosmos DB behind an IP firewall. What enables it?
Answer: Enable "Accept connections from within Azure datacenters" (adds 0.0.0.0). Why: It permits Azure-internal traffic when the caller lacks a static IP.
Q3. With a strict IP firewall in place, does a caller still need credentials?
Answer: Yes. Why: Network rules are additive; a valid key or Entra token is always required.

5.3.3 Configure Data Encryption for Azure Cosmos DB

Cosmos DB encrypts data in transit (TLS/HTTPS) and at rest (AES-256) automatically. Adding CMK provides double encryption for the main transactional store, but you must understand its boundaries: some components and metadata are not covered by the customer key.

Key facts

  • Encryption in transit (TLS) and at rest (AES-256) are automatic.
  • CMK double encryption covers the main transactional store; the Synapse Link analytical store and continuous backups are not double-encrypted with CMK.
  • Not encrypted with CMK: account/database/container names, stored-procedure names, index property paths, and partition key values (these are metadata).

Exam tips & gotchas

  • CMK protects data, not resource/metadata names or partition key values.
  • Analytical store and continuous backups fall outside CMK double encryption.

Practice

Q1. With CMK enabled, are your container names and partition key values encrypted with the customer key?
Answer: No — that metadata is not encrypted with CMK. Why: CMK protects stored data, but names, index paths, and partition key values are metadata excluded from CMK.
Q2. Does CMK double-encrypt the Synapse Link analytical store?
Answer: No. Why: The analytical store and continuous backups are outside CMK double encryption.

5.3.4 Manage Control Plane Access to Azure Cosmos DB by Using Azure Role-Based Access Control (RBAC)

Cosmos DB has two RBAC planes. The control plane governs management of the account and its resources (metadata) through Azure RBAC — creating accounts, changing throughput, reading region config — but not the data inside. The built-in Cosmos DB Operator role manages accounts while explicitly blocking access to keys, connection strings, and data.

Key facts

  • Control plane = manage resources/metadata via Azure RBAC; it does not grant data access.
  • Cosmos DB Operator manages accounts but cannot read keys/connection strings or data.
  • Custom control-plane roles use Microsoft.DocumentDB/* actions; assign with az role assignment create.
  • Native data-plane RBAC definitions/assignments live inside the account, so creating them requires control-plane permissions.
# Grant an operator management rights without data or key access
az role assignment create `
  --assignee "ops-team@contoso.com" `
  --role "Cosmos DB Operator" `
  --scope $(az cosmosdb show -g rg-cosmos -n cosmos-prod --query id -o tsv)
Key: Control-plane Azure RBAC manages the account; it never grants read/write to documents. Data access needs data-plane RBAC (or keys).

Exam tips & gotchas

  • Cosmos DB Operator ≠ data access — it deliberately excludes keys and data.
  • Control-plane actions live under Microsoft.DocumentDB/*; data-plane uses dataActions.
  • Creating data-plane role assignments requires control-plane rights.

Practice

Q1. You need a team to create databases and change throughput but never see keys or data. Which built-in role fits?
Answer: Cosmos DB Operator. Why: It grants account management while blocking keys, connection strings, and data access.
Q2. A user has the Cosmos DB Operator role. Can they read documents in a container?
Answer: No. Why: Control-plane roles don't grant data-plane access; they'd need a data-plane RBAC assignment or keys.

5.3.5 Manage Control Plane Access to Azure Cosmos DB Data Explorer by Using Azure Role-Based Access Control (RBAC)

5.3.6 Manage Data Plane Access to Azure Cosmos DB by Using Microsoft Entra ID

Native data-plane RBAC lets applications read and write documents using a Microsoft Entra ID identity — no keys. Two built-in role definitions cover most needs, and you can craft custom roles from data actions. Disabling local auth forces every data operation through Entra ID.

Key facts

  • Built-in role definitions: Cosmos DB Built-in Data Reader and Cosmos DB Built-in Data Contributor.
  • Custom roles list dataActions such as .../readMetadata, .../containers/*, and .../items/*; there is no notDataActions support.
  • Manage with az cosmosdb sql role definition create/list and az cosmosdb sql role assignment create --scope / (scope can be account, database, or container).
  • Set disableLocalAuth = true to disable keys and require Entra ID.
# Assign the built-in Data Contributor role to an app's managed identity
$principalId = az ad sp show --id <app-client-id> --query id -o tsv
az cosmosdb sql role assignment create `
  --resource-group rg-cosmos `
  --account-name cosmos-prod `
  --role-definition-id "00000000-0000-0000-0000-000000000002" `
  --principal-id $principalId `
  --scope "/"

# Enforce Entra ID by disabling account keys
az cosmosdb update -g rg-cosmos -n cosmos-prod --disable-local-auth true
Key: Data-plane RBAC = keyless data access via Entra ID. Pair it with disableLocalAuth to eliminate key-based access entirely.

Exam tips & gotchas

  • Data-plane roles are Cosmos-specific definitions stored in the account — not Azure RBAC roles.
  • No notDataActions — you can only allow, not deny, specific data actions.
  • disableLocalAuth kills keys; disableKeyBasedMetadataWriteAccess only blocks metadata writes via keys.

Practice

Q1. An app should read and write documents without ever using an account key. What do you configure?
Answer: Assign the Cosmos DB Built-in Data Contributor role to the app's Entra identity (and optionally disable local auth). Why: Data-plane RBAC grants keyless read/write via Entra ID.
Q2. You must guarantee that keys can no longer be used for data access. Which property do you set?
Answer: disableLocalAuth = true. Why: It disables key-based authentication, forcing Entra ID.

5.3.7 Configure Cross-Origin Resource Sharing (CORS) Settings

5.3.8 Manage Account Keys by Using Azure Key Vault

When you still need account keys, don't scatter them through app settings — store them as secrets in Azure Key Vault and retrieve them at runtime using a managed identity, so no secret lives in code or config. Better still, prefer disabling keys entirely and using Entra ID.

Key facts

  • Store account keys / connection strings as Key Vault secrets; fetch them at runtime via managed identity (no secrets in code).

# Store a Cosmos DB key in Key Vault; app reads it via managed identity

Exam tips & gotchas

  • Key Vault + managed identity keeps secrets out of code/config.
  • The stronger answer is usually keyless Entra ID access; Key Vault is the fallback.

Answer: Store keys as Key Vault secrets and retrieve them at runtime with a managed identity. Why: It removes secrets from code/config and centralizes rotation.
Q2. What's an even stronger alternative to storing keys in Key Vault?

5.3.9 Implement Customer-Managed Keys for Encryption

Implementing CMK means creating a key in Key Vault, granting Cosmos DB the rights to wrap/unwrap it, and setting the key URI at account creation. Key Vault must have Soft Delete and Purge Protection enabled, and you accept a small RU overhead on operations.

Key facts

  • Key in Azure Key Vault (RSA ≥ 3072). Key Vault requires Soft Delete + Purge Protection.
  • Grant the Cosmos DB principal Get / WrapKey / UnwrapKey (access policy) or the Key Vault Crypto Service Encryption User role (RBAC).
  • Set the Key URI at account creation. Revoke by disabling the key → after revoke, only account deletion is possible. Rotate via a new key version or a new Key URI.
  • Overhead is roughly +5% for point reads, +6% for writes, +15% for queries/change feed.
# Create a CMK-enabled account referencing a Key Vault key
az cosmosdb create `
  --resource-group rg-cosmos --name cosmos-cmk `
  --locations regionName=eastus `
  --key-uri "https://kv-cosmos.vault.azure.net/keys/cosmos-cmk/<version>" `
  --assign-identity [system]
Gotcha: Revoking (disabling) the CMK makes the account unreadable — the only remaining action is to delete the account. Never revoke without a plan.

Exam tips & gotchas

  • Key Vault needs Soft Delete + Purge Protection or CMK setup fails.
  • CMK is set at account creation; grant Get/WrapKey/UnwrapKey to Cosmos DB.
  • Expect measurable RU overhead, largest on queries/change feed.

Practice

Q1. CMK account creation fails referencing the Key Vault. Which two Key Vault settings are required?
Answer: Soft Delete and Purge Protection. Why: Cosmos DB CMK requires both to protect against irreversible key loss.
Q2. A security incident requires immediately blocking all data access via the customer key. What happens after you disable the key?
Answer: The account becomes inaccessible and can only be deleted. Why: Revoking the CMK removes unwrap capability, leaving deletion as the sole option.

5.3.10 Implement Always Encrypted

Always Encrypted protects specific sensitive fields with client-side encryption — the service never sees plaintext or the keys. It uses a database-level data encryption key (DEK) wrapped by a customer-managed key (CMK / column master key) in Key Vault, and a container encryption policy that maps each property to a key and encryption type.

Key facts

  • Field-level, client-side encryption; the service can't decrypt.
  • Uses a DEK (database-scoped, stored in Cosmos DB) wrapped by a CMK in Key Vault.
  • The container's encryption policy is immutable — set at creation — and maps each property to a DEK + encryption type.
  • Deterministic encryption allows equality filters (same plaintext → same ciphertext, weaker); randomized is stronger but not queryable.
  • Implemented with the .NET (Microsoft.Azure.Cosmos.Encryption) or Java packages using a KeyResolver + managed identity.
// Client-side Always Encrypted with a Key Vault-backed key resolver
var keyResolver = new KeyResolver(new DefaultAzureCredential());
var client = new CosmosClient(endpoint, credential)
    .WithEncryption(keyResolver, KeyEncryptionKeyResolverName.AzureKeyVault);

// Container encryption policy: SSN deterministic (queryable), Salary randomized
var paths = new List<ClientEncryptionIncludedPath>
{
    new() { Path = "/ssn",    ClientEncryptionKeyId = "cek1",
            EncryptionType = "Deterministic",
            EncryptionAlgorithm = "AEAD_AES_256_CBC_HMAC_SHA256" },
    new() { Path = "/salary", ClientEncryptionKeyId = "cek1",
            EncryptionType = "Randomized",
            EncryptionAlgorithm = "AEAD_AES_256_CBC_HMAC_SHA256" }
};

Exam tips & gotchas

  • Deterministic = queryable by equality; randomized = strongest but no querying.
  • The encryption policy is immutable once the container is created.
  • Always Encrypted is field-level and client-side; CMK-at-rest is account-level and server-side. Different features.

Practice

Q1. You must encrypt an SSN field but still filter by exact SSN value. Which encryption type do you choose?
Answer: Deterministic. Why: Deterministic encryption produces identical ciphertext for identical plaintext, enabling equality queries (randomized doesn't).
Q2. After creating a container you need to change which properties are encrypted. Is that allowed?
Answer: No — the encryption policy is immutable and set at container creation. Why: You must recreate the container (and migrate data) to change encrypted paths.

5.4.2 Move Data by Using Client SDK Bulk Operations

Setting AllowBulkExecution = true on the client changes how the SDK dispatches concurrent point operations: it groups them into batched service calls per partition, dramatically increasing import throughput. You fire many operations concurrently (for example, with Task.WhenAll) and let the SDK optimize the batching.

Key facts

  • new CosmosClient(..., new CosmosClientOptions { AllowBulkExecution = true }); the setting is immutable once the client is built.
  • The SDK batches concurrent point operations per partition into fewer service calls.
  • Use Task.WhenAll to submit many operations at once.
  • Fastest import: temporarily raise RU/s and reduce indexing (e.g., exclude paths) during the load.
var options = new CosmosClientOptions { AllowBulkExecution = true };
using var client = new CosmosClient(endpoint, key, options);
var container = client.GetContainer("SalesDB", "Orders");

var tasks = orders.Select(o =>
    container.CreateItemAsync(o, new PartitionKey(o.CustomerId)));
await Task.WhenAll(tasks);   // SDK batches these per partition

Exam tips & gotchas

  • AllowBulkExecution is set at client construction and can't change later.
  • Bulk trades per-request latency for aggregate throughput — great for imports, not for chatty low-latency apps.
  • Raise RU/s and trim indexing during the load, then restore both.

Practice

Q1. How do you enable bulk mode in the .NET SDK, and when is it fixed?
Answer: Set AllowBulkExecution = true in CosmosClientOptions at client creation; it's immutable afterward. Why: Bulk batching is a client-construction-time option.
Q2. Beyond enabling bulk, what two changes speed a large import?
Answer: Temporarily increase RU/s and reduce indexing (exclude paths or mode none). Why: More throughput and less index write cost accelerate ingest.

5.4.6 Move Data by Using the Azure Cosmos DB Spark Connector

The Cosmos DB Spark connector exposes two data source formats. cosmos.oltp reads and writes the transactional store (consuming RUs), while cosmos.olap reads the analytical store (RU-free, read-only, via Synapse Link). Only Spark can write back to the transactional store from the analytics layer.

Key facts

# Read the analytical store (RU-free) and write results to the transactional store

      .option("spark.cosmos.container", "Orders")

(df.write.format("cosmos.oltp")
   .option("spark.cosmos.container", "OrdersCurated")
   .mode("append")
   .save())

Exam tips & gotchas

  • cosmos.olap = analytical store, RU-free, read-only; cosmos.oltp = transactional, costs RUs.

5.5 Implement a DevOps Process for an Azure Cosmos DB Solution

5.5.1 Choose When to Use Declarative Versus Imperative Operations

Declarative tools (ARM, Bicep, Terraform) describe the desired end state and are idempotent, repeatable, and source-controllable — ideal for CI/CD and environment parity. Imperative tools (Azure CLI, PowerShell, SDK) issue step-by-step commands — ideal for one-off operations, scripting, and ad-hoc automation.

Key facts

  • Declarative = desired-state, idempotent, repeatable, version-controlled → best for CI/CD and consistent environments.
  • Imperative = explicit commands → best for one-off ops, quick fixes, and scripting.
  • The control-plane API is not designed for high-volume calls — use it sparingly.

Exam tips & gotchas

  • "Repeatable, source-controlled, environment parity" → declarative (ARM/Bicep).
  • "One-time change" or "quick script" → imperative (CLI/PowerShell).
  • Some operations (like throughput migration) are imperative-only.

Practice

Q1. A team wants identical Cosmos DB environments provisioned repeatably through a release pipeline. Which approach?
Answer: Declarative (ARM/Bicep) templates. Why: They're idempotent, version-controlled, and enforce consistent desired state across environments.
Q2. An operator needs to make a single ad-hoc change to one account right now. What fits best?
Answer: Imperative CLI/PowerShell. Why: Imperative commands suit one-off, immediate operations.

5.5.2 Provision and Manage Azure Cosmos DB Resources by Using Azure Resource Manager Templates

ARM (and Bicep) templates provision and manage the whole account surface declaratively — account, databases, containers, throughput, indexing policy, RBAC, firewall rules, CORS, and backup policy — deployed through az deployment group create and driven from a CI/CD pipeline for repeatable, reviewable infrastructure.

Key facts

  • ARM/Bicep manage account, databases, containers, throughput, indexing policy, RBAC, firewall (ipRules), CORS, and backup policy.
  • Deploy with az deployment group create --template-file; a quickstart samples repo is available.
  • Templates are the backbone of consistent CI/CD and environment parity.
resource account 'Microsoft.DocumentDB/databaseAccounts@2024-11-15' = {
  name: 'cosmos-prod'
  location: location
  properties: {
    databaseAccountOfferType: 'Standard'
    locations: [ { locationName: location, failoverPriority: 0 } ]
    consistencyPolicy: { defaultConsistencyLevel: 'Session' }
  }
}

resource db 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases@2024-11-15' = {
  parent: account
  name: 'SalesDB'
  properties: { resource: { id: 'SalesDB' } }
}
# Deploy the template into a resource group
az deployment group create `
  --resource-group rg-cosmos `
  --template-file ./cosmos.bicep `
  --parameters location=eastus

Exam tips & gotchas

  • ARM/Bicep can manage indexing policy, firewall, CORS, RBAC, and backup policy — not just the account shell.
  • Throughput migration (manual↔autoscale) is NOT done through ARM — it's CLI/PowerShell only.

Practice

Q1. You want a container's indexing policy defined in source control and deployed with the rest of the infrastructure. How?
Answer: Include the indexingPolicy in the ARM/Bicep template and deploy via the pipeline. Why: Templates manage indexing policy declaratively alongside other resources.
Q2. Which command deploys a Bicep/ARM template to a resource group?
Answer: az deployment group create --template-file. Why: It submits the template for resource-group-scoped deployment.

5.5.3 Migrate Between Standard and Autoscale Throughput by Using PowerShell or Azure CLI

Switching a database or container between manual (standard) and autoscale throughput is supported only through the portal, Azure CLI, or PowerShell — not the SDK and not ARM. When you enable autoscale, the system chooses a starting max RU/s based on current manual RU/s and storage.

Key facts

  • Migration paths: portal, CLI, or PowerShell only (NOT SDK, NOT ARM template).
  • CLI: az cosmosdb sql database|container throughput migrate --throughput-type 'autoscale'|'manual'.
  • Enabling autoscale sets a starting max RU/s derived from the current manual value and storage.
# Migrate a container from manual to autoscale throughput
az cosmosdb sql container throughput migrate `
  --resource-group rg-cosmos `
  --account-name cosmos-prod `
  --database-name SalesDB `
  --name Orders `
  --throughput-type autoscale
Gotcha: You cannot migrate throughput type via an ARM/Bicep template — use CLI or PowerShell (or the portal).

Exam tips & gotchas

  • Throughput migration is imperative-only (CLI/PowerShell/portal).
  • Autoscale's initial max RU/s is auto-chosen from current manual RU/s and storage.

Practice

Q1. A pipeline tries to switch a container to autoscale via an ARM template and fails. Why?
Answer: Throughput-type migration isn't supported through ARM — only portal, CLI, or PowerShell. Why: The migration operation is imperative-only.
Q2. Which CLI command migrates a database to manual throughput?
Answer: az cosmosdb sql database throughput migrate --throughput-type manual. Why: It converts the database's throughput type imperatively.

5.5.4 Initiate a Regional Failover by Using PowerShell or Azure CLI

For single-write-region accounts you can trigger a manual failover by reordering failover priorities with az cosmosdb failover-priority-change, promoting a read region to the new write region. Service-managed failover, when enabled, handles regional outages automatically.

Key facts

  • az cosmosdb failover-priority-change --failover-policies <region>=<priority> reorders priorities / triggers manual failover of the write region.
  • Manual failover applies to single-write-region accounts; priority 0 is the write region.
  • Service-managed (automatic) failover reacts to outages without manual action.
# Promote West US to write region (priority 0), East US becomes priority 1
az cosmosdb failover-priority-change `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --failover-policies westus=0 eastus=1

Exam tips & gotchas

  • Priority 0 = the write region; changing priorities triggers/plans the failover.
  • Manual failover = single-write accounts; multi-region-write accounts don't need it the same way.
  • Service-managed failover is automatic; manual failover is deliberate via CLI/PowerShell.

Practice

Q1. You need to deliberately move the write region to West US for a planned maintenance test. Which command?
Answer: az cosmosdb failover-priority-change with westus=0. Why: Setting a region to priority 0 promotes it to the write region.
Q2. What does priority 0 represent in the failover policy?
Answer: The current write region. Why: Failover priority 0 is always the write region for single-write accounts.

5.5.5 Maintain Indexing Policies in Production by Using Azure Resource Manager Templates

Keeping indexing policies under source control means storing each container's indexingPolicy in the ARM/Bicep template and rolling changes through CI/CD. Because index changes trigger an online transformation, production queries keep working while the new policy is applied.

Key facts

  • Store the container's indexingPolicy in the ARM/Bicep template; deploy changes through the pipeline.
  • Index transformation runs online (track with the index transformation progress) — no write downtime.
  • This keeps production indexing consistent, reviewable, and reproducible.
resource container 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases/containers@2024-11-15' = {
  parent: db
  name: 'Orders'
  properties: {
    resource: {
      id: 'Orders'
      partitionKey: { paths: [ '/customerId' ], kind: 'Hash' }
      indexingPolicy: {
        indexingMode: 'consistent'
        includedPaths: [ { path: '/*' } ]
        excludedPaths: [ { path: '/notes/?' } ]
      }
    }
  }
}

Exam tips & gotchas

  • Index changes deployed via template still trigger an online transformation — no downtime.
  • Source-controlling indexing policy prevents config drift between environments.

Practice

Q1. A production container's indexing policy keeps drifting from what's in Git after manual portal edits. How do you enforce consistency?
Answer: Define indexingPolicy in the ARM/Bicep template and deploy only through CI/CD. Why: Declarative deployment makes the template the single source of truth.
Q2. Will deploying a new indexing policy to production block writes?
Answer: No — the index transformation runs online with no write-availability impact. Why: Cosmos DB applies index changes asynchronously in place.
