A. The five consistency levels

LevelGuaranteeRelative RU (reads)Typical use
StrongLinearizable — a read always sees the latest committed writeHighest (2× a normal read)Financial ledgers, inventory where stale reads are unacceptable
Bounded stalenessReads lag the write by at most K versions or t timeHigh (2×)Global apps needing near-strong with defined lag
Session (default)Read-your-own-writes within a client session (via session token)Normal (1×)Most web/mobile apps — great balance
Consistent prefixReads never see out-of-order writes (no gaps), but may be staleNormal (1×)Timelines/feeds where order matters, freshness less so
EventualNo ordering guarantee; converges eventuallyLowest (1×)Likes/counts, non-critical telemetry — lowest latency & cost
Decider: The account has a default level; a request can only relax it (weaker), never strengthen it beyond the account default via request options. Strong and Bounded staleness constrain your region topology; Session is the default and the usual right answer.

C. Partition-key decision quick-reference

GoalRule of thumb
Spread load evenlyPick a key with high cardinality (many distinct values) so requests and storage distribute across physical partitions.
Avoid a hot partitionAvoid low-cardinality or monotonically increasing keys (e.g., a single status or the current date) that concentrate traffic.
Enable transactionsItems that must participate in a transactional batch or a transactional stored procedure must share the same logical partition (same partition-key value).
Low-cardinality fieldBuild a synthetic key — concatenate fields or add a random/hashed suffix to raise cardinality.
Multi-tenant / > 20 GB per prefixUse a hierarchical (sub)partition key (up to 3 levels) so a prefix like TenantId can exceed the 20 GB logical-partition limit and prefix queries stay targeted.
Multiple access patternsConsider separate containers/materialized views (via change feed) keyed differently, since a container has one partition key.
Gotcha: A single logical partition is capped at 20 GB and its own RU ceiling. The partition key is immutable once items are written — choosing it wrong means re-creating the container and migrating data.

D. Request Units & the 429 playbook

RU essentials

  • A 1 KB point read costs ~1 RU; writes, larger items, more indexed paths, and cross-partition queries cost more.
  • Read the actual charge from the x-ms-request-charge response header (SDK: RequestCharge).
  • Exceeding provisioned RU/s returns HTTP 429 (Request rate too large) with a x-ms-retry-after-ms hint; the SDK retries automatically up to a configurable limit.

When you see 429s

  • Check the Normalized RU Consumption metric — if one partition sits at 100% while others idle, you have a hot partition (fix the partition key).
  • If all partitions are saturated, raise RU/s (or switch to autoscale).
  • Trim indexing on write-heavy containers; use bulk mode for large imports; increase SDK MaxRetryAttemptsOnRateLimitedRequests / MaxRetryWaitTimeInSeconds.

E. Indexing cheat sheet

SituationDo this
DefaultEvery property is indexed (includedPaths: /*) — great for reads, costs RU on every write.
Write-heavyExclude paths you never filter on to cut write RU; keep only what queries need.
Read-heavyKeep broad indexing; add composite indexes for multi-property ORDER BY and multi-filter queries.
Multi-property sortA composite index is required; its property order and ASC/DESC directions must match the query.
GeospatialAdd a spatial index for ST_DISTANCE / ST_WITHIN queries.

G. Security — control plane vs data plane

Control planeData plane
ControlsManage the account: regions, throughput, firewall, keys, backupRead/write items inside containers
MechanismAzure RBAC roles (e.g., Cosmos DB Operator)Cosmos DB data-plane RBAC via Microsoft Entra ID (Data Reader/Contributor role definitions + assignments)
Keys?Can disable key-based metadata write accessPrefer Entra ID tokens over primary keys; keys grant full data access

Other security layers

  • Network: IP firewall, VNet service endpoints, and Private Link / private endpoints.
  • Encryption at rest: on by default with service-managed keys; use customer-managed keys (CMK) in Key Vault for control over the key lifecycle.
  • Always Encrypted: field-level encryption so sensitive properties are protected even from the database engine.
  • Key management: store connection strings/keys in Azure Key Vault; rotate regularly.

H. NoSQL SQL query quick reference

-- Project nested fields, filter, aggregate, order
SELECT c.category, COUNT(1) AS items, AVG(c.price) AS avgPrice
FROM c
WHERE c.inStock = true AND ARRAY_LENGTH(c.tags) > 0
GROUP BY c.category

-- Work with an array using a self-join + correlated subquery
SELECT c.id,
       (SELECT VALUE COUNT(1) FROM t IN c.tags WHERE t.type = "promo") AS promoTags
FROM c
WHERE EXISTS (SELECT VALUE t FROM t IN c.tags WHERE t.name = "sale")

-- Type-checking & functions
SELECT c.id
FROM c
WHERE IS_DEFINED(c.discount) AND IS_NUMBER(c.discount)
  AND STARTSWITH(c.sku, "NA-") AND c.updated >= "2026-01-01T00:00:00Z"
Key: A cross-partition query fans out to every physical partition. Add the partition key to the WHERE clause (or set the SDK partition-key option) to keep it single-partition and cheap.

J. Glossary

Terms to know cold

  • RU/s (Request Unit per second): the normalized currency of throughput; a 1 KB point read ≈ 1 RU.
  • Logical partition: all items sharing one partition-key value; capped at 20 GB.
  • Physical partition: the backend host that stores one or more logical partitions; split automatically as data/RU grow.
  • Synthetic partition key: a composed key (concatenation or suffix) that raises cardinality.
  • Hierarchical partition key: up to three key levels for sub-partitioning and > 20 GB prefixes.
  • ETag / optimistic concurrency: conditional write using If-Match to prevent lost updates.
  • Session token: the marker that provides read-your-own-writes under session consistency.
  • Change feed: a persistent, ordered log of inserts/updates per partition, consumed by the processor or Azure Functions.
  • Analytical store: an auto-synced column store for HTAP analytics that doesn't consume transactional RU/s.
  • PITR: point-in-time restore, enabled by continuous backup.
  • PPAF: per-partition automatic failover.
