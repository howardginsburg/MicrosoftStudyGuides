This domain is about talking to Copilot effectively: how a prompt is structured, what context Copilot pulls in automatically, when to give it examples (zero-shot vs few-shot), the best-practice principles behind good prompting (the 4 Ss), and the inbound/outbound process flow a prompt travels through — including how to keep long Chat conversations efficient. Expect "which principle applies," "how does Copilot get context," and process-order questions.

4.1 Craft effective prompts

4.1.1 Describe prompt structure and context

Prompt engineering is the practice of crafting clear instructions that guide GitHub Copilot to produce code tailored to your project. The aim is output that is syntactically, functionally, and contextually correct — not just code that compiles, but code that fits your problem.

A good prompt has a discernible structure: a clear goal (the single thing you want), supporting context/details (comments, related open files, constraints), and — when helpful — examples Copilot can pattern-match against. The "prompt" Copilot actually reasons over is the assembled combination of your text plus the surrounding context it gathers.

Key facts

  • Effective prompts combine a goal, context, and optional examples.
  • Be explicit: "Write a Python function to filter and return even numbers from a given list" is single-focused and specific.
  • Adding comments at the top of a file supplies context and improves the quality of suggestions.
  • Keep the detail useful but concise — richer context helps, but overloading the prompt hurts clarity.
# Goal: parse an ISO-8601 timestamp string and return a timezone-aware datetime.
# Context: use only the standard library; raise ValueError on malformed input.
# Example input: "2026-07-15T09:30:00+00:00"
def parse_timestamp(value: str):
    ...
Tip: Think "goal + context + example." A one-line comment that names the inputs, the constraint, and a sample value usually beats a long, vague sentence.

Exam tips & gotchas

  • The "prompt" is the assembled prompt (your text + gathered context), not only the literal comment you typed.
  • "Contextually correct" is a distinct goal from "syntactically correct" — Copilot aims for code that fits your project, not just valid code.

Practice

Q1. A developer writes a vague comment // do the thing and gets poor suggestions. What single change most improves the prompt structure?
Answer: State a clear, specific goal with the relevant inputs/constraints (e.g., what to compute, expected types, edge cases). Why: a clear single goal plus context is the core of an effective prompt.
Q2. Which three elements make up the structure of an effective Copilot prompt?
Answer: A goal, supporting context/details, and optional examples. Why: Learn frames good prompting as clarity of goal plus context, optionally reinforced with examples.

4.1.2 Understand how context is determined

Copilot doesn't just read the comment under your cursor — while it processes a prompt it gathers context from your editor session so it can turn a high-level request into a concrete coding task. Knowing what it collects tells you how to steer it: open the right files, name things well, and put helpful comments nearby.

Key facts

  • Context Copilot collects includes: code before and after the cursor, the filename and file type, adjacent open tabs, project structure and file paths, and the languages and frameworks in use.
  • Copilot uses a Fill-in-the-Middle (FIM) technique — it considers both the preceding and following code, widening the model's view.
  • Copilot also draws on parallel open tabs in the editor to learn more about your code's requirements.
  • Comments (especially at the top of the file) and a descriptive filename are direct levers you control to enrich context.
Key: Keeping relevant files open in nearby tabs is a real technique — open tabs are part of the context Copilot gathers, so closing distracting/irrelevant files can sharpen suggestions.

Exam tips & gotchas

  • FIM = code after the cursor counts too. A classic distractor is "Copilot only uses code above the cursor."
  • Adjacent open tabs influence suggestions — this maps to the "Surround" principle.
  • Descriptive filenames/file types shape suggestions (e.g., a *.test.js file nudges Copilot toward test code).

Practice

Q1. A developer wants Copilot to follow the conventions in a helper module. What's the simplest way to feed that context?
Answer: Keep the helper file open in an adjacent tab. Why: Copilot uses parallel open tabs as context when generating suggestions.
Q2. Which technique lets Copilot factor in the code that comes after the cursor, not just before it?
Answer: Fill-in-the-Middle (FIM). Why: FIM considers preceding and following code to expand context.
Q3. Name three context sources Copilot gathers besides the current line.
Answer: Any three of: code before/after the cursor, filename/type, adjacent open tabs, project structure/file paths, languages/frameworks. Why: these are the context details Copilot collects during input processing.

4.1.3 Use zero-shot and few-shot prompting

How many examples you give Copilot changes the kind of help you get. With none, Copilot leans entirely on its training; with a few, it learns your specific pattern. Learn describes a spectrum: zero-shot, one-shot, and few-shot.

Key facts

  • Zero-shot: no examples — Copilot relies solely on its foundational training. Best for common patterns / standard functionality (e.g., a Celsius↔Fahrenheit converter from just a comment).
  • One-shot: a single example — helps Copilot follow your specific patterns and conventions for consistent implementations.
  • Few-shot: several examples — balances zero-shot unpredictability against the precision of fine-tuning; good for sophisticated code covering multiple scenarios and edge cases.
// Few-shot: show the pattern with examples, then ask for the next case.
// greet(6)  => "Good morning"
// greet(14) => "Good afternoon"
// greet(20) => "Good evening"
// Now implement greet(hour) covering all times of day:
function greet(hour) {
  // Copilot follows the demonstrated pattern
}

Exam tips & gotchas

  • Zero-shot = zero examples; few-shot = several examples. One-shot sits between (exactly one).
  • Reach for few-shot when output must follow a precise pattern or handle edge cases; use zero-shot for boilerplate/standard tasks.
  • Few-shot is described as a middle ground between zero-shot unpredictability and full fine-tuning — it does not retrain the model.

Practice

Q1. You need Copilot to output objects that exactly match an existing schema used across your repo. Which approach fits best?
Answer: Few-shot (provide several example objects). Why: multiple examples steer Copilot toward a precise, consistent pattern and edge cases.
Q2. Generating a standard "reverse a string" utility from a one-line comment with no examples is what kind of prompting?
Answer: Zero-shot. Why: no examples are supplied; Copilot relies on its foundational training for a common pattern.

4.1.4 Apply best practices for prompt crafting

Beyond the core principles, Learn calls out a handful of advanced best practices that make each interaction with Copilot more productive. They build directly on the 4 Ss and turn prompting into an efficient dialogue rather than a one-shot guess.

Key facts

  • Provide enough clarity — be explicit (builds on Single + Specific).
  • Provide enough context with details — add descriptive comments, keep related files open (the "Surround" principle); more context = more fitting suggestions.
  • Provide examples for learning — examples clarify expectations and help Copilot learn patterns quickly, cutting revision cycles; great for boilerplate, test templates, and repetitive code.
  • Assert and iterate — if the first output isn't right, don't restart; erase it, enrich the comment with detail/examples, and prompt again so each turn builds on Copilot's understanding.
  • Role prompting — tell Copilot to "Act as a …" (security expert, performance-optimization expert, testing specialist) to get targeted, domain-aware code on the first try.
  • Keep good, representative sample code open so Copilot can mirror your conventions.
Tip: Treat prompting as a conversation. Iterating on an enriched comment usually reaches production-ready code faster than hand-editing a mediocre first suggestion.

Exam tips & gotchas

  • "Assert and iterate" means refine the prompt, not manually rewrite the output from scratch.
  • Role prompting ("Act as a security expert…") is an explicit Learn technique for specialized tasks.
  • Breaking a large task into smaller, focused prompts pairs with the "Single" principle.

Practice

Q1. Copilot's first suggestion is close but misses an edge case. What's the recommended next step?
Answer: Assert and iterate — remove the suggestion, add the missing detail/example to your comment, and prompt again. Why: iterating on an enriched prompt is faster than manual refinement and builds on prior context.
Q2. A team wants security-hardened code from the first attempt. Which best practice helps most?
Answer: Role prompting — "Act as a cybersecurity expert…". Why: assigning an expert role produces more targeted, domain-aware suggestions.

4.2 Engineer prompts for performance

4.2.1 Explain prompt engineering principles

Learn distills prompt engineering into four memorable principles — the 4 Ss. They are the foundation the advanced best practices are built on, so knowing each one and what it targets is high-yield.

Key facts (the 4 Ss)

  • Single — focus each prompt on a single, well-defined task or question; clarity yields accurate responses.
  • Specific — make instructions explicit and detailed; specificity leads to more precise code.
  • Short — keep prompts concise; balance detail without overloading Copilot.
  • Surround — use descriptive filenames and keep related files open to supply rich context.
Key: "Specific" and "Short" pull in opposite directions on purpose — be detailed enough to be unambiguous, but trim anything that doesn't help.

Exam tips & gotchas

  • Memorize the mapping: Single→one task, Specific→explicit detail, Short→concise, Surround→filenames + open files.
  • "Surround" is the principle behind open-tabs/context behavior — a common answer to "why do open files matter?"
  • The advanced best practices (clarity, context, examples, iterate) are extensions of the 4 Ss, not separate principles.

Practice

Q1. Which of the 4 Ss explains why keeping related files open improves suggestions?
Answer: Surround. Why: Surround = descriptive filenames and keeping related files open for richer context.
Q2. A prompt asks Copilot to "build the whole billing system" in one comment. Which principle is violated?
Answer: Single (and arguably Short). Why: a prompt should focus on one well-defined task, not a broad multi-part goal.

4.2.2 Describe prompt process flow and chat history usage

A prompt travels through an inbound path (your request + context → filters → model) and an outbound path (model response → validation → delivery). Understanding the ordered stages explains where safety filtering happens and why some suggestions are truncated or withheld.

In Chat, each turn builds on prior turns, so the conversation history becomes part of the context — powerful, but it grows and gets more expensive, so managing it matters for performance.

Key facts — process flow

  • Inbound 1 — Transmission & context: prompt sent over HTTPS; Copilot gathers context and applies FIM.
  • Inbound 2 — Proxy filter: a proxy in a GitHub-owned Microsoft Azure tenant blocks prompt-hacking / manipulation.
  • Inbound 3 — Toxicity filtering: removes hate speech/inappropriate content and personal data before intent extraction.
  • Inbound 4 — LLM generation: the filtered prompt goes to the model, which produces suggestions.
  • Outbound 5 — Post-processing/validation: checks for code quality (e.g., XSS, SQL injection) and an optional matching-public-code filter (blocks suggestions over ~150 characters that closely match public code); failing content is truncated or discarded.
  • Outbound 6 — Delivery & feedback loop: only responses passing all filters are delivered; acceptances/edits/rejections feed improvement.
  • Outbound 7 — Repeat for subsequent prompts.

Key facts — chat history

  • Each Chat turn builds on previous context (e.g., create function → add error handling → add tests → add docs → optimize); full history keeps growing.
  • Cost: long prompts carrying full history can consume 2–3 PRUs per turn; summarizing or resetting can keep it closer to 1 PRU per request.
  • Manage it: summarize context when conversations get long; reset / start a new conversation with focused context for a new feature; use concise references to prior work instead of repeating full implementations.
Gotcha: Carrying a huge conversation forward doesn't just cost more — irrelevant history can dilute context. Start a fresh conversation when you switch tasks.

Exam tips & gotchas

  • Know the order: transmission → proxy → toxicity → LLM (inbound), then post-processing → delivery → repeat (outbound).
  • The matching-public-code check is optional and admin-enabled, applies to responses over ~150 characters, and happens on the outbound path.
  • Managing chat history (summarize/reset) is a performance lever — fewer PRUs per turn and cleaner context.

Practice

Q1. Put these inbound stages in order: LLM generation, proxy filter, secure transmission & context, toxicity filtering.
Answer: Secure transmission & context → proxy filter → toxicity filtering → LLM generation. Why: context is gathered first, then filtered through proxy and toxicity checks before reaching the model.
Q2. A Chat session has run for 20 turns and feels sluggish and pricey. What's the recommended fix?
Answer: Summarize the relevant context or start a new conversation with focused context. Why: this reduces PRU cost per turn and keeps context clean.
Q3. On which path (inbound or outbound) does the optional public-code matching filter run?
Answer: Outbound, during post-processing/response validation. Why: it inspects the model's generated response before delivery.
