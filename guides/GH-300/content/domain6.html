Domain 6 — Configure privacy, content exclusions, and safeguards 10–15%

This domain covers the administrative controls that keep sensitive code private and steer Copilot's output: content exclusions (repo- and org-level), editor settings, who owns and is responsible for accepted suggestions, the suggestions-matching-public-code duplication filter, and how to troubleshoot missing suggestions or exclusions that seem not to apply. Expect "where do I configure this," "which plan has it," and "why isn't it working" questions.

6.1 Manage privacy settings and exclusions

6.1.1 Configure content exclusions and editor settings

Content exclusions let repository administrators and organization owners stop specific files, directories, or repositories from being used to inform Copilot's code-completion suggestions. The goal is to protect sensitive material (secrets, proprietary logic) from being surfaced or used as context.

Exclusions are configured in organization or repository settings — not toggled per-user inside the editor. They are a Business/Enterprise capability and apply to members licensed under those plans.

Key facts

  • Repository level: repo main page → Settings → (Code & automation) Copilot → Repositories and paths to exclude → list the files/directories.
  • Organization level: profile photo → Your organizations → org Settings → Copilot > Content exclusion → enter files or repositories to exclude.
  • You can specify content exclusions only in org or repository settings; there is no editor-side field to add them.
  • Exclusions set on an org/repo within an enterprise apply to all members licensed as GitHub Copilot Business or Enterprise.
  • What exclusion does: code completion is disabled in the affected files; those files won't inform completions in other files; and they won't inform Copilot Chat responses.
  • Trade-off: excluding files can yield more secure/compliant output but reduces available context, which can lower suggestion accuracy (e.g., excluding a key config file removes context dependent code needs).
# Illustrative path patterns (configured in org/repo Copilot settings).
# "*" applies to all repos; a "owner/repo" key scopes rules to one repo.
"*":
  - "/**/secrets/**"        # any secrets directory, anywhere
  - "/**/*.env"             # environment files
"my-org/payments-service":
  - "/config/keys/*.pem"    # certs/keys in this repo
  - "/src/legacy/**"        # keep a sensitive area out of context
Gotcha: Content exclusions are Business/Enterprise only — Free, Pro, and Pro+ have no content-exclusion feature. A distractor is "any user can exclude files from the editor."

Exam tips & gotchas

  • Two configuration scopes: repository (Repositories and paths to exclude) and organization (Content exclusion). Both live in Settings → Copilot, not in the IDE.
  • Exclusion has three effects: no completion in the file, file not used as context elsewhere, and file not used in Chat. If asked "what does it stop," name all three.
  • Limitations: the @github chat participant in VS Code/Visual Studio is not covered; semantic info (types, hover definitions) can still leak via a non-excluded file; scope is limited to the configuring org's members.
  • Excluding content reduces context — it can make suggestions worse, which is an intentional privacy/quality trade-off.

Practice

Q1. A team lead on Copilot Pro wants to exclude a /secrets folder from Copilot. Can they, and where?
Answer: No — content exclusions require Copilot Business or Enterprise; they are configured in organization or repository Settings → Copilot, not on a personal Pro plan. Why: Content exclusions are a Business/Enterprise policy feature set in org/repo settings.
Q2. After excluding a file, name the three things Copilot stops doing with it.
Answer: (1) No code completion in that file, (2) it won't inform completions in other files, (3) it won't inform Copilot Chat responses. Why: Exclusion removes the file from completion and from context/Chat.
Q3. Why might excluding a critical configuration file reduce the quality of Copilot suggestions elsewhere?
Answer: Excluded files no longer provide context, so dependent code loses the information Copilot would have used, lowering accuracy. Why: Exclusion trades context for privacy/security.

6.1.2 Describe ownership and limitations of outputs

Copilot is a generative-AI assistant, so its output can be wrong, biased, or resemble existing code. The developer keeps ownership of and responsibility for the code they accept and commit — human oversight, review, and validation remain essential regardless of plan.

Organizational plans add contractual protections on top of that responsibility, most notably IP indemnity, but those protections come with conditions rather than removing the need to validate.

Key facts

  • AI systems can make hard-to-interpret decisions and unintended/harmful outputs (bias, privacy issues); mitigation relies on governance, transparency, and human oversight — you validate before accepting.
  • IP indemnity is included with Business and Enterprise. If a Copilot suggestion is challenged as infringing third-party IP, GitHub assumes legal responsibility.
  • Condition: for GitHub to assume that responsibility, the Matching public code setting must be set to Block.
  • A Data Protection Agreement (DPA) and the GitHub Copilot Trust Center document data-protection measures and security/privacy/compliance/IP safeguards.
  • Plan tiers: the public code filter exists on Free, Pro/Pro+, Business, Enterprise; IP indemnity, content exclusions, enterprise-grade security, data excluded from training by default, user management, and usage metrics are Business/Enterprise.
  • Individual subscribers can choose whether GitHub collects and retains their prompts and suggestions.
Key: You own and are responsible for the code you accept. Business/Enterprise IP indemnity only applies when Matching public code = Block — indemnity and the public-code filter are linked.

Exam tips & gotchas

  • IP indemnity is not on Free/Pro/Pro+ — it's a Business/Enterprise contractual protection.
  • Remember the dependency: indemnity requires blocking matching public code. If a question says an org wants indemnity, the correct action includes setting the filter to Block.
  • Contractual protection ≠ correctness guarantee — you still must review, test, and validate outputs.
  • Only Individual plans expose the choice to collect/retain prompts; org plans exclude data from training by default.

Practice

Q1. An enterprise wants GitHub to stand behind Copilot suggestions against third-party IP claims. What must be true?
Answer: They must be on Business/Enterprise (which include IP indemnity) and have Matching public code set to Block. Why: Indemnity requires the public-code filter to be blocked.
Q2. Who is responsible for a security bug in code a developer accepted from Copilot?
Answer: The developer/organization — you own and are accountable for accepted code and must validate it. Why: Copilot is an assistant; human oversight and review remain required.

6.2 Apply safeguards and troubleshoot

6.2.1 Enable suggestions matching public code filtering

The suggestions matching public code filter (duplication detection) identifies code suggestions that closely match publicly available code on GitHub and can Allow or Block them. Blocking helps keep a codebase original and reduces the risk of importing nonsecure or noncompliant look-alike code.

Where the setting lives and who controls it depends on the plan and whether the seat is personal or organization-provided.

Key facts

  • Two choices: Allow or Block suggestions matching public code.
  • The duplication filter targets longer matches — roughly 150+ characters that closely resemble public code are blocked/truncated.
  • Organization (Business/Enterprise): admins can Block for all members. Blocking is required to activate IP indemnity.
  • Personal, individually paid (Free/Pro/Pro+): the user fully controls the Allow/Block toggle under Copilot → Features → Privacy.
  • Personal, org-provided seat: the toggle may be locked and reflects the organization's policy.
  • Org admin path: profile → Your enterprises/Your organizations → Settings → Copilot → Features → Privacy → Suggestions matching public code → Block → Save.
Tip: The public-code filter is available on all plans (Free, Pro/Pro+, Business, Enterprise) — but only org admins can enforce Block for everyone, which is what unlocks IP indemnity.

Exam tips & gotchas

  • Setting location to memorize: Copilot → Features → Privacy → Suggestions matching public code.
  • Personal-paid users control their own toggle; an org-provided seat may have it locked to org policy.
  • The filter targets ~150+ character matches — short snippets aren't the target.
  • Block is the value that enables IP indemnity at Business/Enterprise (ties back to 6.1.2).

Practice

Q1. A Pro user with a seat provided by their organization can't change the "Suggestions matching public code" toggle. Why?
Answer: For an org-provided seat the toggle may be locked and reflects the organization's policy. Why: Only individually paid personal users fully control it; org policy governs org-assigned seats.
Q2. Where does an org admin enable blocking of public-code matches for all members?
Answer: Org/enterprise Settings → Copilot → Features → Privacy → Suggestions matching public code → Block → Save. Why: That's the organizational path, and Block enables IP indemnity.
Q3. Roughly what size of matching snippet does the filter act on?
Answer: About 150 characters or more that closely match public code. Why: Duplication detection targets longer look-alikes.

6.2.2 Resolve issues with suggestions and content exclusions

Two common support scenarios: Copilot shows no suggestions, or a content exclusion doesn't seem to apply. Most causes are environmental (connection, extension version, IDE support, sign-in) or timing/scope-related (propagation delay, org membership). Knowing the checklist and the status-bar signal resolves the majority of cases.

Key facts

  • No suggestions — checklist: verify a stable internet connection; update the Copilot extension; confirm IDE compatibility; and review content exclusions (an excluded file won't show suggestions).
  • Exclusions not applying — propagation: changes can take up to 30 minutes to take effect in IDEs that already loaded the settings; reload the content-exclusion settings in the IDE to apply immediately.
  • Exclusions not applying — scope: exclusions cover only members of the org where they were configured; make sure affected teammates are in scope.
  • Status-bar signal: if an exclusion applies to the current file, the Copilot icon has a diagonal line through it; hover to see whether an org or the parent repository disabled Copilot for that file.
  • IDE-specific limits: in some IDEs, exclusions don't apply to certain features such as Copilot Chat (e.g., the @github participant).
  • Poor-quality suggestions: add clear context (descriptive comments, meaningful names), use Copilot commands (e.g., Ctrl+Enter in VS Code), and adjust prompt length/detail.
Tip: When an exclusion "isn't working," check the status-bar icon (diagonal line = excluded) and remember the up-to-30-minute propagation window — reload settings to apply changes immediately.

Exam tips & gotchas

  • Missing-suggestion checklist keywords: connection, extension version, IDE compatibility, content exclusions (and being signed in).
  • The 30-minute delay applies to already-loaded IDE sessions; a reload forces immediate application.
  • Scope gotcha: exclusions only protect the configuring org's members — outsiders with file access still get suggestions.
  • The diagonal-line status icon is the fastest way to confirm a file is excluded; hover reveals org vs. parent-repo cause.

Practice

Q1. An admin added a content exclusion, but a developer with the IDE already open still gets suggestions in the excluded file. What's the likely cause and fix?
Answer: Propagation delay — changes can take up to 30 minutes in an already-loaded IDE; reload the content-exclusion settings in the IDE to apply immediately. Why: Exclusion changes aren't instant in loaded sessions.
Q2. Copilot shows no inline suggestions at all. List two first checks.
Answer: Any two of: internet connection, Copilot extension is up to date, IDE compatibility, content exclusions applying to the file (and being signed in). Why: These are the standard missing-suggestion causes.
Q3. How can a developer quickly tell a file is under a content exclusion?
Answer: The Copilot status-bar icon shows a diagonal line; hovering says whether an org or the parent repo disabled Copilot for the file. Why: The status bar surfaces exclusion state per file.
