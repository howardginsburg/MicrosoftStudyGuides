Domain 2 — Use GitHub Copilot features 25–30%

This is the largest domain. It covers using Copilot inside the IDE, the Copilot CLI in the terminal, the advanced feature surface (Agent Mode, Copilot Edits, MCP, the Cloud Agent, code review, Spaces, and Chat), and how administrators govern Copilot across an organization.

2.1 Use GitHub Copilot in the IDE

2.1.1 Enable Copilot in the IDE

Before Copilot can suggest code, you sign in with a GitHub account that has a Copilot entitlement and install the extension for your editor. Copilot runs in several environments, so setup differs slightly per surface, and you can enable or disable it at any time.

Key facts

  • Access Copilot from your GitHub profile under Settings → Copilot, in the Code, planning, and automation section.
  • Supported surfaces: GitHub.com (no extension needed), VS Code, Visual Studio, JetBrains IDEs, and Neovim.
  • In VS Code: install the GitHub Copilot extension from the Marketplace, then Sign in to GitHub when prompted.
  • Toggle Copilot from the status icon in the bottom pane; you can disable it globally or per language.
  • Inline completions are controlled under Settings → Extensions → GitHub Copilot → Editor: Enable Auto Completions (also per language).
  • The Free tier includes 2,000 code completions and 50 chat messages per month; educators, students, and select open-source maintainers can qualify for Copilot Pro at no cost.
Tip: If Copilot won't connect, run GitHub Copilot: Collect Diagnostics and open the extension log (Developer: Open Log File). Network firewalls or proxies are common culprits.

Exam tips & gotchas

  • GitHub.com needs no extension; every other supported environment does.
  • Disabling completions offers a choice between globally and for the current language — know both options.
  • The Free tier's limits (2,000 completions / 50 chat messages per month) are frequently tested numbers.

Practice

Q1. A developer wants to use Copilot on GitHub.com to explain code in a repository. What must they install first?
Answer: Nothing — GitHub.com requires no extension. Why: Copilot is built into the web interface; extensions are only needed for IDEs like VS Code, Visual Studio, JetBrains, and Neovim.
Q2. A user on the Free tier is surprised Copilot stopped completing code mid-month. What is the most likely cause?
Answer: They exhausted the monthly Free allowance (2,000 completions). Why: Free is usage-limited; upgrading to Pro removes those caps.
Q3. Where in VS Code do you disable Copilot for only JavaScript files?
Answer: Click the Copilot status icon and choose Disable completions for the language, or set per-language auto completions in Settings. Why: Copilot supports language-scoped enable/disable, not just a global switch.

2.1.2 Trigger Copilot through inline suggestions, chat, CLI, and agent mode

Copilot offers escalating levels of engagement. Inline suggestions autocomplete as you type; chat answers questions in natural language; the CLI helps in the terminal; and agent mode autonomously drives multi-step tasks. Choosing the right mode for a task is a core skill.

Key facts

  • Inline suggestions appear as grey ghost text. Accept with Tab (or →); reject with Esc or by continuing to type.
  • Multiple suggestions: cycle with Alt+] / Alt+[ (Option on macOS) or the light-bulb control.
  • Command palette (Ctrl+Shift+P / Cmd+Shift+P) exposes actions like Explain This and Generate Unit Tests.
  • Copilot Chat is a project-aware panel for natural-language questions; inline chat opens with Ctrl+I / Cmd+I.
  • Comments-to-code: describe intent in a comment and press Enter to have Copilot generate the implementation.
  • Copilot Edits applies changes across multiple files; Agent Mode works autonomously and can run terminal commands; the Copilot CLI brings the assistant into the terminal.

Exam tips & gotchas

  • Match the trigger to the task: ghost text for boilerplate, chat for questions, inline chat for a focused edit, agent mode for multi-file/multi-step work.
  • Inline chat and slash commands (/explain, /tests) are distinct from the main Chat panel.
  • Agent Mode runs terminal commands and iterates; static inline suggestions do not.

Practice

Q1. A developer highlights a function and wants a quick, in-place explanation without leaving the code. Which trigger fits best?
Answer: Inline chat (Ctrl+I) with /explain. Why: Inline chat gives context-specific help at the cursor without switching to the full Chat panel.
Q2. Which mode can install dependencies and run tests on its own as part of completing a task?
Answer: Agent Mode. Why: Agent Mode autonomously determines files, executes edits, and runs terminal commands, iterating until the task succeeds.

2.1.3 Configure content exclusions for specific files or repositories

Content exclusions tell Copilot to ignore specific files, folders, or repositories so sensitive material never informs suggestions. They are configured by repository administrators and organization owners and are a Business/Enterprise capability.

Key facts

  • Configure at the repository level: repo Settings → Copilot → Repositories and paths to exclude.
  • Configure at the organization level: org Settings → Copilot → Content exclusion.
  • Effect on excluded files: no code completion in them; their content won't inform completions in other files or Chat responses.
  • Available only on Copilot Business and Copilot Enterprise (not Free/Pro); settings apply to those licensed members.
  • Limitations: exclusions are not applied to the @github chat participant in VS Code/Visual Studio; semantic info (types, hover definitions) can still surface from non-excluded files; the policy applies only to members of the configuring organization.
Gotcha: Content exclusions reduce the context Copilot has, which can lower suggestion quality — exclude deliberately, balancing security against usefulness.

Exam tips & gotchas

  • Only organization and repository settings define exclusions — there is no per-user IDE exclusion list.
  • Exclusions do not stop the @github participant, and outsiders with repo access still see suggestions.
  • This is a Business/Enterprise-only control — a common distractor is offering it on Pro.

Practice

Q1. A security team excludes secrets/ from Copilot but notices Chat still references a symbol defined there. Why?
Answer: Copilot can still use semantic information (type/hover definitions) surfaced from a non-excluded file, and the @github participant ignores exclusions. Why: Exclusions block completions/context from the file itself but have documented IDE limitations.
Q2. Which plans support content exclusions?
Answer: Copilot Business and Copilot Enterprise. Why: Content exclusion is a management feature not available on Free or Pro.

2.2 Use GitHub Copilot CLI

2.2.1 Define GitHub Copilot CLI and how it benefits developers

GitHub Copilot CLI brings Copilot into the terminal. It explains shell commands, suggests commands from natural-language descriptions, and helps you work interactively and safely with your files — reducing guesswork on the command line.

Key facts

  • Runs in the terminal; can explain commands and suggest shell commands from natural language.
  • Uses GitHub authentication and runs independently from the GitHub CLI (gh), though it reuses your existing credentials.
  • Benefits both newcomers and experienced developers by speeding everyday workflows and reducing manual lookups.

Exam tips & gotchas

  • Copilot CLI is separate from gh but shares your GitHub sign-in — they complement each other.
  • Its two headline capabilities are explain and suggest.

Practice

Q1. Is Copilot CLI the same tool as the GitHub CLI (gh)?
Answer: No. Why: Copilot CLI runs independently but authenticates with your existing GitHub credentials; you can combine it with gh for repo/issue tasks.

2.2.2 Identify the steps for installing GitHub Copilot CLI

Copilot CLI installs via a package manager or an official script, then launches into an interactive session. On first launch it asks whether you trust the current folder, because it may read, modify, or execute files there.

Key facts

  • Install with Homebrew (macOS/Linux) or the official install script.
  • Launch interactive mode with copilot; run a one-shot prompt with copilot -i "…".
  • On first launch Copilot asks you to trust the files in the current folder — only proceed in trusted locations.
# Install (choose one)
brew install copilot-cli
curl -fsSL https://gh.io/copilot-install | bash

# Launch interactive mode
copilot

# One-shot prompts (no full session)
copilot -i "explain brew install git"
copilot -i "suggest find large files and delete them"

Exam tips & gotchas

  • copilot = interactive; copilot -i "…" = one-shot.
  • The first-run trust prompt exists because Copilot may read/modify/execute files in the directory.

Practice

Q1. You just want a single explanation without entering a full session. Which command?
Answer: copilot -i "explain <command>". Why: The -i one-shot flag returns an answer without starting interactive mode.

2.2.3 Describe key GitHub Copilot CLI features and commands

Inside an interactive session you mix natural-language prompts with explicit slash commands. Slash commands control the session and configuration and cannot be replaced by natural language.

Key facts

  • Natural language handles explain, suggest, and revise requests.
  • Common slash commands: /help, /explain <command>, /suggest <task>, /revise, /feedback, /exit.
  • Configuration slash commands: /model <model>, /theme, /skills, /mcp, /list-dirs, /reset-allowed-tools, /sandbox.
  • Slash commands are the only way to control session settings — natural language can't substitute for them.

Exam tips & gotchas

  • Know the split: NL for explain/suggest/revise; slash commands for control/config.
  • /model selects the AI model; /mcp manages MCP servers; /skills manages enhanced capabilities.

Practice

Q1. A developer types "please switch to a different model" in the session but nothing changes. Why?
Answer: Model switching requires the /model slash command. Why: Session settings and configuration can only be changed with slash commands, not natural language.

2.2.4 Use GitHub Copilot CLI interactively and in sessions

Copilot CLI supports interactive sessions where you attach files as context and, for safety, run commands inside sandboxed environments. Sandboxing can be local (on your machine) or in the cloud (a GitHub-hosted, isolated Linux environment).

Key facts

  • Use @ to select a file as context in an interactive session.
  • Local sandboxing (/sandbox enable) restricts filesystem, network, and OS access on your machine.
  • Cloud sandboxing (copilot --cloud) runs in a fully isolated GitHub-hosted Linux environment, built on Azure Container Apps Sandboxes.
  • Cloud sessions have three states — Active, Stopped, Deleted — and a stopped session snapshots its state so you can resume across devices.
  • Organization/enterprise admins must enable the Cloud Sandbox policy before members can use cloud sessions.

Exam tips & gotchas

  • Local sandbox = your machine's resources; cloud sandbox = GitHub-hosted, device-independent, parallelizable.
  • Only cloud sessions can be resumed on another device; they require an admin-enabled policy.

Practice

Q1. A developer wants to start work on one laptop and continue on another without moving files. Which option supports this?
Answer: Cloud sandboxing (copilot --cloud). Why: Cloud sessions run in GitHub-hosted infrastructure and preserve state so they can be resumed from any device.
Q2. Members can't start a cloud sandbox session. What's the most likely governance cause?
Answer: The org/enterprise Cloud Sandbox access policy isn't enabled. Why: Admins must enable that policy before members can use cloud sandboxes.

2.2.5 Generate scripts and manage files with GitHub Copilot CLI

Copilot CLI can turn natural language into shell commands and scripts and act on files, but you stay in control through permissions. Access is governed by trusted directories, tool permissions, path permissions, and URL permissions, and you should always review a suggested command before running it.

Key facts

  • Configuration comes from permission prompts, command-line flags, and local config files.
  • Controls include trusted directories, tool permissions (--allow-tool / --deny-tool), path permissions, and URL permissions.
  • Suggest a command from natural language, then revise it with follow-up prompts.
  • Combine Copilot CLI with the gh CLI for repo and issue management; always review commands before execution.
Key: Copilot CLI never runs destructive commands unattended by design — it proposes, you review, then it executes with your permission.

Exam tips & gotchas

  • Permissions are scoped by directory, tool, path, and URL — know these four axes.
  • Sandboxing plus review-before-execute is the recommended pattern for unfamiliar or destructive operations.

Practice

Q1. How do you stop Copilot CLI from being able to run a particular tool during a session?
Answer: Use the --deny-tool flag (tool permissions). Why: Tool permissions explicitly allow or restrict what Copilot can execute on your behalf.

2.3 Use GitHub Copilot features and capabilities

2.3.1 Use Agent Mode, Copilot Edits, and MCP; manage Agent Sessions and delegate to Sub-Agents

Beyond completions and chat, Copilot offers autonomous capabilities. Copilot Edits applies coordinated multi-file changes; Agent Mode plans and iterates on complex tasks in your IDE; the Model Context Protocol (MCP) connects Copilot to external tools and data; and the Copilot Cloud Agent works as a background teammate inside GitHub, running agent sessions you can track and steer.

Key facts

  • Engagement ladder: inline suggestions → Chat → Copilot Edits (multi-file) → Agent Mode (autonomous, iterative).
  • Agent Mode analyzes the whole workspace, determines relevant files and dependencies, runs terminal commands (build/install/test), and self-heals — detecting failures, correcting, and re-running until tests pass, while keeping you in control.
  • MCP is an open standard ("USB-C for AI tools") that connects models to tools/data. The GitHub MCP Server is hosted at https://api.githubcopilot.com/mcp/; add it via MCP: Add server → HTTP, then authorize with OAuth (or use a PAT, or a local Docker image). It exposes 30+ tools.
  • The Cloud Agent (Pro/Pro+/Business/Enterprise) runs in a GitHub Actions environment. Assign it an issue and it adds an 👀 reaction, creates a copilot/ branch, opens a draft PR, runs an agent session, pushes commits, then posts "Copilot finished work" and requests review.
  • Iterate by mentioning @copilot in a PR comment (write permission required). Draft PRs need human approval; the requester can't approve their own; Actions run only after Approve and run workflows; sessions time out after 1 hour.
  • The Cloud Agent uses Actions minutes + PRUs; customize its environment with .github/workflows/copilot-setup-steps.yml and extend it with MCP tools (tools only; remote OAuth servers aren't supported). Defaults include the GitHub and Playwright MCP servers.
// VS Code: connect the hosted GitHub MCP Server over HTTP + OAuth
{
  "servers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    }
  }
}
Key: The Cloud Agent runs autonomously inside GitHub (Actions-powered), while Agent Mode / Copilot Edits makes autonomous edits locally in your IDE. Don't confuse the two.

Exam tips & gotchas

  • Cloud Agent PRs are always draft, need independent approval, and the requester cannot approve their own — this preserves required-reviews rules.
  • MCP for the Cloud Agent supports tools only (not resources/prompts) and no remote OAuth servers.
  • Comments made on the original issue after assignment aren't seen by the agent — add new info as comments on its PR.
  • Premium reasoning roughly doubles PRU consumption versus the standard model.

Practice

Q1. A developer assigns an issue to Copilot, then adds clarifying details as a comment on the issue, but the agent ignores them. Why?
Answer: The agent only receives the issue content at assignment time; later issue comments aren't seen. Why: To guide it after it starts, mention @copilot on the agent's pull request instead.
Q2. Why can't the developer who asked Copilot to open a PR approve that PR themselves?
Answer: The requester is blocked from approving to preserve the repository's required-review rules. Why: An independent human reviewer must approve before merge.
Q3. Which protocol lets Copilot reach external APIs and enterprise tools in a standardized way?
Answer: The Model Context Protocol (MCP). Why: MCP is an open standard that connects models to external tools and data, strengthening agent mode with agentic loops.

2.3.2 Use Copilot for code review and coding assistance

Copilot acts as a collaborative reviewer: it drafts PR summaries, flags security and style issues, explains unfamiliar code, and suggests fixes — in your IDE before a PR and on GitHub.com as an assigned reviewer. Its role is advisory; humans still make the final call.

Key facts

  • Review features: PR summaries, security fixes (integrated with Code Scanning), line-by-line explanations, drafted review comments, and reviews inside the IDE before opening a PR.
  • On GitHub.com: add Copilot from the Reviewers menu; reviews typically finish in under 30 seconds and land as line comments.
  • Copilot's review is advisory: it does not approve or reject a PR and its comments don't count toward required approvals.
  • Guide it with .github/copilot-instructions.md (repo-wide) and path-specific .github/instructions/*.instructions.md files using an applyTo glob.
  • Automate at scale via rulesets (Request pull request review from Copilot) at repo/org level, paired with status checks; individuals on Pro/Pro+ can enable Automatic Copilot code review.
  • PRUs unlock deeper, context-rich analysis of large diffs.
# .github/instructions/security.instructions.md (frontmatter)
applyTo: "**/*.ts,**/*.tsx"
# Below the frontmatter, plain-language guidance, e.g.:
# Flag unsafe string interpolation and require input validation.

Exam tips & gotchas

  • Copilot review is advisory only — it never blocks a merge or satisfies a required approval.
  • Repo-wide guidance lives in copilot-instructions.md; path-scoped guidance uses *.instructions.md with applyTo.
  • To require Copilot on every PR to a protected branch, use a ruleset, not a personal setting.

Practice

Q1. A team wants Copilot to automatically review all PRs targeting main in a production repo. What should an admin configure?
Answer: A branch ruleset with Require a pull request before merging and Request pull request review from Copilot. Why: Rulesets enforce automatic Copilot review at the repo/org level; the personal auto-review toggle only covers one user's PRs.
Q2. Does a Copilot review count toward a branch's required approvals?
Answer: No. Why: Copilot's comments are advisory; a human reviewer must still approve.

2.3.3 Utilize Spaces, Spark, pull request summaries, and instructions files

These features let you ground and customize Copilot. Copilot Spaces package curated context into a reusable, scoped workspace; PR summaries auto-draft descriptions; instructions files encode team conventions; and GitHub Spark builds apps from natural-language descriptions.

Key facts

  • Copilot Spaces: create at github.com/copilot/spaces; owned by you or an organization. Add Instructions (free text) and Attachments — repo files/folders, linked issues/PRs, uploads, or free text.
  • Spaces always reference the latest version of the main branch and inherit GitHub permissions — a Space grants no new access; viewers only see content they're already entitled to.
  • Governance: keep one job per Space, assign an owner, use clear naming conventions, and set a review cadence.
  • PR summaries: from the PR description editor, use the Copilot icon to generate a summary/outline; this consumes PRUs.
  • Instructions files: repo-wide .github/copilot-instructions.md; path-specific *.instructions.md with an applyTo glob.
  • GitHub Spark: build and deploy applications from natural-language descriptions.
Tip: Don't paste secrets into a Space's free-text notes — link version-controlled files instead so normal review and permissions apply.

Exam tips & gotchas

  • Spaces inherit existing repo/issue/PR permissions and grant no new access — a key governance point.
  • Spaces track the main branch's latest version; keep scope narrow ("one job per Space").
  • Instructions files come in two forms: repo-wide copilot-instructions.md and path-scoped *.instructions.md.

Practice

Q1. A manager worries that sharing a Space will expose private repos to people who lack access. Is that a risk?
Answer: No. Why: Spaces inherit GitHub permissions and grant no new access — only users already entitled to the linked content see it reflected in answers.
Q2. Which file makes Copilot apply security review rules only to TypeScript files?
Answer: A path-specific *.instructions.md file with applyTo: "**/*.ts,**/*.tsx". Why: The applyTo glob scopes instructions to matching paths; copilot-instructions.md would apply repo-wide.

2.3.4 Understand Copilot Chat limits, options, feedback, commands, and prompt files

Copilot Chat becomes far more effective when you scope questions, use slash commands, pick the right model, and give feedback. Prompt/instruction files keep its behavior consistent, and usage limits and model costs are worth knowing.

Key facts

  • Scope references: point at a file with #file:name; use participants @terminal, @vscode, and @github.
  • Slash commands: /doc, /explain, /fix, /generate, /optimize, /tests, and /new (scaffold a project).
  • Model selection: standard models (e.g., GPT-4o) cost ~1 PRU per request; premium reasoning models (e.g., o1-preview/o1-mini) cost ~2 PRUs.
  • Feedback: rate responses with thumbs up / thumbs down.
  • Limits: the Free tier includes 50 chat messages per month; content exclusions are not applied to the @github participant.
  • Prompt files / instructions: custom prompt files keep Copilot's behavior consistent, including when working with specific MCP servers.

Exam tips & gotchas

  • Premium models roughly double PRU cost — reserve them for complex reasoning.
  • The @github participant bypasses content exclusions — a security caveat.
  • #file: scopes to a file; @participants scope to a tool/context.

Practice

Q1. A developer needs deep reasoning on a complex multi-threaded bug and is watching PRU usage. What's the trade-off of choosing a premium model?
Answer: Better reasoning, but roughly double the PRU cost (~2 vs ~1). Why: Premium models consume more PRUs, so reserve them for genuinely complex tasks.
Q2. How do you focus a chat question on a single file named controller.js?
Answer: Reference it with #file:controller.js. Why: The #file: scope tells Chat to focus on that file's contents.

2.4 Manage organization-wide settings and policies

2.4.1 Configure organization-wide policy management and Copilot Code Review policies

Administrators govern Copilot through plan-tier features, privacy controls, and rulesets. The plan you choose determines which management capabilities you get, and policies cascade from the enterprise down through organizations and repositories.

Key facts

  • Plans: Free, Pro/Pro+, Business, Enterprise. Business/Enterprise add user management, data excluded from training by default, enterprise-grade security, IP indemnity, content exclusions, SAML SSO, and usage metrics. Require GitHub Enterprise Cloud is Enterprise-only.
  • Suggestions matching public code: org admins (Business/Enterprise) set it at Settings → Copilot → Features → Privacy and choose Block — which is required for IP indemnity. Personal (Free/Pro/Pro+) users can toggle Allow/Block, but an org-provided seat may be locked to the org policy.
  • Code Review policies: create rulesets that Require a pull request before merging and Request pull request review from Copilot, applied to target branches/repos; optionally Require conversation resolution before merging.
  • The public code filter is available on all plans; content exclusions are Business/Enterprise only.
Gotcha: IP indemnity applies only when the Matching public code setting is set to Block. Allowing matching suggestions forfeits indemnity coverage.

Exam tips & gotchas

  • IP indemnity and content exclusions are Business/Enterprise features; the public code filter exists on every plan.
  • Blocking matching public code is a precondition for IP indemnity.
  • An org-provided seat can have its privacy toggle locked by org policy.

Practice

Q1. An enterprise legal team requires IP indemnity coverage for all Copilot suggestions. What setting is mandatory?
Answer: Set Suggestions matching public code to Block for the organization. Why: IP indemnity applies only when matching public code is blocked.
Q2. A Pro user wants organization-level usage metrics and content exclusions. Can they get them?
Answer: No — those require Business or Enterprise. Why: Usage metrics and content exclusions are management features not available on Free or Pro.
