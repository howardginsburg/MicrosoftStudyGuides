Domain 1 — Design and Implement Data Models 35–40%

This is the largest domain on DP-420, and it is the heart of building on Azure Cosmos DB for NoSQL. It runs from data modeling (schema-agnostic containers, embedding vs. referencing, TTL, versioning) through partitioning strategy (logical vs. physical partitions, choosing a key, synthetic and hierarchical keys) and sizing and scaling (RUs, serverless vs. provisioned vs. free tier, database-level throughput) to the hands-on SDK skills: connectivity, the SQL query language, point/batch/bulk operations, concurrency and consistency controls, and server-side JavaScript. 1.1 Design and Implement a Non-Relational Data Model for Azure Cosmos DB for NoSQL

1.1.1 Store Multiple Entity Types in the Same Container

Cosmos DB containers are schema-agnostic: unlike a relational table, a single container can hold items of completely different shapes. The resource hierarchy is account → database → container → item, and throughput is provisioned at the database or container level — so co-locating entity types can mean fewer throughput buckets to pay for.

You tell entity types apart with a discriminator property, commonly named type. When related types share a partition key value, they land in the same logical partition and can be read together in a single, cheap query.

Key facts

  • One container can store heterogeneous items (e.g., book and review) — there is no fixed schema.
  • Add a discriminator property (e.g., type/entityType) so application code and queries can filter by shape.
  • Co-locating types that share a partition key value enables single-partition reads across those types.
  • Fewer containers can reduce the number of separately provisioned throughput allocations.
-- Two entity types share partition key /bookId, distinguished by c.type
SELECT * FROM c WHERE c.bookId = "B-100" AND c.type = "review"
Key: Co-locate entity types that are read together and share a partition key value; use a type discriminator to separate them in queries.

Exam tips & gotchas

  • "Multiple types in one container" is a modeling advantage in Cosmos DB — not something to normalize away.
  • The benefit only appears when the types share a partition key value; otherwise reads still fan out.
  • Don't confuse a discriminator (type) with the partition key — they are separate properties.

Practice

Q1. You want to read a book and all of its reviews in a single, low-cost request. What two design choices make this possible?
Answer: Store books and reviews in the same container, both partitioned by the same value (e.g., /bookId), and add a type discriminator. Why: Shared partition key value = one logical partition = a single-partition query returns both types.
Q2. How does an application distinguish a "book" item from a "review" item when they live in the same container?
Answer: By a discriminator property such as type that each item carries. Why: Containers have no schema, so the app relies on a known property to branch logic and filter queries.

1.1.2 Store Multiple Related Entities in the Same Document (Embedding)

Embedding nests related data — arrays and sub-objects — inside a single JSON item. Because Cosmos DB has no server-side joins across items, an embedded (denormalized) aggregate is returned by a single point read, which is the fastest and cheapest operation available.

Key facts

  • Embed for contained / one-to-few relationships, data that changes together, and data that is read together.
  • Embedding generally gives the best read performance (one operation, no follow-up reads).
  • Do not embed unbounded (1:many that grows without limit) or high-write child collections — an item has a hard 2 MB size limit.
{
  "id": "order-42",
  "customerId": "C-1001",
  "type": "order",
  "lines": [
    { "sku": "A1", "qty": 2 },
    { "sku": "B7", "qty": 1 }
  ]
}

Exam tips & gotchas

  • Trigger words for embed: "read together", "one-to-few", "bounded", "changes together".
  • Trigger words against embed: "unbounded growth", "written independently", "very large child list".
  • The 2 MB item ceiling is the concrete reason unbounded embedding fails.

Practice

Q1. An order has at most a handful of line items always displayed with the order. Embed or reference?
Answer: Embed the line items in the order document. Why: Bounded, one-to-few, always read together — a single point read returns everything.
Q2. A social post can accumulate millions of comments over time. Why is embedding the comments a poor choice?
Answer: Unbounded growth would eventually exceed the 2 MB item limit and make every post read expensive. Why: Unbounded 1:many should be referenced, not embedded.

1.1.3 Develop a Model That Denormalizes Data Across Documents

Denormalization copies frequently read fields into other documents so a read doesn't need a second lookup. It trades cheaper reads for more expensive writes (duplicated data must be kept in sync) and extra storage — an excellent bargain for read-heavy, rarely-changing data.

Key facts

  • Duplicate stable fields (e.g., author name, thumbnail) into each document that displays them, avoiding a join/second read.
  • Keep duplicated copies consistent with the change feed (e.g., an Azure Function) when the source of truth changes.
  • Precomputed aggregates (e.g., countOfReviews) are a denormalization pattern that optimizes read-heavy workloads.
{
  "id": "book-100",
  "type": "book",
  "title": "Cosmos in Depth",
  "authorId": "A-9",
  "authorName": "Jordan Lee",   // denormalized copy
  "reviewCount": 128            // precomputed aggregate
}
Tip: Denormalize the few, stable, read-together fields — not the whole related entity. Sync via change feed when the master changes.

Exam tips & gotchas

  • Denormalization is right when reads dominate and the duplicated field rarely changes.
  • The cost is write fan-out; the change feed is the standard mechanism to propagate updates.

Practice

Q1. A book listing shows the author's name on every card, and author names almost never change. What denormalization improves read performance, and how do you keep it correct?
Answer: Copy authorName into each book document; on the rare author rename, use the change feed to update the copies. Why: Stable, read-together data denormalizes well; change feed handles the infrequent sync.

1.1.4 Develop a Design by Referencing Between Documents (Normalization)

Referencing stores a related item's id (a soft foreign key) instead of embedding it. Each entity is its own item, so the app issues additional reads to follow references. Cosmos DB enforces no foreign keys — referential integrity is the application's (or a trigger/change-feed's) responsibility.

Key facts

  • Reference for one-to-many unbounded, many-to-many, and frequently changing / large related data.
  • Normalizing generally gives better write performance but can require more round trips on read.
  • No enforced constraints — no cascading deletes; the app maintains integrity.
  • Hybrid models are the norm: embed the few, reference the many.

Exam tips & gotchas

  • Embed vs. reference is the single most tested modeling decision — anchor on read pattern, cardinality, and volatility.
  • "Frequently updated independently" or "unbounded" → reference; "small, stable, read together" → embed.

Practice

Q1. Publishers each have thousands of books, and book records change often. Should a book embed its publisher, or reference it?
Answer: Reference the publisher (store publisherId). Why: One-to-many-unbounded and volatile data is referenced; embedding would bloat items and duplicate volatile data.
Q2. Who enforces that a referenced publisherId actually exists?
Answer: The application (optionally aided by triggers or change-feed processing). Why: Cosmos DB has no foreign-key constraints or referential integrity.

1.1.5 Identify the Partition Key, id, and Unique Keys

An item is uniquely identified by the combination of its partition key value + id. The id only has to be unique within a logical partition. Unique keys add extra uniqueness constraints on other property paths — but, like id, they are enforced only within a single logical partition.

Key facts

  • id is a required string, unique within a logical partition, and case-sensitive.
  • The partition key value is immutable after an item is created — to "change" it you delete and recreate the item.
  • Unique keys are defined only at container creation and are immutable thereafter.
  • Uniqueness is scoped to a logical partition (the same email can exist under two different partition key values).
  • Limits: up to 16 paths across unique-key definitions and up to 10 unique-key constraints; sparse is not supported; unique keys add a small write RU overhead.
// Unique key on /email, scoped to each logical partition (set at creation only)
var props = new ContainerProperties("users", partitionKeyPath: "/tenantId");
props.UniqueKeyPolicy = new UniqueKeyPolicy
{
    UniqueKeys = { new UniqueKey { Paths = { "/email" } } }
};
await database.CreateContainerAsync(props, throughput: 400);
Gotcha: Unique keys are enforced per logical partition — not globally. They cannot be added or changed after the container exists.

Exam tips & gotchas

  • Partition key + id = the unique identity of an item; a point read needs both.
  • You cannot alter the partition key path, the id uniqueness scope, or the unique-key policy after creation.

Practice

Q1. You need every user's email to be unique per tenant, and the container is partitioned by /tenantId. How do you enforce it, and when must you configure it?
Answer: Add a unique key on /email at container creation. Why: Unique keys are scoped to the logical partition (here, a tenant) and can only be set when the container is created.
Q2. A developer wants to change an item's partition key value after inserting it. What must they do?
Answer: Delete the item and recreate it with the new partition key value. Why: The partition key value is immutable once the item exists.

1.1.6 Identify Data and Associated Access Patterns

In Cosmos DB you model around how the application reads and writes, not around normalized entity relationships. List the queries and mutations first — their frequency, latency needs, and volume — then choose document shape (embed vs. reference) and partition key to serve the top ones with cheap single-partition operations.

Key facts

  • Read-heavy workloads favor embedding/denormalization; write-heavy favor referencing/normalization.
  • Enumerate access patterns first; then shape items and pick the partition key to make the most frequent/critical query a single-partition read.

Exam tips & gotchas

  • "Query-first" (access-pattern-driven) design is the mindset the exam rewards — do not start from a normalized ER diagram.

Practice

Q1. Before choosing embed vs. reference and a partition key, what should you enumerate first?
Answer: The application's read and write access patterns (queries, frequency, latency, volume). Why: Cosmos DB modeling is driven by access patterns, not entity normalization.

1.1.7 Specify a Default TTL on a Container for a Transactional Store

Time to Live (TTL) auto-expires items so you don't pay to store or delete stale data manually. The container's DefaultTimeToLive master-switch decides whether TTL is on at all, and what the default expiry is; individual items can override it.

Key facts

  • DefaultTimeToLive: null = TTL off (no expiry); -1 = TTL on, no default expiry (items only expire if they set their own ttl); n = items expire n seconds after last modification (_ts).
  • An item's ttl overrides the container default, but only takes effect if the container's DefaultTimeToLive is not null.
  • Expired items vanish from queries immediately; a background process deletes them using leftover RUs (provisioned) or billed RUs (serverless).
  • TTL requires indexing — it cannot be enabled on an indexing-mode-none container. Max TTL = 2,147,483,647 seconds.
// Enable TTL with a 30-day default; items can override with their own ttl
var props = await container.ReadContainerAsync();
props.Resource.DefaultTimeToLive = 60 * 60 * 24 * 30; // 2,592,000 seconds
await container.ReplaceContainerAsync(props.Resource);
Key: -1 means "TTL is enabled but nothing expires by default" — the switch that lets per-item ttl work without expiring everything.

Exam tips & gotchas

  • Item ttl does nothing if the container's DefaultTimeToLive is null.
  • Distinguish the three container values: null (off), -1 (on, no default), positive n (on, default n seconds).
  • You can't use TTL with indexing mode none.

Practice

Q1. You want most items to live forever but a few to self-expire via their own ttl. What container DefaultTimeToLive value do you set?
Answer: -1. Why: It enables TTL without a blanket expiry, so only items that set their own ttl expire.
Q2. Items set a ttl property but never expire. The container's DefaultTimeToLive is null. What's wrong?
Answer: TTL is disabled at the container level; item ttl is ignored until DefaultTimeToLive is set to -1 or a positive number. Why: The container switch must be non-null for item TTL to work.

1.1.8 Develop a Design for Versioning Documents

To keep a history of an entity, don't overwrite in place — store prior states as separate versioned items (snapshot, or event-sourcing snapshot+delta). The "current" document stays a fast point read; historical versions are queried on demand and can be aged out with TTL.

Key facts

  • Keep each version as its own item (e.g., id = "order-42_v3") or in a dedicated history container fed by the change feed.
  • Use per-item ttl (with container TTL -1) to automatically age out old versions.
  • The change feed is a natural trigger to write a history record whenever the source item changes.

Exam tips & gotchas

  • Versioning = new items, not in-place mutation; that's what preserves history.
  • Combine with the change feed (to capture changes) and TTL (to retire old versions).

Practice

Q1. A compliance rule requires keeping the last 12 months of edits to each contract but the app must still read the current contract with one point read. Outline a design.
Answer: Keep the current contract as one item; on each change, write a versioned history item (change-feed triggered) with a 12-month ttl. Why: Current stays a fast point read; history is retained and auto-expired.

1.1.9 Develop a Design for Document Schema Versioning

Because containers are schema-free, item shapes can evolve over time. Add a schemaVersion property to each item so code can branch on the shape it reads, supporting forward/backward compatibility while you migrate lazily (on read/write) or in bulk.

Key facts

  • Tag items with schemaVersion (or _schema) so consumers know how to interpret them.
  • Migrate lazily (upgrade an item when it's next written) or with a one-time bulk pass.
  • Mixing versions in one container is fine — the schema-free engine doesn't care.

Exam tips & gotchas

  • Schema versioning (tracking item shape) is different from document versioning (history of an entity's data).

Practice

Q1. Your item shape must change but you can't migrate all existing items at once. What single property lets old and new code coexist?
Answer: A schemaVersion property on each item. Why: Consumers branch on it, so mixed shapes coexist and you can migrate lazily.

1.2 Design a Data Partitioning Strategy for Azure Cosmos DB for NoSQL

1.2.1 Choose a Partitioning Strategy Based on a Specific Workload

A logical partition is the set of all items sharing a partition key value; it is also the scope of transactions. Cosmos DB maps logical partitions onto internally managed physical partitions and splits them automatically as data and throughput grow — you never manage physical partitions directly. Your job is to pick a key that spreads storage and requests evenly.

Key facts

  • Logical partition limits: up to 20 GB of storage and 10,000 RU/s per partition key value.
  • Physical partitions are managed internally (each up to ~50 GB and ~10,000 RU/s); splits are automatic and non-disruptive.
  • Provisioned RU/s is divided evenly across physical partitions.

Exam tips & gotchas

  • Memorize the two ceilings: 20 GB per logical partition; ~10,000 RU/s per physical partition.
  • You never create/split physical partitions; the service does. You only choose the partition key.

Practice

Q1. A single partition key value is projected to store 35 GB. What limit does this violate and what must change?
Answer: It exceeds the 20 GB logical-partition limit; choose a higher-cardinality, synthetic, or hierarchical key so no single value grows that large. Why: A logical partition (one key value) cannot exceed 20 GB.

1.2.2 Choose a Partition Key

The partition key is the most consequential design decision. A good key has high cardinality (many distinct values), spreads both storage and request volume evenly, and appears as an equality filter in the most frequent and critical queries so those queries stay single-partition.

Key facts

  • Ideal key: high cardinality, immutable, even storage + RU distribution, aligned to common query filters.
  • Anti-patterns: low-cardinality fields (status, country, type); a monotonically increasing value like a timestamp (write hotspot); a random GUID that's never used in filters (all queries fan out).
  • Partition key value length ≤ 2048 bytes (≤ 101 bytes without the large-partition-key setting).
Gotcha: A perfectly distributed key that never appears in your query filters is still bad — every query fans out. Alignment to query filters matters as much as cardinality.

Exam tips & gotchas

  • Balance two forces: even distribution (avoid hot partitions) and filter alignment (avoid fan-out).
  • A timestamp key concentrates all current writes on one partition — classic hotspot.

Practice

Q1. Most queries filter by customerId, and customers are numerous and evenly active. Why is /customerId a strong partition key?
Answer: High cardinality + even load + it's the common query filter, so those queries are single-partition. Why: It satisfies distribution and filter-alignment together.
Q2. Why is partitioning telemetry by /eventDate (today's date) a poor choice for a write-heavy workload?
Answer: All of today's writes hit a single logical partition — a hot partition that throttles. Why: A monotonically changing key concentrates current traffic.

1.2.3 Plan for Transactions When Choosing a Partition Key

Every Cosmos DB transaction — a transactional batch or a stored-procedure/trigger — is scoped to a single logical partition. If a group of items must be modified atomically, they must share the same partition key value. This constraint often drives the partition key choice.

Key facts

  • Transactional batches and server-side scripts operate within one partition key value only.
  • Choose a key that co-locates items that need atomic multi-item changes.

Exam tips & gotchas

  • "Atomic across items" is a hard requirement for shared partition key value — there are no cross-partition transactions.

Practice

Q1. A funds transfer must debit and credit two accounts atomically. What partition-key constraint does this impose?
Answer: Both account items must share the same partition key value (e.g., partition by /transferGroupId) so a transactional batch can update them atomically. Why: Transactions are single-logical-partition only.

1.2.4 Evaluate the Cost of a Cross-Partition Query

A query that omits the partition key (or spans many values) is a cross-partition (fan-out) query: the engine runs it on every physical partition and merges results client-side, raising RU cost and latency. Including the partition key (or a hierarchical-key prefix) routes the query to a subset of partitions.

Key facts

  • Fan-out adds roughly a few RU per extra physical partition plus merge latency; single-partition queries are cheapest.
  • ORDER BY results are merge-sorted across partitions and COUNT/aggregates are summed across partitions on fan-out.
  • Add the partition key to the WHERE clause to make a query single-partition.
-- Single-partition (cheap): partition key in the filter
SELECT * FROM c WHERE c.customerId = "C-1001" AND c.status = "open"

-- Cross-partition (fans out to all physical partitions)
SELECT * FROM c WHERE c.status = "open"

Exam tips & gotchas

  • The single biggest RU lever on reads is whether the partition key is in the filter.
  • Cross-partition isn't forbidden — just costlier; the SDK parallelizes it with MaxConcurrency.

Practice

Q1. The container is partitioned by /customerId. Which query is cheaper: filtering by customerId and status, or by status alone? Why?
Answer: Filtering by customerId (and status) is cheaper — it's single-partition. Status alone fans out to every physical partition. Why: Including the partition key avoids fan-out.

1.2.5 Evaluate Data Distribution Based on Partition Key Selection

Uneven partition key values create storage skew: one value grows toward the 20 GB logical-partition ceiling while others stay small. Estimate the storage each key value will accumulate, and use the Azure Monitor "storage per logical partition key" view to detect skew before it becomes a wall.

Key facts

  • A single logical partition (one key value) cannot exceed 20 GB.
  • Evaluate value frequency/volume; a few "whale" values (a huge tenant) signal skew.
  • Remedy skew with a synthetic or hierarchical key when a value grows too large.

Exam tips & gotchas

  • Storage skew → 20 GB wall; monitor the per-logical-partition storage metric.
  • Hierarchical keys specifically let a first-level value exceed 20 GB by subpartitioning.

Practice

Q1. Partitioning multi-tenant data by /tenantId, one enterprise tenant is heading toward 20 GB. What's a good remedy?
Answer: Use a hierarchical partition key (e.g., /tenantId then /userId) so the large tenant is subpartitioned beyond 20 GB. Why: HPK lets a top-level value exceed the single-partition limit.

1.2.6 Evaluate Throughput Distribution Based on Partition Key Selection

Provisioned RU/s is split evenly across physical partitions. If requests concentrate on one partition key value, that partition can throttle (429) even while the container's total RU/s sits well under budget — a hot partition wastes the rest of your throughput.

Key facts

  • RU/s ÷ number of physical partitions = the budget available to each; skew starves the hot one.
  • A hot partition causes 429s despite spare aggregate RU/s.
  • Aim for even RU consumption across partition key values.

Exam tips & gotchas

  • "429 even though total RU/s looks fine" = hot partition / uneven request distribution.

Practice

Q1. A container provisioned at 40,000 RU/s throttles requests to one popular product while overall usage is low. Diagnose it.
Answer: A hot partition — that product's key value receives disproportionate requests and exceeds its per-partition RU slice. Repartition (synthetic/hierarchical key) to spread the load. Why: RU/s is divided evenly across partitions; concentration causes 429s.

1.2.7 Construct and Implement a Synthetic Partition Key

When no single property distributes well, build a synthetic partition key: concatenate several properties, append a random/bucketed suffix (to spread writes), or precompute a hash. You store the composed value as a real property on each item and partition on it.

Key facts

  • Techniques: concatenate properties (tenantId-deviceId), add a random suffix (e.g., 1–N buckets), or hash values.
  • Raises cardinality and spreads writes, avoiding hotspots.
  • Downside: queries that filter on only one component may fan out unless every component value is supplied.
// Compose and store a synthetic key before insert
item.PartitionKey = $"{item.TenantId}-{item.DeviceId}";
await container.CreateItemAsync(item, new PartitionKey(item.PartitionKey));

Exam tips & gotchas

  • A random-suffix synthetic key spreads writes but makes reads that need all items scan every bucket.
  • The synthetic value must be a real, stored property — you can't partition on a computed-only value.

Practice

Q1. Devices all report under a handful of tenant IDs, causing hot partitions. What synthetic key improves distribution while keeping tenant queries efficient?
Answer: Concatenate tenantId with deviceId (e.g., tenantId-deviceId) and store it as the partition key. Why: It raises cardinality and spreads writes; queries supplying both components stay targeted.

1.2.8 Design and Implement a Hierarchical Partition Key (Subpartitioning)

A hierarchical partition key (HPK) uses up to three ordered levels (e.g., /tenantId → /userId → /sessionId). Because data is subpartitioned, a single first-level value can exceed the 20 GB / 10,000 RU/s single-partition limits, while queries that supply the full key or a prefix still route to a small subset of partitions.

Key facts

  • Up to 3 levels; defined as kind: MultiHash, version: 2; set only at container creation.
  • Lets a first-level prefix exceed 20 GB and 10,000 RU/s by spreading it across physical partitions.
  • Queries by the full key or a leading prefix are targeted; filtering only a middle or lower level fans out.
  • The first level should have high cardinality; supported in newer .NET/Java/Python/JS SDKs.
// Create a container with a 3-level hierarchical partition key
var props = new ContainerProperties(
    id: "sessions",
    partitionKeyPaths: new List<string> { "/tenantId", "/userId", "/sessionId" });
await database.CreateContainerAsync(props, throughput: 400);

// Prefix read: all items for a tenant+user route to a partition subset
var pk = new PartitionKeyBuilder().Add("tenant-9").Add("user-3").Build();
Key: HPK's superpower is letting a first-level value exceed 20 GB while keeping prefix queries efficient. Filtering only a lower level still fans out.

Exam tips & gotchas

  • Prefix must be leading: filtering /userId without /tenantId fans out.

1.3 Plan and Implement Sizing and Scaling for a Database Created with Azure Cosmos DB

1.3.1 Evaluate Throughput and Data Storage Requirements

Throughput is measured in Request Units (RU/s), a currency that normalizes CPU, memory, and IOPS. A 1-KB point read costs about 1 RU; writes and queries cost more. The same operation on the same data always costs the same number of RUs, which makes capacity planning deterministic.

Key facts

  • Estimate RU/s = operations/second × RU-per-operation, with headroom; storage is billed per GB.
  • RU cost rises with item size, number of indexed properties, query complexity, and stronger consistency (strong/bounded reads ≈ 2× the RU of weaker levels).
  • The capacity planner converts a workload profile into RU/s and storage estimates.

Exam tips & gotchas

  • 1 KB point read ≈ 1 RU is the anchor number; writes cost several times more.
  • Stronger consistency roughly doubles read RU cost — a planning factor.

Practice

Q1. Roughly how many RUs does a 1-KB point read consume, and what raises that number?
Answer: About 1 RU; larger items, more indexed paths, complex queries, and stronger consistency increase it. Why: RUs normalize the total resource cost of an operation.

1.3.2 Choose Between Serverless, Provisioned, and Free Tier

Cosmos DB offers three capacity models. Provisioned throughput reserves RU/s (manual or autoscale) and suits sustained, predictable traffic and global distribution. 1.4.1 Choose a Connectivity Mode (Gateway vs. Direct)

The SDK reaches Cosmos DB in one of two modes. 1.5.1 Write Queries with Arrays, Nested Objects, Aggregation, and Ordering

The NoSQL query language operates over JSON. You navigate nested properties with dot/bracket notation, unwind arrays with an intra-item JOIN (JOIN t IN c.tags), aggregate with COUNT/SUM/AVG/MIN/MAX (optionally with GROUP BY), and sort with ORDER BY.

Key facts

  • JOIN in Cosmos DB is a self-join within a single item to unwind arrays — not a cross-document join.
  • Aggregates: COUNT, SUM, AVG, MIN, MAX; grouping via GROUP BY.
  • ORDER BY defaults to ascending; multi-property or mixed-direction ordering needs a composite index.
SELECT c.id, COUNT(1) AS tagCount
FROM c
JOIN t IN c.tags
WHERE t = "sale"
GROUP BY c.id
-- ORDER BY on multiple fields requires a composite index

Exam tips & gotchas

  • JOIN unwinds an array inside one item; it never joins two different documents.
  • Multi-field or mixed asc/desc ORDER BY fails without a matching composite index.

Practice

Q1. In Cosmos DB NoSQL, what does JOIN t IN c.tags actually do?
Answer: It unwinds the tags array within each item (an intra-item self-join). Why: There are no cross-document joins; JOIN flattens arrays inside a single document.
Q2. An ORDER BY status ASC, price DESC query fails. What's missing?

1.5.2 Implement a Correlated Subquery

A correlated subquery is nested inside the outer query and references outer values; it's evaluated per outer item. It's the idiomatic way to filter or aggregate over an array once and reuse the result, and it can dramatically cut the cost of array JOINs by reducing the cross-product.

Key facts

  • Used to filter arrays, run scalar aggregate subqueries, and back EXISTS/ARRAY expressions.

1.6.1 Choose Between a Point Operation and a Query Operation

A point read fetches a single item by id + partition key. It is the cheapest and lowest-latency operation available (≈ 1 RU for 1 KB) and bypasses the query engine. Use a query only when you don't know the item's id or need to filter across items.

Key facts

  • Point read (ReadItemAsync with id + partition key) ≈ 1 RU for ~1 KB; lowest latency.
  • Query cost varies with fan-out, filters, and result size — always ≥ a point read.
  • Prefer a point read whenever the id and partition key are known.

Exam tips & gotchas

  • "Know the id? → point read." Even a single-item query is more expensive than a point read.

1.6.2 Implement Point Operations That Create, Update, and Delete Items

The SDK exposes point operations for full-item CRUD, each taking the item's partition key for efficient routing. Know the semantic differences: Create fails if the id exists, Replace overwrites an existing item, and Upsert does whichever applies.

Key facts

  • CreateItemAsync — insert; fails with 409 if the id already exists in the partition.
  • ReplaceItemAsync — full overwrite; the id must already exist.
  • UpsertItemAsync — create or replace.
  • ReadItemAsync / DeleteItemAsync — read/delete by id + partition key.
await container.CreateItemAsync(order, new PartitionKey(order.CustomerId));   // 409 if exists

await container.UpsertItemAsync(order, new PartitionKey(order.CustomerId));   // create or replace

  • Always pass the partition key for efficient, correctly-routed operations.

1.6.3 Update an Item by Using a Patch Operation (Partial Document Update)

Patch sends only the changed paths to the server rather than the whole document. It's cheaper and lower-bandwidth than a full replace and avoids read-modify-write races because you never round-trip the full item.

Key facts

TransactionalBatch groups several point operations into a single all-or-nothing transaction — but only across items sharing one logical partition key. If any operation fails, the whole batch rolls back.

Key facts

  • All operations must target the same partition key value.
  • Bounded by 100 operations or 2 MB payload per batch.
  • Atomic: one failure rolls back all; per-operation status codes are returned (424 = a skipped/dependent op).
var pk = new PartitionKey("C-1001");
var batch = container.CreateTransactionalBatch(pk)
    .CreateItem(orderHeader)
    .CreateItem(orderLine1)
    .CreateItem(orderLine2);
using var resp = await batch.ExecuteAsync();
if (!resp.IsSuccessStatusCode) { /* whole batch rolled back */ }
Gotcha: Every item in a batch must share the same partition key value. There is no cross-partition transactional batch.

Exam tips & gotchas

  • Limits to memorize: same partition key, ≤ 100 ops, ≤ 2 MB.
  • Batch = atomic; bulk = high-throughput but not atomic (see 1.6.5).

Answer: TransactionalBatch, with all four items sharing the same partition key value. Why: Batches are atomic but single-logical-partition only.

1.6.5 Perform a Multi-Item Load Using Bulk Support in the SDK

1.6.6 Implement Optimistic Concurrency Control Using ETags

Every item carries an _etag that changes on each write. To avoid lost updates without locking, pass the etag you read via IfMatchEtag; if the stored item changed in the meantime, the write fails with HTTP 412 (Precondition Failed), letting you re-read and retry.

Key facts

  • Set ItemRequestOptions.IfMatchEtag to the item's _etag on update/delete.
  • Mismatch → 412; the caller re-reads (getting the fresh etag) and retries.
  • Lock-free — good for read-modify-write without blocking.
var opts = new ItemRequestOptions { IfMatchEtag = current.ETag };
try {
    await container.ReplaceItemAsync(current, current.Id,
        new PartitionKey(current.CustomerId), opts);
} catch (CosmosException e) when (e.StatusCode == HttpStatusCode.PreconditionFailed) {
    // 412: someone else changed it — re-read and retry
}

Exam tips & gotchas

  • 412 is the "someone beat you to it" code for optimistic concurrency — handle by re-reading.

1.6.11 Handle Transient Errors and Request Rate Limiting (429)

Key facts

1.7.1 Write, Deploy, and Call a Stored Procedure

Stored procedures are JavaScript functions that run on the server against a container, registered under container.Scripts and invoked with a partition key value. They access data through the context/collection object and run inside an implicit transaction.

Key facts

  • Registered per container; use getContext(), getCollection(), and getResponse().
  • Executed with a required partition key value; the RU charge is estimated up front (averaged from prior runs).
  • All operations run in one implicit, partition-scoped transaction.
function createItem(doc) {
    var context = getContext();
    var container = context.getCollection();
    var accepted = container.createDocument(container.getSelfLink(), doc,
        function (err, created) {
            if (err) throw err;
            context.getResponse().setBody(created.id);
        });
    if (!accepted) throw new Error("Not enough time/RUs; retry.");
}

Exam tips & gotchas

  • A stored proc always runs against a single partition key value you supply at execution.
  • Collection methods return a boolean; false means the operation was queued out of budget — stop and let the client retry.

Answer: A partition key value. Why: Stored procedures run against a single logical partition, so the runtime needs the key to scope execution.

1.7.3 Implement and Call Triggers

Triggers are JavaScript that runs around an operation. Pre-triggers run before a write (validate or mutate the incoming item via getRequest().getBody()); post-triggers run after, in the same transaction. Triggers are not automatic — the caller must name them in request options.

Key facts

  • Pre-triggers take no input parameters and manipulate the request body; post-triggers run after the operation, in the same transaction.

// Pre-trigger: stamp a createdAt timestamp on the incoming item

}

Domain 2 — Design and Implement Data Distribution 5–10%

Azure Cosmos DB is a globally distributed database, and this domain covers how you spread data across Azure regions, choose the right consistency guarantee, plan for regional failures, and enable active-active writes. Although only 5–10% of the exam, these topics generate some of the most nuanced questions: the five consistency levels and their trade-offs in latency, RU cost, and availability; the difference between service-managed and manual failover; and how conflict resolution works when every region can accept writes. Master the consistency table and the failover behaviours and you will bank these points quickly.

2.1 Design and implement a replication strategy

2.1.1 Choose when to distribute data

An Azure Cosmos DB account can transparently replicate its data across any number of Azure regions. You associate regions with the account to place data near your users, and Azure Cosmos DB keeps every region synchronised according to the account's consistency level. Because distribution is a configuration setting rather than an application concern, you can add or remove regions at any time with no application redeploy and no downtime.

Distributing data is not free: each region carries its own provisioned throughput (RU/s) and storage charge, so you distribute deliberately — to cut user latency, survive regional outages, or satisfy data-residency requirements.

Key facts

  • Reasons to distribute: lower read/write latency (data near users), business continuity during a regional outage, global read/write scale, and data-residency compliance.
  • Regions can be added or removed dynamically without redeploying or pausing the application; the account stays highly available during the change.
  • Throughput (RU/s) is billed per region, so each added region multiplies cost — distribute intentionally.
  • Serverless accounts are single-region only. Global distribution requires provisioned (manual or autoscale) throughput.
  • A single-region account can still enable availability zones (zone redundancy) for intra-region resiliency without adding a second region.
Key: Serverless cannot be globally distributed. If a scenario needs multiple regions, the account must use provisioned throughput.

Exam tips & gotchas

  • Adding/removing a region requires no code change and no downtime — watch for distractors claiming redeployment is needed.
  • Serverless = one region. Global distribution ⇒ provisioned throughput.
  • Every extra region multiplies RU/s cost; distribution improves latency/availability but raises the bill.

Practice

Q1. An architect wants a single Cosmos DB account to serve low-latency reads from three continents and survive a regional outage. The current account is serverless. What must change first?
Answer: Convert the account to provisioned (or autoscale) throughput. Why: Serverless accounts are single-region only, so global distribution is impossible until the account uses provisioned throughput; only then can additional regions be associated.
Q2. A team is told that adding a fourth region to their Cosmos DB account will require an application redeployment and a maintenance window. Is this correct?
Answer: No — regions can be added at any time with no redeploy and no downtime. Why: Azure Cosmos DB begins replicating to the new region while the account stays available; the only planning needed is for the added RU/s and storage cost.

2.1.2 Define automatic (service-managed) failover policies for regional failure

For a single-write-region (single-master) account, service-managed failover lets Azure Cosmos DB automatically promote a read region to the write region if the write region suffers an outage. You configure it in two steps: enable service-managed failover on the account, then define a failover priority order that ranks each region. During a write-region outage, the highest-priority available region is promoted to write.

Service-managed failover trades control for automation. Because Microsoft must first declare the outage, promotion can take an hour or more, so it is a safety net rather than a fast-recovery tool.

Key facts

  • Configure by (1) enabling service-managed failover and (2) setting a failover priority for each region; the top available priority becomes the new write region.
  • Triggering can take ~1 hour or more because Microsoft must declare the outage first — for faster recovery, perform a manual (forced) failover instead.
  • Unless the account uses strong consistency (RPO 0), any writes not yet replicated when the region failed can be lost.
  • The recovered region is not promoted back to write automatically — you must change the write region back manually when it is safe.
  • Per-partition automatic failover (PPAF) is a separate, faster (public preview) option that fails over individual partitions automatically, typically within ~3 minutes.
  • Configure via the Azure portal, PowerShell, Azure CLI, or ARM/Bicep.
Gotcha: Service-managed failover is not instantaneous. If a scenario demands the fastest possible write recovery during an outage, the answer is a manual forced failover, not service-managed failover.

Exam tips & gotchas

  • Two steps: enable service-managed failover and set failover priorities. Missing either means no automatic promotion.
  • Promotion may take an hour+; use forced failover for speed.
  • Recovered region does not auto-return to write — you flip it back manually.
  • Only strong consistency guarantees zero data loss on write-region failover.

Practice

Q1. A single-write-region account has service-managed failover enabled but no failover priority order configured. The write region goes down. What happens?
Answer: Nothing is promoted automatically — you must set a failover priority order in advance. Why: Automatic promotion needs both the feature enabled and a ranked priority list; with no ranked target, the reliable fix during the outage is a manual forced failover to a healthy region.
Q2. After a service-managed failover, the original write region comes back online. Operations assume it will automatically resume as the write region. Are they right?
Answer: No — a recovered region rejoins as a read region and is not auto-promoted back to write. Why: An operator must explicitly change the write region back once it is safe; doing so on healthy regions causes no data or availability loss.

2.1.3 Perform manual failovers to move single-master write regions

A manual (forced) failover — also called a customer-managed failover or offline-region operation — lets you take a region offline on demand. It is the recommended way to quickly restore write availability during an outage, because you do not wait for Microsoft to declare the outage. Taking the write region offline promotes the next-highest-priority read region to write.

When all regions are healthy, a related operation — change write region (planned failover) — deliberately moves writes to another region with zero data loss, because replication catches up before the switch. Planned failover requires healthy regions, so it cannot be used during an actual outage.

Key facts

  • Forced failover = you trigger it; recommended for fast recovery during an outage. A forced failover of the write region can lose unreplicated writes.
  • Change write region (planned failover) = deliberate move when regions are healthy; no data loss (replication catches up first); requires healthy regions so it cannot run during an outage.
  • Manual failover is the standard way to run disaster-recovery drills: temporarily disable service-managed failover, force a failover, observe the app, then fail back.
  • During an outage, do not run control-plane operations (change write region, change consistency, modify networking) on the affected region — they delay recovery.
  • Trigger via Azure portal, PowerShell, or Azure CLI.
Tip: Distinguish the two operations by health state — forced failover is for an outage (may lose data), change write region (planned) is for healthy regions (no data loss).

Exam tips & gotchas

  • Forced failover = outage recovery, possible data loss. Planned change write region = healthy regions, zero data loss.
  • Planned failover cannot be used during an outage (it needs region-to-region connectivity for a consistency check).
  • Use forced failover to simulate a region-down scenario in DR testing.
  • Avoid control-plane changes on an affected region mid-outage.

Practice

Answer: Perform a manual forced failover to a healthy region. Why: Service-managed failover can take an hour or more to trigger, whereas a forced failover restores write availability quickly; any unreplicated writes may be lost unless the account uses strong consistency.

2.1.4 Choose a consistency model

Azure Cosmos DB offers five well-defined consistency levels, from strongest to weakest: Strong, Bounded staleness, Session, Consistent prefix, Eventual. Each is a precise contract about how current and how ordered your reads are. The account has a default consistency level of Session, set at the account scope and applied to every database and container.

You can override the account default per request, but only to a weaker level — never stronger than the account setting. This lets a single account serve strict reads by default while relaxing to eventual for cheap, high-throughput reads where staleness is acceptable.

Key facts

  • Strong — linearizability: a read always returns the latest committed write; never a partial or uncommitted value.
  • Bounded staleness — reads lag by at most K versions (updates) or T time, whichever comes first. Ideal for single-write-region accounts spanning multiple regions; an anti-pattern for multi-region writes.
  • Session (default) — within a client session, guarantees read-your-writes and write-follows-reads via a partition-bound session token.
  • Consistent prefix — reads never see writes out of order; a transactional batch becomes visible together.
  • Eventual — weakest; no ordering guarantee; replicas eventually converge.
  • Set the default at the account level; override per request only to a weaker level.
Key: Session is the default. A per-request override can only weaken consistency relative to the account default, never strengthen it.

Exam tips & gotchas

  • Memorise the order: Strong → Bounded staleness → Session → Consistent prefix → Eventual.
  • Bounded staleness is bounded by both K (versions) and T (time) — whichever is hit first.
  • Session consistency is client/session-scoped and relies on the session token.
  • You cannot request a stronger level than the account default at request time.

Practice

Q1. Which consistency level guarantees that a read never returns a write that is out of order, yet still allows some replication lag?
Answer: Consistent prefix. Why: It guarantees reads never observe writes out of sequence (and a transactional batch appears together) while still permitting the replica to lag; Strong would eliminate lag entirely and Eventual gives no ordering guarantee.
Q2. An account's default consistency is Session. A developer wants a specific read to return the absolute latest committed value by requesting Strong on that one call. Will it work?
Answer: No. Why: Per-request overrides can only relax consistency to a weaker level than the account default, never strengthen it; to get Strong reads the account default itself must be raised to Strong (which also requires a single write region).

2.1.5 Identify use cases for different consistency models

Choosing a consistency level is a business decision as much as a technical one. The stronger the level, the more current and ordered your data — but at higher latency, more RU cost for reads, and lower availability during failures. Map each level to the correctness the scenario actually needs rather than defaulting to the strongest.

Key facts

  • Strong — financial ledgers, inventory counts, or ordering where a stale read is unacceptable. Single write region only.
  • Bounded staleness — globally distributed reads that must be near-current within a known bound, e.g. leaderboards or near-real-time dashboards, on a single-write-region account.
  • Session (default) — user-centric apps needing read-your-writes: profiles, shopping carts, social timelines. Best overall balance.
  • Consistent prefix — status transitions or comment threads where order matters but some lag is fine.
  • Eventual — like counts, retweet counts, non-threaded reactions: maximum throughput/availability, ordering irrelevant.
Tip: When a question stresses "the user must immediately see their own change," the answer is almost always Session — the default and the read-your-writes guarantee.

Exam tips & gotchas

  • "Read your own writes" ⇒ Session. "Absolutely latest, no staleness" ⇒ Strong.
  • "Near-current within a defined lag, globally" ⇒ Bounded staleness.
  • "Order matters but lag is OK" ⇒ Consistent prefix. "Counts/likes, order irrelevant" ⇒ Eventual.
  • Don't over-provision consistency — Strong costs more RUs and lowers availability.

Practice

Q1. A social app must ensure a user always sees their own most recent post immediately, while other users can tolerate slight delays. Which level is the best fit and why?
Answer: Session consistency. Why: It guarantees read-your-writes within the user's own session via the session token, so the author always sees their new post, while keeping low latency and high throughput for everyone else; it is also the account default.
Q2. A "likes" counter on a media site needs the highest possible read throughput and availability; being off by a few for a second or two is acceptable. Which level fits?
Answer: Eventual consistency. Why: Exact ordering and immediate currency are unnecessary for a like count, and Eventual gives the lowest latency, lowest read RU cost, and highest availability; stronger levels would waste RUs and reduce availability for no benefit.

2.1.6 Evaluate the impact of consistency on availability and request unit cost

Consistency affects both your bill and your uptime. Stronger read consistency requires the service to consult more replicas, which doubles the read RU cost for the two strongest levels. Stronger consistency also reduces availability during failures, because more replicas/regions must agree before a read or write succeeds.

Key facts

  • Read RU cost: Strong and Bounded staleness reads use a local minority quorum of two replicas, costing roughly 2× the RUs of Session, Consistent prefix, and Eventual reads (single replica).
  • Write RU cost is identical across all consistency levels for a given operation.
  • Strong is not supported with multi-region writes; it needs a single write region and 2+ regions for dynamic quorum, and can lose write availability if the account drops below quorum.
  • Weaker levels (Session/Consistent prefix/Eventual) offer higher availability and throughput.
  • RPO on region outage: Strong = 0; Bounded staleness = K & T; Session/Consistent prefix/Eventual = < 15 minutes (multi-region accounts).
Gotcha: Only read RU charges change with consistency. Write RU charges are the same at every level — a common exam trap.

Exam tips & gotchas

  • Strong & Bounded staleness reads ≈ 2× read RUs (two-replica quorum).
  • Write RU cost does not vary by consistency level.
  • Strong ⇒ single write region, lower availability, RPO 0.
  • Weaker levels ⇒ cheaper reads, higher availability/throughput, small RPO.

Practice

Q1. A team switches their read-heavy account from Session to Strong consistency and is surprised the RU consumption on reads roughly doubles. Why?
Answer: Strong (and Bounded staleness) reads consult a two-replica local quorum, so they cost about 2× the RUs of single-replica reads. Why: That extra replica read is required to guarantee the latest committed value; write RU charges, by contrast, are unchanged by consistency level.
Q2. An architect proposes Strong consistency on an account that uses multi-region writes for maximum correctness. What's the problem?
Answer: Strong consistency is not supported with multi-region writes. Why: Strong requires a single write region and cross-region synchronous commits; to keep multi-region writes the team must choose Session, Consistent prefix, or Eventual (Bounded staleness is also an anti-pattern here) and handle write conflicts.

2.1.7 Evaluate the impact of consistency on performance and latency

All consistency levels deliver single-digit-millisecond latency at the 99th percentile — except Strong across multiple regions. Because a strong write must commit in every region before it is acknowledged, its latency scales with the distance between regions, which is why Azure Cosmos DB blocks strong consistency for very widely separated regions by default.

Key facts

  • Reads: < 10 ms at P99 (≈ 4 ms average) for all levels.
  • Writes: < 10 ms at P99 (≈ 5 ms average) for all levels except strong across multiple regions.
  • Strong multi-region write latency ≈ 2× the round-trip time (RTT) between the two farthest regions + 10 ms at P99.
  • Strong consistency for regions more than 5,000 miles (8,000 km) apart is blocked by default (contact support to enable) because write latency becomes too high.
  • Stronger consistency ⇒ higher latency and higher read RU; weaker consistency ⇒ lower latency and higher throughput.
Key: The performance penalty of Strong shows up on writes across regions. Within a single region, Strong writes are still single-digit-ms.

Exam tips & gotchas

  • Only Strong multi-region writes break the < 10 ms P99 guarantee.
  • Strong write latency ≈ 2× RTT + 10 ms between the farthest regions.
  • Strong is disabled by default for regions > 5,000 mi / 8,000 km apart.
  • Reads stay < 10 ms P99 at every level.

Practice

Q1. A global account with regions in Australia and Brazil cannot enable Strong consistency in the portal. Why, and what determines the latency if it were enabled?
Answer: The regions are more than 5,000 miles (8,000 km) apart, so Strong is blocked by default. Why: If enabled via support, Strong write latency at P99 would be roughly 2× the round-trip time between the farthest regions plus ~10 ms — far above the usual single-digit-ms target.
Q2. A single-region account uses Strong consistency and the team worries writes will be slow. Are they right?
Answer: No. Why: The latency penalty of Strong applies only to writes that must commit across multiple regions; in a single region there is no cross-region synchronous commit, so Strong writes still meet the < 10 ms P99 target.

2.1.8 Specify application connections to replicated data

The SDK, not the service, decides which region a client talks to. You give the client an ordered list of preferred (application) regions; it reads from the first healthy region in the list and automatically reroutes to the next one if a region becomes unavailable — with no code change. This "multi-homing" is how an application stays responsive and available during a regional failure.

Key facts

  • Configure a prioritised region list in the SDK: .NET v3 ApplicationRegion (single) or ApplicationPreferredRegions (ordered list); Java v4 preferredRegions; Python preferred_locations.
  • If a preferred read region is down, the SDK detects it via backend response codes and reroutes to the next region automatically — no application change required.
  • For multi-region-write accounts, point each client at the region it is hosted in for low-latency local writes.
  • Front globally distributed apps with a global load balancer (Azure Front Door / Traffic Manager) to route users to a healthy region.
  • Setting preferred regions correctly means most read-region outages are handled entirely client-side, with no service-side failover.
// Configure an ordered preferred-region list (.NET v3 CosmosClientOptions)
{
  "ApplicationPreferredRegions": [
    "West US 2",
    "East US",
    "North Europe"
  ]
}
// The client reads from West US 2; if it is unavailable it
// automatically falls back to East US, then North Europe.
Tip: A correctly ordered preferred-region list lets the SDK absorb most read-region outages transparently — no forced failover and no code change.

Exam tips & gotchas

  • Region routing is an SDK client setting (ApplicationPreferredRegions), not a server setting.
  • SDK reroutes reads to the next preferred region automatically on outage.
  • Order the list by proximity/priority; the first healthy region wins.
  • Pair with Azure Front Door / Traffic Manager for end-to-end regional routing.

Practice

Q1. During a read-region outage, an application keeps trying to read from the failed region and errors out, even though other regions are healthy. What is the most likely cause?
Answer: The SDK's preferred-regions list is misconfigured or empty. Why: With ApplicationPreferredRegions set correctly, the client detects the outage from backend response codes and reroutes reads automatically; fixing the preferred-region configuration resolves the errors without any service-side failover.
Q2. A multi-region-write app hosted in East US and West Europe should minimise write latency for users on each continent. How should the clients be configured?
Answer: Set each client's application/preferred region to its local hosting region (East US clients → East US, West Europe clients → West Europe). Why: With multi-region writes enabled, each client then commits locally with low latency while Azure Cosmos DB replicates asynchronously between the regions.

2.2 Design and implement multi-region writes

2.2.1 Choose when to use multi-region writes

Multi-region writes (formerly "multi-master") make every associated region both readable and writable — an active-active topology. This delivers low-latency local writes for globally distributed users and the highest availability tier: 99.999% for reads and writes. The cost is complexity: strong consistency is unavailable, and concurrent writes to the same item in different regions can conflict.

Key facts

  • Every region accepts writes (active-active), giving local low-latency writes and 99.999% read+write availability.
  • Choose it when writes must be fast for users worldwide, or when writes must stay available even if a region fails.
  • Strong consistency is not supported with multi-region writes; Bounded staleness is an anti-pattern.
  • Concurrent same-item writes across regions can conflict, so you must define a conflict-resolution strategy.
  • Each additional write region adds cost.
Gotcha: Turning on multi-region writes rules out Strong consistency. If a scenario needs both global writes and Strong, it is not achievable — one requirement must give.

Exam tips & gotchas

  • Multi-region writes = active-active, 99.999% RW availability, local write latency.
  • Strong is unsupported and Bounded staleness is an anti-pattern with multi-region writes.
  • Cross-region write conflicts are possible ⇒ a conflict-resolution policy is mandatory.
  • "Multi-master" is the old name for the same feature.

Practice

Q1. A global app needs users on every continent to experience fast local writes and to keep writing even if one region fails. Which capability delivers this, and what constraint comes with it?
Answer: Multi-region writes (active-active), which give local low-latency writes and 99.999% read+write availability. Why: The trade-off is that Strong consistency cannot be used and concurrent same-item writes across regions may conflict, so the design must include a conflict-resolution policy.
Q2. A requirement states "global low-latency writes AND strong consistency." Can a single Cosmos DB account satisfy both?
Answer: No. Why: Strong consistency requires a single write region, which is incompatible with multi-region (active-active) writes; the team must relax one requirement — keep a single write region for Strong, or accept a weaker level (e.g., Session) to keep multi-region writes.

2.2.2 Implement multi-region writes

Enabling multi-region writes is a two-part job: turn on multiple write regions at the account level, then configure the application so each client writes to its local region. Behind the scenes a hub region arbitrates write conflicts, and replication between write regions is asynchronous, with lag governed by the account's consistency level.

Key facts

  • Enable multiple write regions on the account (portal, PowerShell, CLI, ARM) — at creation or later.
  • Configure the app's region in the SDK so each client writes locally (.NET v3 ApplicationRegion / ApplicationPreferredRegions).
  • A hub region acts as the arbiter for write conflicts.
  • Cross-region replication is asynchronous; lag depends on the consistency level.
  • Avoid repeatedly updating or recreating the same document ID (e.g., right after a TTL expiry) — it multiplies conflicts and degrades replication.
Key: Enabling multi-region writes on the account is only half the job — the application/SDK must also be pointed at the local write region to actually get local write latency.

Exam tips & gotchas

  • Two steps: enable multiple write regions on the account and set the client's application region.
  • Replication between write regions is asynchronous — expect some lag.
  • The hub region arbitrates conflicts.
  • Hot document IDs (frequent re-create/update, TTL churn) worsen conflicts and replication.

Practice

Q1. An account has multiple write regions enabled, but clients in Europe still see the same write latency as before. What was likely missed?
Answer: The application/SDK was not configured with the local region. Why: Enabling multiple write regions on the account is necessary but not sufficient — each client must set its application/preferred region (e.g., ApplicationRegion) to its local region so writes commit locally; until then clients keep routing to the original region.
Q2. A multi-region-write workload repeatedly deletes and recreates documents with the same IDs after their TTL expires, and replication performance is degrading. Why?
Answer: Reusing hot document IDs (recreating the same ID right after TTL expiry) generates many cross-region write conflicts. Why: The system must resolve each conflict, and that extra load degrades replication performance; redesigning to avoid reusing hot IDs relieves the pressure.

2.2.3 Implement a custom conflict resolution policy for Azure Cosmos DB for NoSQL

When two regions write the same item concurrently, Azure Cosmos DB resolves the clash using the container's conflict-resolution policy. There are two policy modes — Last Write Wins (LWW) and Custom — and the choice can only be made at container creation time; it cannot be changed afterward. Conflicts come in three flavours: insert, replace, and delete.

Key facts

  • Last Write Wins (LWW) — the default. Uses the system timestamp _ts by default; for API for NoSQL you can name a custom numeric property (the conflict-resolution path) whose highest value wins. A delete always wins over insert/replace. LWW conflicts do not appear in the conflict feed.
  • Custom — register a merge stored procedure invoked automatically (exactly-once) on conflict. API-for-NoSQL only.
  • If no merge procedure is registered, or it throws, conflicts are written to the conflict feed for manual resolution in application code.
  • Policy is set only at container creation and cannot be modified later.
  • The conflict feed also surfaces unreplicated writes recovered after a region outage, which you reconcile with app-specific logic.
// Container with a custom numeric LWW path (API for NoSQL)
{
  "id": "orders",
  "partitionKey": { "paths": ["/customerId"], "kind": "Hash" },
  "conflictResolutionPolicy": {
    "mode": "LastWriterWins",
    "conflictResolutionPath": "/version"   // highest /version wins
  }
}
// Custom mode instead registers a merge stored procedure:
// { "mode": "Custom", "conflictResolutionProcedure": ".../sprocs/resolver" }
Gotcha: The conflict-resolution policy is fixed at container creation. If a question asks how to change an existing container's policy, the answer is to create a new container with the desired policy and migrate data — you cannot alter it in place.

Exam tips & gotchas

  • LWW is the default and uses _ts; NoSQL can override with a custom numeric path where the highest value wins.
  • Delete always beats insert/replace in LWW.
  • Custom policy = merge stored procedure; NoSQL only; if it fails or is absent, conflicts land in the conflict feed.
  • Policy is set at container creation and is immutable afterward.

Practice

Q1. A NoSQL container must resolve concurrent updates by keeping the record with the highest application-supplied version number, with no server-side code. Which policy configuration fits?
Answer: Last Write Wins with a custom numeric conflict-resolution path pointing at the version property (e.g., /version). Why: In LWW the item with the highest value at that path wins with no stored procedure; it must be set at container creation because the policy cannot be changed later.
Q2. A team set a Custom conflict-resolution policy but forgot to register the merge stored procedure. Two regions update the same item. What happens to the conflict?
Answer: The conflict is written to the conflict feed rather than auto-resolved. Why: With no merge procedure registered (or if it throws), Custom mode surfaces the conflict for the application to resolve in code; automating resolution requires registering a valid merge stored procedure, but the policy mode itself can't be changed on the existing container.

3.1.1 Configure Azure Cosmos DB Mirroring for Microsoft Fabric

Mirroring in Microsoft Fabric continuously replicates an Azure Cosmos DB database into Fabric OneLake in near real-time, giving you a no-ETL path to analytics. The replica is stored in the open-source Delta Lake / Parquet format, so every Fabric engine — Power BI (Direct Lake), the SQL analytics endpoint, and Spark — can read it without you building or maintaining a pipeline.

The critical selling points for the exam: replication is free (Fabric compute for replication and OneLake storage cost nothing), it has no performance impact on your transactional workload, and it doesn't consume Request Units. Mirroring is GA and is now the recommended replacement for Azure Synapse Link.

Key facts

  • Data replicates into OneLake in near real-time as Delta/Parquet; no ETL and no RU consumption on the source account.
  • Continuous backup is a prerequisite — enable 7-day (free, recommended) or 30-day continuous backup before mirroring.
  • Each mirrored item auto-creates a read-only SQL analytics endpoint and a default Power BI semantic model; the source is viewable read-only in Data Explorer.
  • Replication compute and OneLake storage are free; you pay only for query compute (SQL, Power BI, Spark) against Fabric capacity.
  • Currently supports API for NoSQL only; not in sovereign clouds; multiple write regions are not supported.
  • Mirroring does not use the analytical store or change feed as its source — those remain independently available.
  • Nested data surfaces as JSON strings in the SQL endpoint; expand with OPENJSON, CROSS APPLY, or OUTER APPLY.
Key: Mirroring gives HTAP analytics with zero RU cost and zero transactional impact. If a scenario asks for near real-time BI on Cosmos DB data without ETL and without touching provisioned throughput, mirroring is the answer.

Exam tips & gotchas

  • Continuous backup is mandatory — mirroring won't start without it. 7-day is free.
  • NoSQL only, single write region only. Multi-write-region accounts can't be mirrored.
  • The SQL analytics endpoint is read-only — you cannot write to Cosmos DB from Fabric.
  • Mirroring replaces Synapse Link for new projects but is a separate mechanism from analytical store/change feed.

3.1.2 Choose Between Azure Cosmos DB Mirroring and the Azure Cosmos DB Spark Connector

Both bring Cosmos DB data into an analytics environment, but they solve different problems. Mirroring is fully managed and no-code: it replicates the whole database into OneLake for read-only analytics, free of RU cost. The Spark connector is code-first and can both read and write — but reading the transactional store (cosmos.oltp) consumes RUs.

Choose mirroring for BI and analytics with no pipeline to maintain. Choose the Spark connector when you need programmatic transformations, data-engineering/ML workflows, or the ability to write results back to Cosmos DB.

Key facts

  • Mirroring: managed, no-code, near real-time, read-only, free compute, no RU cost; best for BI/reporting.
  • Spark connector: code-first; the cosmos.oltp format reads/writes the transactional store and consumes RUs; best for data engineering, ML, and write-back.
  • The Spark cosmos.olap format reads the analytical store (via Synapse Link) — read-only and RU-free — an alternative to cosmos.oltp when you don't need the freshest transactional data.
  • Only the Spark connector can write data back to Cosmos DB; mirroring is one-way and read-only.

Exam tips & gotchas

  • Need write-back or custom transformations? Spark connector. Need managed, RU-free BI? Mirroring.
  • cosmos.oltp = transactional store, costs RUs. cosmos.olap = analytical store, RU-free, read-only.
  • Mirroring is one-directional into OneLake; don't pick it when the scenario needs to update Cosmos DB.

Practice

Answer: Spark connector. Why: Only the Spark connector supports write-back to the transactional store (cosmos.oltp); mirroring is read-only into OneLake.
Q2. A BI team wants Power BI dashboards over Cosmos DB with the least operational overhead and no RU impact. Which do you recommend?

3.1.3 Enable the Analytical Store on a Container

The analytical store is a fully isolated, auto-managed column store that mirrors your container's operational data in a format optimized for large analytical scans and aggregations. Because it is physically separate from the row-based transactional store, analytical queries have no impact on transactional performance and require no RU allocation.

You enable it per container by setting Analytical TTL (ATTL) to -1 (infinite retention) or a positive number of seconds; 0 disables it. Enabling analytical store depends on Azure Synapse Link being enabled at the account level first.

Key facts

  • Enable via Analytical TTL: -1 = keep all history; positive n = expire after n seconds; 0 = disabled.
  • Auto-sync propagates inserts/updates/deletes to the column store, usually within ~2 minutes (up to ~5 min for shared-throughput databases with many containers).
  • No RU cost and no transactional impact — the column store is billed separately (storage + analytical read/write operations).
  • Turning on Synapse Link at the account does not auto-enable analytical store; you set ATTL on each container.
  • Configure via portal, Azure CLI (--analytical-storage-ttl), PowerShell (-AnalyticalStorageTtl), ARM, or SDK (AnalyticalStoreTimeToLiveInSeconds / analytical_storage_ttl).
  • Data tiering: set ATTL ≥ transactional TTL to keep longer history cheaply in analytical store while hot data expires from the transactional store.
  • Schema limits: 1000 properties, 127 nesting levels; columns can't be removed; no schema reset.
# Create a Synapse Link-enabled container with analytical store (infinite retention)
container = db.create_container(
    id="orders",
    partition_key=PartitionKey(path="/customerId"),
    analytical_storage_ttl=-1   # -1 = keep all history; positive n = expire after n seconds
)
Tip: ATTL is set at the container level only. Transactional TTL can be per-item or per-container, but analytical TTL is container-wide.

Exam tips & gotchas

  • ATTL -1 = infinite, 0 = disabled, positive n = seconds of retention. Memorize these three values.
  • Analytical store adds no RU cost — distractors claiming it consumes provisioned throughput are wrong.
  • Enabling Synapse Link on the account is a prerequisite but is not enough on its own; set ATTL per container.
  • For NoSQL, full-fidelity vs well-defined schema is chosen when Synapse Link is first enabled and can't be changed later.

Practice

Q1. You want operational data to expire from the transactional store after 30 days but retain two years of history for analytics. How do you configure TTL?
Answer: Set transactional TTL to 30 days and analytical TTL to two years (ATTL ≥ TTTL). Why: Analytical TTL controls the column store's retention independently, so history persists cheaply in analytical store after it expires from the transactional store.
Q2. After enabling Synapse Link on the account, a developer expects analytical queries to work but the analytical store is empty. What's missing?
Answer: Analytical store wasn't enabled on the container — analytical TTL is still unset. Why: Enabling Synapse Link at the account level doesn't turn on analytical store; you must set analytical TTL (-1 or a positive value) on each container.
Q3. Does enabling analytical store increase the RU/s you must provision for transactional queries?
Answer: No. Why: The analytical store is isolated and billed separately (storage plus analytical operations); auto-sync doesn't consume the container's provisioned RUs.

3.1.4 Enable a Connection to an Analytical Store and Query from Azure Synapse Spark or Azure Synapse SQL

Key facts

When you need the freshest data or capabilities the analytical store can't provide, Spark can read the transactional store directly using the cosmos.oltp format. Unlike cosmos.olap, reading the transactional store consumes provisioned RUs and can affect OLTP performance, so use it deliberately.

Key facts

      .option("spark.synapse.linkedService", "myCosmosLinkedService")
      .option("spark.cosmos.container", "orders")
      .load())

4.1 Optimize Query Performance When Using the API for Azure Cosmos DB for NoSQL

4.1.1 Adjust Indexes on the Database

Every container has an indexing policy. By default Cosmos DB automatically indexes every property of every item using range indexes for strings and numbers — great for correctness, but you pay write RUs for paths you never query. Tuning the policy means choosing an indexing mode, deciding which paths to include or exclude, and adding specialized index kinds where a query needs them.

Path notation matters: a scalar path ends in /?, array elements use /[], and the wildcard /* matches everything below a node. The root /* must be either included or excluded, and the more precise (deeper) path wins when rules overlap.

Key facts

  • Indexing modes: consistent (index updated synchronously with writes) and none (no secondary index — a pure key/value store that disables queries; used to speed bulk loads). New containers cannot use lazy indexing.
  • Include /* and exclude unqueried paths (recommended — new properties auto-index); or exclude /* and include only specific paths.
  • Changing the policy triggers an online, in-place index transformation: asynchronous, consumes RUs at lower priority, and has no write-availability impact.
  • Adding an index takes time (transformation must finish before it's used); removing an index takes effect immediately.
  • System properties id and _ts are indexed in consistent mode; _etag is excluded by default.
  • TTL requires indexing — you cannot set TTL on a none-mode container.
  • Track progress with the SDK's index transformation progress or the portal.
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/description/?" },
    { "path": "/metadata/*" }
  ]
}
Key: Default = index everything. To cut write cost, keep /* included and exclude the paths you never filter on — new properties still get indexed automatically.

Exam tips & gotchas

  • Root /* must appear in exactly one of included or excluded paths.
  • Removing an index is instant; adding one requires the transformation to complete first — add before you remove when swapping.
  • Mode none disables all queries and blocks TTL; use it only for bulk ingest, then switch back to consistent.
  • If you exclude /*, remember to include the partition key path (unless it's /id) or PK-filter queries do full scans.

Practice

Q1. A write-heavy container indexes many large text fields that are never queried, and write RU charges are high. What is the least-effort fix that keeps future properties searchable?
Answer: Keep the included path /* and add the unqueried fields to excluded paths. Why: Excluding specific paths stops indexing those properties (lowering write RUs) while /* continues to auto-index any new properties.
Q2. An engineer removes a property path from the indexing policy and expects queries to slow down only after a long transformation. When does the removal take effect?
Answer: Immediately. Why: Index removals take effect right away; only index additions require an asynchronous transformation to complete before they're used.
Q3. A team wants the fastest possible bulk load and will run no queries during load. Which indexing mode should they set temporarily?
Answer: none. Why: Indexing mode none removes secondary-index write overhead for the fastest ingest; they switch back to consistent afterward (note TTL can't be used while in none).

4.1.2 Calculate the Cost of the Query

Every operation in Cosmos DB is normalized to Request Units (RU) — a currency abstracting CPU, IOPS, and memory. A 1-KB point read is about 1 RU; a query's charge depends on how many documents it must load, whether it hits one partition or fans out, and how well the index serves the filter.

The big cost lever is the partition key filter. A query scoped to a single partition key is far cheaper than a cross-partition query, which executes on every physical partition and merges results client-side.

Key facts

  • Read cost, cheapest to most expensive: point read (id + partition key) → query with a partition-key filter → query with an equality/range filter on any property → query with no filter.
  • Cross-partition (fan-out) queries have higher latency and RU cost; ORDER BY results are merge-sorted and COUNT is summed across partitions.
  • Query execution metrics expose where cost goes: RetrievedDocumentCount, OutputDocumentCount, IndexLookupTime, and IndexHitRatio (matched ÷ loaded — a low ratio signals a scan).
  • PopulateIndexMetrics reveals which indexes were used and which are missing (adds overhead — debug only).
  • RU cost also grows with result size, projections, UDFs, and system functions like LOWER() that force scans.
-- Single-partition: cheap, served from the index
SELECT * FROM c WHERE c.customerId = "C-1001"

-- Cross-partition scan: LOWER() defeats the index, retrieves many, returns few
SELECT c.name FROM c WHERE LOWER(c.name) = "alice"
Tip: If RetrievedDocumentCount is far larger than OutputDocumentCount (low IndexHitRatio), the query is scanning — add an index or rewrite the filter to use one.

Exam tips & gotchas

  • Always prefer a point read when you know the id and partition key — it's the cheapest operation.
  • Adding a partition-key predicate turns a fan-out query into a single-partition query and slashes RU cost.
  • Functions that transform a property (LOWER, UPPER) prevent index use and cause a full scan.
  • Higher provisioned RU/s doesn't lower a query's charge — it just lets you run more of them before rate limiting (429).

Practice

Q1. A query SELECT * FROM c WHERE c.status = 'active' runs against a container partitioned by /tenantId and is expensive. What single change most reduces its RU cost?
Answer: Add a c.tenantId = @id predicate so the query targets one partition. Why: Filtering on the partition key converts a cross-partition fan-out into a single-partition query, dramatically lowering RUs and latency.
Q2. Query metrics show RetrievedDocumentCount = 5000 and OutputDocumentCount = 12 with a very low IndexHitRatio. What does this indicate?
Answer: The query is scanning documents rather than using an index. Why: A large retrieved-vs-output gap and low IndexHitRatio mean the filter isn't index-served; add the needed index or remove the scan-forcing function.

4.1.3 Retrieve the Request Unit Cost of a Point Operation or Query

You measure RU cost by inspecting the response of an operation — Cosmos DB returns the exact request charge for every call. In the portal, Data Explorer's Query Stats shows it; in code, every SDK response exposes it, and the underlying HTTP header is x-ms-request-charge.

Key facts

  • .NET v3: response.RequestCharge (a double).
  • Java: response.getRequestCharge(); Go: resp.RequestCharge.
  • Node.js: result.headers["x-ms-request-charge"].
  • Python: container.client_connection.last_response_headers["x-ms-request-charge"].
  • Portal: Data Explorer → run query → Query Stats shows the request charge.
  • For paged queries, sum RequestCharge across every page to get the total.
// .NET SDK v3 — point read
ItemResponse<Product> read = await container.ReadItemAsync<Product>(
    id: "P-1001",
    partitionKey: new PartitionKey("electronics"));
Console.WriteLine($"Point read cost: {read.RequestCharge} RU");

// Query — accumulate charge across pages
double total = 0;
using FeedIterator<Product> it = container.GetItemQueryIterator<Product>(
    "SELECT * FROM c WHERE c.category = 'electronics'");
while (it.HasMoreResults)
{
    FeedResponse<Product> page = await it.ReadNextAsync();
    total += page.RequestCharge;
}
Console.WriteLine($"Query total: {total} RU");

Exam tips & gotchas

  • The HTTP header name to remember is x-ms-request-charge.
  • Node.js and Python read the charge from response headers; .NET/Java/Go expose a typed RequestCharge property/method.
  • A single query can span multiple pages — one page's charge is not the whole cost.

Q2. In the Node.js SDK, where is the request charge exposed?
Answer: In the response headers, under x-ms-request-charge. Why: The Node.js and Python SDKs surface the raw HTTP headers; the RU charge is the x-ms-request-charge value.

4.1.4 Implement Azure Cosmos DB Integrated Cache

The integrated cache is an in-memory cache built into the dedicated gateway. It has an item cache (point reads) and a query cache (query result sets). The headline benefit: a cache hit costs 0 RU, making read-heavy workloads with repeated reads dramatically cheaper.

To use it you must connect through the dedicated gateway, use gateway connection mode, and use session or eventual consistency. Stronger consistencies (strong, bounded staleness, consistent prefix) bypass the cache and are served from the backend.

Key facts

  • Requires a provisioned dedicated gateway, the dedicated-gateway connection string, gateway connection mode, and session or eventual consistency.
  • Cache hits return 0 RU; read-through / write-through with LRU eviction.
  • MaxIntegratedCacheStaleness sets the maximum acceptable staleness per request — default 5 minutes, min 0, max 10 years; higher = more hits.
  • Each dedicated gateway node has its own independent cache; multi-page queries aren't guaranteed the same node.
  • Best for read-heavy, repeated point reads/queries and hot read partitions; not for write-heavy or rarely repeated reads.
  • Transactional batch and bulk operations do not populate the item cache.
  • Metrics: IntegratedCacheItemHitRate, IntegratedCacheQueryHitRate, DedicatedGatewayRequests, IntegratedCacheEvictedEntriesSize.
// Connect via the dedicated gateway in Gateway mode
var options = new CosmosClientOptions { ConnectionMode = ConnectionMode.Gateway };

4.2.3 Manage the Number of Change Feed Instances by Using the Change Feed Estimator

The change feed estimator measures lag — the gap between the last item your processor has handled (from the lease state) and the latest change in the container. Rising lag means your processing rate is below your change rate, signaling that you should scale out the processor.

Key facts

  • Push model: GetChangeFeedEstimatorBuilder(name, delegate, interval) pushes the count of pending changes to your delegate (default interval 5 seconds).
  • On-demand model: GetChangeFeedEstimator(name, leaseContainer).GetCurrentStateIterator() returns per-lease EstimatedLag plus the owning InstanceName.
  • The estimator must share the same lease container and processor name as the processor it monitors.
  • Deploy it on an independent instance; each estimation consumes RUs (a ~1-minute frequency is a good starting point).
  • Works in both change feed modes; the estimate is approximate, not an exact count.
// On-demand: detailed lag per lease and owning instance
ChangeFeedEstimator estimator = monitoredContainer
    .GetChangeFeedEstimator("orderProcessor", leaseContainer);

using FeedIterator<ChangeFeedProcessorState> it = estimator.GetCurrentStateIterator();
while (it.HasMoreResults)
{
    foreach (ChangeFeedProcessorState s in await it.ReadNextAsync())
    {
        Console.WriteLine($"Lease {s.LeaseToken} owned by {s.InstanceName}: lag {s.EstimatedLag}");
    }
}

Exam tips & gotchas

  • The estimator and processor must match on processor name + lease container, or the lag reading is meaningless.
  • Consistently high or growing lag → add processor instances; near-zero lag → you may be over-provisioned.
  • The estimator itself consumes RUs; don't poll it too aggressively.
  • Use the on-demand iterator when you need per-lease detail and which host owns each lease.

Practice

4.2.4 Implement Denormalization by Using a Change Feed

Cosmos DB favors denormalized, read-optimized data. The change feed is the engine that keeps those denormalized copies in sync: as items change in a source container, a processor propagates the relevant fields into other containers or partitions shaped for specific read patterns. This is a data-movement pattern with eventual consistency.

Key facts

  • Use the change feed to replicate/copy data into a container with a different partition key optimized for a query.
  • Denormalization trades write amplification (maintaining copies) for cheaper, single-partition reads.
  • Replication through the change feed is eventually consistent — copies lag the source briefly.
  • Order is guaranteed only within a partition key value, so choose a key that keeps related updates ordered.
// Denormalize product name into an orders-by-customer view

        await ordersView.PatchItemAsync<dynamic>(p.Id, new PartitionKey(p.Id), patch);

}

Exam tips & gotchas

  • Denormalization via change feed is eventually consistent — don't rely on instant propagation.
  • Pick the destination partition key for the target query, not the source.
  • Order guarantees exist only within a single partition key value.

4.2.5 Implement Referential Enforcement by Using a Change Feed

Cosmos DB has no foreign-key constraints, so referential integrity is enforced in application logic. An event-computing pattern on the change feed watches for changes and cascades related updates or deletes across containers — for example, removing or flagging child items when a parent is deleted.

Key facts

  • There are no server-side FK constraints; referential rules run in a change-feed consumer or Function.
  • On each change, validate references and cascade updates/deletes to dependent items in other containers.
  • Latest version mode doesn't emit deletes — model deletes as a soft-delete flag so the consumer can act on them.
  • Order is guaranteed only within a partition key value; choose a key that groups related events for correct sequencing.
// Cascade a soft-deleted customer to their orders
static async Task HandleChangesAsync(IReadOnlyCollection<Customer> changes, CancellationToken ct)
{
    foreach (Customer c in changes)
    {
        if (c.Deleted)
        {
            // mark or remove dependent order documents
            await ArchiveOrdersForCustomerAsync(c.Id);
        }
    }
}

Exam tips & gotchas

  • Because latest version mode drops deletes, use a deleted:true marker to trigger cascades.
  • Enforcement is eventually consistent — a brief window exists where references are stale.
  • Keep parent and related events under a partition key that preserves their order when sequence matters.

Practice

4.2.6 Implement Aggregation Persistence by Using a Change Feed

Key facts

  • Process change-feed events, compute an aggregate, and write the result back to a summary container/item.

  • Well suited to leaderboards, counters, and rolling totals updated in near real-time.
  • Make updates idempotent (at-least-once delivery) and keep related events under one partition key for correct ordering.

4.3 Define and Implement an Indexing Strategy for Azure Cosmos DB for NoSQL

4.3.1 Choose When to Use a Read-Heavy Versus Write-Heavy Index Strategy

Indexing is a trade-off. Every indexed path speeds reads but adds RU cost to writes. The right strategy depends on your workload's read/write mix: index generously for read-heavy workloads, and trim indexed paths for write-heavy ones.

Key facts

  • Write-heavy: exclude paths you never query so each write updates fewer index entries — lower write RUs and latency.
  • Read-heavy: index the paths you filter and order on, and add composite indexes to make queries cheaper and faster.
  • The default (index everything) favors reads at the expense of write cost — tune it toward your actual pattern.
  • Measure with query metrics (read cost) and observe write RU charges to validate the trade-off.
Key: More indexes = cheaper reads, pricier writes. Match the number of indexed paths to whether the container is read-heavy or write-heavy.

Exam tips & gotchas

  • An ingestion/logging container that's rarely queried is a prime candidate for aggressive path exclusion.
  • A reporting/lookup container benefits from more (including composite) indexes.
  • Don't blindly disable all indexing — you still need indexes for the queries you do run, and for TTL.

Practice

Q1. A telemetry container ingests millions of writes per hour and is queried only occasionally on two fields. What indexing strategy minimizes write cost?
Answer: Exclude /* and include only the two queried paths (a write-heavy strategy). Why: Indexing only the paths you actually query minimizes per-write index maintenance, cutting write RUs on the high-ingest workload.
Q2. A read-heavy catalog runs many multi-field filters and sorts. What should you add?
Answer: More indexes, including composite indexes for the multi-field queries. Why: Read-heavy workloads benefit from richer indexing that serves filters and ORDER BY efficiently, lowering query RUs.

4.3.2 Choose an Appropriate Index Type

Cosmos DB offers several index kinds, each suited to a class of query. Picking the right kind — and adding the ones that aren't created by default — is central to an effective indexing strategy.

Key facts

  • Range: equality and range comparisons, ORDER BY on a single property, and string prefixes — the default for strings and numbers.
  • Spatial: geospatial queries over Point, Polygon, MultiPolygon, and LineString — not created by default, so add it explicitly.
  • Composite: required for ORDER BY on two or more properties and efficient for multi-property filters.
  • Specialized kinds also exist (vector, full-text, tuple) for AI/search scenarios.

Exam tips & gotchas

  • Range indexes cover most everyday queries; you rarely add them manually since they're default.
  • Spatial and composite indexes must be defined explicitly — they're not in the default policy.
  • If a scenario mentions sorting by more than one field, the answer is a composite index.

Q2. Which index kind serves single-property equality, range, and ORDER BY by default?
Answer: A range index. Why: Range indexes are the default for strings and numbers and handle equality, range, prefix, and single-property ORDER BY.

4.3.4 Implement a Composite Index

A composite index spans two or more properties and is required for ORDER BY on multiple properties. It also makes multi-property filters efficient. Each composite index is an ordered set of path + sort-order entries, and the sequence and order must match the query.

Key facts

  • Defined under compositeIndexes as arrays of { path, order }; each path has an implicit /? and no /* wildcard is allowed.
  • Multi-property ORDER BY: the composite path sequence and ASC/DESC must match the ORDER BY; the fully reversed order also works.
  • Multi-property filters: equality filters come first; at most one range filter (>, <, >=, <=, !=) and it must be last. Multiple range filters need multiple composite indexes.
  • Filter + ORDER BY: list filtered properties first in the ORDER BY definition.
  • By default no composite indexes exist; after you add one, existing range indexes serve queries until the transformation completes.
{
  "compositeIndexes": [
    [
      { "path": "/category", "order": "ascending" },
      { "path": "/price", "order": "descending" }
    ]
  ]
}
/* Serves: SELECT * FROM c
           WHERE c.category = 'books'
           ORDER BY c.category ASC, c.price DESC */
Gotcha: A composite index for (category ASC, price DESC) serves that exact ORDER BY and its full reverse (category DESC, price ASC) — but not a different mix like (category ASC, price ASC).

Exam tips & gotchas

  • ORDER BY on 2+ properties fails or is inefficient without a matching composite index.
  • Only one range filter per composite index, and it must be the last path.
  • Wildcards (/*) aren't allowed in composite index paths.
  • Path order matters: equality props first, range prop last; for filter + sort, filtered props lead the ORDER BY.

Practice

Q1. A query ORDER BY c.lastName ASC, c.firstName ASC returns an error about needing an index. What must you add?
Answer: A composite index on (lastName ASC, firstName ASC). Why: ORDER BY across two or more properties requires a composite index whose path sequence and sort orders match the query.
Q2. A query filters on equality of status and a range on createdDate. How should the composite index be ordered?
Answer: status first (equality), createdDate last (the single range filter). Why: Composite indexes place equality filters first and allow one range filter, which must be the last path.
Q3. Does a composite index on (a ASC, b ASC) serve ORDER BY a DESC, b DESC?
Answer: Yes. Why: A composite index serves the defined order and its complete reverse, so (a ASC, b ASC) also covers (a DESC, b DESC).

4.3.5 Optimize Index Performance

Optimizing indexing means getting the most query benefit for the least write overhead, and using metrics to find what's missing or wasted. The core moves: exclude unqueried paths, add composite indexes for real query shapes, index the partition key path when needed, and batch policy changes.

Key facts

  • Keep /* included and exclude large unqueried subtrees to lower write RUs while auto-indexing new fields.
  • Add composite indexes for multi-property ORDER BY and multi-filter queries.
  • If you exclude /*, index the partition key path (unless it's /id) to avoid full scans on partition-hierarchy filters.
  • Group multiple policy edits into one transformation, and add before removing when swapping indexes.
  • Use index metrics and IndexHitRatio to spot missing indexes (scans) and unused ones (wasted write cost).
// Surface index recommendations while tuning a slow query
var options = new QueryRequestOptions { PopulateIndexMetrics = true };
using FeedIterator<dynamic> it = container.GetItemQueryIterator<dynamic>(
    "SELECT * FROM c WHERE c.category = 'books' ORDER BY c.price DESC", requestOptions: options);
FeedResponse<dynamic> page = await it.ReadNextAsync();

Exam tips & gotchas

  • A low IndexHitRatio (many retrieved, few output) points to a missing index or a scan-forcing query.
  • Unused indexes cost write RUs for no read benefit — remove them.
  • Add-before-remove keeps queries served during an index swap; removals are instant, additions aren't.
  • PopulateIndexMetrics has overhead — enable it only while debugging.

Practice

Q2. You need to replace one composite index with another without slowing queries during the change. What's the safe sequence?
Answer: Add the new composite index, wait for the transformation to finish, then remove the old one. Why: Index additions require a transformation before use while removals are instant; add-before-remove avoids a gap in index coverage.

5.1.1 Evaluate Response Status Code and Failure Metrics

Azure Monitor collects platform metrics for every Cosmos DB account automatically — no configuration, one-minute granularity, and short-term retention out of the box. The most useful failure signal is the Total Requests metric split by the StatusCode dimension, which lets you see exactly which HTTP responses your account is returning and how often.

Server-side telemetry (throughput, storage, availability, latency, consistency) answers "is the service healthy?"; client-side diagnostics (request charge, activity ID, substatus, diagnostic string) answer "why did this request fail?".

Key facts

  • Metrics are collected with no setup at 1-minute granularity; dimension values such as container names are case-insensitive.
  • Split Total Requests by StatusCode to count failures over time.
  • Status codes to know: 429 (request rate too large / RU budget exceeded), 401/403 (auth or firewall), 404 (not found), 449 (retriable transient write conflict), 410 (partition split — retry), 503 (service unavailable).
  • A 429 is expected under bursts; the SDK auto-retries, but sustained 429s mean you need more RU/s or a better partition key.
  • Client diagnostics expose activity ID, substatus code, and a diagnostics string that pinpoints the failing region/partition.
// Count data-plane responses by status code over the last hour
CDBDataPlaneRequests
| where TimeGenerated > ago(1h)

Key: A rising 429 count is the number-one troubleshooting signal — it means at least one physical partition ran out of its per-second RU budget.

5.1.2 Monitor the Normalized RU Consumption Metric by Using Azure Monitor

Normalized RU Consumption is a 0–100% metric that reports how close you are to fully using provisioned throughput. Crucially, it is the maximum utilization across all partition key ranges (and regions) — not an average — so it exposes a single hot partition even when the account as a whole looks idle.

Key facts

  • Value 0–100%; 100% means at least one physical partition's per-second RU budget is fully consumed and further requests to it are throttled (429).
  • Because it's a MAX, a single hot partition can pin it at 100% while total account RU usage is low → classic skew signal.
  • Split by PhysicalPartitionId and Region to locate the hot partition.
  • Sustained near-100% with 429s → scale up, switch to autoscale, or redesign the partition key.
// Correlate normalized RU with per-logical-partition RU spend
CDBPartitionKeyRUConsumption
| where TimeGenerated > ago(1h)
| summarize TotalRU = sum(RequestCharge) by PartitionKey
| top 10 by TotalRU desc
Tip: If Normalized RU Consumption is high but Total Request Units is low, you have a hot partition — investigate data/access skew rather than adding throughput blindly.

Exam tips & gotchas

  • It's a maximum across partitions, not an average — the exam loves this distinction.
  • Adding RU/s does not help a hot partition much; each physical partition gets an equal slice of total RU/s.
  • Split by PhysicalPartitionId to prove skew.

Q2. Which dimension do you split Normalized RU Consumption by to identify which partition is hot?
Answer: PhysicalPartitionId (and Region for multi-region). Why: It pinpoints the specific partition and location driving the utilization.

5.1.3 Monitor Server-Side Latency Metrics by Using Azure Monitor

The server-side latency metric measures time spent inside the Cosmos DB service and deliberately excludes network round-trip. Comparing server-side latency to the total latency your client observes tells you whether a slowdown is the database's fault or the network/SDK's.

Key facts

  • Split by operation type, region, and connection mode.
  • High server-side latency → cross-partition/fan-out queries, unindexed queries, or large response payloads.
  • Low server-side latency but high client latency → network, SDK config, gateway vs direct mode, or client-side resource pressure.

// Find the slowest data-plane operations server-side

| where TimeGenerated > ago(1h)
| summarize p99 = percentile(DurationMs, 99), reqs = count() by OperationName
| order by p99 desc

  • Direct mode usually lowers latency versus gateway mode for high-throughput apps.
  • Cross-partition queries and unindexed scans are the usual server-side latency culprits.

5.1.5 Configure Azure Monitor Alerts for Azure Cosmos DB

Alerts turn passive metrics and logs into proactive notifications. Cosmos DB supports three alert types: metric alerts (static or dynamic thresholds), log (KQL) alerts over Log Analytics, and activity-log alerts for control-plane events. Azure even ships recommended alert rules for the most common failure modes.

Key facts

  • Recommended rules: rate limiting on RU (metric alert on StatusCode = 429), region failed over (count > threshold), and account keys rotated (activity-log alert).
  • Metric alerts fire on thresholds or dynamic baselines; log alerts run a saved KQL query on a schedule.
  • A dedicated how-to covers alerting when a logical partition key approaches the 20-GB storage limit.
  • Alerts route to action groups (email, SMS, webhook, Logic Apps, Functions).
# Metric alert: fire when 429 (throttled) requests exceed a threshold
$acct = Get-AzResource -ResourceGroupName "rg-cosmos" `
  -ResourceType "Microsoft.DocumentDB/databaseAccounts" -Name "cosmos-prod"

$condition = New-AzMetricAlertRuleV2Criteria `
  -MetricName "TotalRequests" `
  -DimensionSelection (New-AzMetricAlertRuleV2DimensionSelection `
      -DimensionName "StatusCode" -ValuesToInclude "429") `
  -TimeAggregation Total -Operator GreaterThan -Threshold 100

Add-AzMetricAlertRuleV2 -Name "cosmos-429-alert" -ResourceGroupName "rg-cosmos" `
  -WindowSize 00:05:00 -Frequency 00:01:00 -TargetResourceId $acct.ResourceId `
  -Condition $condition -Severity 2 -ActionGroupId $actionGroupId

Exam tips & gotchas

  • Key rotation and region failover are control-plane events → use activity-log alerts, not metric alerts.
  • Throttling (429) is a metric alert on Total Requests split by StatusCode.
  • Log alerts require a diagnostic setting sending logs to Log Analytics first.

Practice

Q1. You want to be paged when someone rotates the account keys. Which alert type do you configure?
Answer: An activity-log alert on the key-rotation operation. Why: Key rotation is a control-plane event captured in the activity log, not a metric.
Q2. You need an alert when throttling becomes frequent. What signal and split do you use?
Answer: A metric alert on Total Requests split by StatusCode = 429. Why: Throttling surfaces as 429 responses in the request metric.
Q3. A logical partition is nearing its storage ceiling. How do you get warned early?
Answer: Configure the alert on logical partition key storage size approaching 20 GB (using PartitionKeyStatistics logs). Why: The 20-GB logical partition limit is hard; alerting early prevents write failures.

5.1.6 Implement and Query Azure Monitor Resource Logs for Azure Cosmos DB

Unlike metrics, resource (data-plane) logs are not collected until you create a diagnostic setting that routes them to Log Analytics, a storage account, or Event Hubs. Once flowing to Log Analytics you query them with KQL to inspect individual requests, query runtime statistics, and per-partition data.

Key facts

5.1.7 Monitor Throughput Across Partitions

Key facts

5.1.8 Monitor Distribution of Data Across Partitions

5.2 Implement Backup and Restore for an Azure Cosmos DB Solution

5.2.1 Choose Between Periodic and Continuous Backup

Cosmos DB backs up your data automatically in one of two modes. Periodic (the default) takes full snapshots on an interval and requires a Microsoft support ticket to restore. Continuous gives you self-service point-in-time restore (PITR) to any second within the retention window. Both always restore into a new account.

Key facts

  • Periodic: full backup every 4 hours by default; last two copies retained (interval/retention configurable); stored in geo-redundant blob storage; restore ONLY via a support request; deleted db/container snapshots kept 30 days.
  • Continuous: self-service PITR to any second; two tiers — 7-day (free) and 30-day (paid); you trigger restore via portal/CLI/PowerShell/ARM.
  • Restore always targets a new account (you cannot overwrite the source in place).
  • Backups don't consume your provisioned RUs.
Key: Need self-service, "restore to any second" recovery? Choose continuous backup. Periodic requires opening a support ticket.

Exam tips & gotchas

  • Continuous = self-service PITR; periodic = support-ticket restore. This is the classic exam decision.
  • Restore always creates a new account, in both modes.
  • Continuous tiers are 7-day (free) and 30-day (paid) — you can't restore beyond the window.

Practice

Q1. A team must recover to the exact second before an accidental bulk delete, without contacting support. Which backup mode?
Answer: Continuous backup with point-in-time restore. Why: Only continuous mode provides self-service restore to an arbitrary timestamp.
Q2. With periodic backup, how is a restore initiated?
Answer: By opening a Microsoft support ticket. Why: Periodic-mode restores are performed by Microsoft support, into a new account.
Q3. After any restore, where does the data land?
Answer: In a new Cosmos DB account. Why: Both backup modes restore into a new account, never overwriting the source.

5.2.2 Configure Periodic Backup

For periodic mode you tune two knobs — the backup interval and the retention (how many copies to keep) — plus the storage redundancy of the backup blobs. Two backup copies are free; extra copies are billed, and backups never touch your RU budget.

Key facts

  • Configure interval + retention via portal, CLI, PowerShell, or ARM.
  • Backup storage redundancy: Geo (default), Zone, or Local.
  • What a restore does not bring back: account keys (new ones are generated), firewall/VNet/private-endpoint config, RBAC assignments, stored procedures/triggers/UDFs, TTL-expired documents, analytical store, and materialized views. The restored account is single-region (the write region).
# Set periodic backup interval, retention, and redundancy
az cosmosdb update `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --backup-interval 240 `
  --backup-retention 16 `
  --backup-redundancy Zone

Exam tips & gotchas

  • Interval is in minutes; retention is a count of copies (or hours depending on tool) — read the parameter carefully.
  • Restored accounts don't include keys, network rules, RBAC, or server-side scripts — you reconfigure those.
  • Backup redundancy (Geo/Zone/Local) is independent of the account's own data redundancy.

Practice

Q1. After a periodic restore, apps fail to authenticate with the old connection string. Why?
Answer: The restored account has new keys; account keys are not restored. Why: Restores generate fresh keys, so connection strings must be updated.
Q2. You must keep backup blobs within a single region for data-residency reasons. Which redundancy do you pick?
Answer: Local (locally redundant) backup storage. Why: It keeps backup copies within the region rather than geo-replicating them.

5.2.3 Configure Continuous Backup and Point-in-Time Restore

Continuous backup is enabled at account creation or by a one-way migration from periodic. You pick the 7-day (free) or 30-day (paid) tier, and mutations are captured asynchronously (within roughly 100 seconds), enabling restore to any point in the retention window.

Key facts

  • Enable at creation or migrate periodic → continuous (irreversible). You can switch tiers (7 ↔ 30 day).
  • Mutations are backed up asynchronously within ~100 seconds.
  • Supported APIs: NoSQL, MongoDB, Gremlin, Table — not Cassandra.
  • Not restored: firewall/VNet/private-endpoint config, data-plane RBAC, stored procedures/triggers/UDFs, and all source regions (restore is single-region).
  • CMK + continuous backup requires a user- or system-assigned managed identity (not the first-party identity).
# Create an account with continuous 7-day backup
az cosmosdb create `
  --resource-group rg-cosmos `
  --name cosmos-pitr `
  --locations regionName=eastus `
  --backup-policy-type Continuous `
  --continuous-tier Continuous7Days

# Migrate an existing periodic account to continuous (one-way)
az cosmosdb update `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --backup-policy-type Continuous
Gotcha: Migrating periodic → continuous is one-way; you cannot switch back to periodic. And Cassandra API does not support continuous backup.

Exam tips & gotchas

  • Continuous supports NoSQL, MongoDB, Gremlin, Table — memorize that Cassandra is excluded.
  • Migration to continuous is irreversible.
  • CMK accounts need a managed identity to use continuous backup.

Practice

Q1. Which API cannot use continuous backup?
Answer: API for Apache Cassandra. Why: Continuous backup supports NoSQL, MongoDB, Gremlin, and Table only.
Q2. A team migrated to continuous backup and now wants to revert to periodic. Can they?
Answer: No — the migration is one-way. Why: Periodic → continuous cannot be reversed.

5.3.1 Choose Between Service-Managed and Customer-Managed Encryption Keys

Encryption at rest is always on in Cosmos DB (AES-256) with no toggle and no extra cost, using service-managed keys by default. If your compliance regime demands that you control the key lifecycle, you can add customer-managed keys (CMK) stored in Azure Key Vault, giving a second layer of encryption you manage.

Key facts

  • CMK gives you key rotation and revocation control, at the cost of a small RU overhead and Key Vault dependency.

Exam tips & gotchas

  • Choose CMK only when compliance requires customer key control — it adds operational burden.

Q2. If you do nothing, is your data encrypted at rest?

5.3.2 Configure Network-Level Access Control for Azure Cosmos DB

Network controls decide from where requests may reach the account, layered on top of (never instead of) the auth token. The three mechanisms are the IP firewall, VNet service endpoints, and Private Link / private endpoints, with publicNetworkAccess = Disabled as the strongest lockdown.

Key facts

  • IP firewall (ipRules / CIDR): non-listed IPs get 403. "Accept connections from within Azure datacenters" adds 0.0.0.0 (needed for some Azure services with no fixed IP).
  • VNet service endpoints allow specific subnets to reach the account over the Azure backbone.
  • Private endpoints (Private Link) assign a private IP in your subnet; combine with publicNetworkAccess = Disabled, which takes precedence over all firewall rules.
  • A valid auth token is ALWAYS required regardless of network rules — network controls are additive.
# Restrict to specific IP ranges and Azure services, keep public access on
az cosmosdb update `
  --resource-group rg-cosmos --name cosmos-prod `
  --ip-range-filter "13.88.56.148,40.65.0.0/16,0.0.0.0"

# Lock down entirely: disable public network access (use private endpoints)
az cosmosdb update `
  --resource-group rg-cosmos --name cosmos-prod `
  --public-network-access Disabled
Key: Network rules never replace authentication — even from an allowed IP or private endpoint, a request still needs a valid key or Entra token.

Exam tips & gotchas

  • publicNetworkAccess=Disabled overrides firewall rules — nothing public gets in.
  • 0.0.0.0 in the IP filter is the special "allow Azure datacenter traffic" entry, not "allow the world."
  • Private endpoints give private IPs; service endpoints keep public IPs but restrict by subnet.

Q1. You want the account reachable only through a private IP in your VNet, with no public exposure. What do you configure?

Q2. An Azure service with no fixed outbound IP must reach Cosmos DB behind an IP firewall. What enables it?
Answer: Enable "Accept connections from within Azure datacenters" (adds 0.0.0.0). Why: It permits Azure-internal traffic when the caller lacks a static IP.
Q3. With a strict IP firewall in place, does a caller still need credentials?
Answer: Yes. Why: Network rules are additive; a valid key or Entra token is always required.

5.3.4 Manage Control Plane Access to Azure Cosmos DB by Using Azure Role-Based Access Control (RBAC)

The control plane governs management of the account and its resources (metadata) through Azure RBAC — creating accounts, changing throughput, reading region config — but not the data inside. The built-in Cosmos DB Operator role manages accounts while explicitly blocking access to keys, connection strings, and data.

Key facts

  • Control plane = manage resources/metadata via Azure RBAC; it does not grant data access.
  • Cosmos DB Operator manages accounts but cannot read keys/connection strings or data.

5.3.9 Implement Customer-Managed Keys for Encryption

Implementing CMK means creating a key in Key Vault, granting Cosmos DB the rights to wrap/unwrap it, and setting the key URI at account creation. Key Vault must have Soft Delete and Purge Protection enabled, and you accept a small RU overhead on operations.

Key facts

  • Key in Azure Key Vault (RSA ≥ 3072). Key Vault requires Soft Delete + Purge Protection.
  • Grant the Cosmos DB principal Get / WrapKey / UnwrapKey (access policy) or the Key Vault Crypto Service Encryption User role (RBAC).
  • Set the Key URI at account creation. Revoke by disabling the key → after revoke, only account deletion is possible. Rotate via a new key version or a new Key URI.
  • Overhead is roughly +5% for point reads, +6% for writes, +15% for queries/change feed.
# Create a CMK-enabled account referencing a Key Vault key
az cosmosdb create `
  --resource-group rg-cosmos --name cosmos-cmk `
  --locations regionName=eastus `
  --key-uri "https://kv-cosmos.vault.azure.net/keys/cosmos-cmk/<version>" `
  --assign-identity [system]
Gotcha: Revoking (disabling) the CMK makes the account unreadable — the only remaining action is to delete the account. Never revoke without a plan.

Exam tips & gotchas

  • Key Vault needs Soft Delete + Purge Protection or CMK setup fails.
  • CMK is set at account creation; grant Get/WrapKey/UnwrapKey to Cosmos DB.
  • Expect measurable RU overhead, largest on queries/change feed.

Practice

Q1. CMK account creation fails referencing the Key Vault. Which two Key Vault settings are required?
Answer: Soft Delete and Purge Protection. Why: Cosmos DB CMK requires both to protect against irreversible key loss.
Q2. A security incident requires immediately blocking all data access via the customer key. What happens after you disable the key?
Answer: The account becomes inaccessible and can only be deleted. Why: Revoking the CMK removes unwrap capability, leaving deletion as the sole option.

5.3.10 Implement Always Encrypted

Always Encrypted protects specific sensitive fields with client-side encryption — the service never sees plaintext or the keys. It uses a database-level data encryption key (DEK) wrapped by a customer-managed key (CMK / column master key) in Key Vault, and a container encryption policy that maps each property to a key and encryption type.

Key facts

  • Field-level, client-side encryption; the service can't decrypt.
  • Uses a DEK (database-scoped, stored in Cosmos DB) wrapped by a CMK in Key Vault.
  • The container's encryption policy is immutable — set at creation — and maps each property to a DEK + encryption type.
  • Deterministic encryption allows equality filters (same plaintext → same ciphertext, weaker); randomized is stronger but not queryable.
  • Implemented with the .NET (Microsoft.Azure.Cosmos.Encryption) or Java packages using a KeyResolver + managed identity.
// Client-side Always Encrypted with a Key Vault-backed key resolver
var keyResolver = new KeyResolver(new DefaultAzureCredential());
var client = new CosmosClient(endpoint, credential)
    .WithEncryption(keyResolver, KeyEncryptionKeyResolverName.AzureKeyVault);

// Container encryption policy: SSN deterministic (queryable), Salary randomized
var paths = new List<ClientEncryptionIncludedPath>
{
    new() { Path = "/ssn",    ClientEncryptionKeyId = "cek1",
            EncryptionType = "Deterministic",
            EncryptionAlgorithm = "AEAD_AES_256_CBC_HMAC_SHA256" },
    new() { Path = "/salary", ClientEncryptionKeyId = "cek1",
            EncryptionType = "Randomized",
            EncryptionAlgorithm = "AEAD_AES_256_CBC_HMAC_SHA256" }
};

Exam tips & gotchas

  • Deterministic = queryable by equality; randomized = strongest but no querying.
  • The encryption policy is immutable once the container is created.
  • Always Encrypted is field-level and client-side; CMK-at-rest is account-level and server-side. Different features.

Practice

Q1. You must encrypt an SSN field but still filter by exact SSN value. Which encryption type do you choose?
Answer: Deterministic. Why: Deterministic encryption produces identical ciphertext for identical plaintext, enabling equality queries (randomized doesn't).
Q2. After creating a container you need to change which properties are encrypted. Is that allowed?
Answer: No — the encryption policy is immutable and set at container creation. Why: You must recreate the container (and migrate data) to change encrypted paths.

5.4.2 Move Data by Using Client SDK Bulk Operations

Setting AllowBulkExecution = true on the client changes how the SDK dispatches concurrent point operations: it groups them into batched service calls per partition, dramatically increasing import throughput. You fire many operations concurrently (for example, with Task.WhenAll) and let the SDK optimize the batching.

Key facts

  • new CosmosClient(..., new CosmosClientOptions { AllowBulkExecution = true }); the setting is immutable once the client is built.
  • The SDK batches concurrent point operations per partition into fewer service calls.
  • Use Task.WhenAll to submit many operations at once.
  • Fastest import: temporarily raise RU/s and reduce indexing (e.g., exclude paths) during the load.
var options = new CosmosClientOptions { AllowBulkExecution = true };
using var client = new CosmosClient(endpoint, key, options);
var container = client.GetContainer("SalesDB", "Orders");

var tasks = orders.Select(o =>
    container.CreateItemAsync(o, new PartitionKey(o.CustomerId)));
await Task.WhenAll(tasks);   // SDK batches these per partition

Exam tips & gotchas

  • AllowBulkExecution is set at client construction and can't change later.
  • Bulk trades per-request latency for aggregate throughput — great for imports, not for chatty low-latency apps.
  • Raise RU/s and trim indexing during the load, then restore both.

Practice

Answer: Set AllowBulkExecution = true in CosmosClientOptions at client creation; it's immutable afterward. Why: Bulk batching is a client-construction-time option.

5.5.1 Choose When to Use Declarative Versus Imperative Operations

Exam tips & gotchas

  • "Repeatable, source-controlled, environment parity" → declarative (ARM/Bicep).

5.5.3 Migrate Between Standard and Autoscale Throughput by Using PowerShell or Azure CLI

Switching a database or container between manual (standard) and autoscale throughput is supported only through the portal, Azure CLI, or PowerShell — not the SDK and not ARM. When you enable autoscale, the system chooses a starting max RU/s based on current manual RU/s and storage.

Key facts

  • Migration paths: portal, CLI, or PowerShell only (NOT SDK, NOT ARM template).
  • CLI: az cosmosdb sql database|container throughput migrate --throughput-type 'autoscale'|'manual'.
  • Enabling autoscale sets a starting max RU/s derived from the current manual value and storage.
# Migrate a container from manual to autoscale throughput
az cosmosdb sql container throughput migrate `
  --resource-group rg-cosmos `
  --account-name cosmos-prod `
  --database-name SalesDB `
  --name Orders `
  --throughput-type autoscale
Gotcha: You cannot migrate throughput type via an ARM/Bicep template — use CLI or PowerShell (or the portal).

5.5.4 Initiate a Regional Failover by Using PowerShell or Azure CLI

For single-write-region accounts you can trigger a manual failover by reordering failover priorities with az cosmosdb failover-priority-change, promoting a read region to the new write region. Service-managed failover, when enabled, handles regional outages automatically.

Key facts

  • az cosmosdb failover-priority-change --failover-policies <region>=<priority> reorders priorities / triggers manual failover of the write region.
  • Manual failover applies to single-write-region accounts; priority 0 is the write region.
  • Service-managed (automatic) failover reacts to outages without manual action.
# Promote West US to write region (priority 0), East US becomes priority 1
az cosmosdb failover-priority-change `
  --resource-group rg-cosmos `
  --name cosmos-prod `
  --failover-policies westus=0 eastus=1

Exam tips & gotchas

  • Priority 0 = the write region; changing priorities triggers/plans the failover.
  • Manual failover = single-write accounts; multi-region-write accounts don't need it the same way.
  • Service-managed failover is automatic; manual failover is deliberate via CLI/PowerShell.

Practice

Q1. You need to deliberately move the write region to West US for a planned maintenance test. Which command?
Answer: az cosmosdb failover-priority-change with westus=0. Why: Setting a region to priority 0 promotes it to the write region.
Q2. What does priority 0 represent in the failover policy?
Answer: The current write region. Why: Failover priority 0 is always the write region for single-write accounts.

5.5.5 Maintain Indexing Policies in Production by Using Azure Resource Manager Templates

Keeping indexing policies under source control means storing each container's indexingPolicy in the ARM/Bicep template and rolling changes through CI/CD. Because index changes trigger an online transformation, production queries keep working while the new policy is applied.

Key facts

  • Store the container's indexingPolicy in the ARM/Bicep template; deploy changes through the pipeline.
  • Index transformation runs online (track with the index transformation progress) — no write downtime.
  • This keeps production indexing consistent, reviewable, and reproducible.
resource container 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases/containers@2024-11-15' = {
  parent: db
  name: 'Orders'
  properties: {
    resource: {
      id: 'Orders'
      partitionKey: { paths: [ '/customerId' ], kind: 'Hash' }
      indexingPolicy: {
        indexingMode: 'consistent'
        includedPaths: [ { path: '/*' } ]
        excludedPaths: [ { path: '/notes/?' } ]
      }
    }
  }
}

Exam tips & gotchas

  • Index changes deployed via template still trigger an online transformation — no downtime.
  • Source-controlling indexing policy prevents config drift between environments.

C. Partition-key decision quick-reference

GoalRule of thumb
Spread load evenlyPick a key with high cardinality (many distinct values) so requests and storage distribute across physical partitions.
Avoid a hot partitionAvoid low-cardinality or monotonically increasing keys (e.g., a single status or the current date) that concentrate traffic.
Enable transactionsItems that must participate in a transactional batch or a transactional stored procedure must share the same logical partition (same partition-key value).
Low-cardinality fieldBuild a synthetic key — concatenate fields or add a random/hashed suffix to raise cardinality.

Gotcha: A single logical partition is capped at 20 GB and its own RU ceiling. The partition key is immutable once items are written — choosing it wrong means re-creating the container and migrating data.

D. Request Units & the 429 playbook

RU essentials

  • A 1 KB point read costs ~1 RU; writes, larger items, more indexed paths, and cross-partition queries cost more.
  • Read the actual charge from the x-ms-request-charge response header (SDK: RequestCharge).
  • Exceeding provisioned RU/s returns HTTP 429 (Request rate too large) with a x-ms-retry-after-ms hint; the SDK retries automatically up to a configurable limit.

When you see 429s

  • Check the Normalized RU Consumption metric — if one partition sits at 100% while others idle, you have a hot partition (fix the partition key).
  • If all partitions are saturated, raise RU/s (or switch to autoscale).
  • Trim indexing on write-heavy containers; use bulk mode for large imports; increase SDK MaxRetryAttemptsOnRateLimitedRequests / MaxRetryWaitTimeInSeconds.

J. Glossary

Terms to know cold

  • RU/s (Request Unit per second): the normalized currency of throughput; a 1 KB point read ≈ 1 RU.
  • Logical partition: all items sharing one partition-key value; capped at 20 GB.
  • Physical partition: the backend host that stores one or more logical partitions; split automatically as data/RU grow.
  • Synthetic partition key: a composed key (concatenation or suffix) that raises cardinality.
  • Hierarchical partition key: up to three key levels for sub-partitioning and > 20 GB prefixes.
  • ETag / optimistic concurrency: conditional write using If-Match to prevent lost updates.
  • Session token: the marker that provides read-your-own-writes under session consistency.
  • Change feed: a persistent, ordered log of inserts/updates per partition, consumed by the processor or Azure Functions.
  • Analytical store: an auto-synced column store for HTAP analytics that doesn't consume transactional RU/s.
  • PITR: point-in-time restore, enabled by continuous backup.
  • PPAF: per-partition automatic failover.
