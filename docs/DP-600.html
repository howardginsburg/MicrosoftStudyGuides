Domain 1 — Maintain a Data Analytics Solution 25–30%

Domain 1 covers the operational and governance responsibilities of a Fabric analytics solution: securing workspaces and data at every granularity, managing the trustworthiness of published assets, and controlling how code and semantic models move through the development lifecycle into production. Mastery here underpins safe, auditable enterprise analytics at scale.

1.1 Implement Security and Governance

1.1.1 Implement Workspace-Level Access Controls

Every Microsoft Fabric workspace comes with four built-in roles that govern what principals can do inside that workspace. Roles can be assigned to individuals, security groups, Microsoft 365 groups, distribution lists, or Microsoft Entra service principals. When a user belongs to multiple groups, Fabric grants the highest permission from any applicable membership — not the intersection.

Workspace roles are strictly scoped to a single workspace; they do not cascade across workspaces or to capacity administration. OneLake data-access roles can further refine data permissions within a workspace independently of these workspace roles.

Key facts

  • Four roles (highest → lowest): Admin, Member, Contributor, Viewer.
  • Admin only: delete or update the workspace itself, add other Admins, create a workspace identity, and connect the workspace to a Git repository.
  • Admin + Member: add Members or lower-privilege roles to the workspace.
  • Admin + Member + Contributor: create, write, and delete Fabric items; execute notebooks and pipelines; schedule refreshes via an on-premises gateway; read Lakehouse and Warehouse data via OneLake APIs and Spark (ReadAll permission).
  • All four roles (including Viewer): view item content; connect to the SQL analytics endpoint; read Lakehouse and Warehouse data via T-SQL through the TDS endpoint (ReadData permission).
  • Viewer cannot: write data, create items, or read data via OneLake APIs or Spark.
  • Admin and Member can edit OneLake security roles; Contributor and Viewer cannot.
  • Service principals (Entra app registrations) can be assigned workspace roles and behave like users for API-based operations.
  • By default, only Admins can connect or disconnect the workspace from Git. Admins can optionally enable a setting that allows Contributors/Members to also switch branches.
  • Workspace roles do not span workspaces; each workspace is an independent permission boundary.
Tip: When a user belongs to several groups with different roles, Fabric awards the highest permission, not the most restrictive. Verify effective permissions explicitly before assuming a user is limited to Viewer.
Key: RLS in a semantic model applies only to Viewer-level users. Admin, Member, and Contributor workspace roles bypass row-level security and always see all data.

Exam tips & gotchas

  • Viewers can read Lakehouse/Warehouse data through T-SQL (ReadData via the TDS endpoint), but they cannot read through OneLake APIs or Spark (ReadAll) — that requires Contributor or higher.
  • Only a workspace Admin can connect or disconnect the workspace from a Git repository — no exceptions unless the Admin explicitly enables branch-switching for Contributors/Members.
  • Contributors cannot add other Contributors or Members — only Admins and Members can add Members or lower roles.
  • OneLake security roles add a second permission layer; they can restrict data access even for Contributor-level users at the storage level.
  • Workspace roles are scoped to a single workspace; they grant no access to items in other workspaces.

Practice

Q1. A data analyst needs to run Spark notebooks against a Lakehouse and schedule pipeline refreshes, but must not be able to delete the workspace or modify workspace membership. What is the minimum sufficient workspace role?
Answer: Contributor. Why: Contributor can create and execute Fabric items (notebooks, pipelines) and read data via OneLake APIs (ReadAll). Admin and Member grant more rights than necessary (managing membership, connecting Git). Viewer cannot execute items or write data.
Q2. A user connects to a Fabric Lakehouse SQL analytics endpoint using SSMS. The user has only the Viewer workspace role and no OneLake security role. What level of data access does the user have?
Answer: Read-only access via T-SQL (ReadData through the TDS endpoint). Why: All four workspace roles — including Viewer — can connect to the SQL analytics endpoint and query data through T-SQL. Viewers cannot use OneLake APIs or Spark (ReadAll requires Contributor+), but ReadData via T-SQL is permitted for all roles.
Q3. A senior developer with the Member workspace role wants to connect the workspace to Azure DevOps. Can they do this without Admin involvement?
Answer: No. Why: Only workspace Admins can connect or disconnect a workspace from a Git repository. The Member role does not include this capability. The developer must request that an Admin performs the connection, or be elevated to Admin.

1.1.2 Implement Item-Level Access Controls

Item permissions control access to a single Fabric artifact — a semantic model, report, lakehouse, and so on — independently of workspace roles. A user who holds no workspace role can still be granted access to a specific item via a sharing link, reaching only that item without any workspace context. Item permissions and workspace roles coexist; Fabric evaluates both during access checks.

Access evaluation follows a three-layer model: first Microsoft Entra authentication, then Fabric item access (workspace role or item permission), and finally data-layer security such as RLS and OneLake roles.

Key facts

  • Read (default on share): the recipient can view the item's metadata and any reports built on it, but this permission alone does not grant access to underlying data via SQL or OneLake.
  • Build: lets the recipient create new reports, dashboards, and other downstream content on top of a shared semantic model — the key permission for self-service BI.
  • Write: lets the recipient modify the item itself (e.g., edit a semantic model's definition).
  • Reshare: lets the recipient grant access to other users.
  • DirectLake reports shared with Read-only item permission require the user to also hold OneLake data permissions on the underlying Delta tables to see actual data.
  • If a user holds a workspace Viewer role, revoking item-level permissions alone is insufficient — workspace role access remains. Both must be removed to fully block access.
  • The SQL analytics endpoint supports native T-SQL GRANT/DENY for granular column or schema-scoped control beyond item permissions.
  • Security evaluation order: (1) Entra authentication → (2) Fabric access (workspace role or item permission) → (3) data security (RLS, OLS, OneLake roles).
Gotcha: Sharing a semantic model with Read permission grants view access to existing reports but does not allow direct data queries via SQL or OneLake. Grant Build for self-service report creation; use SQL GRANT on the SQL analytics endpoint for direct data queries.

Exam tips & gotchas

  • Build permission is required for a consumer to create new reports on a shared semantic model — Read alone is not sufficient.
  • A user with no workspace role but with item-level Read can access the item via direct link without seeing the workspace.
  • Removing item-level permissions does not revoke access if the user also has a workspace role — both layers must be revoked.
  • The SQL analytics endpoint supports T-SQL GRANT/DENY independently, enabling column- or schema-scoped control that is separate from item-level permissions.

Practice

Q1. A business analyst in a different workspace needs to create new Power BI reports against a published semantic model. The analyst should not be able to modify the semantic model or browse other workspace items. What is the minimum permission needed on the semantic model?
Answer: Build. Why: Build permission lets the analyst create new reports and dashboards on a shared semantic model. It does not grant Write access (to modify the model) or visibility of the source workspace. Read alone does not enable new report creation.
Q2. A user was granted item-level Read permission on a report backed by a DirectLake semantic model. The report loads but shows no data. What additional permission is most likely missing?
Answer: OneLake data-access permission on the underlying Delta tables. Why: DirectLake reads directly from OneLake storage. Item-level Read on the report does not include OneLake storage access. The user needs an OneLake data-access role that grants Read on the relevant tables for data to be returned.
Q3. An admin revokes the item-level share permission from a user for a semantic model. The user still has a Viewer workspace role. Can the user still access the semantic model?
Answer: Yes. Why: Workspace roles are independent of item permissions. A Viewer role grants the right to see all items in the workspace. Removing the item-level share does not override the workspace role; both must be revoked to fully block access.

1.1.3 Implement Row-Level, Column-Level, Object-Level, and File-Level Access Control

Fabric provides four distinct data-layer security mechanisms that operate at different granularities. Row-Level Security (RLS) filters which rows a user sees within a table, using DAX expressions inside role definitions. Object-Level Security (OLS) hides entire tables or individual columns from certain roles, concealing both the data and the object's metadata name. Column-level restriction in the tabular model is technically a form of OLS (CLS via OLS). OneLake data-access roles enforce file- and folder-level RBAC at the storage layer, independently of any semantic model.

These controls activate after a user has passed Fabric item-access checks. They filter or block specific data for users who have already been granted the right to reach the item.

Key facts — RLS (Row-Level Security)

  • RLS restricts which rows a user sees by applying a DAX filter expression on one or more columns within a role definition.
  • Roles are defined in Power BI Desktop → Modeling → Manage Roles; published to the Power BI service, where members are added under the semantic model's Security option.
  • RLS applies only to workspace Viewer-level users (and users who accessed the model via item sharing only). Admin, Member, and Contributor workspace roles bypass RLS and always see all rows.
  • Static RLS: hard-coded filter value — e.g., restrict rows where Region equals a fixed string.
  • Dynamic RLS: uses USERPRINCIPALNAME() or USERNAME() so a single role adapts to each signed-in user via a mapping table — the standard production pattern.
  • Bi-directional cross-filter propagation with RLS must be enabled per relationship and can degrade query performance.
  • For Direct Lake semantic models, RLS is supported; DAX fallback to DirectQuery also respects the RLS filters.
  • Workflow: define roles in Desktop → publish → add members in the service → validate with Test as role or View as.

Key facts — OLS (Object-Level Security) and CLS (Column-Level Security)

  • OLS hides entire tables or individual columns from a role, concealing both the data and the object's metadata name — it appears as if the object does not exist.
  • Applies to tabular models at compatibility level 1400+ (Fabric/Premium, Azure AS, SSAS 2017+).
  • Configured via: TMDL view in Power BI Desktop, open-source Tabular Editor (GUI), TMSL JSON scripts, or TOM API. There is no dedicated OLS dialog in Power BI Desktop's Manage Roles UI.
  • The controlling property is metadataPermission on a tablePermissions or columnPermissions object: none = hidden; read = visible.
  • Column-Level Security (CLS) in the semantic model is implemented via columnPermissions with metadataPermission: none — the same OLS mechanism applied at column granularity.
  • Workspace Admin, Member, and Contributor roles bypass OLS — they can see all secured objects regardless.
  • OLS cannot be applied to a table that would break a required relationship chain — this causes a design-time error.
  • OLS is incompatible with Q&A visuals, Quick insights, Smart narrative, and the Excel Data Types gallery.
  • Combining RLS and OLS rules from different roles on the same user causes a query-time error.

Key facts — OneLake Data-Access Roles (File/Folder Level)

  • OneLake uses a deny-by-default RBAC model: users start with no data access unless explicitly granted by an OneLake role.
  • Each role specifies: Permission (Read or ReadWrite), Scope (specific tables, folders, or schemas), and Members (Entra identities/groups).
  • Supported Fabric items: Lakehouse, Azure Databricks Mirrored Catalog, Mirrored Databases.
  • Two default roles are auto-created per Lakehouse: DefaultReader (Read on all folders; assigned to all ReadAll-permission users) and DefaultReadWriter (ReadWrite on all folders; assigned to all Write-permission users). Modify or remove these to enforce custom access.
  • Workspace Admin, Member, and Contributor automatically have Write access to OneLake — OneLake roles cannot restrict them below this. Viewer has no OneLake access by default; an OneLake role must explicitly grant it.
  • Admin and Member can manage OneLake security roles; Contributor and Viewer cannot.
  • Supported engines (GA): Lakehouse, Spark notebooks, SQL Analytics Endpoint (user-identity mode), Direct Lake on OneLake. Eventhouse RLS: public preview.
  • For non-authorized engines where RLS/CLS is defined, the entire query is blocked rather than partially filtered, to prevent data leakage.
-- Static RLS: restrict rows to a single hard-coded region
-- Applied in the role filter for the Sales table:
[Region] = "West"

-- Dynamic RLS: one role covers all users via a mapping table
-- DimEmployee[UserEmail] must hold the user's UPN (e.g., from Entra ID)

Gotcha: RLS applies only to Viewer-level consumers. Admin, Member, and Contributor workspace roles always see all rows — they are not filtered by any RLS role definition. To test RLS as a developer, use the View as / Test as role feature in the Fabric service.
Key distinction summary:
  • RLS — row filtering via DAX; defined in Desktop Manage Roles; bypassed by Admin/Member/Contributor.
  • CLS (semantic model) — column hiding via columnPermissions metadataPermission: none; same OLS mechanism, same bypass rules.
  • OLS — table and column hiding (data + metadata name); no dedicated Desktop UI — requires TMDL, Tabular Editor, TMSL, or TOM; Admin/Member/Contributor bypass it.
  • OneLake data-access roles — file/folder-level RBAC at the storage layer; deny-by-default; independent of semantic model roles; Admin/Member/Contributor still have Write access and cannot be restricted below it.

Exam tips & gotchas

  • OLS hides the metadata too — a secured column does not appear in field lists, unlike RLS which only filters rows.
  • OLS has no dedicated dialog in Power BI Desktop's Manage Roles UI; configure it via TMDL, Tabular Editor, TMSL, or TOM. RLS does have a "Manage Roles" dialog.
  • Admin, Member, and Contributor bypass both RLS and OLS. Security at these levels must be enforced by limiting who holds those roles.
  • OneLake data-access roles use deny-by-default; Viewers have no OneLake access unless an OneLake role explicitly grants it.
  • Combining RLS and OLS from different roles on the same user generates a query-time error — ensure a user is assigned to only one role that applies both if needed.
  • For Direct Lake models, RLS is enforced even when the query falls back to DirectQuery mode.

Practice

Q1. A report should show each salesperson only the rows for their own territory, dynamically based on their login. A single role must cover all salespeople. Which DAX function enables this pattern?
Answer: USERPRINCIPALNAME(). Why: Dynamic RLS uses USERPRINCIPALNAME() to return the signed-in user's UPN, matched against a column in the model mapping users to territories. One role definition handles all users without needing a role per person. USERNAME() is an alternative but returns the domain\user format.
Q2. A finance semantic model contains a Salary column. It should be completely invisible to most consumers — absent from field lists, report visuals, and metadata. Which security mechanism achieves this?
Answer: Object-Level Security (OLS) / Column-Level Security via columnPermissions with metadataPermission: none. Why: OLS hides both the data and the column name from the role, making it appear nonexistent. RLS would only filter rows — the column name would still be visible in field lists.
Q3. A developer with a Contributor workspace role opens a Power BI report that has RLS defined. Are the RLS filters applied to the developer's session?
Answer: No. Why: RLS applies only to Viewer-level consumers. Admin, Member, and Contributor workspace roles bypass RLS and always see all rows. The developer must use the "View as / Test as role" feature to simulate an end-user experience.
Q4. A Lakehouse owner wants to grant a third-party team read access to only the "SalesData/2024/" folder, not the entire Lakehouse. Which mechanism enables this?
Answer: A custom OneLake data-access role with a folder-scoped permission. Why: OneLake RBAC roles support folder-level scoping. The owner can modify or replace the default roles with a custom role that grants Read permission scoped specifically to "SalesData/2024/", leaving all other folders inaccessible.

1.1.4 Apply Sensitivity Labels to Items

Sensitivity labels in Microsoft Fabric are powered by Microsoft Purview Information Protection. Labels are created and published in the Microsoft Purview compliance portal, after which they surface in Fabric and Power BI Desktop to classify data asset confidentiality — for example, "Public", "Confidential", or "Highly Confidential". Labels can travel with data across supported export paths, extending governance outside the Fabric boundary.

Key facts

  • Labels must be created and published to users via a label policy in the Purview compliance portal before they appear in Fabric. Being in the "allowed users" list alone is insufficient.
  • Licensing: a Microsoft Purview Information Protection license plus a Power BI Pro or PPU license are required to apply labels to Power BI items.
  • Five label capabilities in Fabric:
    1. Manual labeling — user applies a label; supported for all Fabric items.
    2. Default labeling — label applied automatically on item creation or edit; all Fabric items (with limits for non-Power BI items).
    3. Mandatory labeling — users cannot save without a label; fully supported for Power BI items only.
    4. Programmatic labeling — via Power BI admin REST APIs; all Fabric items.
    5. Downstream inheritance — label propagates to dependent items; all Fabric items with limits.
  • Downstream label inheritance rules:
    • Power BI item → Power BI item: ✓ supported.
    • Fabric item → Fabric item: ✓ supported.
    • Fabric item → Power BI item: ✓ supported.
    • Power BI item → Fabric item: ✗ not supported.
  • Inheritance on creation: a notebook created from a Lakehouse inherits the Lakehouse label; a KQL Queryset created from a KQL Database inherits its label.
  • Source-data label inheritance is currently supported only for Power BI semantic models.
  • Supported export paths where labels travel with the file: Export to Excel, PDF, or PowerPoint; Analyze in Excel; PivotTable in Excel (requires Microsoft 365 E3+); download .pbix from Fabric.
  • CSV, TXT, and cross-tenant exports do not carry labels.
  • Sensitivity labels are not supported with Power BI Projects (.pbip) files — manage labels in the Fabric service post-publish.
  • Label coverage can be monitored in the OneLake catalog governance experience.
Gotcha: Sensitivity labels are not supported with the .pbip (Power BI Projects) file format. If your development workflow uses .pbip, labels must be applied or changed in the Fabric service after publishing — not in the project files.

Exam tips & gotchas

  • Labels must be published via a policy in Purview — simply creating a label does not make it available to users in Fabric.
  • Mandatory labeling (blocking save without a label) is fully supported only for Power BI items; other Fabric items have limited mandatory-labeling support.
  • Downstream inheritance does not flow from Power BI items to Fabric items — only the reverse direction (Fabric → Power BI) is supported.
  • CSV and TXT exports strip labels; only structured formats (Excel, PDF, PowerPoint, .pbix) carry labels out of Fabric.
  • .pbip files do not support sensitivity labels — this is a known limitation during the Power BI Projects preview.

Practice

Q1. A sensitivity label is applied to a published semantic model. A report author creates a new report from that semantic model in the same workspace. Will the report automatically inherit the label?
Answer: Yes, through downstream inheritance. Why: A Fabric semantic model to a Power BI report is a supported inheritance path. The new report inherits the label automatically when downstream inheritance is enabled (the default). The inheritance is applied at item creation.
Q2. A compliance officer wants to enforce that all Power BI reports must carry a sensitivity label before they can be saved. Which labeling capability satisfies this requirement?
Answer: Mandatory labeling. Why: Mandatory labeling prevents users from saving a Power BI item without first selecting a sensitivity label. It is configured through the Purview label policy and is fully supported for Power BI items.
Q3. A user exports a report page as a CSV file. Will the report's sensitivity label travel with the exported file?
Answer: No. Why: CSV and TXT exports are not in the list of supported export paths for label persistence. Only structured formats — Excel, PDF, PowerPoint, and .pbix downloads — carry the sensitivity label with the exported content.

1.1.5 Endorse Items

Endorsement is Fabric's mechanism for surfacing trusted, high-quality data assets to consumers. Three levels form a progressive trust hierarchy: Promoted (creator-assessed quality), Certified (organization-approved authority), and Master data (single source of truth for core reference data). Endorsed items display distinctive badges, appear higher in search and discovery lists, and show their badges in lineage view.

Key facts

  • Promoted: Any user with Write permission on the item can self-apply it. Signals the creator believes the item is production-ready and suitable for sharing.
  • Certified: Only users designated by a Fabric administrator (or a delegated domain administrator) can grant Certified status. Any user can request certification. Signals the item meets organizational quality standards and is authoritative.
  • Master data: Only users designated by a Fabric administrator can apply it. Any user can request it. Indicates the item is a core organizational reference source — the single source of truth (e.g., a product code list or customer master record).
  • Certification and Master data endorsement must first be enabled by a Fabric administrator in the admin portal. Without admin enablement, only Promoted is available.
  • Certification authority can be delegated to domain administrators, allowing per-domain certifiers without granting tenant-wide admin rights.
  • What can be endorsed:
    • Promoted and Certified: all Fabric and Power BI items, except Power BI dashboards.
    • Master data: only items that contain data (e.g., lakehouses, semantic models, warehouses). Cannot be applied to reports or dashboards.
  • Endorsement badges are visible in workspace lists, lineage view item cards, and OneLake catalog search results.
  • Endorsement is independent of sensitivity labels; both can coexist on the same item.
Key: Any user can request Certified or Master data status, but only admin-designated individuals can grant those badges. Promoted is the only tier that can be self-applied — by anyone with Write permission on the item.

Exam tips & gotchas

  • Power BI dashboards cannot be endorsed — neither Promoted, Certified, nor Master data applies to dashboards.
  • Master data cannot be applied to reports or dashboards — only to items that contain data.
  • Certified and Master data require admin enablement; if not enabled, only Promoted is available to all users.
  • Any user with Write permission on an item can apply Promoted — no admin involvement is needed for that level.
  • Certification authority can be delegated at the domain level, enabling per-domain governance without full tenant-admin privileges.

Practice

Q1. A data engineer wants to mark a Lakehouse as the authoritative source for all customer reference data across the organization. Which endorsement level is appropriate, and who can apply it?
Answer: Master data, applied by a Fabric administrator or an admin-designated user. Why: Master data is specifically designed for core organizational reference data representing the single source of truth. It cannot be self-applied — only admin-designated individuals can grant it, and the item must contain data (a Lakehouse qualifies).
Q2. A report author with Write permission on a semantic model wants to signal to colleagues that the model is production-ready and safe to use. Which endorsement can they apply without contacting an administrator?
Answer: Promoted. Why: Promoted is the only endorsement tier that can be self-applied by any user with Write permission on the item. Certified and Master data require an admin-designated approver — they cannot be self-granted.
Q3. Which of the following can receive the Master data endorsement: (a) a Power BI report, (b) a Fabric Lakehouse, (c) a Power BI dashboard?
Answer: (b) A Fabric Lakehouse only. Why: Master data applies only to items that contain data. Power BI reports and dashboards are visualization items; neither qualifies. A Lakehouse stores data and is a valid candidate for Master data endorsement.

1.2 Maintain the Analytics Development Lifecycle

1.2.1 Configure Version Control for a Workspace

Fabric Git integration connects a workspace to a branch in a cloud-hosted Git repository — either Azure DevOps or GitHub — so changes made in the workspace can be committed to source control, and changes made in source control can be pulled back into the workspace. Each workspace connects to exactly one branch and one folder within a repository, establishing a clear workspace-to-branch mapping.

The connection enables modern team practices: branching strategies, pull-request reviews, rollback to previous commits, and CI/CD automation. Semantic models are stored as TMDL files and reports as PBIR files, making them fully text-diffable during code review.

Key facts

  • Supported providers: Azure DevOps (cloud), GitHub (cloud), GitHub Enterprise (cloud). On-premises Azure DevOps Server is not supported.
  • Only a workspace Admin can connect or disconnect a workspace from Git.
  • Admins can optionally enable a workspace setting that allows Contributors/Members to also switch Git branches.
  • Two tenant switches must be enabled: (1) Users can synchronize workspace items with their Git repositories; (2) for GitHub, additionally Users can synchronize workspace items with GitHub repositories.
  • Authentication: Azure DevOps uses OAuth2 or Service Principal; GitHub requires a Personal Access Token (fine-grained with Contents read/write, or classic with repo scope).
  • Initial sync logic: if workspace is empty and Git branch has content → content copied to workspace; if both have content → user chooses direction.
  • Day-to-day operations: Commit (workspace → Git), Update from Git (Git → workspace), Undo (revert workspace items to last synced state).
  • Supported items: Lakehouses, Notebooks, Spark Job Definitions, Dataflow Gen2, Pipelines, Eventhouses, KQL Databases, Warehouses (preview), Power BI Reports (preview), Semantic models (preview). Unsupported items are silently ignored during sync — not deleted.
  • Source code format: semantic models use TMDL (YAML-like); reports use PBIR (JSON). Both are human-readable and suitable for line-by-line code review.
  • Workspace folder hierarchy is preserved in the repository.
Key: Git integration is workspace-to-branch, not workspace-to-repository. One workspace maps to exactly one branch (and one folder). To use a multi-branch workflow, map separate workspaces to separate branches and merge via standard Git pull requests.

Exam tips & gotchas

  • On-premises Azure DevOps Server is not supported — only cloud Git providers are supported.
  • Only workspace Admins can connect or disconnect from Git. Members and Contributors cannot, unless explicitly allowed to switch branches via the Admin-controlled workspace setting.
  • Unsupported item types are skipped silently during sync — they are not deleted from the workspace.
  • Semantic models → TMDL; reports → PBIR. Both are plain text and diffable.
  • The GitHub tenant switch must be enabled separately from the general Git sync tenant switch.

Practice

Q1. A team wants to connect their Fabric workspace to a self-hosted, on-premises Azure DevOps Server. Is this configuration supported?
Answer: No. Why: Fabric Git integration supports only cloud-hosted Git providers: Azure DevOps (cloud), GitHub (cloud), and GitHub Enterprise (cloud). On-premises Azure DevOps Server is explicitly not supported.
Q2. A developer with a Contributor workspace role wants to connect the workspace to a GitHub repository. What must happen first?
Answer: A workspace Admin must perform the connection. Why: Only workspace Admins can connect or disconnect a workspace from a Git repository. The Contributor role cannot perform this action regardless of the Git provider. An Admin must also ensure the required tenant switches are enabled.

Why: Both are plain-text, human-readable formats — TMDL is YAML-like and PBIR is JSON-based. They can be reviewed as readable text diffs in pull requests, enabling proper code review and change tracking for analytics assets.

1.2.2 Create and Manage a Power BI Desktop Project (.pbip)

A Power BI Project (.pbip) saves a report and its semantic model as a collection of individual plain-text files in a folder structure, rather than as a single binary .pbix. The format is designed for source-control workflows and team collaboration — files can be opened in any text editor, diffed in Git, and automated with scripts or programmatic tooling.

Key facts

  • Enabled via: File → Options and settings → Options → Preview features → Power BI Project (.pbip) save option.
  • Folder structure created on save:
    • ProjectName.Report/ — report pages and visuals in PBIR (JSON) format.
    • ProjectName.SemanticModel/ — model definition in TMDL (YAML-like) format.
    • .gitignore — auto-created; excludes local cache files (cache.abf, localSettings.json) from version control.
    • ProjectName.pbip — optional pointer file for opening the report in Desktop.
  • Multiple reports can share a single semantic model folder in the same project — each report has its own PBIR folder.
  • The .pbip pointer file is optional; reports can also be opened by opening the definition.pbir file directly.
  • TMDL — YAML-like syntax for the semantic model definition (tables, measures, relationships); supports batch edits.
  • PBIR — JSON format for report pages and visuals; enables programmatic report generation.
  • External tool edits (outside Desktop) require a Desktop restart to take effect — not picked up in real time.
  • Files must be saved in UTF-8 without BOM encoding; configure Git with autocrlf to avoid line-ending diffs.
  • Deploying .pbip to Fabric: via Fabric Git integration (recommended for teams), Fabric REST APIs, or Power BI Desktop Publish.
  • PBIX ↔ PBIP conversion is only possible in Power BI Desktop via File → Save As. No programmatic conversion tool exists.
  • Sensitivity labels are not supported with .pbip files — apply labels in the Fabric service after publishing.
  • Maximum Windows path length (260 characters) can be reached with long table or object names.
  • Cannot save .pbip directly to OneDrive or SharePoint cloud paths — use a locally synced OneDrive folder (with awareness of potential sync conflicts).
Gotcha: Sensitivity labels are not supported with the .pbip format. Labels must be applied in the Fabric service after publishing. Also, .pbip cannot be saved directly to a SharePoint or OneDrive cloud URL — use a locally synced folder.
Tip: Visual Studio Code is the recommended editor for .pbip files — it provides syntax highlighting, validation, IntelliSense for TMDL, and Git diff visualization out of the box.

Exam tips & gotchas

  • The .pbip format requires a Preview feature toggle in Desktop — it is not enabled by default.
  • Unlike .pbix, a .pbip project contains no data cache — only definitions. Data is loaded from sources when the file is opened.
  • External tool edits to TMDL or PBIR files are picked up only after restarting Power BI Desktop, not in real time.
  • Sensitivity labels cannot be applied to .pbip files — manage them post-publish in the Fabric service.
  • For team workflows, deploy to Fabric via Git integration or REST APIs rather than solely relying on the Desktop Publish button.

Practice

Answer: After restarting Power BI Desktop. Why: External edits to TMDL or PBIR files outside Desktop require a Desktop restart to be recognized. Changes made while Desktop is open are not picked up in real time.
Q2. A compliance team needs to apply sensitivity labels to a report being developed in .pbip format. Can labels be applied from within the .pbip project files?
Answer: No. Why: Sensitivity labels are not supported with the .pbip file format. Labels must be applied in the Fabric service after the project has been published. The .pbip files themselves contain no label metadata.
Q3. A team wants multiple report authors to independently develop reports that all use the same underlying semantic model — without duplicating the model definition. How does .pbip support this pattern?
Answer: Multiple report folders can share a single SemanticModel folder within the same .pbip project. Each report has its own PBIR folder pointing to the shared semantic model definition. This enables independent report development without duplicating the model.

1.2.3 Create and Configure Deployment Pipelines

Deployment pipelines provide a structured promotion workflow for Fabric items across environments — typically Development → Test → Production — where each stage is mapped to a distinct Fabric workspace. Content is copied from one stage to the next, and deployment rules allow you to configure stage-specific settings (such as different connection strings or parameter values) without modifying the source items themselves.

Key facts

  • A pipeline can have 2 to 10 stages; the default is 3 (Development, Test, Production). Stages can be added, removed, or renamed.
  • Each stage maps to a Fabric workspace. A Fabric capacity is required — shared capacity is not supported.
  • Deployment copies content (does not move it); paired items in the target stage are overwritten.
  • Item pairing occurs on the first deployment by name. Items added to the source workspace after the workspace was assigned to the pipeline are not auto-paired; they must be included in an explicit deployment to create the pairing.
  • Paired items remain linked even if renamed in one stage — differing names between stages are permitted.
  • Deployment can be triggered manually or via the Deployment Pipelines REST APIs for automation/CI-CD integration.
  • Workspace folder hierarchy is preserved during deployment; items can also be selected from a flat cross-folder list.
  • Deployment rules are configured on the target stage and take effect on the next deployment:
    • Data source rules: replace connection strings between stages (requires same data source type — e.g., SQL Server → SQL Server).
    • Parameter rules: override Power Query parameter values for the target stage.
    • Default lakehouse rules: specify the default Lakehouse for notebooks in the target stage.
  • Rules cannot be created in the Development stage — only Test and Production stages support rules.
  • You must be the item owner to create a deployment rule on that item.
  • If a data source or parameter referenced in a rule is changed or removed in the source stage, the rule becomes invalid and the next deployment fails.
  • Items that support data source rules: Dataflow Gen1, Semantic model, Paginated report, Mirrored database. Notebooks support only default lakehouse rules.
Gotcha: Deployment rules are defined on the target stage (e.g., Production), not the source. They do not take effect until the next deployment — comparing stages before deploying shows them in a "proposed" state.
Key: Rules cannot be created in the Development stage. If environment-specific configuration is needed in Development, use Power Query parameters in the source queries instead.

Exam tips & gotchas

  • A pipeline can have 2–10 stages, not just 3. The exam may test the valid range.
  • Deployment rules are set on the target stage and apply on the next deployment, not immediately.
  • Rules cannot be created in the Development stage; only Test and Production stages are eligible.
  • Data source rules require the same source type; cross-type swaps are not supported — use parameter rules instead for cross-type scenarios.
  • New items added to the source workspace after pipeline assignment are not auto-paired — they require an explicit first deployment to create the pairing.
  • A Fabric capacity is required; deployment pipelines are not available for shared-capacity workspaces.

Practice

Q1. A team wants the Production stage of a deployment pipeline to connect to a different Azure SQL Database than Development. Which deployment rule type enables this, and on which stage is the rule configured?
Answer: A data source rule, configured on the Production (target) stage. Why: Data source rules override connection strings when content is promoted from source to target. They are always defined on the target stage. The rule requires the same data source type in both stages (both must be Azure SQL Database).
Q2. After assigning a Development workspace to a pipeline and deploying once to Test, a developer adds a new notebook to Development. They then run another deployment to Test. The notebook does not appear in Test. Why?
Answer: The notebook was added after the workspace was assigned to the pipeline and has not been paired with the Test stage yet. Why: Items added to a workspace after pipeline assignment are not automatically paired. The developer must include the notebook in an explicit deployment to create the pairing for the first time.
Q3. A deployment pipeline has a data source rule on the Production stage. A developer changes the Development semantic model to use a completely different data source type. What happens at the next deployment?
Answer: The deployment fails. Why: If the data source referenced by a rule is removed or changed to a different type in the source stage, the rule becomes invalid. Fabric validates rules before deploying and will block the deployment with an error indicating the rule cannot be resolved.

1.2.4 Perform Impact Analysis of Downstream Dependencies

Lineage view and impact analysis together expose the full dependency graph of Fabric artifacts across workspaces. Lineage view shows data flowing from sources through lakehouses, dataflows, warehouses, and semantic models to downstream reports and dashboards. Impact analysis, available directly from a semantic model's lineage card, quantifies how many consumers would be affected by a model change and provides 30-day usage metrics to help prioritize communication and remediation effort.

Key facts

  • Access lineage view: workspace list view → View → Lineage.
  • Requires at least Contributor workspace role and a Power BI Pro license. Viewers cannot access lineage view.
  • Lineage view shows both upstream dependencies (data sources, dataflows, lakehouses, warehouses) and downstream dependencies (reports, dashboards) — including external items from other workspaces.
  • Item cards display: last refresh time, endorsement badges, data source type, and gateway information.
  • Selecting the double-arrows icon on a card highlights all related upstream/downstream artifacts and dims the rest.
  • Semantic model impact analysis (via the impact analysis button on the semantic model's card):
    • Shows counts of impacted workspaces, reports, and dashboards.
    • Displays 30-day usage metrics (excluding the current day): distinct viewer count and total view count per downstream artifact.
    • Helps prioritize which downstream consumers are most critical — a report with thousands of viewers warrants more urgent communication than one with a handful.
  • Notify contacts: sends an email under the analyst's name to the contact lists of all affected workspaces — useful for coordinating change communication.
  • Impact analysis can only be performed on semantic models in your own workspace; navigate to the source workspace to analyze external (cross-workspace) semantic models.
  • Items the user cannot access appear as "Limited access" in the impact panel — their existence is shown for count accuracy, but details are withheld.
  • From Power BI Desktop, republishing a changed semantic model displays a pre-publish warning with the count of impacted workspaces, reports, and dashboards.
  • Usage metrics are not supported for personal workspaces (My Workspace).
Tip: Before modifying a widely used semantic model, run impact analysis to identify high-traffic downstream reports. Use the "Notify contacts" feature to proactively inform workspace owners of all affected workspaces about the upcoming change.

Exam tips & gotchas

  • Lineage view requires at least Contributor role — Viewers are excluded.
  • Usage metrics cover the last 30 days, excluding the current day. Metrics older than 30 days are not shown.
  • To analyze an external (cross-workspace) semantic model, you must navigate to its source workspace — you cannot analyze models you do not own from another workspace.
  • Items the user lacks permission to view appear as "Limited access" — their count is included in totals but their details are hidden.
  • Usage metrics are unavailable for My Workspace (personal workspace) items.

Practice

Q1. A data engineer is about to make breaking changes to a shared semantic model. They want to know how many reports and dashboards would be affected and how heavily each is used. Which Fabric feature provides this information?
Answer: Semantic model impact analysis, accessed from the lineage view. Why: Impact analysis shows the count of downstream workspaces, reports, and dashboards, plus 30-day usage metrics (unique viewers and view counts per artifact). This enables the engineer to identify high-priority consumers and use "Notify contacts" for proactive communication.
Q2. A workspace Viewer wants to review the lineage of items in the workspace to understand data dependencies. Can they access lineage view?
Answer: No. Why: Lineage view requires at least a Contributor workspace role and a Power BI Pro license. The Viewer role is insufficient — Viewers are excluded from this feature.
Q3. In the impact analysis panel for a semantic model, several downstream items are listed as "Limited access." What does this indicate?
Answer: The user performing the analysis lacks permission to view those items. Why: Fabric includes these items in the impact count (to give an accurate total) but withholds their names, owners, and usage metrics because the requesting user does not have access to those artifacts.

1.2.5 Deploy and Manage Semantic Models by Using the XMLA Endpoint

The XMLA endpoint exposes the XML for Analysis (XMLA) protocol on Power BI semantic models hosted in Premium, Premium Per User (PPU), or Power BI Embedded workspaces. It enables external client tools — including Tabular Editor, SQL Server Management Studio (SSMS), DAX Studio, and ALM Toolkit — to connect using a standard Analysis Services–compatible connection string for querying, metadata scripting, programmatic refresh, and structural model management. All communication over the XMLA protocol is fully encrypted.

The endpoint is read-only by default. An administrator must explicitly enable read-write mode at the capacity level before tools can push model changes.

Key facts

  • Connection string format: powerbi://api.powerbi.com/v1.0/[tenant name]/[workspace name] — used as the server name in any Analysis Services–compatible client tool. Workspace names must be URI-encoded (spaces as %20).
  • Read-only is the default for all Premium/PPU workspaces. Allows: querying data, reading metadata, viewing model schema, profiling queries.
  • Read-write must be explicitly enabled by an administrator:
    • Premium capacity: Admin portal → Capacity settings → Power BI Premium → [capacity] → Workloads → XMLA Endpoint → select Read Write. Applies to all workspaces on that capacity.
    • PPU: Admin portal → Premium Per User → Semantic model workload settings → XMLA Endpoint → Read Write.
  • Read-write enables: deploying model changes via TMSL scripts, programmatic refresh/processing, advanced structural modifications, batch scripting.
  • Client tool requirements:
    • DAX Studio — read-only (DAX queries, performance profiling).
    • Tabular Editor 2.x — read-only for queries; read-write for metadata and model changes.
    • ALM Toolkit — read-only for comparison; read-write for deployment and model sync.
    • SSMS ≥ 18.9 — read-only for queries; read-write for TMSL scripting and object management.
    • SQL Server Profiler (with SSMS ≥ 18.9) — read-only (trace and event profiling).
    • Excel PivotTables — read-only.
    • Power BI Report Builder — read-only (paginated report design).
  • Client libraries used: MSOLAP, ADOMD.NET, AMO — the same libraries used for Azure AS and SSAS.
  • The Az.AnalysisServices PowerShell module is not supported for Power BI semantic models. Use the SqlServer PowerShell module (version ≥ 21.1.18256) with the powerbi:// connection string instead.
  • TMSL (Tabular Model Scripting Language): JSON-based language for model management operations (create, alter, refresh, process) executed via SSMS or XMLA scripts.
  • For models larger than ~1 GB, enabling the large models storage format significantly reduces write overhead during XMLA operations.
Gotcha: The XMLA endpoint is read-only by default. Tools like Tabular Editor and ALM Toolkit cannot push model changes until a capacity administrator explicitly enables Read Write in the Admin portal under Capacity settings.
Key: Never use the Az.AnalysisServices PowerShell module for Power BI semantic models — it is designed for Azure Analysis Services and is not compatible with Power BI. Use the SqlServer module (≥ 21.1.18256) instead.

Exam tips & gotchas

  • XMLA is read-only by default — read-write requires explicit admin enablement at the capacity level, not the workspace level.
  • The connection string uses the powerbi:// scheme, not a traditional server URL. Workspace names must be URI-encoded.
  • Az.AnalysisServices PowerShell module is not compatible with Power BI — use the SqlServer module.
  • DAX Studio and Excel PivotTables operate read-only. Tabular Editor, ALM Toolkit, and SSMS require read-write to push changes.
  • The XMLA endpoint is available on Premium capacity, PPU, and Power BI Embedded — not on workspaces using only shared/Pro capacity.

Practice

Q1. A capacity administrator wants to enable Tabular Editor to deploy model schema changes to a Premium capacity workspace. What configuration step is required?
Answer: In the Admin portal, navigate to Capacity settings → Power BI Premium → [capacity] → Workloads → XMLA Endpoint → select Read Write. Why: The XMLA endpoint is read-only by default. Enabling Read Write at the capacity level unlocks write operations for all tools connecting via XMLA to workspaces on that capacity, including Tabular Editor.
Q2. A developer connects to a Power BI Premium semantic model via the XMLA endpoint using DAX Studio. What operations are available?
Answer: Read-only operations — running DAX queries against the model, viewing metadata, and performance profiling. Why: DAX Studio uses read-only XMLA connectivity. It cannot modify the model structure or trigger programmatic refreshes. Those operations require a tool like Tabular Editor or SSMS with the endpoint set to Read Write.
Q3. A developer attempts to use the Az.AnalysisServices PowerShell module to refresh a Power BI Premium semantic model via the XMLA endpoint. What is the expected outcome?
Answer: Failure — the Az.AnalysisServices module is not supported for Power BI semantic models. Why: Az.AnalysisServices is designed for Azure Analysis Services and does not support the Power BI XMLA endpoint. The correct approach is to use the SqlServer PowerShell module (version ≥ 21.1.18256) with the powerbi:// connection string.

1.2.6 Create and Update Reusable Assets (.pbit, .pbids, Shared Semantic Models)

Power BI and Fabric provide three complementary mechanisms for reusing analytics assets across teams and projects. A .pbit (Power BI Template) packages a report's structure and data model — without any data — as a starting point for new reports. A .pbids (Power BI Data Source) file distributes connection specifications without credentials, directing report authors to the correct data endpoint. Shared semantic models publish a single governed model to the Fabric service so report authors across workspaces can build their own reports on top of it, supporting the managed self-service BI pattern.

Key facts — .pbit (Power BI Template)

  • Created via: File → Export → Power BI template in Power BI Desktop.
  • Contains: report pages and visuals, data model schema (tables, relationships, measures), all query definitions.
  • Does not contain data — no data cache is included. File size is smaller than a .pbix.
  • Opening a .pbit: Desktop prompts for parameter values (if defined), then credentials and data source location, then loads data. The result is a full .pbix with live data — the template is the starting point.
  • Useful for standardizing report layouts, measure naming, and data model structure across multiple report authors.

Key facts — .pbids (Power BI Data Source)

  • Contains: connection information only (protocol, server, database address). Contains no credentials and no table/schema definitions.
  • Opening a .pbids: Desktop prompts for credentials, then displays the Navigator to select tables/views.
  • Supports specifying "mode" as "DirectQuery" or "Import"; if absent, the user is prompted to choose.
  • Currently supports a single data source per file — specifying multiple sources causes an error.
  • Created by: exporting from an existing .pbix (File → Options → Data source settings → export as PBIDS), or manually authoring JSON in the DSR (Data Source Reference) format with a .pbids extension.
  • Useful for distributing standardized connection specs to report authors without sharing credentials.

Key facts — Shared Semantic Models

  • A semantic model published to the Fabric service can be shared with report authors across workspaces — the foundation of the managed self-service BI pattern.
  • Share via: OneLake catalog or data details page → Share → configure permission level.
  • Key permissions on a shared semantic model:
    • Read: view metadata and existing reports; cannot query underlying data or create new reports.
    • Build: create new reports and dashboards on top of the semantic model — the core self-service BI permission.
    • Write: modify the semantic model definition.
    • Reshare: grant access to other users.
  • Downstream reports built on shared semantic models are visible in lineage view, with the source workspace indicated on the semantic model card.
  • Impact analysis is especially valuable for widely shared semantic models, since changes can affect consumers across many workspaces simultaneously.
// Example .pbids file — Azure SQL Database connection
// Contains only the address; NO credentials, NO table selection
{
  "version": "0.1",
  "connections": [
    {
      "details": {
        "protocol": "tds",
        "address": {
          "server": "myserver.database.windows.net",
          "database": "SalesDB"
        }
      },
      "mode": "DirectQuery"
    }
  ]
}
Key distinction — all four formats:
  • .pbit — report structure + model schema; no data; prompts for credentials and source on open; distribution format.
  • .pbids — connection address only; no credentials, no schema; prompts for credentials and table selection on open; single data source per file.
  • .pbip — full project definition (model + report) as editable text files (TMDL + PBIR); designed for development and source control, not distribution.
  • Shared semantic model — live published model in the service; shared via Build permission; enables cross-workspace self-service BI; change impact is tracked via lineage.

Exam tips & gotchas

  • .pbit contains no data — only structure. Opening it always triggers data load from the live source.
  • .pbids contains no credentials and no schema — only the connection address. The user provides credentials and selects tables at open time.
  • .pbids supports only a single data source per file — attempting multiple sources causes an error.
  • Build permission on a shared semantic model is required for self-service report creation. Read alone is insufficient for authoring new reports.
  • The .pbip format is a development format (source control, team collaboration), not a distribution format like .pbit or .pbids.
  • Impact analysis is especially important for shared semantic models because one change can ripple across report consumers in many workspaces.

Practice

Q1. A BI team wants to distribute a standardized report template — with pre-built measures, a star schema, and consistent branding — to 10 regional report developers. Each developer will connect to their own regional database. Which file format is most appropriate?
Answer: .pbit (Power BI Template). Why: A .pbit packages the report structure, data model, and query definitions without any data. Each developer opens it, provides their own credentials and data source details, and immediately has a fully structured starting point. The template enforces consistent measures and layout without pre-loading any data.
Q2. A data architect wants to ensure all report developers in the company always connect to the correct Azure SQL Database server and database name, without distributing credentials. Which file format achieves this?
Answer: .pbids (Power BI Data Source file). Why: A .pbids file contains only the connection address (protocol, server, database) — no credentials and no table definitions. Authors open it, are prompted for credentials, and then select their own tables. The connection target is standardized without exposing any credentials.
Q3. A report author in Team B wants to build new Power BI reports on top of a semantic model owned by Team A in a different workspace. Team A shares the model but grants only Read permission. Can Team B's author create new reports?
Answer: No. Why: Creating new reports on a shared semantic model requires Build permission. Read permission only allows viewing metadata and existing reports — it does not enable report authoring. Team A must explicitly grant Build permission to enable self-service report creation on the shared model.
Q4. A developer opens a .pbids file and sees the Power BI Desktop Navigator dialog prompting them to select tables. What information did the .pbids file supply, and what did it intentionally omit?
Answer: The .pbids file supplied the connection address (protocol, server, and database name). It intentionally omitted credentials (the developer entered those when prompted) and table/schema definitions (the developer selects tables in the Navigator). The .pbids format is a connection specification only — no credentials, no schema, no data.

2.1.1 Create a data connection

Before any Fabric item (Dataflow Gen2, pipeline, or semantic model) can pull from an external source, a connection must exist that stores the endpoint URL and credentials. Fabric centralises connection management at Settings → Manage connections and gateways, where administrators create, edit, share, and delete both cloud and on-premises connections.

Cloud sources that are publicly reachable (Azure SQL, ADLS Gen2, SharePoint Online, REST APIs) use a cloud connection with no additional software. Private or on-premises sources require a gateway installed on a machine that can reach the source.

Key facts

  • On-premises data gateway (standard mode) — multi-user; supports scheduled refresh, DirectQuery, and live connection. Requires gateway software on a customer-managed machine.
  • On-premises data gateway (personal mode) — single user only; scheduled refresh only; no DirectQuery or live connection support.
  • Virtual network (VNet) data gateway — Microsoft-managed SaaS; multi-user; requires Premium capacity or PPU; no on-premises software to install.
  • Cloud connections are created in the Connections tab (New → Cloud) for publicly accessible sources such as Azure SQL, ADLS Gen2, and SharePoint Online.
  • Authentication options: Basic, OAuth2, Service Principal (Entra app registration). OAuth2 caveat: long-running queries may fail when the access token expires; cross-tenant Entra accounts are not supported.
  • A connection can be made shareable: other workspace members can use it in their items without seeing the stored credentials.
  • Gateway version ≥ 3000.214.2 is required to support Fabric pipelines against on-premises sources; older versions support only Dataflows and semantic-model refresh.
  • Fabric Data Factory supports 26+ on-premises connectors via gateway (SQL Server, Oracle, SAP HANA, PostgreSQL, MySQL, Salesforce, SharePoint, ODBC, and more).
// Create a shareable cloud connection (Fabric REST API)
// POST https://api.fabric.microsoft.com/v1/connections
{
  "displayName": "Azure SQL – SalesDB (shared)",
  "connectivityType": "ShareableCloud",
  "connectionDetails": {
    "type": "AzureSqlDatabase",
    "parameters": [
      { "name": "server",   "value": "myserver.database.windows.net" },
      { "name": "database", "value": "SalesDB" }
    ]
  },
  "credentialDetails": {
    "credentialType": "ServicePrincipal",
    "singleSignOn": false,
    "skipTestConnection": false
  }
}
Tip: Use standard-mode gateway (not personal) whenever DirectQuery or multi-user scheduled refresh is required. Personal mode is a single-user development convenience and must not be used for shared production reports.

Exam tips & gotchas

  • Personal-mode gateway = single user + scheduled refresh only; no DirectQuery or live connection.
  • VNet gateway = Microsoft-managed, no software install, but requires Premium P or Fabric F capacity — it is not free.
  • OAuth2 tokens expire: long-running ETL jobs may fail mid-run when the token lifetime is exceeded.
  • Shareable connections hide credentials — members can use the connection but cannot view the password or SPN secret.
  • Fabric pipeline activities against on-premises sources require gateway ≥ 3000.214.2; older gateways block pipeline execution even when Dataflows work fine.

Q2. A new analyst needs to use an existing Azure SQL connection in their Dataflow without seeing the stored password. What feature enables this?
Answer: Marking the connection as shareable. Why: A shareable connection lets workspace members reference the connection by its ID in their Dataflows and pipelines without exposing the underlying credentials (password or SPN secret).

Answer: The gateway version is below the required minimum (3000.214.2). Why: Older gateway versions support only Dataflows and semantic model scheduled refresh — not pipeline activities. Upgrading the gateway software resolves the issue.

2.1.2 Discover data via OneLake catalog & Real-Time hub

Microsoft Fabric provides two first-class discovery experiences. The OneLake data catalog (previously OneLake data hub) is a searchable, organization-wide inventory of every Fabric item — lakehouses, warehouses, KQL databases, semantic models, Eventstreams, and more — that you have access to. It surfaces endorsement badges, lineage, descriptions, and contact information so teams can locate authoritative data without direct knowledge of workspace names.

The Real-Time hub is a streaming-data marketplace: it lists all Eventstream sources, Azure Event Hubs namespaces, MQTT topics, and other event producers that are accessible to you, and it provides one-click connections from those streams into Fabric workloads.

Key facts

  • OneLake catalog — search or browse by item type, workspace, domain, or endorsement; "All data" tab shows everything you have read access to across the entire tenant.
  • Endorsements: Promoted (workspace members can endorse) and Certified (tenant-admin-designated certifiers only). Certified items rank higher in search and have a gold badge.
  • Clicking any catalog item opens a detail panel with description, contact, freshness, sensitivity label, and lineage; users can request access directly from the panel if they lack permission.
  • Domains (logical groupings of workspaces) appear as filter facets in the catalog, enabling business-unit-level browsing.
  • Real-Time hub — surfaces all event producers visible to you; can create and connect Eventstreams, alert rules (Activator), or KQL eventhouses from a stream in one step.
  • Real-Time hub "My streams" tab shows Eventstreams you own; "All streams" shows streams from the entire organisation you have access to; "Microsoft sources" shows connectable Azure services (Event Hubs, IoT Hub, Service Bus, Azure Blob Storage events).
  • From Real-Time hub you can create an alert (Fabric Activator) directly on a stream without writing code.

2.1.3 Ingest or access data

Fabric offers a layered set of ingestion options ranging from ad-hoc file upload to fully orchestrated pipelines. Choosing the right mechanism depends on data volume, latency needs, transformation complexity, and team skill set. A key architectural principle is that all Fabric storage items ultimately write data to OneLake in Delta-Parquet format, enabling read sharing across engines.

The medallion architecture (Bronze → Silver → Gold) is the recommended organisational pattern: raw data lands in the Bronze layer, cleaned and conformed data in Silver, and business-ready aggregates in Gold. Fabric tools map naturally to each layer.

Key facts

  • Upload — drag-and-drop small CSV/Parquet/JSON files directly into a Lakehouse Files section; not suitable for large or recurring loads.
  • Dataflows Gen2 — low-code Power Query M transforms; writes results to a destination (Lakehouse, Warehouse, Azure SQL, etc.); supports query folding for source-pushdown; scheduled up to 48× per day.
  • Pipelines (Data Factory) — orchestration engine; Copy activity for bulk ingest; supports 90+ connectors; parameterisable; error handling and retry; can call notebooks, Dataflows, stored procedures, and REST APIs.
  • Notebooks / Spark — PySpark or Scala for complex transforms; read Lakehouse files or Delta tables, transform with Spark DataFrame API, write back as Delta; best for large-scale or schema-on-read scenarios.
  • Shortcuts — virtual references to external storage (ADLS Gen2, S3, GCS, Dataverse, Google Drive, S3-compatible) without physically copying data; appear as tables or files in the Lakehouse and are read by any Fabric engine.
  • COPY INTO (Warehouse) — T-SQL bulk load command that reads Parquet or CSV from ADLS Gen2 or OneLake paths directly into Fabric Warehouse tables; no pipeline required for flat loads.
  • Eventstream — real-time ingestion from Event Hubs, IoT Hub, Kafka, CDC sources, or custom apps into Eventhouse KQL database, Lakehouse Delta tables, or Activator alerts with sub-second latency.
  • Pipelines support incremental copy via high-watermark patterns (store last-loaded timestamp in a parameter table, filter source with a WHERE clause in the Copy source query).
-- COPY INTO: bulk-load Parquet from ADLS Gen2 into Fabric Warehouse
COPY INTO dbo.SalesFact
FROM 'https://myaccount.dfs.core.windows.net/raw/sales/2024/*.parquet'
WITH (
    FILE_TYPE = 'PARQUET',
    CREDENTIAL = (IDENTITY = 'Managed Identity')
);

# PySpark: read CSV from Lakehouse Files, clean, write as Delta table
from pyspark.sql import functions as F

df = spark.read.option("header", True).csv(
    "Files/raw/sales/2024/*.csv"

Exam tips & gotchas

  • COPY INTO is a Fabric Warehouse–only command; it is not available at the Lakehouse SQL analytics endpoint.
  • Shortcuts are read-only views of external storage — you cannot INSERT or UPDATE data in the source via a shortcut.
  • Eventstream supports only append semantics; it cannot update existing Eventhouse rows.
  • Dataflows Gen2 can write to a Fabric Warehouse destination (DML upsert/overwrite handled by the runtime) — this is distinct from running COPY INTO yourself.
  • Pipeline Copy activity vs. COPY INTO: Copy activity works for any destination (Lakehouse, Warehouse, Azure SQL); COPY INTO is T-SQL only and targets the Warehouse.

2.1.4 Choose between data stores

Fabric surfaces four primary analytical data stores, each optimised for a different workload profile. Choosing incorrectly leads to limitations that cannot be worked around — for example, a Lakehouse SQL analytics endpoint silently blocks DML writes, and an Eventhouse cannot perform row updates. Understanding the architectural contracts of each store is a core exam competency.

Key facts

  • Lakehouse — stores files and Delta tables in OneLake. Accessed via Spark (full read/write) or the auto-generated SQL analytics endpoint (read-only SELECT; CREATE VIEW, FUNCTION, PROCEDURE allowed; no INSERT/UPDATE/DELETE/CREATE TABLE via SQL endpoint). Best for big-data engineering with Spark.
  • Fabric Data Warehouse — full T-SQL analytical warehouse; supports INSERT, UPDATE, DELETE, MERGE, CREATE TABLE, CTAS, SELECT INTO. Data stored as Delta-Parquet in OneLake. Best for BI teams comfortable with T-SQL who need DML at scale.
  • Eventhouse (KQL database) — append-optimised columnar store for time-series, logs, and telemetry. Primary query language is KQL. Supports very high ingest throughput and sub-second freshness. Rows cannot be updated or deleted in the standard write path (use update policies or soft delete, not SQL-style UPDATE).
  • SQL database in Fabric — OLTP/translytical; full SQL Server-compatible DML; automatically replicates to OneLake Delta for analytical queries; designed for row-level transactional workloads, not large analytical scans.
  • Summary rule: Need UPDATE/MERGE at analytical scale → Warehouse. Need Spark ETL → Lakehouse. Need sub-second time-series streaming ingest + KQL → Eventhouse. Need OLTP row transactions → SQL database.
-- Fabric Warehouse: full DML supported
MERGE INTO dbo.Customer AS target
USING dbo.CustomerStaging AS src
  ON target.CustomerKey = src.CustomerKey
WHEN MATCHED THEN
  UPDATE SET target.Email = src.Email,
             target.UpdatedAt = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
  INSERT (CustomerKey, Name, Email, CreatedAt)
  VALUES (src.CustomerKey, src.Name, src.Email, GETDATE());

-- Lakehouse SQL analytics endpoint: DML is NOT supported
-- The statement below will be REJECTED at the analytics endpoint:
-- UPDATE dbo.Customer SET Email = 'new@example.com' WHERE CustomerKey = 1;
Gotcha: The Lakehouse SQL analytics endpoint is read-only. Writing T-SQL INSERT/UPDATE/DELETE/CREATE TABLE statements against it returns an error. If your workflow requires DML via SQL, you must use a Fabric Warehouse.

Exam tips & gotchas

  • Lakehouse SQL endpoint: SELECT + DDL for views/functions/procedures = OK. DML (INSERT/UPDATE/DELETE) + CREATE TABLE = blocked.
  • Warehouse stores data in OneLake as Delta-Parquet — both Spark and other engines can read Warehouse tables via shortcuts or SQL analytics endpoint.
  • Eventhouse is the only store with native KQL support; SQL can be used for basic queries but KQL is the primary and more powerful interface.
  • SQL database in Fabric auto-replicates to OneLake, but is intended for small-to-medium OLTP loads — not petabyte analytical workloads.
  • Both Lakehouse and Warehouse expose a SQL analytics endpoint, but only the Warehouse endpoint allows full DML.

Practice

Q1. A data engineer runs the T-SQL statement INSERT INTO dbo.Orders SELECT * FROM dbo.OrdersStaging against the Lakehouse SQL analytics endpoint. What happens?
Answer: The statement fails with an error. Why: The Lakehouse SQL analytics endpoint is read-only; INSERT, UPDATE, DELETE, and CREATE TABLE are not supported. The engineer must use a Fabric Data Warehouse for T-SQL DML operations.
Q2. A team ingests millions of IoT sensor events per minute and needs KQL-based time-series analysis with sub-second freshness. Which Fabric data store is the best fit?
Answer: Eventhouse (KQL database). Why: Eventhouse is purpose-built for high-throughput append-only streaming ingest and KQL-based time-series analysis. SQL databases and Data Warehouses are not optimised for sub-second streaming.
Q3. A BI team wants to use T-SQL MERGE to implement SCD Type 2 updates on a dimension table. Which Fabric store should they use?
Answer: Fabric Data Warehouse. Why: MERGE is a DML operation that requires write access. The Fabric Data Warehouse supports full T-SQL DML including MERGE; the Lakehouse SQL analytics endpoint does not.

2.1.5 OneLake integration for Eventhouse and semantic models

OneLake is Fabric's unified storage layer — a single logical data lake for the entire organisation, backed by ADLS Gen2 and accessible via DFS endpoint. All Fabric compute engines read from and write to OneLake, giving analytical items a shared physical foundation. Two items that require special understanding are Eventhouse and semantic models.

Eventhouse OneLake availability lets KQL database data be mirrored to OneLake as Delta-Parquet, making it queryable by Spark, Warehouse, and other engines without an Eventstream pipeline. Direct Lake is the semantic model storage mode in which Power BI reads Parquet files directly from OneLake rather than importing data into memory or relying on DirectQuery — combining the freshness of DirectQuery with near-Import query performance.

Key facts

  • OneLake availability for Eventhouse — enable per-table in the KQL database properties; tables are materialised as Delta-Parquet in OneLake; latency is typically minutes after ingestion.
  • Once enabled, Eventhouse data appears as Delta tables under the Lakehouse Files path and can be queried from Spark notebooks or the Lakehouse SQL analytics endpoint with no data movement.
  • Direct Lake mode — semantic model reads Delta-Parquet files from OneLake (Lakehouse or Warehouse) directly into columnar memory; no import copy or DirectQuery per-query round-trips.
  • Direct Lake has a framing concept: the model frame is anchored to a specific Delta checkpoint (v-order); a refresh operation advances the frame to the latest Delta version, making new data visible.
  • Direct Lake falls back to DirectQuery automatically when the data volume exceeds capacity limits (parquet file row count or column cardinality thresholds).
  • OneLake paths follow the pattern onelake://<workspace>/<item>.Lakehouse/Tables/<tablename>; ADLS Gen2 DFS URL: https://onelake.dfs.fabric.microsoft.com/<workspace-id>/....
  • Fabric Mirroring (for Azure SQL, Snowflake, Cosmos DB) also replicates data to OneLake as Delta-Parquet, enabling the same cross-engine access pattern as Eventhouse OneLake availability.
Key: Direct Lake is a semantic model storage mode, not an ingestion tool. It does not move or transform data — it reads existing Delta files in OneLake and requires no scheduled import refresh for data freshness (only a frame-advance refresh).

Exam tips & gotchas

  • Direct Lake fallback to DirectQuery happens automatically when parquet capacity limits are hit — the user sees slower queries but no errors.
  • Enabling OneLake availability for Eventhouse does not make Eventhouse writable from Spark; it is a one-direction mirror (KQL → Delta).
  • A Direct Lake semantic model does not need an import refresh to see new Lakehouse data — but it does need a frame refresh to advance the Delta checkpoint pointer.
  • OneLake exposes an ADLS Gen2 DFS endpoint, so any tool compatible with ADLS Gen2 (ADF, Synapse, external Spark clusters, local storage explorer) can read Fabric data.

Practice

Q1. A data engineer enables OneLake availability for an Eventhouse KQL database table. What happens to that data?
Answer: The table is mirrored to OneLake as Delta-Parquet and becomes queryable by Spark, the Lakehouse SQL analytics endpoint, and other Fabric engines. Why: OneLake availability exposes KQL table data as Delta files in OneLake without copying it to a separate Lakehouse item.
Q2. A Power BI semantic model in Direct Lake mode begins returning slower query responses than normal. No model changes were made. What is the most likely cause?
Answer: The model has fallen back to DirectQuery because the data volume exceeded the capacity parquet limits. Why: Direct Lake automatically falls back to DirectQuery when column cardinality or row thresholds are breached, resulting in slower per-query SQL round-trips to the underlying store.
Q3. Which statement correctly describes the relationship between a Direct Lake semantic model and its underlying Lakehouse Delta table?
Answer: The semantic model reads Delta-Parquet files directly from OneLake without importing data into memory or issuing per-query SQL. New data written to the Delta table is only visible to the model after a frame refresh that advances the model's Delta checkpoint pointer.

2.2.1 Create views, functions, and stored procedures (T-SQL)

Fabric Warehouse supports the full set of T-SQL persistent database objects: views (named SELECT statements), user-defined functions (scalar and table-valued), and stored procedures (parameterised T-SQL programs). These objects encapsulate business logic, simplify report queries, and reduce duplicated code across notebooks and Dataflows.

The Lakehouse SQL analytics endpoint also allows CREATE VIEW, CREATE FUNCTION, and CREATE PROCEDURE — but it does not allow DML. Views created at the analytics endpoint are read-only projections over the underlying Delta tables.

Key facts

  • Views — virtualise a SELECT; no data stored; reflect current table state at query time; support SELECT, JOIN, WHERE, GROUP BY, subqueries, CTEs.
  • Inline table-valued functions (iTVF) — parameterised views; return a table; can be used in FROM clause; support predicate push-down (query folding from Dataflows).
  • Scalar functions — return a single value; useful in computed columns or expressions; may reduce query performance if applied row-by-row.
  • Stored procedures — T-SQL programs; support parameters (IN), conditional logic (IF/ELSE), loops, transactions, and error handling (TRY/CATCH); return result sets or output parameters.
  • Objects created at the Lakehouse SQL analytics endpoint are not visible from Spark — they live in the analytics-endpoint SQL catalog only.
  • Fabric Warehouse does not currently support EXECUTE AS OWNER or cross-database queries within the same warehouse SQL endpoint.
  • Schemas (CREATE SCHEMA) are supported and recommended to organise objects: dbo for core tables, rpt for report views, etl for staging objects.
-- Create a schema for report objects
CREATE SCHEMA rpt;
GO

-- Create a view that joins fact and dimension
CREATE OR ALTER VIEW rpt.vSalesByProduct AS
SELECT
    p.ProductName,
    p.CategoryName,
    SUM(f.SalesAmount)  AS TotalSales,
    COUNT(f.OrderID)    AS OrderCount
FROM dbo.SalesFact      AS f
JOIN dbo.DimProduct     AS p ON f.ProductKey = p.ProductKey
WHERE f.SalesAmount > 0
GROUP BY p.ProductName, p.CategoryName;
GO

-- Inline table-valued function (parameterised view)
CREATE OR ALTER FUNCTION dbo.fnSalesByRegion
    (@Region NVARCHAR(50))
RETURNS TABLE
AS
RETURN
(
    SELECT f.OrderID, f.SalesAmount, d.RegionName
    FROM   dbo.SalesFact   f
    JOIN   dbo.DimGeography d ON f.GeographyKey = d.GeographyKey
    WHERE  d.RegionName = @Region
);
GO

-- Simple stored procedure with error handling
CREATE OR ALTER PROCEDURE dbo.uspRefreshSalesSummary
    @LoadDate DATE
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRY
        DELETE FROM dbo.SalesSummary WHERE SaleDate = @LoadDate;
        INSERT INTO dbo.SalesSummary (SaleDate, TotalSales)
        SELECT SaleDate, SUM(SalesAmount)
        FROM   dbo.SalesFact
        WHERE  SaleDate = @LoadDate
        GROUP BY SaleDate;
    END TRY
    BEGIN CATCH
        THROW;
    END CATCH;
END;
GO

Exam tips & gotchas

  • Views and functions can be created at the Lakehouse SQL analytics endpoint, but DML inside them (INSERT/UPDATE) is blocked.
  • CREATE OR ALTER is the preferred idempotent syntax in Fabric Warehouse — avoids DROP/CREATE pairs that break dependent objects.
  • Inline TVFs support query folding from Dataflows Gen2 (the WHERE predicate on the TVF is pushed to the source); scalar functions do not fold.
  • Stored procedures are not supported as a Direct Lake data source — they must be called from a pipeline activity or a Dataflow's "invoke stored procedure" step.

Practice

Q1. A Dataflow Gen2 is configured to read from an iTVF in a Fabric Warehouse and filter on a date parameter. A data engineer wants to ensure the filter is applied at the Warehouse rather than in the Dataflow engine. What must be true for query folding to occur?
Answer: The Dataflow must connect to the Warehouse SQL endpoint via a foldable connector (e.g., SQL Server connector), and the filter step must be expressible as a WHERE clause compatible with the source SQL dialect. Inline TVFs support predicate push-down, so folding will occur as long as the connector supports it.
Q2. A data engineer creates a view at the Lakehouse SQL analytics endpoint that performs a JOIN between two Delta tables. Another engineer tries to query the view from a PySpark notebook. What is the result?
Answer: The query fails — or returns an error that the view is not found. Why: Views created at the SQL analytics endpoint exist only in the analytics-endpoint SQL catalog; they are not registered as Delta tables in the Lakehouse metastore and are not visible to Spark.
Q3. Why should a Fabric Warehouse developer prefer CREATE OR ALTER VIEW over DROP VIEW IF EXISTS; CREATE VIEW?
Answer: CREATE OR ALTER is atomic and preserves permissions on the existing object. DROP followed by CREATE breaks the permission chain — any GRANT on the old view object must be re-applied after re-creation.

2.2.2 Enrich data (add columns and tables)

Data enrichment adds computed columns, derived attributes, or reference data that are not in the source but increase analytical value. In Fabric, enrichment happens in three common layers: T-SQL computed columns or ALTER TABLE ADD COLUMN in the Warehouse, PySpark withColumn transforms in notebooks, and Power Query M column-addition steps in Dataflows Gen2.

Key facts

  • T-SQL — ALTER TABLE … ADD column AS expression persisted computed columns; or add a regular column and populate with UPDATE. Computed columns referencing other columns are evaluated at query time.
  • PySpark withColumn — adds a new column from an expression; non-destructive (existing columns are kept); chained calls are lazy-evaluated and optimised by Spark.
  • Power Query M — Table.AddColumn(table, "NewCol", each [OldCol] * 1.1) adds a computed column; evaluated in-engine unless the connector folds it back to the source.
  • Adding a reference/lookup table — import a small CSV or call an API in a pipeline → load to Lakehouse → join in Spark or Warehouse. Common pattern: postal code → territory mapping table enriching orders.
  • Enrichment columns should be added in the Silver or Gold layer, not Bronze — Bronze is the raw landing zone and should mirror the source structure.
# PySpark: enrich with computed columns
from pyspark.sql import functions as F

df_enriched = (df_orders
    .withColumn("ProfitMargin",
        F.round((F.col("Revenue") - F.col("Cost")) / F.col("Revenue"), 4))
    .withColumn("OrderYear",  F.year("OrderDate"))
    .withColumn("OrderMonth", F.month("OrderDate"))
    .withColumn("IsHighValue", (F.col("Revenue") > 10000).cast("boolean"))
)
df_enriched.write.format("delta").mode("overwrite").option("overwriteSchema","true") \
    .saveAsTable("gold.OrdersEnriched")

-- T-SQL: enrich via computed column (Fabric Warehouse)
ALTER TABLE dbo.SalesFact
    ADD GrossProfit AS (SalesAmount - CostAmount);

-- T-SQL: add regular column, then populate from lookup join
ALTER TABLE dbo.SalesFact ADD TerritoryName NVARCHAR(100);
UPDATE f
SET    f.TerritoryName = t.TerritoryName
FROM   dbo.SalesFact   f
JOIN   dbo.DimTerritory t ON f.TerritoryKey = t.TerritoryKey;

Exam tips & gotchas

  • Computed columns in T-SQL are evaluated at query time and cannot be directly indexed (non-persisted). Use persisted computed columns (AS expression PERSISTED) to make them indexable.
  • PySpark withColumn is lazy — it does not execute until an action (write, show, count) is triggered.
  • Power Query Table.AddColumn may fold to the source SQL if the expression is simple (arithmetic, string concat); complex expressions (regex, custom M functions) prevent folding.

Practice

Q1. A data engineer adds a column ProfitMargin to a Lakehouse Delta table using PySpark withColumn. Nothing is written to storage. What must the engineer do to persist the result?
Answer: Call a write action such as df.write.format("delta").mode("overwrite").saveAsTable(...). Why: PySpark transformations are lazy; withColumn only defines the transformation plan. No data is written until an action triggers execution.
Q2. A Dataflow Gen2 uses Table.AddColumn to compute the ratio of two columns already present in a SQL Server source table. Will this step fold?
Answer: Yes, if the SQL Server connector supports folding for arithmetic expressions — which it does. The Dataflow engine translates the column addition to a SQL SELECT col1 / col2 AS Ratio expression pushed to SQL Server, avoiding loading all rows into the Dataflow engine.

2.2.3 Implement a star schema

A star schema organises analytical data into one or more central fact tables surrounded by dimension tables, connected by foreign-key relationships. It is the standard design for the Gold layer of a Fabric lakehouse or warehouse because it simplifies DAX, reduces join complexity for report tools, and leverages VertiPaq compression efficiently in semantic models.

The schema name derives from its visual appearance: the fact table at the center with dimension tables radiating outward like star points. Choosing the right fact type and dimension patterns is a primary exam topic.

Key facts

  • Grain — the level of detail of one row in the fact table (e.g., one row per product per store per day). All measures in the fact must be consistent with the declared grain. Adding columns at a finer grain than declared corrupts the grain.
  • Fact types:
    • Transaction fact — one row per business event (sale, click, call); sparse; most common.
    • Periodic snapshot fact — one row per period per entity (e.g., daily account balance); dense; period inserted even when no transaction occurred.
    • Accumulating snapshot fact — one row per process instance that is updated as the process advances through lifecycle milestones (e.g., order → pick → ship → deliver); contains multiple date-FK columns for each milestone.
  • Surrogate key — integer key generated by the warehouse (IDENTITY or sequence), not from the source system; insulates the warehouse from source key changes and enables SCD history rows.
  • Conformed dimension — shared across multiple fact tables (e.g., DimDate used by both SalesFact and InventoryFact); enables cross-subject-area drill and consistent filtering.
  • Role-playing dimension — a single physical dimension table used multiple times in the same fact table for different date roles (e.g., DimDate used as OrderDate, ShipDate, DueDate); implemented in the semantic model as inactive relationships activated with USERELATIONSHIP().
  • SCD Type 1 — overwrite old value; no history kept; simpler; use when historical attribute is irrelevant (e.g., correcting a typo in a name).
  • SCD Type 2 — add new row with new surrogate key; old row marked inactive (IsCurrent = 0, ExpiredDate = <date>); preserves history; requires surrogate key design; most common for name/address changes.
-- Create DimProduct with surrogate key and SCD columns
CREATE TABLE dbo.DimProduct (
    ProductKey      INT IDENTITY(1,1) PRIMARY KEY,  -- surrogate key
    ProductID       NVARCHAR(20)  NOT NULL,           -- natural/business key
    ProductName     NVARCHAR(200) NOT NULL,
    CategoryName    NVARCHAR(100),
    UnitCost        DECIMAL(10,2),
    -- SCD Type 2 columns
    EffectiveDate   DATE NOT NULL,
    ExpiryDate      DATE NULL,       -- NULL = current row
    IsCurrent       BIT  NOT NULL DEFAULT 1
);

-- SCD Type 2 MERGE: expire old row, insert new row
MERGE INTO dbo.DimProduct AS tgt
USING (
    SELECT ProductID, ProductName, CategoryName, UnitCost
    FROM   stg.ProductStaging
) AS src ON tgt.ProductID = src.ProductID AND tgt.IsCurrent = 1
WHEN MATCHED AND (
        tgt.ProductName  <> src.ProductName   OR
        tgt.CategoryName <> src.CategoryName  OR
        tgt.UnitCost     <> src.UnitCost
) THEN
    UPDATE SET tgt.IsCurrent   = 0,
               tgt.ExpiryDate  = CAST(GETDATE() AS DATE)
WHEN NOT MATCHED BY TARGET THEN
    INSERT (ProductID, ProductName, CategoryName, UnitCost,
            EffectiveDate, ExpiryDate, IsCurrent)
    VALUES (src.ProductID, src.ProductName, src.CategoryName,
            src.UnitCost, CAST(GETDATE() AS DATE), NULL, 1);

-- After expiring old rows, INSERT new versions (for changed rows)
INSERT INTO dbo.DimProduct
    (ProductID, ProductName, CategoryName, UnitCost,
     EffectiveDate, ExpiryDate, IsCurrent)
SELECT src.ProductID, src.ProductName, src.CategoryName, src.UnitCost,
       CAST(GETDATE() AS DATE), NULL, 1
FROM   stg.ProductStaging src
JOIN   dbo.DimProduct      tgt
    ON  tgt.ProductID = src.ProductID AND tgt.IsCurrent = 0
    AND tgt.ExpiryDate = CAST(GETDATE() AS DATE); -- just expired today
Key: Accumulating snapshot facts contain multiple foreign keys to the same dimension (e.g., OrderDateKey, ShipDateKey, DeliveryDateKey — all pointing to DimDate). When implemented in a semantic model, each becomes a separate relationship, with only one active at a time.

Exam tips & gotchas

  • SCD Type 1 = overwrite (simpler, no history). SCD Type 2 = insert new row (history preserved, surrogate key required).
  • Grain must be declared before choosing measures — a "unit price" column at transaction grain is valid; including both transaction-level and period-snapshot measures in the same fact table violates grain consistency.
  • Surrogate key must be generated by the warehouse (IDENTITY), not sourced from the source system — source natural keys can change or be reused.
  • Role-playing dimensions: only one relationship between fact and dimension can be active in a semantic model at a time; the others must use USERELATIONSHIP() in DAX measures.
  • Periodic snapshot facts should include rows for every period even when no activity occurred — otherwise NULL gaps in time-intelligence calculations.

Practice

Q1. A fact table contains one row per product per store per day recording closing inventory level, including days when no sales occurred. What type of fact table is this?
Answer: Periodic snapshot. Why: A periodic snapshot captures the state at the end of each regular period (day, week, month) for every entity — rows exist for every period regardless of whether a transaction occurred. A transaction fact would only have rows when events happen.
Q2. A dimension table for Customer is used in both SalesFact and ServiceRequestFact. When the filter on Customer is applied in a report, it automatically filters both facts. What dimension pattern does this describe?
Answer: Conformed dimension. Why: A conformed dimension has the same keys and attributes across multiple fact tables, enabling consistent cross-subject filtering and drill-through in a single semantic model.
Q3. A product changes its name but keeps the same ProductID. The team needs to analyse historical sales under the old name and future sales under the new name. Which SCD type should be implemented?
Answer: SCD Type 2. Why: Type 2 preserves both old and new attribute values as separate rows with different surrogate keys and effective/expiry dates. Historical sales facts pointing to the old surrogate key will correctly resolve to the old name; new sales get the new row's surrogate key.
Q4. An order tracking fact table records OrderDateKey, PickDateKey, ShipDateKey, and DeliveryDateKey — all foreign keys to DimDate. What fact type is this, and what challenge does it create in a semantic model?
Answer: Accumulating snapshot fact. Challenge: A semantic model can only have one active relationship between the fact and DimDate at a time. The other three date relationships must be marked inactive and activated with USERELATIONSHIP() inside DAX measures.

2.2.4 Denormalize data

Denormalization intentionally relaxes third-normal-form (3NF) rules by merging related tables or pre-computing derived attributes into a single wider table. The goal is to reduce JOIN complexity at query time, which improves report performance at the cost of storage redundancy. In a Fabric Lakehouse or Warehouse Gold layer, denormalized tables are common inputs for Direct Lake semantic models or flat Power BI reports.

Key facts

  • Wide table / flattened fact — combines columns from fact and all related dimensions into one table; eliminates JOIN at query time; increases row width and storage, reduces query latency.
  • Pre-aggregated summary table — stores GROUP BY results; fastest query; must be refreshed on a schedule; risk of stale data.
  • Trade-offs: denormalization reduces query complexity but increases data duplication; update anomalies must be managed by rebuilding or MERGE-updating the denormalised table.
  • In Fabric, CTAS (CREATE TABLE AS SELECT) and SELECT INTO are the T-SQL patterns for creating a denormalised snapshot table from a multi-table JOIN.
  • PySpark equivalent: df_fact.join(df_dim, key).write.format("delta").saveAsTable("gold.WideSales").
-- Fabric Warehouse: create a denormalised wide table (CTAS pattern)
CREATE TABLE gold.WideSalesFact
AS
SELECT
    f.OrderID,
    f.OrderDate,
    f.SalesAmount,
    f.Quantity,
    p.ProductName,
    p.CategoryName,
    c.CustomerName,
    c.CountryRegion,
    d.CalendarYear,
    d.MonthName
FROM dbo.SalesFact      f
JOIN dbo.DimProduct     p ON f.ProductKey   = p.ProductKey
JOIN dbo.DimCustomer    c ON f.CustomerKey  = c.CustomerKey
JOIN dbo.DimDate        d ON f.DateKey      = d.DateKey;

Exam tips & gotchas

  • Denormalization trades storage space for query speed — always valid in analytical (OLAP) scenarios; never appropriate for OLTP writes (update anomalies).
  • A denormalised table must be rebuilt or merged whenever source dimension attributes change (e.g., product name changes); it does not stay in sync automatically.
  • CTAS in Fabric Warehouse (CREATE TABLE … AS SELECT) creates and populates a table in one statement — equivalent to INSERT … SELECT but also creates the table.

Practice

Q1. A reporting team complains that their Power BI report is slow because it must JOIN five dimension tables at runtime. A data engineer creates a denormalised wide table that pre-joins all dimensions. What is the primary trade-off?
Answer: Increased storage and risk of stale data. The wide table stores attribute values redundantly for every row (e.g., ProductName repeated for every sale). If a product name changes, the wide table must be rebuilt or updated; it does not reflect source changes automatically.
Q2. Which T-SQL statement creates a new denormalised Fabric Warehouse table and populates it in a single operation?
Answer: CTAS — CREATE TABLE … AS SELECT. This is the Fabric Warehouse equivalent of a "create-and-fill" operation; it combines table creation with a SELECT query into one atomic step.

2.2.5 Aggregate data

Aggregation summarises detailed data to a coarser grain using functions such as SUM, COUNT, AVG, MIN, MAX. In Fabric, aggregations are performed in T-SQL (GROUP BY with aggregate functions), PySpark (groupBy + agg), Power Query M (Table.Group), and KQL (summarize). Window functions extend aggregation by computing values within a partition without collapsing rows.

Key facts

  • T-SQL GROUP BY — every non-aggregate SELECT column must appear in GROUP BY; filter aggregates with HAVING (not WHERE); use ROLLUP for hierarchical subtotals, CUBE for all combinations.
  • Window functions — SUM() OVER (PARTITION BY … ORDER BY … ROWS …); returns one row per input row (no collapsing); ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, FIRST_VALUE, LAST_VALUE; highly optimised in Fabric Warehouse.
  • PySpark groupBy.agg — df.groupBy("col").agg(F.sum("amount"), F.count("*")); equivalent to GROUP BY; result is a new DataFrame.
  • Power Query Table.Group — Table.Group(table, {"Key"}, {{"Total", each List.Sum([Amount]), type number}}); folds to source GROUP BY when the connector supports it.
  • Aggregation tables in Power BI — pre-aggregated summary tables that Power BI automatically uses for high-level queries, falling back to detail tables for fine-grain queries.
-- T-SQL: GROUP BY with HAVING and ROLLUP
SELECT
    p.CategoryName,
    p.ProductName,
    SUM(f.SalesAmount) AS TotalSales,
    COUNT(DISTINCT f.OrderID) AS Orders
FROM dbo.SalesFact  f
JOIN dbo.DimProduct p ON f.ProductKey = p.ProductKey
GROUP BY ROLLUP(p.CategoryName, p.ProductName)
HAVING SUM(f.SalesAmount) > 1000
ORDER BY p.CategoryName, p.ProductName;

-- Window function: running total and rank within category
SELECT
    ProductName,
    CategoryName,
    SalesAmount,
    SUM(SalesAmount) OVER (PARTITION BY CategoryName
                           ORDER BY SaleDate
                           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
                           AS RunningTotal,
    RANK() OVER (PARTITION BY CategoryName ORDER BY SalesAmount DESC) AS SalesRank
FROM dbo.SalesFact
JOIN dbo.DimProduct ON SalesFact.ProductKey = DimProduct.ProductKey;

# PySpark: groupBy aggregation and window functions
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# Basic aggregation
df_summary = (df_sales
    .groupBy("CategoryName", "ProductName")
    .agg(
        F.sum("SalesAmount").alias("TotalSales"),
        F.countDistinct("OrderID").alias("Orders"),
        F.avg("SalesAmount").alias("AvgSale")
    )
    .filter(F.col("TotalSales") > 1000)
)

# Window function: running total per category
window_spec = Window.partitionBy("CategoryName").orderBy("SaleDate") \
    .rowsBetween(Window.unboundedPreceding, Window.currentRow)
df_running = df_sales.withColumn(
    "RunningTotal", F.sum("SalesAmount").over(window_spec)
)

// Power Query M: Table.Group aggregation
let
    Source = SalesFact,
    Grouped = Table.Group(
        Source,
        {"CategoryName"},
        {
          {"TotalSales",  each List.Sum([SalesAmount]),    type number},
          {"OrderCount",  each Table.RowCount(_),          type number},
          {"AvgSale",     each List.Average([SalesAmount]),type number}
        }
    )
in
    Grouped

Exam tips & gotchas

  • HAVING filters on aggregated values; WHERE filters on row values before aggregation — they are not interchangeable.
  • Window functions do not reduce row count — the original rows are preserved with the computed window value added. Use GROUP BY to collapse rows.
  • PySpark groupBy.agg is lazy; use .write or .show to trigger execution.
  • Table.Group in Power Query M will fold to source GROUP BY SQL if the source connector supports it; complex aggregation functions prevent folding.

Practice

Q1. A T-SQL query uses GROUP BY CategoryName and includes a column ProductName in the SELECT list that is not in GROUP BY. What is the result?
Answer: A syntax error — "Column 'ProductName' is invalid in the select list because it is not contained in either an aggregate function or the GROUP BY clause." Every non-aggregated SELECT column must appear in GROUP BY.
Q2. A report needs to show each sale row alongside a running total of sales for the same product category, without removing any detail rows. Which T-SQL feature is most appropriate?
Answer: A window function — SUM(SalesAmount) OVER (PARTITION BY CategoryName ORDER BY SaleDate ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW). Window functions compute per-row aggregates without collapsing rows.

2.2.6 Merge and join data

Join operations combine rows from two or more tables based on a matching key. In Fabric, joins are performed via T-SQL (INNER/LEFT/RIGHT/FULL OUTER JOIN, CROSS JOIN), PySpark DataFrame join API, and Power Query M (Table.Join / merge queries). The T-SQL MERGE statement extends this to a combined INSERT/UPDATE/DELETE upsert in a single pass, which is critical for dimension loading and incremental ETL.

Key facts

  • INNER JOIN — returns only rows where the key exists in both tables; rows with no match in either table are excluded.
  • LEFT OUTER JOIN — all rows from the left table; NULL for right-table columns when no match; used for "show all customers including those with no orders".
  • FULL OUTER JOIN — all rows from both tables; NULLs on either side when no match; useful for reconciliation.
  • CROSS JOIN — Cartesian product (every row of left × every row of right); no ON clause; use deliberately for calendar/product grid generation.
  • MERGE (upsert) — matches source to target; WHEN MATCHED → UPDATE; WHEN NOT MATCHED BY TARGET → INSERT; WHEN NOT MATCHED BY SOURCE → DELETE (soft-deletes pattern). Single-pass over target table.
  • PySpark join types: inner, left, right, full, left_semi (like INNER but returns only left columns), left_anti (rows in left with no match on right).
  • Power Query merge queries — visual LEFT/INNER/RIGHT/FULL/LEFT ANTI/RIGHT ANTI join; folds to SQL JOIN if source supports it.
-- T-SQL: LEFT JOIN with NULL handling
SELECT
    c.CustomerName,
    c.Email,
    COALESCE(SUM(f.SalesAmount), 0) AS LifetimeSales
FROM dbo.DimCustomer c
LEFT JOIN dbo.SalesFact f ON c.CustomerKey = f.CustomerKey
GROUP BY c.CustomerName, c.Email
ORDER BY LifetimeSales DESC;

-- T-SQL: MERGE upsert pattern (incremental load)
MERGE INTO dbo.DimCustomer AS tgt
USING stg.CustomerStaging  AS src
    ON tgt.CustomerID = src.CustomerID
WHEN MATCHED AND (
    tgt.Email <> src.Email OR tgt.Phone <> src.Phone
) THEN
    UPDATE SET tgt.Email       = src.Email,
               tgt.Phone       = src.Phone,
               tgt.UpdatedDate = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
    INSERT (CustomerID, CustomerName, Email, Phone, CreatedDate)
    VALUES (src.CustomerID, src.CustomerName, src.Email,
            src.Phone, GETDATE());

# PySpark: join types and left-anti for orphan detection
df_sales    = spark.table("silver.SalesFact")
df_customer = spark.table("silver.DimCustomer")

# INNER join: only matched rows
df_inner = df_sales.join(df_customer, "CustomerKey", "inner")

# LEFT join: all sales, NULL customer attributes if unmatched
df_left = df_sales.join(df_customer, "CustomerKey", "left")

# LEFT ANTI join: sales with no matching customer (data quality check)
df_orphans = df_sales.join(df_customer, "CustomerKey", "left_anti")
df_orphans.write.format("delta").mode("overwrite") \
    .saveAsTable("dq.OrphanSales")

Exam tips & gotchas

  • MERGE requires the source to produce at most one matching row per target row — if the source has duplicates on the join key, MERGE throws an error. Deduplicate staging before MERGE.
  • PySpark left_semi — returns left rows where a match exists in right, but only left-table columns; equivalent to SQL EXISTS subquery.
  • PySpark left_anti — returns left rows with no match in right; efficient for finding orphaned fact rows (missing dimension key).
  • Power Query merge creates an "expanded" column — you must explicitly choose which right-table columns to expand; unexpanded columns remain as nested Table objects.
  • CROSS JOIN performance: every row combination is produced, so 1,000 left rows × 365 date rows = 365,000 output rows. Always verify intent before using.

Practice

Q1. A data quality job needs to find all rows in SalesFact that have a CustomerKey value not present in DimCustomer. Which PySpark join type returns exactly these rows?
Answer: left_anti join. Why: df_sales.join(df_customer, "CustomerKey", "left_anti") returns only the sales rows for which there is no matching CustomerKey in the dimension — exactly the orphaned fact rows.
Q2. A MERGE statement fails with "The MERGE statement attempted to UPDATE or DELETE the same row more than once." What is the cause and the fix?
Answer: The source table contains duplicate rows on the join key. MERGE requires each target row to match at most one source row. Fix: deduplicate the source (e.g., SELECT DISTINCT or ROW_NUMBER() OVER (PARTITION BY key) = 1) before the MERGE.
Q3. A T-SQL query must return all customers and all orders, including customers with no orders and orders with no matching customer. Which JOIN type is required?
Answer: FULL OUTER JOIN. Why: LEFT JOIN would exclude orders with no customer; RIGHT JOIN would exclude customers with no orders. FULL OUTER JOIN returns all rows from both tables, placing NULLs on whichever side has no match.

2.2.7 Handle duplicates, missing values, and nulls

Real-world source data frequently contains duplicate rows, missing values (empty strings, sentinel values such as -1 or "N/A"), and SQL NULLs. Addressing these in the Silver transformation layer is a fundamental data engineering practice. Fabric supports dedicated operations across T-SQL (ISNULL, COALESCE, NOT EXISTS deduplication), PySpark (dropDuplicates, na.drop, na.fill), and Power Query M (Table.Distinct, Table.FillDown, Table.ReplaceValue).

Key facts

  • NULL semantics: Any arithmetic with NULL produces NULL. Any comparison with NULL using = or <> produces UNKNOWN (not TRUE/FALSE) — use IS NULL / IS NOT NULL. COUNT(*) counts all rows; COUNT(column) skips NULLs.
  • ISNULL(expr, replacement) — SQL Server/Fabric syntax; returns replacement if expr is NULL; limited to two arguments.
  • COALESCE(expr1, expr2, …) — ANSI SQL; returns the first non-NULL value from the list; preferred when checking multiple fallback values.
  • ROW_NUMBER() deduplication — assign a row number within a group (PARTITION BY natural key ORDER BY freshness DESC); keep row_num = 1; most flexible dedup pattern in T-SQL.
  • PySpark dropDuplicates([cols]) — removes duplicate rows; optional column list restricts dedup key; whole-row or subset dedup.
  • PySpark na.fill(value, subset=[cols]) — fills NULL with a constant or dict of column→value; na.drop(subset=[cols], thresh=n) removes rows with fewer than n non-null values in the subset.
  • Power Query Table.Distinct([cols]) — removes rows where specified columns are identical; without argument, deduplicates on all columns.
  • Table.FillDown — propagates last non-null value downward in a column; useful for sparse hierarchical data exports.
-- T-SQL: COALESCE and ISNULL for null handling
SELECT
    OrderID,
    COALESCE(ShipCity, BillingCity, 'Unknown') AS City,
    ISNULL(Discount, 0)                         AS Discount,
    ISNULL(Notes, '')                            AS Notes
FROM dbo.Orders;

-- T-SQL: ROW_NUMBER deduplication (keep most recent per OrderID)
WITH Dedup AS (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY OrderID
               ORDER BY LastModifiedDate DESC
           ) AS RowNum
    FROM dbo.OrdersStaging
)
DELETE FROM Dedup WHERE RowNum > 1;

-- T-SQL: filter out rows where critical column is NULL
SELECT * FROM dbo.SalesFact
WHERE CustomerKey IS NOT NULL
  AND ProductKey  IS NOT NULL;

# PySpark: dropDuplicates, na.fill, na.drop
from pyspark.sql import functions as F

df = spark.table("bronze.RawOrders")

# 1. Remove full-row duplicates
df_dedup = df.dropDuplicates()

# 2. Remove duplicates on business key, keeping latest
from pyspark.sql.window import Window
w = Window.partitionBy("OrderID").orderBy(F.col("LastModifiedDate").desc())
df_dedup_bk = (df
    .withColumn("_rn", F.row_number().over(w))
    .filter(F.col("_rn") == 1)
    .drop("_rn")
)

# 3. Fill nulls: constant for numeric, dict for mixed types
df_filled = df_dedup.na.fill(0, subset=["Discount", "TaxAmount"])
df_filled = df_filled.na.fill({"ShipCity": "Unknown", "Notes": ""})

# 4. Drop rows missing critical columns
df_clean = df_filled.na.drop(subset=["OrderID", "CustomerKey", "OrderDate"])

df_clean.write.format("delta").mode("overwrite").saveAsTable("silver.Orders")

Exam tips & gotchas

  • NULL ≠ empty string — WHERE Notes = '' does not catch NULLs; use WHERE Notes IS NULL OR Notes = ''.
  • COALESCE short-circuits evaluation (stops at first non-NULL); ISNULL does not — prefer COALESCE for multiple fallbacks.
  • PySpark dropDuplicates() with no arguments deduplicates on all columns — rows with the same business key but different timestamps will be kept.
  • COUNT(column) vs COUNT(*): COUNT(*) counts all rows including NULLs; COUNT(column) counts only non-NULL values in that column.
  • ROW_NUMBER deduplication in T-SQL inside a CTE can be deleted directly with DELETE FROM cte_name WHERE RowNum > 1 — Fabric Warehouse supports this pattern.

Practice

Q1. A T-SQL query filters rows with WHERE Discount <> 0. A row with Discount = NULL is present. Is it included in the result?
Answer: No. Why: NULL compared with any value using = or <> evaluates to UNKNOWN, which is treated as FALSE. The row is excluded. To include NULLs, use WHERE Discount <> 0 OR Discount IS NULL.
Q2. A PySpark job calls df.dropDuplicates() on a DataFrame with an OrderID column and a LastModifiedDate column. Two rows have the same OrderID but different LastModifiedDate values. Are both rows kept?
Answer: Yes, both are kept. Why: dropDuplicates() with no arguments deduplicates on all columns. Since the two rows differ on LastModifiedDate, they are not considered duplicates. Use dropDuplicates(["OrderID"]) to deduplicate by business key only (non-deterministic which row is kept).
Q3. A column ShipCity contains NULLs. A T-SQL expression must return the BillingCity if ShipCity is NULL, or the string 'Unknown' if BillingCity is also NULL. Which expression is correct?
Answer: COALESCE(ShipCity, BillingCity, 'Unknown'). Why: COALESCE evaluates its arguments left to right and returns the first non-NULL. ISNULL only accepts two arguments, so it cannot check three values in one expression.

2.2.8 Convert data types

Source data frequently arrives with incorrect or generic types: dates stored as strings, integers stored as VARCHAR, decimals stored as FLOAT. Converting to the correct analytical type improves storage efficiency, enables correct sorting and arithmetic, and prevents semantic errors in reports. Fabric supports type conversion in T-SQL (CAST, CONVERT, TRY_CAST, TRY_CONVERT), PySpark (cast, withColumn), and Power Query M (type annotations and conversion functions).

Key facts

  • CAST(expr AS type) — ANSI SQL standard; throws an error on invalid conversion (e.g., non-numeric string to INT).
  • TRY_CAST / TRY_CONVERT — "safe" variants; return NULL instead of an error on invalid conversion; essential for dirty data in ETL pipelines.
  • CONVERT(type, expr, style) — SQL Server extension; adds a style parameter for date/time formatting (e.g., style 103 = dd/mm/yyyy); not ANSI standard.
  • Implicit conversion — Fabric Warehouse will silently convert compatible types (e.g., INT to BIGINT); implicit conversion of incompatible types (e.g., VARCHAR to DATE) causes runtime errors.
  • PySpark cast — df.withColumn("Amount", col("Amount").cast("double")); for invalid values, returns NULL (safe by default in Spark).
  • Power Query M — each column has a declared type; changing type via UI generates Table.TransformColumnTypes(table, {{"Col", type date}}); errors can be replaced with Table.ReplaceErrorValues.
  • Date/time conversions are the most error-prone: string date format must exactly match the style parameter (or ISO 8601 yyyy-MM-dd); locale-specific formats (MM/dd/yyyy vs dd/MM/yyyy) cause silent wrong conversions.
-- T-SQL: CAST, TRY_CAST, TRY_CONVERT examples
SELECT
    OrderID,
    -- Safe conversion; returns NULL for invalid strings (not an error)
    TRY_CAST(OrderDateStr  AS DATE)          AS OrderDate,
    TRY_CAST(SalesAmountStr AS DECIMAL(10,2)) AS SalesAmount,
    -- CAST raises error on bad data — use when data is guaranteed clean
    CAST(Quantity AS INT)                    AS Quantity,
    -- CONVERT with style 103 = dd/mm/yyyy input format
    TRY_CONVERT(DATE, EUDateStr, 103)        AS EUDate
FROM dbo.OrdersStaging
WHERE TRY_CAST(SalesAmountStr AS DECIMAL(10,2)) IS NOT NULL;  -- filter bad rows

# PySpark: cast columns to correct types
from pyspark.sql import functions as F
from pyspark.sql.types import IntegerType, DoubleType, DateType

df = spark.table("bronze.RawOrders")

df_typed = (df
    .withColumn("OrderDate",   F.to_date("OrderDateStr", "yyyy-MM-dd"))
    .withColumn("SalesAmount", F.col("SalesAmountStr").cast(DoubleType()))
    .withColumn("Quantity",    F.col("QuantityStr").cast(IntegerType()))
    # Flag rows where cast produced NULL (i.e., invalid source values)
    .withColumn("HasTypeError",
        F.col("SalesAmount").isNull() | F.col("Quantity").isNull())
)

df_valid   = df_typed.filter(~F.col("HasTypeError"))
df_invalid = df_typed.filter( F.col("HasTypeError"))

df_valid.write.format("delta").mode("append").saveAsTable("silver.Orders")
df_invalid.write.format("delta").mode("append").saveAsTable("dq.TypeConversionErrors")

Exam tips & gotchas

  • CAST vs TRY_CAST: use TRY_CAST in ETL pipelines where source data quality cannot be guaranteed; use CAST only when data is validated.
  • FLOAT is an approximate type — avoid it for financial amounts where DECIMAL(p,s) is required for exact precision.
  • PySpark cast to DoubleType() returns NULL (not an error) for invalid string values — always check for unexpected NULLs post-cast.
  • Power Query M type changes applied via the UI generate Table.TransformColumnTypes; this step will throw a step-level error if a value cannot be converted, halting the entire Dataflow refresh.

Practice

Q1. A staging table column SalesAmountStr contains mostly numeric strings but occasionally the value 'N/A'. A T-SQL ETL pipeline must convert this column to DECIMAL(10,2) without failing on 'N/A' rows. Which function should be used?
Answer: TRY_CAST(SalesAmountStr AS DECIMAL(10,2)). Why: TRY_CAST returns NULL for invalid conversions instead of raising an error. CAST would fail the entire query for any 'N/A' value.
Q2. A PySpark notebook casts a column Price from StringType to DoubleType. After the cast, a data quality check finds unexpected NULLs in Price. What most likely caused this?
Answer: The source column contains invalid (non-numeric) string values that Spark could not convert. PySpark's cast returns NULL for values that fail conversion instead of throwing an error, so invalid strings silently become NULLs.

2.2.9 Filter data

Filtering reduces a dataset to only the rows needed for a given transformation step, report, or destination. Applying filters as early as possible in the processing pipeline — ideally at the source system via query pushdown — is the most important performance optimisation in data engineering. This principle is formalised as predicate pushdown in Spark and query folding in Power Query.

Key facts

  • T-SQL WHERE — evaluated after FROM/JOIN, before GROUP BY; can reference any column in scope; supports AND, OR, NOT, IN, BETWEEN, LIKE, IS NULL, EXISTS.
  • T-SQL HAVING — filters on aggregated expressions after GROUP BY; cannot reference aliases defined in SELECT (use CTE or subquery).
  • PySpark filter / where — equivalent methods; accepts Column expressions or SQL strings; lazy (no execution until action); Spark automatically pushes filters to the Delta scan (partition and file-level pruning).
  • Delta partition pruning — if the filter column matches a Delta partition column, Spark skips entire file partitions; dramatically reduces I/O on large tables.
  • Power Query Table.SelectRows — Table.SelectRows(table, each [Amount] > 0 and [Status] = "Active"); folds to source WHERE if connector supports it.
  • Query folding — Power Query sends the filter as a WHERE clause to the data source, reducing rows returned over the network; verify with "View native query" option.
-- T-SQL: WHERE with multiple conditions and IN list
SELECT OrderID, CustomerKey, SalesAmount, OrderDate
FROM   dbo.SalesFact
WHERE  SalesAmount > 0
  AND  OrderDate BETWEEN '2024-01-01' AND '2024-12-31'
  AND  StatusCode IN ('SHIP', 'COMP')
  AND  CancelledDate IS NULL;

-- T-SQL: filter using EXISTS (semi-join — efficient for large tables)
SELECT c.CustomerKey, c.CustomerName
FROM   dbo.DimCustomer c
WHERE  EXISTS (
    SELECT 1 FROM dbo.SalesFact f
    WHERE  f.CustomerKey = c.CustomerKey
      AND  f.OrderDate >= '2024-01-01'
);

# PySpark: filter with Column expressions (partition pruning)
from pyspark.sql import functions as F

df = spark.table("silver.SalesFact")  # partitioned by OrderYear

# Spark pushes the year filter to the Delta scan — skips other partitions
df_filtered = (df
    .filter(F.col("OrderYear") == 2024)           # partition pruning
    .filter(F.col("SalesAmount") > 0)
    .filter(F.col("StatusCode").isin("SHIP","COMP"))
    .filter(F.col("CancelledDate").isNull())
)

df_filtered.write.format("delta").mode("overwrite") \
    .saveAsTable("gold.Sales2024Active")

// Power Query M: Table.SelectRows with query folding
let
    Source    = Sql.Database("myserver.database.windows.net", "SalesDB"),
    SalesFact = Source{[Schema="dbo",Item="SalesFact"]}[Data],
    // This filter folds to WHERE SalesAmount > 0 AND StatusCode IN ('SHIP','COMP') in SQL
    Filtered  = Table.SelectRows(SalesFact,
                    each [SalesAmount] > 0
                    and List.Contains({"SHIP","COMP"}, [StatusCode])
                    and [CancelledDate] = null)
in
    Filtered
Tip: Always filter on Delta partition columns first in PySpark. If a table is partitioned by OrderYear, a filter on OrderYear = 2024 lets Spark skip all other year partitions at the file-system level — often a 10× or greater I/O reduction.

Exam tips & gotchas

  • WHERE vs HAVING: WHERE filters before aggregation; HAVING filters the result of GROUP BY. Using HAVING to filter on a non-aggregate column is legal but less efficient.
  • Query folding in Power Query: if any step breaks folding (e.g., adding a custom M column with a non-foldable function), all subsequent steps also lose folding — even simple filters.
  • PySpark filter and where are identical — choose either for readability.
  • Delta Z-Order clustering (if applied) also helps filter performance by collocating related data within Parquet files, reducing the number of row groups scanned.
  • EXISTS in T-SQL is usually faster than IN with a large subquery list because it short-circuits on the first match.

Practice

Q1. A Dataflow Gen2 reads from an Azure SQL Database and applies a Table.SelectRows filter. A data engineer then adds a custom M column using a regex function. What is the impact on query folding for the filter?
Answer: The filter loses query folding. Why: The custom M column uses a function that cannot be translated to SQL (regex is not a foldable operation). Any step that breaks folding causes all subsequent steps — including earlier filter steps that appear after the break — to execute in the Dataflow engine rather than at the source.
Q2. A PySpark job filters a Delta table on a column ProductCategory, but the table is partitioned by OrderYear. Will Spark apply partition pruning for the ProductCategory filter?
Answer: No. Why: Partition pruning skips entire partition directories based only on the partition columns (here, OrderYear). A filter on ProductCategory cannot skip partitions; instead, Spark uses column statistics and row-group filtering within Parquet files (data skipping), which is less aggressive than partition pruning.
Q3. Which T-SQL filtering technique is most efficient when checking whether a large fact table contains at least one row for each customer in a dimension table, without returning the fact rows themselves?
Answer: EXISTS — WHERE EXISTS (SELECT 1 FROM SalesFact WHERE CustomerKey = c.CustomerKey). Why: EXISTS short-circuits on the first match, so Fabric Warehouse does not need to read all matching fact rows. IN with a subquery or a JOIN that returns all fact rows requires processing more data.

2.3.1 Visual Query Editor and query folding

The Visual Query Editor is a no-code query builder available inside Fabric Warehouse and the Lakehouse SQL analytics endpoint. It presents a drag-and-drop canvas where engineers add tables, apply transforms (filter, group by, join, rename columns), and preview results — all without writing SQL. Each step is translated into a SQL query that is sent to the underlying engine, meaning the engine does all the heavy lifting rather than the client.

Query folding is the mechanism whereby Power Query (Dataflows Gen2, Power BI Desktop) translates M transform steps into a single native query — SQL, OData, SPARQL — that runs at the source rather than in the Mashup engine. Folded queries reduce data movement and use source-side indexes, parallelism, and pushdown capabilities.

Key facts

  • Visual Query Editor in Warehouse produces SQL that can be viewed and copied; the generated query is submitted to the Warehouse SQL engine — no separate compute is used.
  • Transforms available in Visual Query: filter rows, choose/rename/reorder columns, group by, sort, merge (JOIN), append (UNION), aggregate, add column (limited expressions).
  • Query folding indicators — in Power Query Editor, right-click a step: if "View native query" is available (not grayed out), the step folds. A grayed-out option means folding stopped at that step.
  • Connectors that always support folding: SQL Server, Azure SQL, Fabric Warehouse/Lakehouse, OData, SharePoint List, Salesforce.
  • Steps that commonly break folding: custom M functions, invoke custom function, add index column, Table.Buffer, Table.Sort followed by a non-foldable step, and any function without a native SQL equivalent.
  • Best practice: apply all foldable steps (filter, remove columns, rename, type) before any non-foldable step so the maximum filter is pushed to the source.
  • Visual Query results can be saved as a view in the Warehouse without writing SQL manually.
Key: Query folding is a Mashup engine feature, not a storage engine feature. It affects Dataflows Gen2 and Power BI Desktop (Direct Query or Import with Power Query steps) — it does not apply to T-SQL or KQL queries, which always run at the engine directly.

Exam tips & gotchas

  • If a step breaks folding, ALL subsequent steps also lose folding — even a simple rename step after a non-foldable step will execute in the Mashup engine.
  • "View native query" being grayed out confirms folding stopped; use this to diagnose Dataflow performance problems.
  • Visual Query Editor (Warehouse) vs Power Query Editor (Dataflows): both are no-code, but VQE is purely SQL-based and always runs at the engine; Power Query uses M and relies on folding to push work to the source.
  • Table.Buffer() forces data into memory and always breaks folding — avoid unless you have a specific reason (e.g., preventing multiple source evaluations for performance).

Practice

Q1. A Dataflow Gen2 connects to a SQL Server source and applies steps in this order: (1) filter rows, (2) remove columns, (3) add custom M column using Text.BetweenDelimiters, (4) rename columns. Which steps fold to SQL Server?
Answer: Steps 1 and 2 fold. Why: Step 3 (Text.BetweenDelimiters is a non-foldable M function) breaks folding. Step 4 (rename), although normally foldable, comes after the break and therefore also executes in the Mashup engine.
Q2. A data engineer right-clicks a filter step in Power Query Editor and finds "View native query" is grayed out. What does this indicate?
Answer: The filter step does not fold to the source — it executes in the Power Query Mashup engine (client-side). A previous step likely broke folding, causing all subsequent steps to run locally. The engineer should identify and relocate or remove the non-foldable step.
Q3. Which action in a Dataflow Gen2 step sequence always breaks query folding, regardless of the source connector?
Answer: Table.Buffer(). Why: Table.Buffer forces the data into memory in the Mashup engine, terminating the connection to the source query plan. All steps after Buffer execute entirely in memory within the Mashup engine.

2.3.2 Query data with SQL

Fabric Warehouse and the Lakehouse SQL analytics endpoint both expose T-SQL endpoints that clients connect to via TDS (SQL Server wire protocol). Standard SQL tooling — SQL Server Management Studio, Azure Data Studio, Power BI Desktop, and the Fabric web SQL editor — connects with no driver changes. Fabric's T-SQL dialect is based on Azure Synapse Dedicated SQL and supports nearly all ANSI SQL alongside common SQL Server extensions.

Key facts

  • SELECT … FROM … JOIN … WHERE … GROUP BY … HAVING … ORDER BY — all supported; CTE (WITH clause) fully supported.
  • Window functions — ROW_NUMBER, RANK, DENSE_RANK, NTILE, LAG, LEAD, FIRST_VALUE, LAST_VALUE, SUM/AVG/COUNT OVER (PARTITION BY … ORDER BY …) — all supported.
  • CROSS APPLY / OUTER APPLY — supported; used to lateral-join table-valued functions or JSON parsing expressions.
  • Common Fabric-specific unsupported T-SQL: no distributed transactions, no linked servers, no CLR objects, no XML methods on columns (as of GA). Check documentation for the latest support matrix.
  • Queries can reference tables across workspaces using cross-workspace shortcuts exposed as external tables (via shortcuts at the Lakehouse level).
  • Fabric Warehouse supports EXPLAIN (query plan) via the SQL editor; use it to diagnose full-table scans and suboptimal join orders.
  • SELECT … INTO creates a new table (not supported at Lakehouse SQL analytics endpoint; Warehouse only).
-- Window functions in Fabric Warehouse SQL
WITH RankedSales AS (
    SELECT
        c.CustomerName,
        p.CategoryName,
        SUM(f.SalesAmount)   AS TotalSales,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryName
            ORDER BY SUM(f.SalesAmount) DESC
        )                    AS CategoryRank,
        RANK() OVER (
            ORDER BY SUM(f.SalesAmount) DESC
        )                    AS OverallRank,
        LAG(SUM(f.SalesAmount), 1, 0) OVER (
            PARTITION BY p.CategoryName
            ORDER BY MIN(f.OrderDate)
        )                    AS PrevPeriodSales
    FROM dbo.SalesFact  f
    JOIN dbo.DimCustomer c ON f.CustomerKey = c.CustomerKey
    JOIN dbo.DimProduct  p ON f.ProductKey  = p.ProductKey
    GROUP BY c.CustomerName, p.CategoryName
)
SELECT *
FROM   RankedSales
WHERE  CategoryRank <= 10
ORDER  BY CategoryName, CategoryRank;

Exam tips & gotchas

  • ROW_NUMBER returns a unique integer per row (no ties); RANK skips numbers after ties; DENSE_RANK does not skip.
  • LAG/LEAD access previous/next row values within the partition — powerful for period-over-period comparison without a self-join.
  • The Lakehouse SQL analytics endpoint supports SELECT, CREATE VIEW/FUNCTION/PROCEDURE but not DML. The Warehouse supports full DML.
  • Window functions cannot be referenced directly in WHERE or HAVING — wrap in a CTE or subquery (as shown above with WHERE CategoryRank <= 10).

Practice

Q1. A SQL query using a window function assigns the same RANK value to two customers with equal TotalSales. What rank value does the next customer receive?
Answer: The next customer receives rank 3 (if two customers tied at rank 1) — i.e., RANK skips rank 2. Why: RANK assigns the same value to tied rows and then skips the next rank number(s). Use DENSE_RANK to avoid skipping.
Q2. A query must filter on a window function result (e.g., show only top-10 per category). A data engineer writes: SELECT ..., RANK() OVER (...) AS Rnk FROM ... WHERE Rnk <= 10. Does this work?
Answer: No — it throws an error. Why: Window functions are evaluated after WHERE, so they cannot be referenced in the WHERE clause of the same SELECT. Wrap the window function in a CTE or subquery and filter in the outer query.

2.3.3 Query data with KQL

Kusto Query Language (KQL) is the primary query language for Fabric Eventhouse (KQL databases). It uses a pipe (|) syntax where the output of each operator flows into the next, building a readable left-to-right transformation pipeline. KQL is optimised for time-series, log analysis, and real-time telemetry — operations that are cumbersome in SQL but natural in KQL.

KQL is available in the Eventhouse query editor, from a KQL queryset item, from Fabric notebooks via the %%kql magic, and programmatically through the Kusto REST API. Mastering the core operators and the bin() time-bucketing function is essential for the DP-600 exam.

Key facts

  • Pipe syntax — TableName | operator1 | operator2 | ...; each operator receives the previous output as its input table.
  • where — filters rows; equivalent to SQL WHERE; | where SalesAmount > 0 and Region == "West". Use == (not =) for equality comparison in KQL.
  • project — selects and optionally renames columns; drops all others; | project OrderID, Amount = SalesAmount.
  • project-away — drops named columns, keeps all others (opposite of project).
  • extend — adds computed columns without dropping others; | extend ProfitMargin = (Revenue - Cost) / Revenue.
  • summarize — aggregates; collapses rows to one per group-by key; | summarize TotalSales = sum(SalesAmount), Orders = count() by CategoryName, bin(EventTime, 1h). After summarize, only by-columns and aggregate expressions are available.
  • bin(column, interval) — floor to the nearest multiple of interval; essential for time-series bucketing; bin(StartTime, 1d) rounds to day.
  • sort by / order by — synonyms; | sort by TotalSales desc.
  • take / limit — return the first N rows (non-deterministic order unless sorted first).
  • render — chart directive; | render timechart; display hint only — does not change data output.
  • join — | join kind=inner (RightTable | project ...) on Key; join kinds: inner, leftouter, rightouter, fullouter, leftanti, rightanti, leftsemi.
  • make-series — creates time-series arrays per group; useful with time-series ML functions (series_decompose, series_outliers).
  • Type functions: toint(), tolong(), todouble(), tostring(), todatetime(), totimespan().
  • Time helpers: ago(30d), now(), startofday(), startofmonth(), datetime(2024-01-01).
// 1. Basic where + project + sort
SalesFact
| where EventTime > ago(30d)
    and SalesAmount > 0
    and Region == "West"
| project OrderID, CustomerID, SalesAmount,
          EventTime, Region
| sort by SalesAmount desc
| take 100

// 2. extend: add a computed column
SalesFact
| where EventTime between (datetime(2024-01-01) .. datetime(2024-12-31))
| extend ProfitMargin = round((Revenue - Cost) / Revenue, 4)
| project OrderID, Revenue, Cost, ProfitMargin

// 3. summarize with bin() for hourly time-series
SalesFact
| where EventTime > ago(7d)
| summarize
    TotalSales = sum(SalesAmount),
    Orders     = count()
  by CategoryName, bin(EventTime, 1h)
| sort by EventTime asc
| render timechart

// 4. join: enrich fact with dimension
SalesFact
| join kind=leftouter (
    DimProduct
    | project ProductID, ProductName, CategoryName
  ) on ProductID
| project OrderID, ProductName, CategoryName, SalesAmount, EventTime

// 5. Time-range filter patterns
SecurityEvents
| where TimeGenerated between (ago(24h) .. now())
| where EventID in (4624, 4625, 4648)
| summarize FailedLogons = countif(EventID == 4625),
            SuccessLogons = countif(EventID == 4624)
  by Computer, bin(TimeGenerated, 1h)
Gotcha: After summarize, only the columns in the by clause and the aggregation expressions exist in the output. Any attempt to reference a column that was present before summarize but not in the by list will cause an error in a downstream operator.

Exam tips & gotchas

  • KQL uses == for equality (not =). Using = causes a parse error.
  • project drops all columns not named; extend keeps all existing columns and adds new ones. Know which to use to avoid accidental column loss.
  • bin() is required to produce regular time buckets in summarize by — without it, each unique timestamp becomes its own group.
  • render is a display directive, not a data operation. It has no effect on the tabular output and does not exist in programmatic API calls.
  • take N is non-deterministic unless preceded by sort by — do not use take to get "the most recent N rows" without an explicit sort.
  • After summarize, you cannot project a column that was not in the by clause — it no longer exists in scope.

Practice

Q1. A KQL query must count the number of events per hour for the past seven days, grouped by server name. Which operators and functions are required?
Answer: | where TimeGenerated > ago(7d) to filter the time range, then | summarize EventCount = count() by ServerName, bin(TimeGenerated, 1h) to count per hour per server. bin(TimeGenerated, 1h) is essential to floor each timestamp to its hour boundary, creating regular one-hour buckets.
Q2. A data analyst writes this KQL query:
SalesFact | summarize Total = sum(SalesAmount) by CategoryName | project OrderID, CategoryName, Total. Why does this query fail?
Answer: The project references OrderID, which no longer exists after summarize. The summarize operator collapses rows to one per CategoryName and retains only the by-clause columns and aggregate expressions — OrderID is not in scope. Remove OrderID from the project list.
Q3. In KQL, what is the difference between project and extend?
Answer: project selects specific columns and drops all others (like SQL SELECT with explicit column list). extend adds new computed columns while preserving all existing columns (like SQL SELECT *, newcol). Use project when you want to narrow the schema; use extend when you want to enrich without losing columns.

2.3.4 Query data with DAX

Data Analysis Expressions (DAX) is the formula language of Power BI semantic models. For DP-600, DAX is tested at an introductory level in Domain 2 (Prepare data) — you must understand the core concepts of filter context, measures, and the most important functions. Deep DAX mastery (time intelligence, semi-additive measures, row context) belongs to Domain 3 (Analyse and visualise data).

Measures are DAX formulas stored in a semantic model that are evaluated dynamically based on the current filter context — the combination of slicer selections, visual filters, and page/report filters active at the moment the measure is called. Unlike calculated columns, measures produce no stored data and are recalculated on every query.

Key facts

  • Filter context — the set of filters applied to a visual at the moment DAX evaluates a measure; determined by slicers, visual-level filters, page filters, report filters, and cross-filter relationships.
  • CALCULATE(expression, [filter1], …) — the single most important DAX function; evaluates expression after modifying the filter context with the given filters; can add, remove, or replace filters.
  • FILTER(table, condition) — returns a filtered virtual table; used as an argument to CALCULATE; CALCULATE(SUM(Sales[Amount]), FILTER(Sales, Sales[Region] = "West")).
  • DIVIDE(numerator, denominator, [alternateResult]) — safe division; returns alternateResult (default: BLANK()) when denominator is 0; always prefer over the / operator to avoid division-by-zero errors.
  • ALL(table_or_column) — removes filters from the specified table or column; used inside CALCULATE to create "percent of total" measures.
  • Basic aggregations: SUM, AVERAGE, COUNT (counts rows where expression is non-blank), DISTINCTCOUNT, MIN, MAX, COUNTROWS (counts rows in a table).
  • Time intelligence: TOTALYTD, TOTALMTD, TOTALQTD require a properly marked Date table; SAMEPERIODLASTYEAR, DATEADD provide comparison periods.
  • DAX measures are defined in the semantic model (dataset), not in the Warehouse or Lakehouse — they do not affect underlying data storage.
  • A semantic model in Direct Lake mode can still have DAX measures; the measures execute against the in-memory framed data loaded from Delta files.
-- Basic measure definitions
TotalSales = SUM(SalesFact[SalesAmount])

OrderCount = COUNTROWS(SalesFact)

AvgOrderValue = DIVIDE(
    SUM(SalesFact[SalesAmount]),
    COUNTROWS(SalesFact),
    BLANK()    -- return BLANK instead of error when no rows
)

-- CALCULATE: modify filter context
WestRegionSales =
CALCULATE(
    SUM(SalesFact[SalesAmount]),
    DimGeography[Region] = "West"
)

-- Percent of total using ALL()
SalesShareOfTotal =
DIVIDE(
    SUM(SalesFact[SalesAmount]),
    CALCULATE(SUM(SalesFact[SalesAmount]), ALL(SalesFact)),
    0
)

-- FILTER inside CALCULATE for complex conditions
HighValueSales =
CALCULATE(
    SUM(SalesFact[SalesAmount]),
    FILTER(
        SalesFact,
        SalesFact[SalesAmount] > 10000
            && SalesFact[StatusCode] = "COMP"
    )
)

-- Time intelligence: year-to-date sales
SalesYTD =
TOTALYTD(
    SUM(SalesFact[SalesAmount]),
    DimDate[Date]
)
Key: CALCULATE is the gateway to advanced DAX. Every time intelligence function (TOTALYTD, SAMEPERIODLASTYEAR) is implemented by calling CALCULATE internally with a modified date filter. Understanding what CALCULATE does to the filter context is the foundation for understanding all complex DAX patterns.

Exam tips & gotchas

  • DIVIDE(x, y) is always preferred over x / y — the / operator returns a division-by-zero error when y = 0; DIVIDE returns BLANK() or an alternate result.
  • Measures are evaluated in filter context, not row context — a measure referencing SalesFact[SalesAmount] aggregates all matching rows, not a single row.
  • FILTER returns a table; it is used as an argument inside CALCULATE, not as a standalone query function.
  • ALL() removes filters from a column or table. ALLEXCEPT(table, col) removes all filters except the specified column — useful for "percent within category" patterns.
  • Time intelligence functions require a Date table marked as a Date table in the semantic model, with a contiguous date range covering all dates in the fact table.
  • Deep DAX (row context, context transition, EARLIER, RANKX, iterators) is tested in Domain 3 — Domain 2 only requires understanding the concepts above.

Practice

Q1. A measure calculates total sales only for the "West" region, regardless of any region filter applied in the report. Which DAX function is required and how is it structured?
Answer: CALCULATE(SUM(SalesFact[SalesAmount]), ALL(DimGeography[Region]), DimGeography[Region] = "West"). Why: CALCULATE first removes any existing Region filter with ALL(), then reapplies the fixed "West" filter. Without ALL(), the measure would still be affected by the report's active region filter, potentially returning BLANK() when a different region is selected.
Q2. A measure is defined as SharePct = SUM(SalesFact[SalesAmount]) / CALCULATE(SUM(SalesFact[SalesAmount]), ALL(SalesFact)). A report user selects only 2024 in a Year slicer. Does the denominator (total) change?
Answer: No. Why: ALL(SalesFact) removes all filters on the SalesFact table, including the Year filter. The denominator always returns the grand total across all years, making SharePct a true "share of total" regardless of slicer selection.
Q3. Why should a developer use DIVIDE(x, y) instead of x / y in a DAX measure?
Answer: DIVIDE handles division by zero gracefully by returning BLANK() (or a custom alternate result) instead of an error. The / operator raises a "Division by zero" error when the denominator evaluates to 0, which can break visuals. DIVIDE also propagates BLANK() correctly through calculations.

Domain 3 — Implement and manage semantic models 25–30%

This domain covers designing and building semantic models (storage modes, star schema, relationships, DAX, calculation groups, large model format, composite models) and optimizing enterprise-scale models (visual/query performance, DAX tuning, Direct Lake, incremental refresh). 3.1 Design and build semantic models

3.1.1 Choose a storage mode

The storage mode of a table determines whether its data is cached in memory, queried live from the source, or read on demand from OneLake. The right choice balances query speed, data freshness, model size, and licensing.

A single model can mix modes (a composite model), and Fabric adds Direct Lake, which aims to combine Import-class speed with near-real-time freshness.

Key facts

  • Import — data is loaded into the in-memory VertiPaq columnar store (typically ~10× compression). Supports the full DAX surface, calculated tables/columns, Q&A. Requires refresh. Desktop upload limit is 10 GB; can grow past that in the service only when Large semantic model storage format is on.
  • DirectQuery — no data cached (metadata only); every visual sends native SQL/KQL to the source. No calculated tables; DAX limited to translatable functions. Real-time freshness. Refresh caps: 8×/day shared, 48×/day Premium.
  • Direct Lake — Fabric-only. VertiPaq processes queries (Import-comparable speed) but reads columns directly from Delta/Parquet in OneLake on demand (transcoding). Refresh = framing (metadata-only, seconds). Requires a Fabric capacity license.
  • Dual — a table is simultaneously Import and DirectQuery; the engine picks the cheaper path per query. Primary use: dimension tables paired with DirectQuery fact tables for efficient native joins.
  • Composite — one model mixing storage modes (or DirectQuery from multiple sources).
Key: Direct Lake refresh is framing — a metadata-only operation that completes in seconds, not a data reload.

Exam tips & gotchas

  • Only Direct Lake requires a Fabric capacity; Import and DirectQuery do not.
  • Need fastest queries + can tolerate scheduled refresh → Import. Need real-time from a relational source without Fabric → DirectQuery. On Fabric with Delta tables and want both speed and freshness → Direct Lake.
  • Dual is the recommended mode for dimensions in a DirectQuery composite model — it lets Power BI fold a single join to the source.

Practice

Q1. A model must show sub-second visuals over a 40 GB Fabric lakehouse Delta table with data that changes hourly, with minimal refresh cost. Which storage mode?
Answer: Direct Lake. Why: VertiPaq-speed queries with metadata-only framing keeps the model current in seconds without an expensive Import reload, and it reads directly from the lakehouse Delta files.
Q2. In a DirectQuery composite model, which storage mode should the dimension tables use to minimize the number of source queries?
Answer: Dual. Why: Dual dimensions can join the DirectQuery facts in a single native query while still serving cached results when queried alone.

3.1.2 Implement a star schema for a semantic model

A star schema separates dimension tables (business entities used to filter and group) from fact tables (numeric observations to summarize). It is the recommended modeling pattern in Power BI because it produces simple relationships, smaller models, and faster DAX.

Key facts

  • Dimension tables hold a unique key (surrogate key) plus descriptive attributes; they are the "one" side of one-to-many relationships and are optimized for filtering/grouping.
  • Fact tables hold foreign keys + numeric measures; they are the "many" side and are optimized for summarization. Load facts at a single, consistent grain.
  • Surrogate keys: add numeric keys to dimensions (Power Query → Add Index Column) and merge back to the fact.
  • Prefer explicit measures (calculator icon) over implicit auto-aggregations (∑ icon).
  • Relationships rely on a single unique column on the one side.
  • Patterns to know: snowflake (normalized sub-tables — usually denormalize into one dimension), role-playing (one date table reused via inactive relationships), factless fact / bridge (keys only, resolves many-to-many), degenerate dimension (a key attribute like invoice number stored in the fact).
Tip: Denormalize snowflaked dimensions into a single flat dimension table for better compression and simpler DAX.

Exam tips & gotchas

  • Dimensions filter; facts summarize — don't mix both purposes in one table.
  • Role-playing dates (OrderDate/ShipDate/DueDate) reuse one date table via inactive relationships activated with USERELATIONSHIP().
  • Implicit measures are discouraged in enterprise models and are blocked once a calculation group exists.

Practice

Q1. A Product table is split into Product → Subcategory → Category. What should you do for a performant Power BI model?
Answer: Denormalize the three tables into a single Product dimension. Why: A flat (star) dimension avoids snowflake joins, compresses better, and simplifies filtering.
Q2. Sales has OrderDate, ShipDate, and DueDate all referencing one Date table. How do you support all three without duplicating the Date table?
Answer: Create one active and two inactive relationships (role-playing), then use USERELATIONSHIP() in measures to activate the needed one. Why: A model allows only one active relationship between two tables.

3.1.3 Implement relationships, such as bridge tables and many-to-many relationships

Relationships propagate filters between tables. Cardinality and cross-filter direction control how filters flow. Many-to-many needs between two dimensions are best resolved with a bridge (factless fact) table rather than a native many-to-many relationship.

Key facts

  • Cardinality / cross-filter: one-to-many → Single or Both; one-to-one → Both only; many-to-many → Single (either direction) or Both.
  • Filters always propagate automatically from the "one" side.
  • Bi-directional (Both) filtering flows both ways — use sparingly; it can hurt performance and create ambiguity.
  • Bridge-table pattern: create two entity tables + a bridge with one row per association, then two one-to-many relationships into the bridge; set one of them bi-directional so filters reach the fact.
  • Performance order (fast → slow): one-to-many intra-source > many-to-many via bridge + bi-di > native many-to-many cardinality > cross-source relationships.
  • Assume referential integrity (DirectQuery): switches LEFT OUTER to INNER JOIN for speed — only safe when all FKs exist in the PK table.
  • Direct Lake: one-side columns must be unique or queries fail.
-- USERELATIONSHIP: activate an inactive relationship inside CALCULATE
Sales by Ship Date =
CALCULATE (
    SUM ( Sales[SalesAmount] ),
    USERELATIONSHIP ( Sales[ShipDate], 'Date'[Date] )
)

-- CROSSFILTER: change cross-filter direction for this evaluation only
Distinct Countries Sold =
CALCULATE (
    DISTINCTCOUNT ( Customer[CountryRegion] ),
    CROSSFILTER ( Customer[CustomerCode], Sales[CustomerCode], BOTH )
)
Gotcha: Bi-directional filtering can introduce ambiguous filter paths in models with multiple relationships. Prefer measure-level CROSSFILTER/USERELATIONSHIP over global Both where possible.

Exam tips & gotchas

  • To let a dimension filter through a bridge to the fact, exactly one bridge relationship must be bi-directional.
  • USERELATIONSHIP only works inside CALCULATE and cannot be used where the base relationship is already active in the same expression.
  • Native many-to-many cardinality is available but slower than a well-designed bridge.

Practice

Q1. Accounts and Customers have a many-to-many association and each filters a Transactions fact. What is the recommended model?
Answer: Add an AccountCustomer bridge (factless fact) with two one-to-many relationships, and set one to bi-directional. Why: A bridge resolves the M:M cleanly and outperforms native many-to-many cardinality.
Q2. A measure must count distinct customer countries filtering from Sales even though the relationship is single-direction. Which function?
Answer: CROSSFILTER(..., BOTH) inside CALCULATE. Why: It temporarily enables bidirectional filtering for that expression without making the model relationship globally bi-directional.

3.1.4 Write calculations that use DAX variables and functions

Robust DAX combines variables (VAR/RETURN), iterators (SUMX), context transition and filtering (CALCULATE/FILTER/ALL/ALLEXCEPT), time-intelligence and windowing (DATESYTD/WINDOW/OFFSET), and information functions (USERPRINCIPALNAME/HASONEVALUE/SELECTEDVALUE). Understanding filter context is the key to correctness.

Key facts

  • VAR/RETURN: a variable is evaluated once (improving performance and readability) in the outer filter context where it's defined — not where it's used.
  • Iterators (SUMX, AVERAGEX): evaluate an expression per row then aggregate — needed for row-level arithmetic across columns.
  • CALCULATE modifies filter context; prefer Boolean filter arguments (SE-optimized) over FILTER(table, ...). Use FILTER only for multi-column/measure/OR logic.
  • ALL removes filters; ALLEXCEPT removes all filters except the listed columns (percent-of-total, running totals). KEEPFILTERS preserves existing filters instead of overwriting.
  • Time intelligence: DATESYTD('Date'[Date]); add a year-end (e.g. "6-30") for fiscal years. Not supported in DirectQuery calculated columns / RLS.
  • Windowing: WINDOW returns multiple rows (a window); OFFSET returns a single row at a relative position; INDEX returns a single row at an absolute position — usable in measures now, not just visual calculations.
  • Information functions: USERPRINCIPALNAME() (RLS/personalization), HASONEVALUE(), SELECTEDVALUE(col[,alt]) (preferred over IF+HASONEVALUE+VALUES).
  • Use DIVIDE() for safe division (BLANK on divide-by-zero); use / only with a constant denominator.
-- VAR/RETURN + time intelligence (evaluate the prior year once)
Sales YoY % =
VAR CurrentSales  = SUM ( Sales[Amount] )
VAR PriorYear     = CALCULATE ( SUM ( Sales[Amount] ), DATESYTD ( 'Date'[Date] ) )
RETURN
    DIVIDE ( CurrentSales - PriorYear, PriorYear )

-- Iterator: per-row arithmetic then aggregate
Revenue = SUMX ( Sales, Sales[Qty] * Sales[UnitPrice] )

-- CALCULATE + Boolean filter, and ALLEXCEPT for % of category
Red Sales     = CALCULATE ( [Revenue], KEEPFILTERS ( 'Product'[Color] = "Red" ) )
Pct of Category =
    DIVIDE ( [Revenue], CALCULATE ( [Revenue], ALLEXCEPT ( 'Product', 'Product'[Category] ) ) )

-- FILTER + ALL when Boolean won't do (measure comparison)
High Value Sales =
CALCULATE ( [Revenue], FILTER ( ALL ( Sales ), Sales[Amount] > 1000 ) )

-- Windowing: 3-period moving total using OFFSET
Sales vs Prev Period =
VAR Prev = CALCULATE ( [Revenue], OFFSET ( -1, ORDERBY ( 'Date'[Date] ) ) )
RETURN [Revenue] - Prev

-- Information function: personalize by signed-in user (RLS pattern)

3.1.5 Implement calculation groups, dynamic format strings, and field parameters

Calculation groups apply reusable logic (YTD, PY, YoY%) to any measure via SELECTEDMEASURE(), eliminating dozens of near-duplicate measures. Dynamic format strings adjust display formatting while keeping numeric types. Field parameters let report readers swap the dimensions or measures shown in a visual.

Key facts

  • Calculation groups contain calculation items that rewrite how a selected measure is evaluated; the placeholder is SELECTEDMEASURE().
  • Adding a calculation group forces "Discourage implicit measures" — items only apply to explicit measures.
  • A precedence integer orders application when multiple calculation groups exist. Helpers: SELECTEDMEASURENAME(), ISSELECTEDMEASURE().
  • Dynamic format strings can be set per calculation item, or per measure (Measure tools → Format → Dynamic). Unlike FORMAT(), the measure keeps its numeric data type so charts still plot correctly.
  • Field parameters (Modeling → New parameter → Fields) build a calculated table with (display name, NAMEOF() reference, order) per entry; can mix measures and dimensions.
  • Field parameter limits: no implicit measures, no Q&A/AI visuals, can't be drill-through target fields, not with live connection unless a local model is added.
-- Calculation item using SELECTEDMEASURE()
YoY % =
VAR PY = CALCULATE ( SELECTEDMEASURE (), SAMEPERIODLASTYEAR ( 'Date'[Date] ) )
RETURN
    DIVIDE ( SELECTEDMEASURE () - PY, PY )
-- Dynamic format string for this item:  "#,##0.0%;-#,##0.0%"

-- Standalone dynamic format string for a measure (returns a format string)
Measure Format = SELECTEDVALUE ( 'Currency Formats'[Format], "#,##0.00" )

-- Field parameter table
Analyze By = {
    ( "Customer", NAMEOF ( 'Customer'[Customer] ), 0 ),
    ( "Category", NAMEOF ( 'Product'[Category] ), 1 ),
    ( "Color",    NAMEOF ( 'Product'[Color] ),    2 )
}
Key: SELECTEDMEASURE() is the placeholder for whatever measure is in context — it's what makes one calculation item work across all measures.

Exam tips & gotchas

  • Once any calculation group exists, implicit (drag-to-aggregate) measures are discouraged model-wide.
  • Prefer dynamic format strings over FORMAT(), which returns text and breaks numeric visuals/sorting.
  • Field parameters don't support Q&A, AI visuals, or being drill-through link fields.

Q1. You have 12 base measures and need YTD, PY, and YoY% versions of each without authoring 36 more measures. What do you build?

Q2. A YoY% calculation item should display as a percentage while the base measure stays currency. How?

3.1.6 Identify use cases for and configure large semantic model storage format

3.1.7 Design and build composite models

A composite model mixes storage modes (Import, DirectQuery, Dual, Direct Lake) or combines DirectQuery from multiple sources in one model. It aims to deliver Import-class performance for stable data plus DirectQuery freshness for volatile data, and can be layered on top of an existing published semantic model.

Key facts

  • Composite = tables in more than one storage mode, or DirectQuery from different sources.
  • Supported combos vary by tooling: Import+DQ and DQ-from-different-sources in Desktop; Import + Direct Lake in web modeling only; DQ + Direct Lake via XMLA only.
  • Direct Lake on SQL is single-source and cannot join a composite model; Direct Lake on OneLake can combine with Import tables in web modeling.
  • Chaining: live-connect to a published model → "Make changes to this model" converts it to a DirectQuery connection → add local tables on top.
  • Dual tables are both Import and DirectQuery; use Dual for dimensions paired with DirectQuery facts so Power BI folds a single native join.
  • Cross-source relationships default to many-to-many and are slower than same-source; you can't pull one-side values across sources with DAX.
Key: Dual is the enabler for efficient composite joins — it lets a dimension serve cached queries yet still fold into DirectQuery fact queries.

Exam tips & gotchas

  • Direct Lake on OneLake supports composite (with Import) via web modeling; Direct Lake on SQL does not.
  • Chaining a published model converts the live connection to DirectQuery so you can extend it locally.
  • Set dimensions to Dual (not Import) when facts are DirectQuery to avoid extra source round-trips.

Practice

Answer: Choose "Make changes to this model," which converts the live connection to DirectQuery, then add local tables. Why: Chaining allows local extensions on top of the published model.
Q2. Which Direct Lake variant cannot participate in a composite model?
Answer: Direct Lake on SQL analytics endpoint. Why: It is single-source and cannot mix with DQ/Dual/Import; only Direct Lake on OneLake supports composite (with Import).

3.2.2 Improve DAX performance

DAX runs across two engines: the fast, multi-threaded, columnar storage engine (SE) and the single-threaded formula engine (FE). Efficient DAX maximizes SE work and minimizes FE overhead. Performance Analyzer / DAX Studio reveal where time is spent.

Key facts

  • Push work to the SE with Boolean column filters; avoid FILTER(table, ...) as a filter argument for simple conditions (it iterates the table in the FE).
  • Use variables so repeated sub-expressions (e.g., prior-period CALCULATE) evaluate once.
  • Use KEEPFILTERS to add a Boolean filter without discarding existing ones.
  • Prefer DIVIDE() over manual zero-checks; SELECTEDVALUE over IF+HASONEVALUE+VALUES; COUNTROWS over COUNT (ignores blanks, self-documenting).
  • Avoid complex calculated columns evaluated at query time; prefer measures. Avoid scanning large fact tables with FILTER.
-- Slow: prior-period expression evaluated twice
Growth Pct Slow =
DIVIDE (
    [Sales] - CALCULATE ( [Sales], PARALLELPERIOD ( 'Date'[Date], -12, MONTH ) ),
    CALCULATE ( [Sales], PARALLELPERIOD ( 'Date'[Date], -12, MONTH ) )
)

-- Fast: evaluated once via a variable, Boolean filter with KEEPFILTERS
Growth Pct Fast =
VAR Prior = CALCULATE ( [Sales], PARALLELPERIOD ( 'Date'[Date], -12, MONTH ) )
RETURN DIVIDE ( [Sales] - Prior, Prior )

Sales Red = CALCULATE ( [Sales], KEEPFILTERS ( 'Product'[Color] = "Red" ) )
Tip: A high formula-engine time in Performance Analyzer / DAX Studio is a signal to convert table iteration into SE-friendly column filters.

Exam tips & gotchas

  • Boolean filter arguments fold to the storage engine; FILTER() over a table pushes work to the slower formula engine.
  • Variables both speed up and clarify DAX — evaluate once, reuse.
  • COUNTROWS(table) beats COUNT(column) for row counts.

Practice

Q1. A measure computes a prior-year value twice inside DIVIDE and is slow. What is the simplest optimization?
Answer: Store the prior-year value in a VAR and reuse it. Why: The expensive CALCULATE evaluates once instead of twice, cutting engine work.
Q2. Which is faster for "red products": CALCULATE([Sales], 'Product'[Color]="Red") or CALCULATE([Sales], FILTER('Product','Product'[Color]="Red"))?
Answer: The Boolean version. Why: It uses the storage engine directly, while FILTER iterates the table in the formula engine.

3.2.3 Configure Direct Lake, including default fallback and refresh behavior

Direct Lake reads Delta/Parquet columns directly from OneLake, transcoding them into VertiPaq structures on demand. Its "refresh" is framing — a metadata-only operation. On the SQL analytics endpoint variant, queries can fall back to DirectQuery under certain conditions, governed by the DirectLakeBehavior property.

Key facts

  • Framing: reads each table's Delta log, points the model at the newest Parquet files, and sets a new baseline; takes seconds. Queries see data as of the last framing point (important for ETL consistency). Dictionaries are usually kept (incremental framing).
  • Transcoding: needed columns load from OneLake on demand and stay resident until evicted.
  • Automatic updates (on by default) re-frame the model when source Delta tables change; disable for controlled framing during long ETL writes.
  • DirectQuery fallback (Direct Lake on SQL only) triggers on: SQL RLS/OLS/DDM, tables over an unmaterialized SQL view, a single table exceeding capacity guardrails, or a not-yet-framed model.
  • DirectLakeBehavior property: Automatic (default, silent fallback), DirectLakeOnly (query fails instead of falling back — good for dev), DirectQueryOnly (always DQ — baseline testing).
  • Direct Lake on OneLake always runs DirectLakeOnly — it never falls back.
  • Diagnose with EVALUATE TABLETRAITS(); a fallback value of None means Direct Lake is active.
-- Check each table's Direct Lake status / fallback reason
EVALUATE TABLETRAITS ()
Key: Framing is metadata-only and near-instant. It does not copy data — it just repoints the model at the latest Delta files.
Gotcha: Only Direct Lake on SQL falls back to DirectQuery. Enforcing SQL RLS/OLS or querying an unmaterialized view triggers it, silently degrading performance.

Exam tips & gotchas

  • Set DirectLakeBehavior = DirectLakeOnly during development so fallback conditions surface as errors instead of hiding.
  • Disable automatic updates when you need a stable snapshot during long ETL loads.
  • Fixing fallback usually means materializing views to Delta, moving security to model-level RLS/OLS, or OPTIMIZE/VACUUM to meet guardrails.

Practice

Q1. A Direct Lake on SQL model intermittently runs slowly. You discover SQL RLS is enforced on the endpoint. What is happening?
Answer: Queries fall back to DirectQuery because SQL RLS is detected. Why: Move security to semantic-model RLS or accept fallback; the DirectLakeBehavior property controls fallback.
Q2. During development you want any fallback condition to fail loudly rather than silently degrade. Which setting?
Answer: Set DirectLakeBehavior to DirectLakeOnly. Why: Queries error when Direct Lake can't be used, exposing the root cause instead of hiding it behind DirectQuery.
Q3. How long does a Direct Lake refresh (framing) typically take, and what does it do?
Answer: Seconds — it's metadata-only, updating pointers to the newest Delta Parquet files. Why: No data reload occurs; transcoding happens on demand at query time.

3.2.4 Choose between Direct Lake on OneLake and Direct Lake on SQL analytics endpoint

Both variants use the Direct Lake engine, but they differ in data source scope, fallback behavior, composite-model support, and security semantics. Direct Lake on OneLake is the recommended default for new models.

Key facts

  • Data source — OneLake: any Fabric item backed by Delta (lakehouse, warehouse, mirrored DB), and can read from multiple items. SQL endpoint: a single lakehouse/warehouse via its SQL analytics endpoint.
  • Fallback — OneLake: never falls back (always DirectLakeOnly). SQL: falls back to DirectQuery when SQL security/views/guardrails trigger it (controlled by DirectLakeBehavior).
  • Composite models — OneLake: supported (with Import) via web modeling. SQL: not supported (single source; can't mix DQ/Dual).
  • Security — OneLake uses OneLake security (file-level ACLs); SQL-based RLS is not applied, so model-level RLS is recommended. SQL endpoint honors its RLS/OLS/DDM but that triggers DirectQuery fallback.
  • Permissions/discovery — OneLake: OneLake-API permissions. SQL: SQL analytics endpoint for discovery + permissions.
  • Both require a Fabric capacity (F or P SKU). A Fabric fixed-identity cloud connection is recommended when model-level RLS is defined.

Comparison at a glance

  • Data source: OneLake = multiple Fabric Delta items · SQL = single source.
  • DirectQuery fallback: OneLake = never · SQL = yes (property-controlled).
  • Composite support: OneLake = yes (with Import) · SQL = no.
  • Security semantics: OneLake = OneLake ACLs (apply model RLS) · SQL = endpoint RLS/OLS/DDM (causes fallback).
  • Recommendation: OneLake = default for new models · SQL = legacy option.
Key: Direct Lake on OneLake never falls back and supports composite models; Direct Lake on SQL can fall back and cannot be composite.

Exam tips & gotchas

  • Need multi-source or Import+Direct Lake composite → choose OneLake.
  • Need SQL-endpoint RLS/OLS honored → SQL variant (accepting DirectQuery fallback).
  • OneLake exceeding a guardrail causes refresh to fail; SQL silently falls back to DirectQuery with a warning.

Practice

Answer: Direct Lake on OneLake. Why: It supports multiple Fabric sources and composite models with Import tables (via web modeling); the SQL variant is single-source and non-composite.
Q2. Which variant guarantees the model never silently degrades to DirectQuery?
Answer: Direct Lake on OneLake. Why: It always runs in DirectLakeOnly mode and never falls back; the SQL endpoint variant can fall back.

3.2.5 Implement incremental refresh for semantic models

Incremental refresh partitions a table and refreshes only the most recent data, cutting refresh time and resource use and enabling very large models. It relies on two reserved Power Query parameters, a parameterized folding filter, and a refresh policy.

Key facts

  • Create two reserved DateTime parameters, case-sensitive names RangeStart and RangeEnd.
  • Filter the source with them (must be a folding step): >= RangeStart and < RangeEnd.
  • In Desktop → right-click the table → Incremental refresh and real-time data → set the Archive (store rows starting) and Refresh (refresh rows in the last) periods.
  • Optional: Detect data changes (a Last-Modified column skips unchanged partitions) and Get the latest data in real time with DirectQuery (Premium — creates a hybrid table).
  • Query folding is required — the RangeStart/RangeEnd filter must translate to a native source predicate; non-folding steps after the source break it.
  • The service creates partitions on first publish/refresh; the rolling window drops partitions past the archive period. First refresh loads all history (slow); later refreshes are fast.
  • Refresh limits: Pro 2 hours, Premium 5 hours, XMLA endpoint no limit. Real-time DQ partition requires Premium/PPU/Embedded (not Pro).
// Parameterized, foldable filter in Power Query
#"Filtered Rows" =
    Table.SelectRows(
        Source,
        each [OrderDate] >= RangeStart and [OrderDate] < RangeEnd
    )
Gotcha: If the RangeStart/RangeEnd filter doesn't fold to the source, Desktop warns you — incremental refresh will scan everything and lose its benefit.

Exam tips & gotchas

  • Parameter names must be exactly RangeStart and RangeEnd, typed DateTime — spelling/case matters.
  • Query folding is mandatory; SQL/Synapse/Oracle/Teradata sources fold reliably.
  • Detect data changes further reduces refresh by skipping partitions with no newer Last-Modified timestamp.
  • Enabling real-time DirectQuery makes a hybrid table (history = Import, latest = DirectQuery); set dimensions to Dual for best performance.

Practice

Q1. After configuring incremental refresh, Desktop warns that the query can't fold. What is the impact and cause?
Answer: Folding is broken (likely a non-foldable step after the source), so the filter can't push to the source and the whole table is scanned. Why: Incremental refresh requires the RangeStart/RangeEnd filter to fold to a native predicate.
Q2. What are the exact names and types of the two required parameters?
Answer: RangeStart and RangeEnd, both DateTime (case-sensitive). Why: Power BI reserves these specific names to drive partition boundaries.
Q3. You want unchanged historical partitions to be skipped during refresh. Which option?
Answer: Detect data changes, using a Last-Modified DateTime column. Why: Partitions without newer timestamps skip the data reload, shortening refresh.

H. Glossary

  • OneLake — the single, tenant-wide data lake underlying all Fabric items ("one copy" of data in Delta Parquet).
  • Shortcut — a pointer to data in another location (OneLake, ADLS Gen2, S3) with no copy.
  • Medallion architecture — Bronze (raw) → Silver (cleaned) → Gold (curated/modeled) layering.
  • Delta table — open Parquet + transaction log; the standard OneLake table format.
  • SQL analytics endpoint — auto-created read-only T-SQL surface over a lakehouse's Delta tables.
  • Framing — Direct Lake's metadata-only refresh that repoints the model to the latest Delta files.
  • Transcoding — on-demand loading of Delta columns into VertiPaq memory for Direct Lake.
  • Star schema — central fact table(s) surrounded by denormalized dimension tables.
  • SCD — Slowly Changing Dimension; Type 1 overwrites, Type 2 keeps history (new row + validity).
  • Calculation group — a reusable set of calculation items (e.g., YTD, PY) applied to measures.
  • Field parameter — lets report users swap measures/columns on a visual via a slicer.
  • XMLA endpoint — the Analysis Services protocol surface for external tools to read/write semantic models.
  • .pbip / .pbit / .pbids — Power BI project (Git-friendly) / template (structure, no data) / data-source connection file.
  • Real-Time hub — central place to discover and subscribe to streaming data (eventstreams, KQL).
  • Endorsement — trust signal: Promoted, Certified, or Master data.
  • Query folding — Power Query pushing transformation steps down to the source system.
